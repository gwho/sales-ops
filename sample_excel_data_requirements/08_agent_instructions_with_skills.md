# Agent Instructions With Skills

Use this file to guide agents before they generate or modify sample data.

## Starting Rule

Treat `sample_excel_data_requirements/` as planning input. Treat the current Python modules and tests as schema authority:

```text
src/order_validation.py
src/inventory_allocation.py
src/payment_aging.py
src/report_export.py
src/sample_data.py
tests/test_sample_data.py
types/index.ts
context/ui-contract-plan.md
```

Do not change output contracts, UI tables, or business rules just to make a richer dataset feel more realistic.


Decide and document:


The expected recommendation is:

```text
Keep current Python schemas stable.

```

The `/architect` output must include this planning artifact before code starts:

```text
## Implementation Plan - Current-Schema Sample Data

### What we are building
[A deterministic, richer sample-data generator aligned to the current Python schemas.]

### Files to generate
[Exact file list, marking sample_customers.xlsx as optional/reference-only if included.]

### Final schemas
[Exact columns for each Excel file, split into required and optional.]

### Intentional issue design
[Which validation/allocation/payment-aging cases will be deliberately present, with target counts or ratios.]

### Compatibility checks
[Which Python loaders and business functions must successfully process the generated files.]

### UI reflection decision
[Whether the current static Next.js UI should continue using existing contract-fixture mock JSON, or should be refreshed to display outputs computed from the enriched sample Excel files.]

### Decisions made
[Whether generation stays in src/sample_data.py, data scale, seed strategy, date strategy, committed output files.]

### Out of scope
[Any fields/rules rejected as future work.]

### Ordered build steps

```

Do not implement until the user explicitly confirms this plan.

## Current UI Data Reality

The current Next.js pages do **not** read `sample_data/*.xlsx`.

They read committed JSON from:

```text
lib/mock-data/*.json
```

Those JSON files are currently generated by:

```text
scripts/generate_mock_data.py
```

and that script currently reads:

```text
tests/contract_fixtures.py
```

Therefore, enriching `sample_data/*.xlsx` alone will improve Phase 10 upload readiness, but it will **not** change what the current static UI shows.

## If The Enriched Data Must Appear In The Current Static UI

Use `/architect` to decide one of these options before implementation:

```text
Option A - Sample-data-only:
Regenerate richer Excel files and tests, but leave current UI mock JSON unchanged.
Use this if the goal is Phase 10 upload readiness only.

Option B - Refresh mock JSON from enriched workflow outputs:
Run the generated sample Excel files through validate_orders(), allocate_inventory(), calculate_payment_aging(), and report_export, then write the resulting envelopes to lib/mock-data/*.json.
Use this if the current static pages should visibly show the enriched dataset before Phase 10.

Option C - Keep contract fixtures small, add a separate demo-mock-data generator:
Leave tests/contract_fixtures.py as small shape fixtures and add a separate script that derives UI demo JSON from sample_data/*.xlsx.
Use this if preserving tiny contract fixtures matters while still making the static UI show richer data.
```

Recommended choice if the user wants the current UI to reflect the new dataset:

```text
Option C.
Keep contract fixtures as contract-shape examples.
Add or update a build-time demo mock-data generation path that computes lib/mock-data/*.json from the enriched sample_data/*.xlsx files through the real Python business modules.
Do not hand-edit lib/mock-data/*.json.
```

If any `lib/mock-data/*.json`, `lib/mock-data.ts`, `types/index.ts`, `app/`, or `components/` file changes, run:

```text
npm run typecheck
npm run lint
npm run build
```

No `/imprint` is required unless reusable UI components are created or restyled.

## Implementation Guardrails

Agents must preserve these current choices unless explicitly changed by ADR:

```text
active, not is_active
available_qty / reserved_qty, not available_stock / reserved_stock
invoice_id, not invoice_no
payment_terms, not payment_term_days
priority = High | Normal | Low
sample_customers.xlsx is optional/reference-only
lead_time_days is supplier context, not a separate lead-time review rule
forecasting is out of scope
```

## Definition of Done

Sample-data generation is ready when:

```text
all generated workflow files load with current Python loaders
order validation returns valid orders and intentional errors
allocation returns fully allocated, partially allocated, and backordered rows
payment aging returns aging rows, data issues, and draft messages
report export works from generated outputs
tests document the intended issue mix
README_sample_data.md explains fictional data and intentional defects
the plan explicitly states whether current static UI mock data was refreshed
if refreshed, lib/mock-data/*.json is generated through real Python business modules, not hand-edited
no Python business contract changed unless explicitly planned
```
