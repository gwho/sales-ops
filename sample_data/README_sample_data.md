# Sample Data

## Fictional data disclosure

Every file in this folder is 100% fictional demo data generated by [`src/sample_data.py`](../src/sample_data.py). No real company, customer, product, price, or confidential information is used anywhere. The scenario is a fictional regional medical optics / healthcare equipment company — inspired only by the general shape of a Far East regional optics/healthcare operating model (Hong Kong + Mainland China customers, Hong Kong/China/Europe warehouses and suppliers) — not any real business.

Generated on: 2026-07-10 (invoice dates shift automatically relative to whatever `reference_date` is passed to `write_sample_workbooks()`; everything else is fixed).

## Generation approach

Deterministic by construction: every row is an explicit, hand-authored literal value in `src/sample_data.py` (no `random` module, no seed). Regenerating with the same `reference_date` always produces byte-identical output; regenerating with a different `reference_date` only shifts invoice/due dates by the same offset, so the aging-bucket story stays believable no matter when this is run. Regenerate with:

```bash
uv run python -c "from src.sample_data import write_sample_workbooks; write_sample_workbooks()"
```

### `sample_customers.xlsx` — optional, reference-only

Columns: `customer_id`, `customer_name`, `customer_region`, `customer_type`, `contact_person`, `email`, `phone`, `payment_terms`, `credit_status`.

**No current Python business module reads this file.** It exists purely so the demo reads as a believable regional sales-ops operation. It is not a required upload, and no output contract depends on it. Making it required or adding customer-master validation logic is out of scope without a separate ADR.

## Demo workflow

Orders arrive from regional sales colleagues → Order Validation checks them against the product master → valid orders flow into Inventory Allocation across HK/China/Europe warehouses → limited stock produces partial allocations, backorders, and supplier follow-ups → Payment Aging reviews outstanding invoices and drafts follow-up reminders → all three stages can export an Excel report.

## Intentional data issues

### Orders (`sample_orders.xlsx`) — 36 rows, 7 issue-bearing (~19%)

Several rows deliberately combine more than one issue (a plausible "sloppy data-entry day") rather than dedicating a row to every issue type:

| order_id | Issue(s) |
|---|---|
| `SO-2026-010` (2nd occurrence) | Duplicate order_id |
| `SO-2026-030` | Unknown SKU (not in product master) |
| `SO-2026-031` | Inactive SKU referenced |
| `SO-2026-032` | Missing SKU + blank `payment_terms` (warning-only, not a required-field error) |
| `SO-2026-033` | Quantity = 0 + missing `requested_delivery_date` |
| `SO-2026-034` | Negative quantity + invalid `priority` value |
| `SO-2026-035` | Non-whole-number quantity + `requested_delivery_date` before `order_date` |

The remaining 29 rows are clean and validate successfully.

### Inventory (`sample_inventory.xlsx`) — 17 rows across HK/China/Europe warehouses

- `DIAG-TONO-007` (HK Warehouse) starts below its reorder point (`available_qty=2` < `reorder_point=3`) and stays untouched by orders (China Warehouse has more allocatable stock, so demand routes there instead) — remains flagged after allocation.
- `PART-CAL-015` has **no inventory row at all** — the one order that requests it (`SO-2026-019`) fully backorders.
- `DIAG-OCT-005` (HK Warehouse) has only 4 units available; the first order against it (`SO-2026-005`, qty 5) partially allocates.

### Invoices (`sample_invoices.xlsx`) — 27 rows

- `INV-2026-001`: high-priority by both age (>60 days overdue) and amount (≥50,000) — the flagship example.
- `INV-2026-024`: missing `due_date` (data issue).
- `INV-2026-025`: negative `invoice_amount` (data issue).
- `INV-2026-026`: deliberate overpayment — `paid_amount` (8,500) exceeds `invoice_amount` (7,000), documented here rather than treated as a data issue.
- All 5 aging buckets (`Current`, `1-30 Days`, `31-60 Days`, `61-90 Days`, `90+ Days`) and all three payment states (Paid / Partial / Unpaid) are represented. `payment_status` is decorative context only — the business logic in `src/payment_aging.py` derives aging/priority from dates and amounts, never from this column.

## Known intentional mismatches

- `SO-2026-030`'s `sku` (`VIS-SCOPE-099`) does not exist in `sample_product_master.xlsx` — intentional (unknown-SKU case).
- A handful of invoices leave `order_id` blank (`INV-2026-025`, `INV-2026-026`, `INV-2026-027`) — intentional, representing invoices not tied to a single tracked order (e.g. consolidated billing). `order_id` on invoices is optional context only, never read by `src/payment_aging.py`.
