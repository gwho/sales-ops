## Why this order

`context/build-plan.md` Phase 2 exists specifically to catch field-name gaps between the Phase 1 contracts and real demo data before Phases 3-5 implement business rules against them. No business logic was written — `sample_data.py` only builds fictional DataFrames and writes them to disk, and `contract_fixtures.py` only holds hand-authored dicts.

## Key decisions (resolved via `/architect` before implementation)

1. **Date anchoring**: `generate_invoices()` and `write_sample_workbooks()` take `reference_date: date | None = None`, resolved to `date.today()` inside the function body (not as a literal default — a literal `date.today()` default is evaluated once at import time and would freeze). Invoice due dates are authored as offsets from `reference_date`, so the Current/overdue mix stays believable no matter when the workbook is regenerated. Orders and inventory use plain fixed dates since their rules (`OV-005`, allocation) don't depend on "today".
2. **Fixture location**: `tests/contract_fixtures.py`, not inline in test functions and not a new `src/` module — `context/architecture.md`'s module-boundary table doesn't name a fixtures module, and Phase 7 (UI planning) needs to read these values directly, not just see them proven correct by a test run.
3. **`REPORT_MANIFEST_FIXTURES`** is a list of 3 (one per required Phase 6 report), the one deliberate exception to "one fixture per family" — a single manifest example wouldn't demonstrate the family's real reuse pattern.

## Scope boundary held

- `tests/test_sample_data.py` asserts the *data shape* of `generate_invoices()` output (due-date offsets, presence of the missing-due-date row) but never computes `days_overdue`, `aging_bucket`, or `follow_up_priority` — those rules belong to Phase 5 (`payment_aging.py`), which doesn't exist yet.
- `tests/contract_fixtures.py` fixtures are hand-authored, not generated by `sample_data.py` and not proof that any business-rule module is implemented.
