# Phase 2: Sample Data and Contract Fixtures — AI Discussion Topics

## On time-dependent defaults

1. "Why does `def f(x=some_call())` only evaluate `some_call()` once, at import time? What other Python gotchas share this root cause (mutable default arguments)?"
2. "The `None`-sentinel pattern (`x: T | None = None`, then `if x is None: x = default()`) is the standard fix. Are there cases where you'd want the frozen-at-import-time behavior instead?"

## On demo data that ages gracefully

3. "This project's invoice due dates are generated relative to `date.today()` so the demo doesn't 'expire.' What other kinds of portfolio or demo data have this same staleness problem, and how would you detect it before an interview?"
4. "What's the trade-off between committing generated files to git (as this repo does with `sample_data/*.xlsx`) versus generating them fresh on every run? When would you choose differently?"

## On fixtures vs. generated data vs. business logic

5. "`tests/contract_fixtures.py` fixtures are hand-authored, while `sample_data/*.xlsx` is generated by code. Why keep those as two separate mechanisms instead of deriving the fixtures from the generated workbooks?"
6. "The plan explicitly avoids computing `days_overdue`/`aging_bucket` in Phase 2 tests to prevent 'reimplementing Phase 5 early.' What's the risk of a test quietly encoding business logic before the module that owns that logic exists?"

## On testing generated data instead of hardcoded data

7. "`tests/test_sample_data.py` asserts things like 'exactly one duplicate order_id' and 'exactly one row below reorder point' about *generated* data. Is this testing the generator, the data, or both? What would break this test in a way that's actually useful to catch?"
8. "The reorder-point scenario was deliberately authored as a static fact (`available_qty < reorder_point` as written) rather than the result of simulated allocation. What made that the right call here, and when would simulating the real algorithm in a test be worth the coupling?"

## On the overall Phase 2 boundary

9. "Phase 2 explicitly produces no business-rule code, yet the sample data is constructed so that Phase 4's future allocation engine will exhibit partial allocation and backorders once it runs (`MED-LENS-001` demand exceeds supply by design). Is pre-shaping data for a not-yet-written algorithm a form of scope creep, or is it exactly what 'sample data as infrastructure' should do?"
10. "How does having Contract Fixtures ready *before* Phase 7 (UI planning) change what kind of questions a frontend/TypeScript planning session can usefully ask, compared to planning against bare `TypedDict` shapes alone?"
