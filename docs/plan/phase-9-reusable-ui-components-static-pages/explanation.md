# Explanation — Feature phase-9-reusable-ui-components-static-pages: Reusable UI Components and Static Pages

## 1. The build error that forced 3 pages to become Client Components

The `/architect` plan for this phase was explicit that `DataTable` would be the only Client Component among the interactive workflow pieces — everything else, including the pages that use it, was expected to stay a Server Component by default. That assumption broke the first time the shell was wired up and a real page tried to use `DataTable`.

`next build` failed while prerendering `/inventory-allocation` with: `Error: Functions cannot be passed directly to Client Components unless you explicitly expose it by marking it with "use server".` The root cause is a fundamental React Server Components serialization rule, not a bug in this codebase: props passed from a Server Component to a Client Component must be serializable, and JavaScript closures are not serializable — there is no way to send a function reference across the server/client boundary the way a string or number can cross it. `DataTable`'s `columns` prop is an array of `DataTableColumn<T>` objects, and each one carries a `render: (row: T) => ReactNode` function (and optionally `sortValue`). Every page that builds that array — `order-validation`, `inventory-allocation`, `payment-aging` — was, at the moment of the error, still a plain Server Component function calling `<DataTable columns={ERROR_COLUMNS} ... />`, which is exactly the forbidden crossing.

The fix actually applied: mark the three workflow pages themselves `"use client"`, not just `DataTable`. Once a page is a Client Component, passing a function-carrying object to another Client Component is unremarkable — the restriction only applies at the Server→Client boundary, not Client→Client. `dashboard` and `reports` never construct a `DataTable` column array (they only pass primitive values — strings, numbers, `ReportManifest` objects — into `MetricCard` and `ReportCard`), so they were left as Server Components exactly as planned.

The `context/ui-registry.md` `DataTable` entry and this feature's `plan.md` both record the general rule for future readers: **any page that builds a `DataTable` columns config must itself be a Client Component.** The alternative — keeping the page server-rendered and moving the column definitions into a small dedicated Client child that receives only serializable `data` — was considered but not used here, because it would have meant one extra file per table (9 tables across 3 pages) for a phase whose primitives were already deliberately kept minimal. That alternative is worth revisiting in Phase 10 if these pages grow real interactive Server-only concerns (e.g. server-side data fetching) that would benefit from staying server-rendered.

## 2. Why StatusBadge never infers its own tone

`context/ui-rules.md`'s Status Badges section defines which contract fields get badges, but not which color each value should render as — that mapping had to be authored during this phase. The naive design would have been a single lookup function inside `StatusBadge` that takes any status string and returns a tone. That design was rejected during planning because the same literal string means different things in different contracts: `"High"` appears as `ValidOrderRow.priority` / `AllocationResultRow.priority` (an *importance* ranking — a high-priority order isn't a problem, it just deserves attention) and separately as `PaymentAgingRow.follow_up_priority` (an *urgency-of-problem* ranking — a high-priority follow-up means someone is dangerously overdue). A single string-keyed lookup would have to guess which meaning applies, or worse, silently pick one and be wrong half the time.

The resolution: `StatusBadge` itself stays a dumb wrapper around `Badge` — it takes `status` (the label to render) and an explicit `tone` prop, nothing else. Next to it in `components/workflow/StatusBadge.tsx` live six small, typed, exported pure functions — `severityTone`, `allocationStatusTone`, `importancePriorityTone`, `agingBucketTone`, `followUpPriorityTone`, `reportLifecycleTone` — each one taking a specific contract's literal union type as its parameter, so TypeScript itself prevents calling `importancePriorityTone` with a `follow_up_priority` value or vice versa. Every call site in the three workflow pages picks the correct helper explicitly, e.g. `<StatusBadge status={r.priority} tone={importancePriorityTone(r.priority)} />` in `order-validation/page.tsx` versus `<StatusBadge status={r.follow_up_priority} tone={followUpPriorityTone(r.follow_up_priority)} />` in `payment-aging/page.tsx`. The full fixed mapping — decided before any component code was written, during the `/architect` session, specifically so `/imprint` would document implementation rather than in-flight decision-making — is recorded in `context/ui-registry.md`'s `StatusBadge` entry; that table, not this paragraph, is the source of truth if the two ever drift.

Two mappings are worth calling out because they're not derivable from `ui-tokens.md` alone: `RemainingInventoryRow.reorder_alert` is a `"Yes"`/`"No"` string, but the badge only ever renders for `"Yes"` (as the label `"Below Reorder Point"`, tone `warning`) — when `reorder_alert` is `"No"`, the Inventory Allocation page renders plain text (`"No"`) instead of a badge, because a badge implies something worth calling out, and "nothing to report" isn't. Similarly, `"Supplier Follow-up"` as a label is never a value of any contract field — it's rendered only because a `(sku, warehouse)` pair appears in the `supplier_follow_ups` list, i.e. it's derived from list membership, exactly as `ui-contract-plan.md`'s glossary defines a "derived" badge.

## 3. Confirming the shadcn hand-write decision held under real component pressure

Phase 8's review had already flagged that `components.json`'s `baseColor: "slate"` would inject shadcn's own `--primary`/`--card`/`--destructive`/`--ring` variables into `app/globals.css` the first time `npx shadcn add` ran, colliding with this project's `--accent` (brand blue vs. shadcn's gray hover) and the CRITICAL "never modify design tokens without approval" rule. That was a prediction at the time — Phase 8 shipped no primitives, so it was never tested. This phase is where it got tested, by actually needing `Button`, `Card`, `Badge`, and a `Table` shell for the first time.

The decision made in `/architect` planning was to hand-write all four rather than generate-then-strip. In practice this meant reading shadcn's public `new-york`-style reference implementations for API shape and variant conventions (e.g. `cva` for variants, `forwardRef` for `Button`), then rewriting every class list against `ui-tokens.md`'s actual token names instead of shadcn's. The verification step that matters here isn't a subjective read of the code — it's `git diff app/globals.css`, run as part of the final verification pass, which came back empty. That's the concrete proof the collision Phase 8 predicted never happened: no shadcn CLI was ever invoked, so there was nothing for it to inject.

One consequence worth noting for Phase 9's successor: `components.json` still exists with `baseColor: "slate"` unchanged. It was left as-is rather than deleted, because a future primitive that genuinely needs `shadcn add`'s scaffolding speed can still use it — but whoever runs that command must manually revert whatever it injects into `globals.css` before committing, exactly as the Phase 8 review warned. This phase didn't need to exercise that risk at all, since every primitive it needed was small enough to hand-write directly.

## 4. What "static showcase" meant concretely for three specific components

The single biggest product decision in this phase's planning was rejecting a simulated-live-app model (upload → fake processing delay → results appear) in favor of a static showcase (mock data renders immediately; the upload UI is present but doesn't gate anything). That decision had to be translated into specific, sometimes counterintuitive implementation choices for three components:

`ReportCard` supports all 4 lifecycle states (`Needs Input`, `Not Generated`, `Processing`, `Ready`) as a discriminated union prop type — `{ state: "Ready"; manifest: ReportManifest } | { state: Exclude<ReportLifecycleState, "Ready">; reportTypeLabel: string }` — but every actual call site in `app/dashboard/page.tsx` and `app/reports/page.tsx` passes `state="Ready"` with a real manifest from `lib/mock-data.ts`, because in a static showcase every report "has already run." The other three states exist purely so the component's prop API is exercised in principle and captured by `/imprint`, not because any page currently renders them. The `Ready` state also includes a `Button` labeled "Download .xlsx" that is permanently `disabled` with an explanatory `title` — there is no generated `.xlsx` file anywhere in the repo for it to link to (report generation happens at Python runtime, not as a committed artifact), so a working download link was never in scope for this phase; a fake/decorative-but-clickable button would have been more misleading than an honest disabled one.

`WorkflowStepper` was originally scoped in the Figma reconciliation notes as "inherently client-only progress," a description of how the Figma prototype behaved, not a Phase 9 requirement. Once static showcase was confirmed, the component was redesigned as a Server Component that takes `steps: string[]` and `currentStep: number` as plain props — no `useState`, no client bundle cost. Every page calls it with `currentStep={STEPS.length - 1}`, i.e. it always displays the final step ("Review Results") as current, which is the honest representation of "this workflow has already run with static data," not a lie dressed up as a live indicator.

`UploadPanel` is the one piece of this phase that is genuinely, fully interactive — it's a real `<input type="file">` with local `useState` tracking the selected filename, and choosing a file actually updates the visible label. What it deliberately does not do is parse the file, validate it, or use its presence/absence to decide whether to show the tables below it. The tables render from `lib/mock-data.ts` regardless of whether a file was ever chosen. This was a specific instruction from the developer during plan review: no fake async run logic, no fake timers, no client-side workflow state machine — Phase 10's real FastAPI integration is what should introduce genuine gating, and building a fake version of that gating now would be throwaway work.

## 5. DataTable's sort design, and what was deliberately left out

`DataTableColumn<T>` was shaped to resemble what a TanStack Table column definition would eventually need — `key`, `header`, `render`, and optionally `sortable`/`sortValue` — specifically so that if Phase 10 brings real multi-row API data and TanStack Table becomes justified, the page-level column definitions (the parts every page author already had to write by hand) don't need to be rewritten, only the internals of `DataTable` itself. Sorting is intentionally minimal: a single `SortState` (`{ key, direction } | null`) held in local `useState`, toggled by clicking a `sortable` column's header, with a plain three-way comparator (`va < vb ? -1 : va > vb ? 1 : 0`) run through `Array.prototype.sort` on a shallow copy of the row array. There is no multi-column sort, no filtering, and no pagination — all three were explicitly deferred during planning because the mock fixtures generated by `scripts/generate_mock_data.py` are 1-2 rows per table, so none of that complexity has anything real to demonstrate yet; adding it now would mean writing and testing behavior against data that can't actually exercise it.

`DataTable` also owns its own empty-row handling: if `data.length === 0`, it renders `EmptyState` instead of a `Table`, using the `emptyTitle`/`emptyDescription` props each page supplies per table (e.g. Inventory Allocation's Backorders table passes `"No backorders."` / `"Every requested quantity was fully allocated."`). This is real, correct logic — not a fake state — even though none of the current mock fixtures actually produce a zero-row array to exercise it; every mock array in `lib/mock-data/*.json` has at least one row. The check exists because it will matter as soon as real, variable-length API data arrives in Phase 10, and writing it now costs nothing.

## 6. Where the Figma Reference Reconciliation corrections actually show up in code

Phase 8's review of four Figma Make prototypes catalogued several inventions and mismatches that had to be actively avoided while writing this phase's pages, not just remembered in the abstract. Concretely: `components/workflow/StatusBadge.tsx`'s `importancePriorityTone` only accepts `"High" | "Normal" | "Low"` — TypeScript itself rejects Figma's fourth value, `"Critical"`, at the type level, not just by convention. `allocationStatusTone` only accepts `"Fully Allocated" | "Partially Allocated" | "Backordered"` — Figma's `"Invalid SKU"` and `"Supplier Follow-up"` as status values are structurally impossible to pass in, because `AllocationResultRow.status` in `types/index.ts` never included them. `app/inventory-allocation/page.tsx`'s `SUPPLIER_FOLLOW_UP_COLUMNS` renders exactly `sku`, `warehouse`, `remaining_qty`, `reorder_point`, `supplier_name`, `lead_time_days` — Figma's invented `shortageQty`/`suggestedReorderQty`/`followUpPriority` have no columns because `SupplierFollowUpRow` has no such fields to read. `followUpPriorityTone` includes `"Watch"` and `"None"` as real, distinct tone cases (`neutral`), so the Payment Aging table never falls back to Figma's `"—"` placeholder. The $50,000 high-priority threshold itself lives in `src/payment_aging.py` and was never touched by this phase — it surfaces correctly here simply because `PaymentAgingRow.follow_up_priority` and `PaymentAgingSummary.high_priority_count` are read directly from the mock JSON, which was generated from the same Python fixtures. `SidebarNav.tsx`'s `NAV_ITEMS` constant has exactly 5 entries, not Figma's 8-item mega-nav.

The verification pass added a grep for the literal Figma-vocabulary strings (`"Critical"`, `"Invalid SKU"`, `40000`/`40,000`, `"—"`) across `app/`, `components/`, `lib/`, `types/` as a final check. It returned several matches, but all were false positives worth recording so a future run of the same grep isn't misread as a regression: the em-dash matches are all `?? "—"` fallback placeholders for genuinely optional fields (`product_name`, `sales_owner`, `warehouse`, etc.) — unrelated to the Payment Aging priority vocabulary the em-dash check was actually targeting — and the `"Invalid SKUs"` match is `ValidationSummary.invalid_skus`, a real KPI count label, not Figma's allocation-status value `"Invalid SKU"` (singular, different field entirely).

## 7. The mock-data and formatting helpers, and the one genuinely new derived aggregate

`lib/mock-data.ts` and `lib/formatters.ts` were added this phase specifically because `context/architecture.md`'s "Future Frontend Shape" diagram had already named both files back when the scaffold shape was first documented, and Phase 8 had only built the JSON generation half of that plan (`scripts/generate_mock_data.py` → `lib/mock-data/*.json`), not the typed-access half. `lib/mock-data.ts` does one job: `import` each of the four committed JSON files and assert them against the corresponding type from `types/index.ts` (`as OrderValidationResult`, etc.) exactly once, so every page imports typed data from `@/lib/mock-data` rather than repeating the type assertion at every call site.

`lib/formatters.ts`'s `formatDate` has a subtlety worth documenting because it's the kind of bug that's invisible until someone in the wrong timezone loads the page: contract dates like `ValidOrderRow.order_date` are plain `"YYYY-MM-DD"` strings with no time or timezone component. Per the ECMAScript `Date` parsing spec, a date-only ISO string is parsed as UTC midnight, so `new Date("2026-03-31")` displayed via `.toLocaleDateString()` in any timezone west of UTC renders as March 30, not March 31 — an off-by-one that would have been easy to ship unnoticed in a demo running in UTC and only surface for a reviewer in, say, US Pacific time. `formatDate` avoids this entirely by splitting the string into `[year, month, day]` and constructing `new Date(year, month - 1, day)` using the local-time constructor overload, which has no UTC-conversion step. `formatDateTime`, by contrast, is given `ReportManifest.generated_at`, an ISO string that *does* include a time component but no timezone suffix (`"2026-07-09T09:15:00"`) — per the same spec, a date-time string without an explicit offset is parsed as local time already, so `new Date(isoDateTime)` is safe there without the same manual-parsing workaround.

The one place this phase introduced a genuinely new (if small) piece of derived logic is `lib/mock-data.ts`'s `ninetyPlusDaysAmount()`. `context/ui-contract-plan.md`'s KPI mapping calls for a "90+ Days Amount" tile on the Payment Aging page, but `PaymentAgingSummary.aging_bucket_counts` is a *count* breakdown (`Record<AgingBucket, number>`), not an amount breakdown — there is no field anywhere that already holds this number. `ninetyPlusDaysAmount()` filters `aging_rows` to `aging_bucket === "90+ Days"` and sums `outstanding_amount`, which is exactly what `ui-contract-plan.md`'s "Derived Display-Only Aggregates" section defines as an acceptable client-side aggregation — it reuses existing fields with an explicit, documented grouping rule, and introduces no new business interpretation the Python core doesn't already produce.

## 8. Verification, and what each check was actually chosen to catch

The verification pass run for this phase deliberately went beyond "the build succeeded." `npm run build`/`lint`/`typecheck` catch type errors and lint violations but say nothing about whether the pages actually render real data — so after the RSC function-prop failure was fixed and the build was green, the production server was started on a spare port and each route was curled twice: once for a 200/307 status check, and a second time with the response body grepped for real contract content (`"Total Orders"`, `"Fully Allocated"`, `"Total Outstanding"` on `/dashboard`; `"OV-003"`, `"Validation Errors"`, `"Valid Orders"` on `/order-validation`) to confirm the pages render actual mock data rather than lingering Phase 8 placeholder text. The raw-color/hex grep (reused verbatim from the Phase 8 review's pattern) returned no matches across `app/`, `components/`, `lib/`, `types/`. `git diff app/globals.css` returning empty is the concrete proof, not just an inference, that the shadcn hand-write decision (see section 3) held. `uv run pytest` was re-run and still reports 152 passed, confirming the Python core was genuinely untouched by an entire phase of frontend work.
