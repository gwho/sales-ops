# Explanation — Feature phase-11-deployment-baseline: Deployment Baseline

## 1. Why this phase exists as "Deployment Baseline" rather than the originally planned SQL-reporting phase

The Phase 11 slot in `context/build-plan.md` was not always this feature. A prior planning session (`/grill-with-docs` + `/architect`) had already produced a fully approved, decision-complete plan for a different Phase 11: a SQLite-backed `Demo Reporting Database` seeding a live `GET /api/dashboard` endpoint, replacing `/dashboard`'s static mock JSON with real queries. That work was actually started — schema (`backend/reporting_db/schema.py`), hand-written SQL queries and a `DashboardResponse` TypedDict (`backend/reporting_db/queries.py`), a router, and a `src/demo_constants.py` pinned-date module were all written and smoke-tested against real `sample_data/*.xlsx` output before the session ended, including a genuine mid-build correction: the original plan had been written assuming pre-Phase-10.2's ~15-tile dashboard, and had to be revised down to 5 tables/trimmed fields once it became clear Phase 10.2 had already consolidated the dashboard to one 5-card Overview row plus 2 charts plus 2 tables.

That work was then explicitly paused by the user, mid-session, with a directional instruction: deploy the app to a public URL first, and build a database-backed dashboard later on Postgres instead of SQLite. This is a case where "started implementing" and "shipped" diverged — the SQLite code was real and working, not abandoned for being broken, but was set aside for a sequencing reason (a working demo with a public URL is worth more, right now, than a more sophisticated dashboard nobody outside the repo can see). Rather than deleting real, tested work, it was preserved with `git stash push -u -m "Phase 11 SQLite reporting WIP (paused): schema, queries, dashboard endpoint, lifespan seed hook, demo_constants" -- <pathspec>`, using explicit pathspecs so that files needing to stay in the working tree (see §2) weren't swept into the stash along with the ones that needed to leave it. It sits today as `stash@{0}`, inspectable via `git stash show -p stash@{0}`, named specifically so a future session skimming `git stash list` understands what it is and why it's there without needing to read this document first.

"Phase 11" was then reassigned to Deployment Baseline — the feature this document describes — and the old SQL-reporting plan was renamed in intent (not in its file paths) to become input for a *future* Phase 12, which will target Postgres instead of SQLite. The original plan file (`/Users/jessejames/.claude/plans/yes-the-plan-aligns-smooth-barto.md`) and its ADR (`docs/adr/0007-sql-reporting-seeds-from-tested-workflow-outputs.md`) were left untouched as accurate historical records of a real planning session, not deleted or rewritten to pretend they never happened.

## 2. Splitting one file's diff across a stash and a clean rewrite: the `backend/main.py` problem

The paused SQLite work had already modified `backend/main.py` twice over: once to add a `lifespan` hook that called `rebuild_demo_reporting_database()` on startup, and once to register the new dashboard router. Both of those changes needed to leave the working tree along with the rest of the SQLite feature. But Deployment Baseline *also* needed changes to that same file — the CORS and health-check work described in §3 — and those needed to stay.

`git stash push -u` operates on whole files' worth of diff, not sub-file hunks, so there was no way to stash "half" of `backend/main.py`'s change set in one command. The approach taken: stash the *entire* `backend/main.py` diff first (which reverts the file to its pre-session `HEAD` state, discarding both the lifespan hook and the CORS/health changes together), then re-apply only the CORS/health portion by hand, writing it fresh against the clean, reverted file rather than trying to cherry-pick lines out of a diff. This is more reliable than attempting a manual partial-diff edit (which risks silently leaving stray context lines or mismatched hunks) at the cost of a small amount of retyping. The result, verified afterward: `backend/main.py` in the working tree has zero import or call dependency on anything inside `backend/reporting_db/` — confirmed by running `uv run pytest` (still 196 passing) and a manual `TestClient` check that `GET /api/dashboard` correctly 404s (proving the dashboard router genuinely isn't registered, not just that its import was silently swallowed).

## 3. Why `CORS_ALLOWED_ORIGINS` is an env var read once at startup, not per-request

Locally, the app had always had `allow_origins=["http://localhost:3000"]` hardcoded directly into the `CORSMiddleware` constructor call inside `backend/main.py`. That's fine when there's exactly one possible frontend origin and it never changes. Once the frontend has a real deployed URL (`https://sales-ops-gamma.vercel.app`) that's different between local dev and production, that hardcoded value has to become configurable without touching source code per environment — the standard reason for pushing a value out to an environment variable.

The implementation, `_cors_origins()`, reads `os.environ.get("CORS_ALLOWED_ORIGINS", _DEFAULT_CORS_ORIGINS)`, splits on commas, strips whitespace from each piece, and drops any resulting empty strings — so a stray trailing comma or extra whitespace in the Render dashboard's env var field (an easy typo to make when pasting a URL) never produces a blank or malformed origin entry that could either silently do nothing or, worse, evaluate to something unexpectedly permissive. The important constraint to know about `CORSMiddleware` itself: `add_middleware()` is called once, at app-construction time, as a side effect of importing `backend/main.py` — it is not re-evaluated per request. That means `_cors_origins()` runs exactly once, whatever the environment looks like at process start. Editing the `CORS_ALLOWED_ORIGINS` value in Render's dashboard and clicking save does *not* change the running process's behavior; only a redeploy (or process restart) picks up the new value. This is a real operational gotcha worth remembering the next time an env var change on Render doesn't seem to take effect.

The `Origin` header's exact-string-match nature is also why the CORS trailing-slash bug (§6) was a real risk here, not a theoretical one.

## 4. Why `GET /health` deliberately checks nothing

Render's Web Service configuration has a "Health Check Path" field that it polls periodically; if that endpoint stops responding successfully, Render can conclude the service is unhealthy and restart it. The natural instinct once a project has *any* form of persisted state is to make the health check verify that state is reachable (e.g., a database ping) — but this app, in this phase, has no database and no external dependency to check. `backend/routers/health.py`'s own module docstring states the reasoning directly: this is deliberately scoped to Phase 11, and if a future phase's health check ever needs to verify something like the (not-yet-built) `Demo Reporting Database`, that's a decision for whichever phase introduces that dependency, not something to speculatively add now. Making `GET /health` a zero-dependency, always-succeeding-if-the-process-is-up check keeps it doing exactly one job — proving the process is alive — and avoids a scenario where a slow or flaky dependency check causes Render to kill and restart an otherwise perfectly healthy API process.

## 5. `.env.example` being silently gitignored, and how it was caught

Immediately after creating `.env.example` with its two documented (but empty) variables, a routine `git status` before staging showed the new file simply wasn't listed — neither as untracked nor as anything else. That's the signature of a file being gitignored rather than merely unstaged: `git add .env.example` followed by `git status` again confirmed nothing had actually been staged. Running `git check-ignore -v .env.example` pinpointed the exact rule: the pre-existing blanket pattern `.env*` in `.gitignore` (there specifically to keep any real `.env` file with actual secrets out of version control) was broad enough to also match the literal filename `.env.example`, which was never meant to be excluded — it's a template with empty values, meant to be committed so other developers (or a future session) know which env vars exist at all.

The fix is the standard gitignore negation pattern: adding `!.env.example` on the line immediately after the `.env*` rule. Gitignore evaluates patterns in file order, and a later negation pattern can re-include a path that an earlier pattern excluded — but only if no *directory* in the path was excluded (irrelevant here, since `.env.example` lives at the repo root) and only if the negation appears after the exclusion it's meant to override. Placing it anywhere before the `.env*` line would have had no effect, since `.gitignore` patterns are evaluated top-to-bottom and a later blanket pattern would simply re-exclude it again.

## 6. The CORS trailing-slash bug: caught in local notes before it ever reached production

While the user was filling in the Render dashboard's `CORS_ALLOWED_ORIGINS` field, they had pasted the Vercel URL with a trailing slash into a local scratch note first — `https://sales-ops-gamma.vercel.app/` rather than `https://sales-ops-gamma.vercel.app`. This was caught before it was ever entered into Render, by reasoning through what `_cors_origins()` actually does with its input: it trims whitespace and commas from each comma-separated entry, but it has no logic to strip a trailing slash. Browsers, meanwhile, never include a trailing slash in the `Origin` header they send on a cross-origin request — `Origin` is always scheme + host + optional port, with no path component at all, trailing slash or otherwise. So a `CORS_ALLOWED_ORIGINS` value ending in `/` would never exact-string-match any real browser's `Origin` header, and every cross-origin request from the deployed frontend would have been silently rejected by `CORSMiddleware` — a failure mode that would have looked, from the browser console, like a generic CORS error with no indication that a single trailing character was the entire cause.

The value was corrected to the slash-free form before being entered into Render, and the fix was verified directly rather than assumed: a live `curl` `OPTIONS` preflight request from the deployed frontend's actual origin against the deployed backend returned `access-control-allow-origin: https://sales-ops-gamma.vercel.app`, an exact match with no trailing slash on either side. This is the kind of bug that's cheap to prevent by reasoning about the exact comparison being performed, and expensive to diagnose after the fact, since the resulting browser error gives no hint that "the value looked right" was the whole problem.

## 7. Deploy sequencing resolves a genuine circular dependency between the two hosts

The two services need each other's URLs before either can be fully configured: the frontend's `NEXT_PUBLIC_API_BASE_URL` needs to point at the backend's real Render URL, but the backend's `CORS_ALLOWED_ORIGINS` needs to list the frontend's real Vercel URL — and neither URL exists until that service has been deployed at least once. This was resolved with a fixed order, not a guess: deploy Render first (its URL is a stable function of the service name Render assigns, independent of any frontend config, so it can go out with a placeholder or default CORS value); take that URL and set it as Vercel's `NEXT_PUBLIC_API_BASE_URL`, then deploy Vercel; take *that* URL and set it as Render's `CORS_ALLOWED_ORIGINS`, then redeploy Render so the new env var value is actually picked up (per the once-at-startup constraint in §3). Getting this order backwards — deploying Vercel first while pointed at no backend, or deploying Render with CORS still pointed at `localhost:3000` — would have produced a working-looking deploy that silently fails the moment a real user's browser tries to call the API.

`lib/api-client.ts`'s existing fallback, `process.env.NEXT_PUBLIC_API_BASE_URL ?? "http://127.0.0.1:8000"`, meant the frontend could still be built and deployed to Vercel even transiently without the env var set (it would just point at a URL unreachable from a real user's browser) — useful in practice, since it meant the very first Vercel deploy didn't have to be blocked on having the Render URL in hand at the exact moment of clicking "Deploy," only before real end-to-end testing.

## 8. Choosing two independent hosted services over a single container, and why Render specifically

The plan explicitly considered and rejected packaging both the Next.js frontend and the FastAPI backend into a single container/host. The reasoning: the local development shape this project has used since Phase 10 is already two independent processes talking over HTTP with CORS (`127.0.0.1:8000` and `localhost:3000`) — mirroring that shape in production means the deployment topology is the *same* topology already being developed and tested against locally, rather than introducing a third, deploy-only topology (a single process serving both, or a reverse proxy in front of both) that only gets exercised once, at deploy time, with no local equivalent to catch problems early.

Within "two services," Vercel for the Next.js frontend was close to a default choice — zero-config, and this repo's `package.json` already sits at the repo root, so Vercel's build auto-detection works without any special root-directory configuration despite the unrelated `backend/`, `src/`, and `sample_data/` directories living alongside it. The backend needed a host that actually runs a long-lived Python process (ruling out purely static or edge-function-oriented hosts) — Render's Free Web Service tier was chosen after weighing it against other Python-capable free tiers considered during the earlier `/architect` session, on cost/complexity grounds, with the known and accepted trade-off that Render's free tier sleeps a service after roughly 15 minutes of inactivity, adding up to about a minute of cold-start latency to the first request after an idle period. That trade-off is documented for the end user in `README.md`'s Live Demo section rather than engineered away with a scheduled keep-alive ping or a paid tier — a deliberate choice to accept a real, minor rough edge on a portfolio demo rather than add ongoing cost or a moving part (a cron job hitting `/health` every 10 minutes) whose only purpose is to work around a free-tier limitation.

Render's dashboard auto-suggested a build command using `uv` directly (`uv sync --frozen --no-dev && uv cache prune --ci`) once the Python runtime was selected — confirming empirically that `uv` is natively available in Render's Python build environment, which meant the plan's originally-considered fallback step (`python -m pip install uv` before `uv sync`) turned out to be unnecessary in practice, discovered by simply trying the suggested command rather than pre-emptively including the extra step.

## 9. Verifying the live deployment with direct `curl`, not by assuming a green dashboard means it works

Both Render's and Vercel's dashboards showed successful deploy status once each service finished building, but a dashboard reporting "deployed" only confirms the build and start commands didn't error — it says nothing about whether the running process actually serves correct responses, has the right CORS configuration, or renders real content rather than an error page. Verification instead ran a small, direct sequence of `curl` commands against the live URLs themselves: `GET /health` on the Render URL, confirmed `200 {"status":"ok"}`; `GET /` on the Vercel URL, confirmed a `307` redirect to `/dashboard` (matching this app's existing, pre-Phase-11 root-redirect behavior — not a new behavior introduced by deployment) followed by a `200` on `/dashboard` itself, with the response body checked for real rendered content specific to this app's known dataset (`<title>Sales Admin Automation Toolkit</title>` plus chart titles reflecting the actual known allocation counts: 26 fully allocated, 1 partially allocated, 1 backordered) rather than just a `200` status code, which alone wouldn't distinguish a correctly rendered page from a blank error boundary; and an `OPTIONS` preflight request with the deployed frontend's `Origin` header set, confirmed to receive back an exactly matching `access-control-allow-origin` value (see §6). Each of these three checks targets a different possible failure mode — process not running, wrong content being served, and cross-origin misconfiguration — and passing all three together is meaningfully stronger evidence of a working deployment than a dashboard's "Live" badge alone.
