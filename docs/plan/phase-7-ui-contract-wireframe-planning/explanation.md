# Explanation — Feature phase-7-ui-contract-wireframe-planning: UI Contract and Wireframe Planning

## 1. Why this phase is a documentation-correction task as much as a documentation-creation task

Phase 7's stated job, per `build-plan.md`, is to define TypeScript interfaces, routes, table columns, KPI cards, status badges, report cards, and empty/loading/error states from "the Phase 1 contracts and Phase 2 contract fixtures." On paper that sounds like pure creation: read the contracts, write the plan. In practice, three planning surfaces already existed before this phase started — `context/ui-rules.md`'s Status Badges section, and the three guidance folders (`sales_admin_automation_toolkit_ui_specs/`, `ui_reference_to_figma_workflow/`, `ui_prompts_for_agents_mcp/`) — and all of them were written before Phases 3–6 existed, meaning before the real field names, real status values, and real envelope shapes existed. `ui_specs`' table plans used `requested_quantity`; the real field, locked in by Phase 4, is `requested_qty`. `ui-rules.md`'s badge vocabularies for Order Validation and Reports referenced concepts (`Missing Field`, `Needs Input`) that don't correspond to any field in the actual `ValidationErrorRow` or `ReportManifest` shapes.

This meant the actual first step of Phase 7 wasn't writing new content — it was diffing old planning prose against `src/contracts.py`, `src/order_validation.py`, `src/inventory_allocation.py`, and `src/payment_aging.py` as they actually exist today, and treating any drift as a bug to fix, not an inspiration to build on. `context/ui-contract-plan.md` is the new-content half of this phase; the `context/ui-rules.md` Status Badges rewrite is the correction half.

## 2. The Status Badges rewrite: how a partially-right answer got caught

The first draft of the plan corrected only Order Validation and Reports — the two groups research had flagged as having *no* real-field backing at all. Inventory Allocation and Payment Aging were left as "already correct," because spot-checking a few labels (`Fully Allocated`, `Current`, `High`) against `AllocationResultRow.status` and `PaymentAgingRow.aging_bucket`/`follow_up_priority` confirmed they were real values.

The user's correction on this point is worth walking through carefully, because the failure mode it caught is subtle: those labels *were* real, but the list containing them also contained labels that weren't — `Below Reorder Point` (a display transform of `RemainingInventoryRow.reorder_alert`, not a value that field ever literally holds) and `Supplier Follow-up` (not a field value at all, but a signal that a row exists in the separate `supplier_follow_ups` list). None of this was flagged as different from the real values sitting next to them. Similarly, Payment Aging's old list mixed the two genuinely-controlled-vocabulary fields (`aging_bucket`, `follow_up_priority`) with four labels (`Overdue`, `Due Soon`, `High Priority`, `Paid`) that were never derived from anything — no code, no rule, no fixture ever produced a label called `"Overdue"`.

The fix applied the same direct/derived split to all four groups uniformly, so every label in `ui-rules.md`'s Status Badges section now states which category it is. This is a stricter standard than "the labels are correct" — it's "the *reason* each label exists is stated," which is what actually prevents an invented label from being copied into Phase 8 code unnoticed.

## 3. The Reports lifecycle: catching an inconsistency between the two docs written in the same session

While writing `context/ui-contract-plan.md`'s Report Cards section, a four-state client lifecycle was designed independently: `not_generated` → `generating` → `ready` → `error`. This was written before the corresponding `ui-rules.md` edit, and used different label text (`generating` vs. the existing `Processing`, `error` as a new state vs. the existing `Needs Input`) than what the approved plan had actually specified — the plan's decision 2 said to *keep* `ui-rules.md`'s existing four labels (`Ready`, `Not Generated`, `Processing`, `Needs Input`) and only annotate them as client-side state, not replace them with a new four-state model.

This was caught by a straightforward cross-check after both files were written: reading `ui-contract-plan.md`'s Report Cards section side-by-side with the freshly-rewritten `ui-rules.md` Status Badges section and noticing the label sets didn't match. The fix reconciled both documents to a single lifecycle, reusing all four original labels but giving them an explicit order and meaning: `Needs Input` (no result envelope yet — the workflow hasn't run) → `Not Generated` (envelope exists, export not yet called) → `Processing` (export in flight) → `Ready` (manifest received). A failed export reverts the card to `Not Generated` with an error message shown, rather than introducing a fifth persisted state that neither original list nor the plan asked for.

The general lesson: writing two related documents in the same session doesn't guarantee they agree with each other by default — verifying cross-document terminology at the end of a docs-writing pass caught a real inconsistency that git diffs alone wouldn't surface (both edits looked individually correct; only comparing them side by side revealed the mismatch).

## 4. The `ReportManifest` fixture/exporter discrepancy: a pre-existing bug found, documented, but deliberately not fixed

The plan's original design called for grounding every contract example in `tests/contract_fixtures.py`, since that file exists specifically to give Phase 7 realistic sample data (per a decision recorded in `docs/plan/phase-2-sample-data/plan.md`). Applying that rule to `ReportManifest` surfaced a real bug: `REPORT_MANIFEST_FIXTURES`'s `report_id` values (e.g. `"rpt-order-validation-20260709-001"`) don't match what `src/report_export.py`'s `_build_manifest()` function actually produces (`f"rpt-{report_type}-{generated_at:%Y%m%d%H%M%S}"`, e.g. `"rpt-order_validation-20260709091500"`). The fixture uses a hyphenated `report_type` and a date-only timestamp with a fake sequence suffix; the real code uses the literal underscored `report_type` string and a full `HHMMSS` timestamp with no sequence at all.

This is exactly the kind of drift the Field Scope Boundary discipline exists to catch — but fixing it would mean editing `tests/contract_fixtures.py`, a Python test-data file, during a phase whose explicit mandate (`build-plan.md`: "must not create production frontend code") and whose broader spirit (don't silently touch `src/`/`tests/` during a docs phase) both argue against it. The resolution was to source `ReportManifest` examples from the real exporter and its test file instead, document the mismatch explicitly in `ui-contract-plan.md` with both formats shown side by side, and flag it as a follow-up for whoever next touches Phase 2 or Phase 6 test data — rather than silently working around it or silently fixing it without dedicated review.

## 5. Why the field-name verification used a script instead of manual re-reading

After drafting all of `context/ui-contract-plan.md`'s TypeScript interfaces by hand from the source files, verification needed to confirm zero typos or drift — the exact failure mode found in the pre-existing guidance folders (`requested_quantity` vs. `requested_qty`). Manually re-reading 67 field names against four source files is exactly the kind of check that's easy to do carelessly under the illusion that "I just wrote this, it must be right." Instead, a small Python script extracted every field name from the TypeScript code blocks via regex, extracted every field name from the four Python source files (both `name: type` TypedDict syntax and `"name":` dict-literal syntax, since fixtures and dict literals in the business modules construct rows with string-keyed dicts) via regex, and diffed the two sets. Zero TS fields were missing from the Python set — meaning every field this doc claims exists really does exist, verified mechanically rather than by re-reading.

This matters as a pattern for any future doc that claims field-level correctness: an automated diff against source is strictly more trustworthy than a second manual read-through of the same material, because a manual re-read shares the same blind spots as the original writing pass.

## 6. Why the git preflight happened before any file was touched, not after

The approved plan included an explicit "item 0" — a git preflight check — before any Phase 7 content was written. This ordering was deliberate: `phase/6-excel-report-export` was found to be one commit ahead of `origin` (a post-review fix, `d21d787`), and PR #5 was open against that branch. Doing the preflight first, and getting an explicit user decision (push `d21d787`, then branch, rather than branch first and push later) meant the Phase 7 branch point was settled and confirmed before any content existed to potentially need rebasing or re-branching later. This mirrors the standing pattern from Phases 4–6 (documented in this project's session `memory.md`): always check live branch/push state rather than assuming the local tip matches origin, and resolve that before starting new work rather than after.
