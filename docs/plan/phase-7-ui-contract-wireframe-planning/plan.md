## Key invariants

- **Every field name in `context/ui-contract-plan.md`'s TypeScript interfaces must exist verbatim in `src/contracts.py` or one of the 3 business-module envelope definitions** (`OrderValidationResult` in `order_validation.py`, `InventoryAllocationResult` in `inventory_allocation.py`, `PaymentAgingResult` in `payment_aging.py`). This was verified this session via an automated field-name diff (67 TS fields, 0 mismatches) ŌĆö if a future contract field is renamed or added, `ui-contract-plan.md` must be updated in the same change, not left to drift the way the original `ui_specs` guidance folders did (`requested_quantity` vs. the real `requested_qty`).
- **A status badge label is either a verbatim contract-field value or an explicitly-documented derivation rule ŌĆö never an unlabeled invented label.** This is the rule the Status Badges section rewrite exists to enforce; any future badge addition to `ui-rules.md` or `ui-contract-plan.md` must state which category it is.
- **`ReportManifest` examples must come from `src/report_export.py`'s `_build_manifest()` / `tests/test_report_export.py`, not from `tests/contract_fixtures.py`'s `REPORT_MANIFEST_FIXTURES`.** The fixture's `report_id` format (`"rpt-order-validation-20260709-001"`) is stale against the real exporter's format (`"rpt-order_validation-20260709091500"`) ŌĆö this mismatch is documented but the fixture itself was deliberately left unfixed this phase (docs-only mandate). Do not copy `report_id` examples from the fixture file until it's corrected.
- **`CONTEXT.md` was deliberately not edited this phase.** UI-planning/process terms (`Status Badge`, `Derived Display Value`) live in `context/ui-contract-plan.md`'s glossary, not the root business-domain glossary. Future UI-planning terms should follow the same placement rule unless they describe an actual business-domain concept.
- **The Reports badge/report-card lifecycle (`Needs Input` ŌåÆ `Not Generated` ŌåÆ `Processing` ŌåÆ `Ready`) is purely client-side session state.** No Python field tracks report generation status ŌĆö `ReportManifest` only exists as the return value of a successful export call. Any future implementation must not add a `status`/`report_status` field to the `ReportManifest` contract to represent this; it belongs in frontend state.
