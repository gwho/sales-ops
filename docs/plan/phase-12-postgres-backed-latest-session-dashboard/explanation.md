## 1. The end-to-end data flow

Start from a browser that has never visited the app before. `lib/session-id.ts`'s `getOrCreateSessionId()` hasn't been called yet, so `localStorage` holds nothing under the key `salesOpsAnonymousSessionId`. The visitor lands on `/order-validation`, uploads a file, and clicks "Run Validation." That click calls `runValidation`, which calls `postJSON("/api/orders/validate", formData)` in `lib/api-client.ts`. Before the `fetch` fires, `postFormData` calls `sessionHeaders()`, which calls `getOrCreateSessionId()` â€” this is the *first* time the ID is generated: `crypto.randomUUID()` produces a value, it's written to `localStorage`, and it's returned. `sessionHeaders()` wraps it as `{ "X-Session-Id": <uuid> }` and the fetch goes out with that header attached.

On the backend, `orders.py`'s `validate_orders_endpoint` has `session_id: Annotated[uuid.UUID | None, Depends(get_session_id)]` as a parameter. FastAPI resolves that dependency before the function body runs: `get_session_id` (in `backend/session.py`) sees the header, parses it with `uuid.UUID(value)`, and returns the parsed object. The route body then calls `_load_and_validate(...)` â€” real business logic, `src/order_validation.py`'s `validate_orders`, completely unaware any of this session machinery exists â€” and gets back an `OrderValidationResult`. Only after that computation is done does `persist_workflow_result(response, repo, session_id, "order_validation", result)` run (`backend/persistence.py`). It calls `repo.save(session_id, "order_validation", result, CONTRACT_SCHEMA_VERSIONS["order_validation"])`. Inside `WorkflowResultsRepository.save` (`backend/repositories/workflow_results.py`), the result is serialized with `json.dumps(result, allow_nan=False)`, a connection is borrowed from the pool (`self._pool.connection(timeout=3.0)`), and an `INSERT ... ON CONFLICT (session_id, workflow_type) DO UPDATE` runs. The `with pool.connection() as conn:` block commits automatically on clean exit â€” there's no explicit `conn.commit()` anywhere in this codebase, because psycopg3's pool-connection context manager does it for you. `save()` returns `True`, and `persist_workflow_result` sets `response.headers["X-Persisted"] = "true"`. The route returns `result` as-is; FastAPI serializes it and merges in the header that was set on the injected `Response` object.

Back in the browser, `postJSON`'s caller in `order-validation/page.tsx` now has `{ data: result, persisted: "true" }`. Since `persisted !== "false"`, no `PersistenceNotice` renders â€” the page looks exactly like it did before Phase 12 existed.

Now the same visitor navigates to `/dashboard`. `app/dashboard/page.tsx` is a Server Component and renders instantly on the server with no knowledge of any of this â€” it renders its static shell and `<DashboardLiveSections />`. `DashboardLiveSections` is a Client Component; once it mounts in the browser, its `useEffect` calls `getDashboardResults()`, which again calls `sessionHeaders()` â€” this time `localStorage` already has the ID from the earlier run, so the *same* UUID goes out on the `GET /api/dashboard` request. `backend/routers/dashboard.py`'s `get_dashboard` resolves the same `get_session_id` dependency, gets a non-`None` `uuid.UUID`, and calls `repo.get_latest(session_id)`. That method fetches up to three rows for that `session_id`, runs each through `_is_usable()` (schema version + TTL check), and returns an envelope where `order_validation` is the real saved result and `inventory_allocation`/`payment_aging` are still `None` (nothing saved for those types yet). Back in `DashboardLiveSections`, `liveOrderValidation` is now non-null, so `orderValidationIsSample` is `false` â€” the Overview row's Total Orders and Invalid Orders `MetricCard`s render without the `sample` badge, while Allocation Status, Outstanding by Aging Bucket, and both tables still show the "Sample data" chip, because their underlying workflow types are still `null`.

## 2. Library installation: `psycopg[binary,pool]`, and why it had to be quoted

`uv add psycopg[binary,pool]` (unquoted) fails at the shell level before `uv` ever sees it, because `[...]` is a glob/bracket-expansion metacharacter in both `bash` and `zsh` â€” the shell tries to expand it as a character class over the literal text `binary,pool`, which doesn't match anything, and most shells either error or pass through something that isn't what was intended. The fix is `uv add "psycopg[binary,pool]"` â€” quoting stops the shell from touching the brackets at all, and `uv` receives the extras specifier exactly as written. This is a general pip/uv gotcha whenever a dependency spec includes an extras marker, not specific to psycopg.

The two extras themselves: `binary` pulls in `psycopg-binary`, a pre-built wheel with the C extension already compiled â€” avoiding a from-source build that would otherwise need PostgreSQL's dev headers installed locally. `pool` pulls in `psycopg-pool`, which is a **separate package** from core `psycopg` (unlike some other DB libraries where pooling is bundled) â€” `ConnectionPool` lives in `psycopg_pool`, imported separately in `backend/db.py`.

## 3. The `app.state.db_pool` `AttributeError`, and why it only showed up in pre-existing tests

The first time the full test suite ran after wiring `backend/main.py`'s new `lifespan` hook, every one of the *pre-existing* backend tests (`test_backend_orders.py`, `test_backend_inventory.py`, `test_backend_payments.py` â€” none of them touched by this feature) failed with `AttributeError: 'State' object has no attribute 'db_pool'`, raised from inside `get_workflow_results_repository`'s `request.app.state.db_pool` lookup.

The root cause: those test files construct `TestClient(app)` once at module level (`client = TestClient(app)`) and never use it as a context manager (`with TestClient(app) as client:`). Starlette's `TestClient` only runs an app's `lifespan` handlers â€” startup and shutdown â€” when entered as a context manager. A bare `TestClient(app)` skips lifespan entirely, so `app.state.db_pool`, which was only ever set inside the `lifespan` function, was never set at all for these tests, and any code path reading it raised.

This wasn't a bug in the new code's logic â€” `open_pool()` and the lifespan hook were both correct in isolation. It was a mismatch between a *new* assumption (there's always a `db_pool` attribute) and an *existing, unrelated* test convention this feature didn't choose and shouldn't have needed to change. The fix was a one-line default set immediately after `app = FastAPI(...)`, before the lifespan is ever a factor: `app.state.db_pool = None`. This makes "no lifespan ran" and "lifespan ran but `DATABASE_URL` was unset" produce the identical, already-handled state â€” a `None` pool, persistence cleanly disabled â€” rather than introducing a third, undefined state. No pre-existing test file needed to change.

## 4. The `.env` shell-parsing error, and the quoting fix

Loading environment variables locally with `source .env` (a common pattern used repeatedly during this feature's live verification) failed with `.env:5: parse error near '&'` the first time a real Neon connection string was written unquoted:

```
DATABASE_URL=postgresql://neondb_owner:...@host/neondb?sslmode=require&channel_binding=require
```

`&` is a shell control character (it backgrounds a command) outside of quotes. `source` reads a `.env` file as literal shell script, so an unquoted value containing `&` parses as "run everything before the `&` in the background, then try to parse `channel_binding=require` as a separate command" â€” which fails. The fix was to quote the value: `DATABASE_URL="postgresql://...&channel_binding=require"`. This is specific to `source`-based `.env` loading (treating the file as shell script); a dedicated dotenv-parsing library (Next.js's own `.env` support, Python's `python-dotenv`) would not have hit this, since those parse `KEY=value` as data, not as shell syntax. Every Postgres connection string with more than one query parameter will contain an unquoted `&` by default, since query strings are `&`-joined â€” this will recur for any future env var with the same shape and should be quoted preemptively.

## 5. The `persist_workflow_result` gap found while writing tests, not while writing the feature

`WorkflowResultsRepository.save()` was written from the start to catch its own exceptions internally and return `False` rather than propagate â€” a self-contained "never raises" guarantee, documented in its own docstring. The three router endpoints were originally written to trust that guarantee completely: call `repo.save(...)`, branch on the boolean it returns, no `try/except` at the call site.

This looked complete until the route-orchestration test suite was being written and one specific test was drafted: "a raising repository call should still return `200` with the full result" (matching the best-effort contract from ADR 0007). Writing a `FakeRepository` whose `save()` method raises on purpose, and running it through the real route, produced a `500` â€” because the route had never actually been exercised against a repository that violates its own contract, and had no independent defense against that case. The gap wasn't visible by reading the route code in isolation (it looked correct, because it correctly used the return value of a function documented to never raise) â€” it only surfaced by deliberately testing the failure mode the design promised to handle.

The fix was to extract the `if session_id is None: ... else: saved = repo.save(...); ...` block that had been duplicated near-identically across all three routers into one function, `backend/persistence.py`'s `persist_workflow_result`, with its own `try/except Exception` around the `repo.save(...)` call, logging and treating any exception as `saved = False`. This is defense in depth applied correctly, not redundantly: the repository still promises never to raise (so normal callers don't need to think defensively), but the one call site that actually matters for the user-facing guarantee doesn't bet the whole request on that promise holding forever â€” including against a future repository implementation that might not uphold it as carefully. All three routers now call the same helper instead of repeating the same six lines three times, which was also a straightforward reuse win independent of the correctness fix.

## 6. Details that look wrong at first glance but are correct

**A `GET /api/dashboard` request with no `X-Session-Id` header at all returns `200` with everything `null`, not a `400` or a `401`.** This looks like it should require authentication or at least reject an anonymous request, but it's the single most common real-world case: a first-time visitor with an empty `localStorage`. Treating "no session" as an error would make the dashboard's default, expected state a broken one.

**`POST /api/orders/validate/report` silently ignores an `X-Session-Id` header if one is sent.** It doesn't validate it, doesn't 400 on a malformed one, doesn't set `X-Persisted`. This isn't an oversight â€” report endpoints are explicitly, deliberately untouched by Phase 12 (see `docs/adr/0007-...md`'s "Report endpoints are entirely unchanged" section). The frontend's `lib/api-client.ts` still attaches the header on report calls (since `postFormData` is the single function both `postJSON` and `postReport` funnel through), and that's fine â€” the backend router for `.../report` never declares a `session_id` dependency at all, so the header is simply never read.

**A TTL-expired row is not deleted â€” `GET /api/dashboard` just stops returning it.** Querying `workflow_results` directly for an old session will still show the row, with its original `saved_at`, indefinitely. This is intentional ("Display Expiry," not deletion) and is a real, load-bearing distinction, not a partially-implemented feature â€” see `explanation.md` in the paired grilling docs (`docs/grilling/phase-12-.../explanation.md`) for the full reasoning.

## 9. The `.env.example` secret-leak near-miss

Mid-session, real Neon connection strings â€” including the database password â€” were written directly into `.env.example`, the **tracked, committed** template file, instead of `.env` (already gitignored via the repo's existing blanket `.env*` rule). This was caught immediately via `git status --porcelain .env .env.example` and `git diff --cached --stat`, which confirmed nothing had been staged or committed yet. The fix: the real values were moved into a newly created `.env` (confirmed gitignored via `git check-ignore -v .env`), and `.env.example` was restored to its established empty-template convention (a pattern already documented in this project from an earlier phase, where `.env.example` deliberately never carries real values â€” only the Vercel/Render dashboard configs and `README.md`'s Live Demo section do). No credential was ever actually exposed on GitHub, since nothing was pushed or even committed, but this is the exact class of mistake that a `git status` check before every commit exists to catch.

## 10. A false-negative in the live-verification script, and how it was distinguished from a real bug

The throwaway Playwright verification script's final check ("Outstanding by Aging Bucket still shows Sample data") failed on first run, while 7 of its 8 checks â€” including the structurally identical adjacent check for "Allocation Status still shows Sample data" â€” passed. Rather than assume the product had a real bug, the DOM was inspected directly with a separate one-off script using an explicit `xpath=ancestor::div[contains(@class, "rounded-xl")][1]` traversal from the "Outstanding by Aging Bucket" heading, which confirmed the "Sample data" text *was* present in that card's actual rendered HTML. The original script's locator (`.locator("..")`, a single-level parent traversal from a text match) simply didn't land on a common-enough ancestor to catch both the heading and the badge in that specific card's DOM structure, even though the same pattern happened to work for the sibling card. The takeaway generalized from this: a test failure should be independently confirmed against the real system before being treated as a defect in the thing under test, especially when a structurally similar sibling assertion passed â€” the more likely explanation, born out here, was a bug in the *verification* code, not the *product* code.
