# Phase 4: Inventory Allocation Core — Technical Explanation

## Why warehouse choice compares `allocatable_qty`, not raw `available_qty`

IA-007 V1's text says to allocate from "the warehouse with the highest available quantity for that SKU." Taken completely literally, that reads as the raw `available_qty` column from `inventory.xlsx`. But IA-002, one rule earlier, defines `allocatable_qty = available_qty - reserved_qty` as *the* quantity that's actually usable for allocation this round. If warehouse selection used the raw column, it's possible to pick a warehouse with `available_qty=50` but `reserved_qty=48` (2 allocatable) over a warehouse with `available_qty=20`, `reserved_qty=0` (20 allocatable) — routing an order to the warehouse that can least fulfill it. Since `allocatable_qty` is already the concept the rest of the module uses to decide *how much* to allocate, using the same value to decide *where* to allocate from keeps one definition of "usable stock" instead of two competing ones.

## Why allocation has to be stateful and sequential, not a batch calculation per SKU

A naive per-SKU calculation (sum all orders for a SKU, sum all inventory for a SKU, distribute) can't reproduce IA-001's priority ordering: if a High-priority order and a Normal-priority order both want the same SKU and there's only enough stock for one, the High order must win outright, not receive a proportional share. This only falls out correctly if orders are processed one at a time, in IA-001's exact sort order, with each line's allocation immediately reducing the shared `allocatable_qty` balance that the next line will see. `_allocate_order_line` mutates `chosen["allocatable_qty"]` and `chosen["allocated_qty"]` in place on the same dict stored in `inventory_by_sku`, so later lines automatically see the depleted state — including a warehouse that no longer has the highest `allocatable_qty` for a SKU once an earlier line has consumed enough of it (see `test_warehouse_choice_switches_as_stock_depletes`).

## Why a fully-backordered line still records a `warehouse`

A `BackorderRow` with `allocated_qty=0` didn't have any stock actually move — so leaving `warehouse=""` would be defensible on the grounds that "nothing was fulfilled from anywhere." But the pre-existing `BACKORDER_ROW_FIXTURE` in `tests/contract_fixtures.py` (authored during Phase 2, before Phase 4's rules were implemented) already showed a populated `warehouse` value on a zero-allocation row. Reading that fixture as intentional rather than an oversight, the rule became: `warehouse` records *which* warehouse IA-007 selected as the best candidate for this SKU, independent of whether that candidate had enough stock to fulfill the line. This gives the eventual UI/report a "where would this ship from once restocked" answer even for backorders — genuinely useful operationally, since supplier follow-up is per-warehouse. The `warehouse=""` case is reserved for the narrower situation where the SKU has *no* inventory rows at all, so there's no candidate to name.

## Why `InvalidInventoryDataError` raises instead of degrading gracefully like `order_validation.py`'s malformed-data handling

Phase 3 converts every malformed order value (bad quantity, unparseable date, invalid active flag) into a business-readable `ValidationErrorRow` rather than raising, because `contracts.py` defines exactly the output surface for that: a row can be flagged and excluded from `valid_orders` while everything else in the batch still processes. Phase 4 has no equivalent contract — there's no `InventoryDataIssueRow` family, and nothing in `02_demo_inventory_allocation.md` describes a "soft" inventory data-quality report. Silently defaulting a malformed `available_qty` to 0 would make a warehouse look phantom-empty without telling anyone why, which is worse than failing loudly — especially since `available_qty` directly drives every downstream allocation decision. Raising `InvalidInventoryDataError` (in the same spirit as `excel_io.py`'s `MissingColumnsError`) surfaces the problem immediately rather than producing a plausible-looking but wrong allocation result. Optional fields (`reserved_qty`, `reorder_point`, `lead_time_days`) don't get this treatment because the spec already defines their blank behavior (`reserved_qty` missing = 0; `reorder_point` missing = the SKU/warehouse simply can't trigger a reorder alert) — there's no ambiguity to fail loudly about.

## Why `low_stock_sku_count` isn't just `len(supplier_follow_ups)`

`AllocationSummary.low_stock_sku_count` is meant to answer "how many distinct products need supplier attention," which is a merchandising-level question. `len(supplier_follow_ups)` instead answers "how many warehouse-level restock actions are needed," which double-counts any SKU stocked (and alerting) in more than one warehouse — exactly `MED-LENS-001` in the sample data, which alerts in both HK and SG Warehouse and would otherwise report as "2 low-stock SKUs" when it's actually one SKU with two restock actions. Tracking a `set()` of alerting SKUs (`low_stock_skus`) rather than counting follow-up rows keeps the summary KPI and the operational follow-up list answering two different, both-useful questions instead of collapsing them into one number that's misleading for either purpose.
