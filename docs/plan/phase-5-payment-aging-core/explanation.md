# Phase 5: Payment Aging Core — Technical Explanation

## Why the result envelope collapses the spec's suggested three-function shape into one

Section 11 of `03_demo_payment_aging.md` suggests `load_invoices()`, `calculate_payment_aging(invoices_df, as_of_date) -> tuple[pd.DataFrame, pd.DataFrame]`, and a separate `create_follow_up_messages(aging_df) -> pd.DataFrame`, called in sequence. Taken literally, a caller would run all three and manually stitch the results together. But CLAUDE.md's Output Contracts section is explicit that Python modules must return JSON-serializable structures with snake_case keys, and Phase 3 and Phase 4 already established the pattern for that: one function, one dict envelope, containing every output family the module produces. Splitting `payment_aging.py` into three functions would mean the module has no single "the payment aging result" object — a future FastAPI route handler would need to know to call three functions in the right order and merge their outputs manually, which is exactly the kind of route-layer business logic CLAUDE.md's Python Module Boundaries table forbids drifting into. `calculate_payment_aging()` therefore does everything in one pass over `invoices_df` — computing aging, spotting data issues, and building draft messages together — and returns `PaymentAgingResult` (`summary`/`aging_rows`/`data_issues`/`draft_messages`). `load_invoices()` stays a separate thin loader, since that part of the spec's shape does match the established Phase 3/4 convention exactly.

## Why Paid invoices stay in `aging_rows` instead of being filtered out

PA-002's exact text is: "If `outstanding_amount <= 0`, mark invoice as `Paid` and exclude it from overdue follow-up." The ambiguity is what "overdue follow-up" scopes over. Section 6's "Aging output columns" table (which becomes `PaymentAgingRow`) has no carve-out for Paid rows, and the downloadable workbook's suggested sheet 3 is literally named "All Invoices with Aging" — a name that only makes sense if paid invoices are still present, since "all invoices" would otherwise silently mean "all unpaid invoices." Section 7's suggested-actions table also has an explicit `Paid → "No action required"` row, which only has anywhere to live if Paid rows exist in the output at all. Reading PA-002 narrowly — "excluded from overdue follow-up" means excluded from ever getting a High/Medium/Low/Watch priority or a draft reminder, not excluded from the aging table — resolves all three signals consistently. This is why `_follow_up_priority()` checks `outstanding_amount <= 0` first and returns `"None"` immediately, before any of the day-count branches run: a Paid invoice can never fall through into Watch or High even if its due date happens to be far in the past or near-future.

## Why `outstanding_amount` is clamped to zero instead of allowed to go negative

PA-001's formula (`invoice_amount - paid_amount`) has no explicit floor, so an overpaid invoice (`paid_amount > invoice_amount`) would mathematically produce a negative outstanding amount. Nothing in the spec discusses overpayment, but PA-002's condition is written as `outstanding_amount <= 0`, not `== 0` — a strong hint the spec's author anticipated the value could go negative and wanted the `<=` to catch it defensively. Rather than carry a negative number into the contract's `outstanding_amount: float` field (a "remaining unpaid amount" that reads as more confusing than useful when negative — is `-50` a $50 credit? a bug?), `_build_data_issue`... no, this is in `calculate_payment_aging()` itself — clamps with `max(invoice_amount - paid_amount, 0.0)`. This has a side effect worth noting: it makes PA-002's own `<= 0` check redundant in practice, since a clamped value is never negative, only ever `0.0` or positive. The `_follow_up_priority()` check is still written as `outstanding_amount <= 0` rather than `== 0.0` to preserve intent-matching with the spec text and stay robust to a future change in the clamping decision, but as implemented today the two are equivalent. `tests/test_payment_aging.py::test_overpayment_clamps_outstanding_to_zero` locks this in explicitly (`invoice_amount=100, paid_amount=150 → outstanding_amount=0.0`), so a future refactor that "simplifies" the clamp away breaks a named test instead of silently reintroducing negative outputs.

## Why `days_overdue` keeps its sign instead of flooring at 0 for future due dates

PA-003's formula is `days_overdue = today_date - due_date`, with a separate sentence — "If `days_overdue <= 0`, the invoice is not overdue" — describing what to *do* with a non-positive value, not redefining the value itself. The two readings that matter here are: (a) store the raw signed day count for every row, including negative values for invoices not yet due, or (b) floor every non-overdue row's `days_overdue` at `0`, since "not overdue" arguably means "there's no overdue day count to report." Flooring was rejected because PA-005's Watch rule — "not overdue but due within 7 days" — cannot be expressed as a simple range check on a floored value; a floored `0` collapses "due in 2 days" and "due in 200 days" into the same number, which would force a second, hidden `days_until_due` calculation that never surfaces in the contract. Keeping the raw signed value means `_follow_up_priority()` can express Watch as one condition, `-7 <= days_overdue <= 0`, using the exact same field the rest of the function already computed. The tradeoff is that `PaymentAgingRow.days_overdue` can be negative in the JSON output, which a future UI/report layer needs to handle by phrasing (e.g. "due in 5 days" for negative values) rather than assuming the field is always a plain "days late" count — this is called out explicitly in `plan.md`'s key decisions so it isn't rediscovered as a surprise later.

## Why High priority can fire on a not-yet-due invoice

PA-005's High condition is `days_overdue > 60 or outstanding_amount >= 50000` — an unconditional `or`, with no accompanying "and is overdue" clause. Combined with decision 5 above (signed, unfloored `days_overdue`), this has a real consequence: an invoice due in the future with a large enough outstanding amount (`>= 50000`) evaluates to `follow_up_priority = "High"` even though its `aging_bucket` is `"Current"` and it hasn't missed its due date. This was implemented literally rather than "fixed" to require `days_overdue > 0` as well, because the spec's own §12 test table has a directly analogous case — "Large overdue amount... outstanding amount >= 50000 → High priority" — stated as a standalone rule independent of the days-overdue test cases, suggesting the amount-only trigger is intentional, not an oversight. The one place this literal reading is *not* allowed to leak through is draft messages: decision 6's guard (`days_overdue > 0` required) means a High-priority-by-amount-alone invoice that isn't actually overdue yet still gets no draft reminder, since section 8 only describes reminders "for each overdue customer" — so the priority and the reminder-eligibility rules diverge on purpose. `test_large_outstanding_amount_forces_high_priority_at_low_day_count` in `tests/test_payment_aging.py` locks in the priority side of this; the draft-message guard is exercised indirectly through `test_draft_message_generated_only_for_overdue_high_medium_low`.

## Why invalid `paid_amount` is treated differently from invalid `invoice_amount`

PA-007's text names one field explicitly: "If invoice amount is missing or less than 0, flag as error." PA-001 treats `paid_amount` as optional and only documents one failure mode — "If `paid_amount` is missing, treat it as 0" — with no equivalent "and if invalid, flag as error" clause. The two fields therefore get genuinely different error-handling paths in `calculate_payment_aging()`: `invoice_amount` failing `_parse_amount()` (blank, non-numeric, or the value being `< 0`) produces a PA-007 `PaymentDataIssueRow` and excludes the row from `aging_rows` entirely; `paid_amount` failing the same parse (including negative values, which the spec doesn't discuss at all) just falls back to `0.0` via `paid_amount_parsed if paid_amount_parsed is not None and paid_amount_parsed >= 0 else 0.0`, and the row proceeds normally. This mirrors the same asymmetry principle Phase 4 already established for optional inventory fields (`reserved_qty`/`reorder_point`/`lead_time_days` degrade gracefully) versus required ones (`sku`/`warehouse`/`available_qty` raise) — except Phase 5 has a `PaymentDataIssueRow` contract to report the required-field failure *softly* instead of raising a hard exception like Phase 4's `InvalidInventoryDataError`, because `docs/adr/0005-payment-data-issue-row-contract.md` added that contract specifically so PA-006/PA-007 failures don't need to halt the whole batch. `test_invalid_paid_amount_degrades_to_zero_no_data_issue` and `test_negative_paid_amount_degrades_to_zero_no_data_issue` both assert `data_issues == []` for their row, which is the test-level guardrail against a future agent "tightening" `paid_amount` validation to match `invoice_amount`'s and silently expanding scope beyond what PA-001/PA-007 actually specify.

## Why the draft message currency format deviates from the pre-existing fixture, and what that fixture's mistake reveals

`tests/contract_fixtures.py`'s `DRAFT_MESSAGE_ROW_FIXTURE` was hand-authored in Phase 2, before `payment_aging.py` existed, and rendered `message_text` with a hardcoded `$58,000.00`. The invoice that fixture describes (`INV-2026-001`) is generated by `sample_data.py`'s `generate_invoices()` with `currency: "HKD"` — so the original fixture's `$` was simply wrong for its own example row, not a deliberate design choice; nobody had implemented the aging module yet to catch it. `DraftMessageRow`'s contract fields (`invoice_id`, `customer_name`, `outstanding_amount`, `days_overdue`, `message_text`) don't include `currency` as a structured field — adding one would violate the Field Scope Boundary, since section 8 of the spec never defines a `currency` output field, only a `[Outstanding Amount]` placeholder in free-form message text. That leaves two honest choices for the string inside `message_text` itself: always use a fixed symbol (wrong for two of the three sample currencies), or read the invoice's own `currency` input column — which the row already has in hand, since `calculate_payment_aging()` iterates the same `invoices_df` row that carries it — and prefix it onto the formatted number. `_format_amount()` implements the second option, and the pre-existing fixture was corrected from `$58,000.00` to `HKD 58,000.00` to match, which is also what made the Phase 4-style sanity check (computed `INV-2026-001` output vs. the fixture) match field-for-field instead of just close.

## Why the fixture's `invoice_date`/`due_date` needed a one-day correction, and why that isn't a logic bug

Running `calculate_payment_aging()` against the real committed `sample_data/sample_invoices.xlsx` with `as_of_date=date(2026, 7, 9)` (the date `generate_invoices()` was actually run with, confirmed by `2026-07-09 - 70 = 2026-04-30` matching the file's real `due_date` for `INV-2026-001`) reproduced `days_overdue=70`, `outstanding_amount=58000.0`, `aging_bucket="61-90 Days"`, `follow_up_priority="High"`, and `suggested_action="Call or email customer urgently"` exactly against `PAYMENT_AGING_ROW_FIXTURE` — but `invoice_date` and `due_date` were each off by one calendar day (`"2026-04-01"`/`"2026-05-01"` in the fixture vs. the real `"2026-03-31"`/`"2026-04-30"`). Every value that depends on business logic matched; only the two literal date strings, which the fixture's author had to type by hand before `sample_data.py`'s exact offset arithmetic existed, were off. This is the same category of finding Phase 4's sanity check made with `MED-LENS-001` — a hand-authored Phase 2 fixture serving as an independent check on Phase 5's logic — except this time the check caught a pre-existing one-day typo in the fixture rather than confirming a clean match immediately. Both date strings were corrected in `tests/contract_fixtures.py` so the fixture is an accurate real-data example again, not because `payment_aging.py`'s date math needed to change.
