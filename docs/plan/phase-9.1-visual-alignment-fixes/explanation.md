# Explanation — Feature phase-9.1-visual-alignment-fixes: Visual Alignment Fixes

## 1. Why the sidebar bug needed rendered HTML, not just source reading, to find

The original review of Phase 9 started as a code-only audit — read every page and component, cross-reference against `ui-tokens.md`/`ui-rules.md`. That pass alone would not have caught the sidebar bug. `components/layout/SidebarNav.tsx` had `h-screen` (fixed 100vh) and `components/layout/AppShell.tsx`'s outer wrapper had `flex min-h-screen` (grows to fit content, no cap) — both individually reasonable-looking lines, and neither obviously wrong when read in isolation. The bug only becomes visible when you reason about what happens when the *row's* height is forced past 100vh by the *other* flex child (the content column), which has no height cap at all: `SidebarNav`'s explicit `h-screen` overrides the flex row's default `align-items: stretch`, so it stays pinned at exactly one viewport while the row (and the content column next to it) keeps growing. The fix was verified two ways, and only the second one actually proved it: reading the compiled CSS classes in the real production build's HTML output (`npm run build && npm run start`, then `curl` each route and grep for the literal `h-screen`/`min-h-screen` class strings) confirmed the mismatched height model was actually shipping, not just theoretically present in source. This is the concrete reason the Phase 9.1 review method was upgraded from "read the code" to "read the code, then verify claims against the real rendered output" — and that same two-step method caught two more issues (the brand-name string mismatch and the `/reports` width difference) that a source-only read would have missed just as easily, since both individually looked like ordinary, unremarkable lines of JSX.

The fix itself (`AppShell`'s wrapper becomes `flex h-screen overflow-hidden`, `SidebarNav` becomes `h-full`) isn't a guess — it's the exact pattern found in the original Figma Overview Dashboard prototype's own root (`className="flex h-screen overflow-hidden"`, read via Figma MCP during the earlier Phase 8/9 reconciliation work), which independently confirms it's the standard, correct way to build a fixed-sidebar-with-internally-scrolling-content layout. After the fix, the whole page no longer scrolls as a document — only `<main>` scrolls internally, bounded by the capped shell.

## 2. Why filtering came back to `DataTable`'s ecosystem without touching `DataTable` itself

Phase 9's original decision was "sort only, no filtering" for `DataTable`, reasoned from the fact that the mock fixtures were 1-2 rows per table and filtering had nothing real to demonstrate. Phase 9.1 revised that decision — via an explicit developer instruction, not a rediscovered constraint — because table-heavy pages needed filter affordances to read as a credible "operations dashboard" rather than a demo shell. The interesting design question wasn't *whether* to add filtering, but *where the filtering logic should live*. The chosen answer, confirmed in a later addendum's review pass, was: filtering stays entirely outside `DataTable`. `FilterToolbar` and `FilterSelect` are new, separate components that each page composes above its `DataTable`, managing their own `useState` filter fields and computing an already-filtered array via a page-local `useMemo`, which is then passed into `DataTable`'s existing `data` prop unchanged. `DataTable` itself gained zero new filtering-related props or logic — it still only ever asks "is this array empty?" This keeps `DataTable`'s responsibility exactly as narrow as it was in Phase 9 (render rows, sort rows, show `EmptyState` when there are none), while the pages get real filtering. The alternative — teaching `DataTable` to accept filter predicates or filter state directly — was never implemented, and would have been the wrong call: it would have coupled a generic rendering component to per-page business logic (which fields are filterable, what "All" means for each one) that only the page actually knows.

## 3. The `wrap: true` column flag that got added, then removed, in the same session

An early addendum asked for long text (`error_message`, `suggested_action`) to wrap onto multiple lines inside cells, so a `wrap?: boolean` column flag was added to `DataTableColumn` and set on those two columns. A later, more detailed addendum in the same session specified the opposite treatment for the same two columns: single-line with `truncate` and a native `title` attribute for the full value on hover, explicitly distinguishing table cells (which should stay single-line and dense) from `DraftMessageRow.message_text` (which lives in a card panel, not a table, and is allowed to wrap). The `wrap` flag itself stayed in `DataTable`'s type — it's a real, useful escape hatch — but every column that had briefly used it was changed to the truncate+`title` pattern instead. As of the end of this phase, no column in the app actually sets `wrap: true`; the flag exists for a future column that might genuinely need it, not because anything currently uses it. A developer grepping for `wrap: true` and finding zero matches should not assume the flag is dead code — it's intentionally unused right now, documented as such in `ui-registry.md`.

## 4. Why the chart components only import a type from `StatusBadge.tsx`, never its functions

`DonutBreakdownChart` and `VerticalBucketBarChart` both take `tone: Tone` per data point as a prop, and a design review flagged a real question: does importing `Tone` from `components/workflow/StatusBadge.tsx` create an awkward "visual component importing another visual component" dependency? The answer, arrived at by actually tracing what gets imported rather than guessing, is no — both chart components do `import type { Tone } from "@/components/workflow/StatusBadge"`, a type-only import that disappears entirely at runtime (TypeScript strips `import type` during compilation). The actual tone-*resolution* functions, `allocationStatusTone` and `agingBucketTone`, are called exactly once each, inside `app/dashboard/page.tsx`, to compute the `tone` value before it's passed down as a prop. The chart components themselves never call these functions and never import them as values. This was already the established pattern for `MiniBar.tsx` and `SegmentedBar.tsx` before this addendum — the review question surfaced it explicitly rather than introducing something new, and confirmed the existing pattern was safe to extend rather than requiring a new shared file (a `components/workflow/status-tones.ts` was considered and explicitly rejected as unnecessary indirection).

The practical payoff: because tone is always resolved by the same two functions regardless of whether the result ends up in a `StatusBadge` pill or a chart segment, a table badge and a chart segment for the same status can never show a different color by accident. If `agingBucketTone("61-90 Days")` is ever changed from `warning` to something else, every consumer — the Payment Aging table's badge column, the dashboard's follow-up items table, and the dashboard's bar chart — picks up the new color automatically, with no risk of one of them being hand-edited and left stale.

## 5. The donut chart's math, and its zero-value guard

`DonutBreakdownChart` builds a ring from stacked SVG `<circle>` strokes rather than a `<path>` arc, which is the simplest correct technique for a donut chart with no library: each segment is a full circle (`r={35}`, `viewBox="0 0 100 100"`) whose `stroke-dasharray` is set to `"{segmentLength} {circumference - segmentLength}"` — a dash of exactly the segment's arc length, followed by a gap of everything else, so only that fraction of the ring's circumference is actually painted. `stroke-dashoffset` is a running negative cumulative sum of every prior segment's length, which rotates each subsequent circle's visible arc forward around the ring. The whole `<svg>` carries a `-rotate-90` Tailwind class so the first segment starts at 12 o'clock instead of 3 o'clock (the default start point for an unrotated circle's dash pattern).

This math divides by `total` (`segment.value / total`) to get each segment's share of the circumference, which is an unguarded division — if every segment's value is 0, `total` is 0 and the calculation produces `NaN`. A design review flagged this as a real risk before it shipped (not discovered by hitting it in practice — the mock data always has a non-zero total), and the fix is a short-circuit: `if (total === 0)`, skip the whole per-segment loop and render one full-circumference circle in a neutral stroke color with the center text hardcoded to `"0"`. The same category of guard exists in `VerticalBucketBarChart` for its `value / max` height calculation — if `max` is 0, it renders an `EmptyState` caption instead of a row of bars that would otherwise all compute `0/0` heights.

## 6. Why the Payment Aging dashboard chart is "amount by bucket," not "count by bucket"

`PaymentAgingSummary.aging_bucket_counts` already exists as a Python-computed field and is exactly what the pre-existing `AgingBucketBars` component visualizes on `/payment-aging` itself — a *count* of invoices per bucket. When the same visualization was first placed on `/dashboard`, it reused that same count-based data. A later addendum asked specifically for an *amount*-based version instead — "outstanding amount by aging bucket," not invoice count — because a dollar-amount breakdown is more informative for a payment-aging summary than a raw row count, and this framing was confirmed to match the amount-led presentation found in the original Figma Payment Aging prototype (read earlier in the project's Figma reconciliation work). Since no contract field produces this amount breakdown directly, `lib/mock-data.ts` gained a new function, `amountByAgingBucket()`, which groups `PaymentAgingResult.aging_rows` by `aging_bucket` (in the fixed 5-bucket order) and sums `outstanding_amount` per bucket — the same "derived display-only aggregate" pattern already established for `ninetyPlusDaysAmount()` in the same file, and documented in `context/ui-contract-plan.md`'s Derived Display-Only Aggregates table before being used, per this project's standing rule that every derived value needs a written source-and-rule entry there before it ships in a component. `/payment-aging`'s own count-based `AgingBucketBars` was deliberately left untouched — the amount-vs-count distinction is dashboard-specific, not a full replacement of the count visualization everywhere it appears.

## 7. The subtitle wording that avoids implying a currency

`formatAmount()` (in `lib/formatters.ts`) has never rendered a currency symbol — it's a plain `toLocaleString` call with two decimal places, a decision made back in Phase 5 of the Python core, because no output contract carries a currency code field (orders, invoices, and payments in the sample data span HKD, SGD, and TWD, and inventing a single currency symbol in the UI would misrepresent that). The `VerticalBucketBarChart`'s subtitle prop was initially drafted as `"Total Outstanding: $165,000.00"` while writing the dashboard wiring, and a review pass caught that the `$` and "Total Outstanding" framing together visually imply USD even though the underlying number has no currency attached. The subtitle was changed to a neutral caption, `"Outstanding amount by bucket"`, with no symbol and no per-currency framing — consistent with every other amount displayed in this app, all of which are plain formatted numbers, never prefixed with a currency symbol.

## 8. Verification method for this phase, and where it genuinely relied on manual screenshots

Every fix in this phase was verified through the same layered method used in the original review: `npm run build`/`lint`/`typecheck` and `uv run pytest` at every checkpoint (clean throughout, 152 Python tests unaffected by an entirely frontend-scoped phase); a raw-color/hex regex scan reused verbatim from the Phase 9 verification pass; and a "reject-vocabulary" scan for the specific Figma-invented strings this project has repeatedly had to guard against (`Critical`, `Invalid SKU`, `On Hold`, `Pending Review`, `Cancelled`) — every match found across the whole phase was a confirmed false positive (e.g. `ValidationSummary.invalid_skus`'s real KPI label "Invalid SKUs," plural, versus the rejected Figma allocation-status value "Invalid SKU," singular).

The one piece these mechanical checks cannot verify is whether the page actually *looks* right — a `grep` can confirm a `bg-surface-subtle` class string exists in the file, not that the resulting panel is visually distinct from its neighbors. For that, the developer was asked for real before/after screenshots partway through the phase after an attempt to automate the capture via the Figma MCP's page-capture tool hit that account's Starter-plan monthly rate limit (6 read-tool calls/month, already spent earlier in the session reading the original prototype source) after only 3 of the planned 5 "before" routes were captured. The fallback — 8 manually captured screenshots per pass, matching a fixed checklist covering every route's key sections — is the same fallback this project's own instructions specify for exactly this situation ("if Figma page capture is unavailable, ask the developer to manually provide screenshots... do not rely on code/HTML audit alone for visual sign-off"). The after-screenshots at the end of this phase were compared against that checklist and confirmed every fix rendered as designed with no visual regressions — including details a grep genuinely cannot check, like the donut chart's zero-value segment correctly being omitted from the ring while still appearing in the legend, and the sidebar visibly spanning the full page height on the tallest routes.
