# Explanation — Feature phase-8-nextjs-frontend-foundation: Next.js Frontend Foundation

## 1. Why scaffold in a scratchpad and transplant, instead of running `create-next-app` at the root

The repo root is not empty. It holds the Python project (`src/`, `tests/`, `pyproject.toml`, `uv.lock`), the whole `context/` design system, `docs/`, `skills/`, and — critically — real `AGENTS.md`, `CLAUDE.md`, `README.md`, and `.gitignore` files that the whole workflow depends on. `create-next-app` writes its own versions of several of those. When it was run in the scratchpad, it generated its own `AGENTS.md` and `CLAUDE.md` (Next 16's scaffold ships agent-guidance files), plus `README.md` and `.gitignore`. Had it run at the root, it would have clobbered the project's authoritative guidance files with Next boilerplate.

So the sequence was: scaffold into `<scratchpad>/next-scaffold`, inspect what it produced, then copy only the Next.js-specific files to the root — `app/`, `public/`, `next.config.ts`, `next-env.d.ts`, `tsconfig.json`, `eslint.config.mjs`, `package.json` — and deliberately leave behind the generated `AGENTS.md`/`CLAUDE.md`/`README.md`/`.git`/`node_modules`/`.next`/`package-lock.json`. The `.gitignore` was merged by hand rather than copied, so the Python ignore rules survived. After the transplant, `git status --short AGENTS.md CLAUDE.md README.md` returned empty, proving none of the real guidance files were touched. This guardrail was explicitly requested during planning and paid off immediately — the generated `AGENTS.md`/`CLAUDE.md` were exactly the clobber it was meant to prevent.

## 2. Why Tailwind is pinned to 3.4 and set up the "v3 way," and how that interacts with the scaffold

`context/library-docs.md` and `context/ui-tokens.md` both hard-require Tailwind CSS 3.4 and forbid v4. This matters because Tailwind v4 is a different product for configuration purposes: v4 is CSS-first (`@import "tailwindcss"`, `@theme` blocks, `@tailwindcss/postcss`), has no `tailwind.config.ts` by default, and does not read a v3 JS config unless explicitly told to. `create-next-app@latest` (this run produced Next 16.2.10) defaults its `--tailwind` flag to **v4**. Downgrading a v4 scaffold to v3 means undoing v4's CSS-first wiring — messier and more error-prone than simply not asking for Tailwind at all.

So the scaffold was created with `--no-tailwind`, and Tailwind 3.4 was added manually via the v3 path documented in the installed Next docs (`node_modules/next/dist/docs/01-app/02-guides/tailwind-v3-css.md`): `tailwindcss@^3.4 postcss autoprefixer`, a `postcss.config.mjs` with the `tailwindcss` + `autoprefixer` plugins, `@tailwind base/components/utilities` directives in `globals.css`, and a `tailwind.config.ts` with `content` globs and `theme.extend`. After `npm install`, both `npm ls tailwindcss` and the lockfile confirmed `tailwindcss@3.4.19`, and `@tailwindcss/postcss` (the v4 marker) was absent — the plan's explicit "verify package.json AND lockfile are 3.4.x" check.

One observation worth recording: even though the dev script was scaffolded with `--no-turbopack`, `next build` still reported "Next.js 16.2.10 (Turbopack)". In Next 16 the production build uses Turbopack regardless; the flag only affects the `dev` default. The build and the Tailwind PostCSS pipeline worked together with no extra configuration, so no action was needed — but a future reader who expects a webpack build should not be surprised.

## 3. Why shadcn is "plumbing only" this phase, and the token-name collision that forced it

The approved plan listed "shadcn init + add button/card/table/badge/input/select primitives." That step could not be executed as written, and the reason is a genuine design conflict discovered only when `context/ui-tokens.md`'s exact token names were read during implementation.

`ui-tokens.md` defines the project's own semantic tokens: `--surface`, `--text-primary`, `--accent` (the brand **blue**, `221 83% 53%`), `--success`, `--danger`, and so on, exposed as Tailwind classes like `bg-surface`, `text-text-primary`, `bg-accent`. shadcn/ui primitives, by contrast, are authored against shadcn's own token set: `--primary`, `--card`, `--muted`, `--destructive`, `--input`, `--ring`, and its own `--accent` — which in shadcn means a **subtle gray hover background**, the direct semantic opposite of this project's brand-blue `--accent`. Dropping shadcn's `button.tsx` in as-is would either render unstyled (its tokens don't exist in our config) or require adding shadcn's parallel token set into `tailwind.config.ts`, at which point `colors.accent` would have two conflicting definitions.

Two project skills made this decisive rather than a judgment call. `skills/tailwind-best-practices/references/rules/tokens-no-modification.md` is tagged **CRITICAL**: "Never Modify Design Tokens... without explicit approval." Adding shadcn's tokens to make its primitives render is exactly that forbidden modification. `library-docs.md` independently says "Do not let generated shadcn code introduce raw color classes that conflict with tokens." And separately, the main `skills/tailwind/SKILL.md` turned out to be HyperFrames/Tailwind-v4-browser-runtime guidance — explicitly *not* for a Next.js + `tailwind.config.js` + `@tailwind`-directives project — so it and the v4 rules docs were disregarded; only `tailwind-configuration` and the token rules applied.

The resolution, confirmed with the user: Phase 8 sets up only shadcn *plumbing* — the `cn()` helper (`lib/utils.ts`), `components.json`, and the runtime deps (`clsx`, `tailwind-merge`, `class-variance-authority`, `lucide-react`). No primitive components land, so no colliding tokens get introduced. Because Phase 8 renders only stub pages, nothing actually needed a shadcn primitive yet, so the deferral costs nothing. Phase 9 will add each primitive only when a real component needs it, reconcile its classes to `ui-tokens.md`, and capture the pattern via `/imprint`. `components.json` still declares `baseColor: slate` / `cssVariables: true` for a future `shadcn add`; Phase 9 owns the reconciliation of whatever that generates.

## 4. The mock-data flow, and why the app must never import `tests/`

The frontend needs realistic example data to build against in Phase 9, and the single best source of correct, already-tested example values is `tests/contract_fixtures.py` — the same fixtures the Python contract tests assert against. But a shipped Next.js app importing from a `tests/` directory would be wrong on multiple axes: it couples production to test-only code, and there is no runtime Python bridge anyway.

The flow is therefore build-time-only and one-directional. `scripts/generate_mock_data.py` inserts the repo root on `sys.path` (so `tests.contract_fixtures` resolves when run as a standalone script rather than under pytest), imports the fixtures, assembles them into the three result envelopes (`OrderValidationResult`, `InventoryAllocationResult`, `PaymentAgingResult`) plus the `ReportManifest` list, and writes them as JSON to `lib/mock-data/`. The envelopes mirror the local `*Result` types in `types/index.ts` exactly: e.g. `inventory-allocation.json` carries `summary`, `allocation_results` (the result row plus the backorder row), `backorders`, `remaining_inventory`, and `supplier_follow_ups`. The fixtures are single-row examples, so the envelopes are intentionally small — enough to prove the shape end to end, not a full demo dataset. At runtime, Phase 9 components will read the committed `lib/mock-data/*.json`; nothing under `app/`, `components/`, or `lib/*.ts` imports `tests/`.

A concrete payoff of sourcing from the fixtures rather than hand-writing JSON: the generated `reports.json` carries `"report_id": "rpt-order_validation-20260709091500"` — the corrected exporter format. Earlier this same session the fixture's `report_id` had a stale, hand-authored format (`rpt-order-validation-20260709-001`) that never matched the real exporter; hand-transcribing mock JSON would have risked reintroducing exactly that drift. Generating from the (now-fixed, regression-tested) fixture makes the mock data correct by construction.

## 5. Why `types/index.ts` keeps snake_case, and the routing/redirect details

`types/index.ts` is a verbatim copy of the TypeScript block already written into `context/ui-contract-plan.md` during Phase 7, which itself mirrors `src/contracts.py`. The field names are snake_case (`requested_qty`, `follow_up_priority`, `aging_bucket_counts`) because that is exactly the JSON the Python core emits — the Phase 7 decision was to preserve snake_case verbatim and add no camelCase adapter, so the UI consumes the Python output with zero mapping. This keeps three artifacts — Python contracts, mock JSON, and TS types — in lockstep, and means a field rename is a single coordinated change rather than a rename plus adapter churn. Payment Aging badges follow the same discipline: the types expose only the two real controlled-vocabulary fields (`aging_bucket`, `follow_up_priority`); no derived "Paid"/"Overdue" shorthand was introduced.

Routing is plain App Router file convention. The root `app/page.tsx` calls `redirect("/dashboard")` from `next/navigation` — `/dashboard` is the intended landing surface, so the bare root just forwards to it. Each of the five workflow routes is a directory with a `page.tsx` default-exporting a Server Component (no `"use client"` — there is no interactivity yet, matching `library-docs.md`'s "Server Components by default"). The stub pages deliberately use semantic token classes (`bg-surface`, `text-text-primary`, `border-border`, `hover:border-accent`) rather than placeholder markup, so that building and serving them actually exercises the token pipeline end to end.

## 6. How the token pipeline was verified end to end

"It compiled" is weaker than "the tokens actually render," so verification went past `next build`. The production server was started on a spare port and every route curled: the root returned 200 after redirect, and all five workflow routes returned 200. The dashboard HTML was grepped and confirmed to contain the semantic token classes (`bg-surface`, `text-text-primary`, `border-border`, `hover:border-accent`) rather than raw palette classes. Then the built CSS on disk (`.next/static/chunks/*.css`) was grepped directly — more reliable than curling once the server was down, and it revealed both halves of the pipeline: the `:root` variable definitions (`--accent:221 83% 53%`, `--surface:0 0% 100%`, `--text-primary:222 47% 11%` — minified, no space after the colon) and the compiled utility classes (`.bg-surface`, `.text-text-primary`, `.border-border`, `.hover\:border-accent`). That closes the loop from `ui-tokens.md` → `globals.css` `:root` vars → `tailwind.config.ts` mapping → utility classes → rendered pages.

`npm run typecheck` (`tsc --noEmit`, a script added to `package.json` because the scaffold provided none) passed under strict mode. `npm run lint` initially flagged one warning — `postcss.config.mjs` used an anonymous default export (`import/no-anonymous-default-export`); it was fixed by assigning the config to a `const` first, after which lint was clean. Finally `uv run pytest` was re-run and returned 152 passed, and `git status` on `src/ tests/ context/... pyproject.toml AGENTS.md CLAUDE.md CONTEXT.md README.md` (excluding the intentional `context/` doc edits) confirmed the scaffold left the Python side untouched.

## 7. The Figma reference reconciliation, and why it is documentation-only this phase

This session was the first with the Figma MCP server connected (remote `mcp.figma.com`; `whoami` confirmed the authenticated account). Four Figma Make prototype links were inspected via `get_design_context` using the Make path (`nodeId="0:1"`), which returns each prototype's full React/TSX source rather than a rendered frame. No MCP error occurred, so no PNG/screenshot fallback was needed.

The prototypes are AI-generated and, as expected, drift from the real contracts in specific ways that were catalogued into a new "Figma Reference Reconciliation" section in `context/ui-contract-plan.md`. The important corrections, all resolved in favor of the contracts: the Inventory prototypes use a `Critical | High | Medium | Low` priority vocabulary and treat `Invalid SKU` / `Supplier Follow-up` as allocation-status values — but the real `AllocationResultRow.status` has exactly three values, priority is `High | Normal | Low`, invalid SKUs never reach allocation (caught upstream by OV-003), and supplier follow-up is a separate list. The Payment Aging prototype states a $40,000 high-priority threshold; the real rule (`payment_aging.py`) is $50,000, and its priority set includes `Watch`/`None`, not a `"—"`. The Supplier panel invents `shortageQty`/`suggestedReorderQty`/`followUpPriority` fields that do not exist on `SupplierFollowUpRow`. The Overview Dashboard shows an 8-item ERP mega-nav and trend deltas backed by no contract or history. All four prototypes use raw Tailwind palette classes and ad-hoc hex.

None of this changed any Phase 8 code, because Phase 8 builds no screens — the reconciliation is a Phase 9 input. It is captured now, while the inspection is fresh, so Phase 9 builds from the corrected classification instead of copying the prototypes. The chrome decision was also settled here: only the Dashboard prototype uses the mandated sidebar nav, so `ui-rules.md`'s existing sidebar decision wins uniformly across all five routes, and the other prototypes remain reference for their *content* (stepper, filter bar, upload panel), not their chrome.
