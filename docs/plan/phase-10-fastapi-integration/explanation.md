# Explanation ŌĆö Feature phase-10-fastapi-integration: FastAPI Integration

## 1. End-to-end data flow: a click on "Run Validation"

Walk through exactly what happens when a user clicks "Run Validation" on `/order-validation` with both files already selected, since every one of the three workflow pages follows this same shape.

`UploadPanel`'s native `<input type="file">` fires `onChange`, which calls `onFileChange?.(file)` ŌĆö a prop added this phase specifically so the page, not just the component, gets a handle on the real `File` object. The page's `setOrdersFile`/`setProductMasterFile` store those `File`s in component state. Clicking "Run Validation" calls `handleRunValidation()`, a thin wrapper that guards on `canSubmit` (both files present) and calls `runValidation(ordersFile, productMasterFile)` ŌĆö a separate async function that takes the files as explicit parameters rather than reading them from closure state. (Section 10 explains why that separation exists.)

`runValidation` sets `status = "submitting"` (which flips `LoadingState` on), builds a `FormData` with `orders_file`/`product_master_file` keys matching the backend's `Annotated[UploadFile, File()]` parameter names exactly, and calls `postJSON<OrderValidationResult>("/api/orders/validate", formData)` from `lib/api-client.ts`. That function does a plain `fetch(..., { method: "POST", body: formData })` ŌĆö no manual `Content-Type` header, since the browser sets the correct `multipart/form-data` boundary automatically when the body is a `FormData` instance; setting it manually is a classic mistake that breaks the boundary parameter FastAPI needs to split the parts.

On the server, `backend/routers/orders.py`'s `validate_orders_endpoint` receives the two `UploadFile`s, passes each through `backend/uploads.py`'s `read_xlsx_upload()` (extension check, then delegates to `src/order_validation.py`'s own `load_orders`/`load_product_master`, which run `load_excel` + `validate_required_columns`), and if both load cleanly, calls `validate_orders(orders_df, product_master_df)` ŌĆö the exact same tested function `tests/test_order_validation.py` already exercises. The route's return type annotation is `OrderValidationResult` (a `TypedDict` imported straight from `src/order_validation.py`), and FastAPI/Pydantic serializes that dict to JSON automatically (see section 3).

Back in the browser, `postJSON` resolves with the parsed JSON, `runValidation` calls `setCurrentResult(result)` and `setStatus("succeeded")`, and the page re-renders its Summary/Validation Errors/Valid Orders sections from `currentResult` instead of a static import. Nothing about this chain is cached, stored, or remembered by the server after the response ŌĆö the next click re-runs the entire chain from the uploaded bytes.

## 2. Why `fastapi[standard]` instead of pinning `fastapi` + `uvicorn` + `python-multipart` separately

The first implementation draft declared three separate dependencies. This was caught during plan review, not during implementation: `pip`/`uv`'s `fastapi[standard]` extra already bundles `uvicorn[standard]` (the dev server), `python-multipart` (required for `Annotated[UploadFile, File()]`/`Annotated[str, Form()]` parsing ŌĆö FastAPI raises a runtime error at multipart-parse time without it, not an import error, so the failure mode is a confusing 500 the first time someone actually uploads a file, not an obvious startup crash), plus `httpx` and `jinja2`. Declaring the sub-dependencies separately risks them drifting to incompatible versions from what `fastapi[standard]` itself expects internally; declaring the extra instead makes `fastapi`'s own release matrix responsible for keeping them compatible. `uv sync` after the change resolved and installed 42 packages including `fastapi==0.139.0` and `pydantic==2.13.4` ŌĆö the Pydantic major version matters directly for section 3.

## 3. TypedDict as a FastAPI return type ŌĆö verified, not assumed

The plan called for route handlers to return the real `src/`-module `TypedDict`s (`OrderValidationResult`, etc.) directly, with no parallel Pydantic `BaseModel`. This is only safe because Pydantic v2 (confirmed installed: `2.13.4`) can build a `TypeAdapter` for a `TypedDict` and validate/serialize it natively ŌĆö a capability that didn't exist the same way under Pydantic v1. Rather than trust that from documentation alone, this was verified empirically before committing to the pattern across all three routers: a throwaway `TestClient` script posted the real `sample_data/sample_orders.xlsx` + `sample_data/sample_product_master.xlsx` to `/api/orders/validate` and printed the response. It returned `200` with `summary.total_orders: 36`, `summary.valid_orders: 28` ŌĆö matching the numbers already known from `context/progress-tracker.md`'s Phase 3/Sample-Data-Enrichment history. That one successful round trip was the actual confirmation; the formal `tests/test_backend_orders.py` suite was written afterward, once the pattern was known to work.

## 4. Why `read_xlsx_upload` takes a `loader` callable instead of its own column lists

`backend/uploads.py`'s `read_xlsx_upload(file, label, loader)` doesn't know what columns are required for an orders file versus an inventory file ŌĆö it takes the actual `load_orders`/`load_inventory`/etc. function from the relevant `src/` module as its third argument and calls `loader(file.file)`. The alternative ŌĆö duplicating `ORDERS_REQUIRED_COLUMNS`, `INVENTORY_REQUIRED_COLUMNS`, etc. inside `backend/` ŌĆö would create a second copy of information `src/order_validation.py` and `src/inventory_allocation.py` already own, and the two copies could silently drift (add a required column to the Python core, forget to update the FastAPI-side list, get a confusing "missing column" error that doesn't match what the loader actually checks). `read_xlsx_upload` only owns the *file-level* concerns ŌĆö extension check, catching whatever `loader` raises ŌĆö while `loader` itself owns exactly which columns are required and what `MissingColumnsError` says. This is the same "each concept has exactly one place it lives" principle the module boundaries in `context/architecture.md` already establish; Phase 10 just extends it across the new `backend/`Ōåö`src/` boundary.

## 5. The Inventory Allocation endpoint runs two business functions, not one

`POST /api/inventory/allocate` takes three files ŌĆö `orders_file`, `product_master_file`, `inventory_file` ŌĆö even though the Phase 9 static-showcase UI only ever had two upload panels ("Valid Orders File" and "Inventory File"). This wasn't a UI accident to preserve; it's a direct, necessary consequence of `allocate_inventory()`'s actual signature, which requires `valid_orders_df: pd.DataFrame` ŌĆö a DataFrame of orders that have already passed validation, not raw uploaded orders. There's no way to satisfy that from a single raw orders file; something has to run `validate_orders()` first. `backend/routers/inventory.py`'s `_run_allocation()` helper does exactly that: `validate_orders(orders_df, product_master_df)` produces a result whose `valid_orders` list gets converted via `pd.DataFrame(validation_result["valid_orders"])` into the DataFrame `allocate_inventory()` expects. This is the identical pattern `scripts/generate_mock_data.py` already used to build the committed mock JSON (`valid_orders_df = pd.DataFrame(order_validation["valid_orders"]); inventory_allocation = allocate_inventory(valid_orders_df, inventory_df)`) ŌĆö Phase 10 didn't invent a new chaining approach, it made the live API do what the mock-data generator already proved works. The frontend's Inventory Allocation page was updated from 2 upload panels to 3 (Orders File, Product Master File, Inventory File) to match what the endpoint actually needs, with a caption explaining the two-stage chain.

## 6. A `TestClient` gotcha: testing the generic 500 handler required `raise_server_exceptions=False`

`tests/test_backend_errors.py`'s test for the unexpected-exception path monkeypatches `validate_orders` to raise a plain `RuntimeError`, then asserts the response is a `500` with the generic message ŌĆö exercising `backend/errors.py`'s `@app.exception_handler(Exception)` handler. The first version of this test failed with the raw `RuntimeError` propagating straight into pytest as an uncaught exception, not as an HTTP response at all. The cause: Starlette's `TestClient` defaults to `raise_server_exceptions=True`, which is specifically designed to re-raise exceptions for debugging convenience during test-writing ŌĆö even when the app has a registered handler that would have converted the exception into a real response in production (under `uvicorn`, that handler runs correctly; it's `TestClient`'s own behavior that bypasses it by default). The fix was creating a second `TestClient(app, raise_server_exceptions=False)` instance just for that one test, which lets the request actually flow through `backend/errors.py`'s registered handler and return the real `500` JSON response instead of re-raising. This is a properly documented Starlette/FastAPI testing behavior, not a bug in the handler itself ŌĆö the handler was correct the whole time; only the test needed adjusting.

## 7. CORS `expose_headers`: present on the wire, invisible to JavaScript without it

The report endpoints return `X-Report-Id` and `X-Generated-At` as custom response headers alongside the `.xlsx` bytes. The first CORS configuration only set `allow_origins`. This was caught in plan review before implementation, not discovered as a runtime bug, but it's worth recording precisely why it would have failed: the browser's CORS model only exposes a small "safelisted" set of response headers (`Content-Type`, `Content-Length`, a few others) to cross-origin JavaScript by default ŌĆö everything else, including any custom `X-...` header, is present in the actual HTTP response (visible in a browser devtools Network tab) but returns `null` from `response.headers.get("X-Report-Id")` in `fetch()`-based code unless the server explicitly lists it in the `Access-Control-Expose-Headers` response header. FastAPI's `CORSMiddleware` accepts this as its `expose_headers` parameter. The final `backend/main.py` sets `expose_headers=["Content-Disposition", "X-Report-Id", "X-Generated-At"]` ŌĆö all three headers `lib/api-client.ts`'s `postReport()` actually reads.

## 8. ESLint's `exhaustive-deps` warning on `currentResult?.x ?? []`

All three workflow pages originally derived their table data as `const errors = currentResult?.errors ?? [];` directly in the component body. ESLint's `react-hooks/exhaustive-deps` rule flagged every downstream `useMemo` that depended on these values, because `?? []` constructs a brand-new empty array literal on every single render when `currentResult` is `null` ŌĆö so even though the *value* is conceptually "the same empty list," the *reference* changes every render, which would make a `useMemo` depending on it recompute every render regardless of memoization. The fix was wrapping each derivation in its own `useMemo(() => currentResult?.errors ?? [], [currentResult])`, so the fallback empty array is only reconstructed when `currentResult` itself actually changes. This isn't a correctness bug (the old code would still render right), but it defeats the point of the `useMemo`s reading from it ŌĆö a real, if minor, effectiveness issue the linter caught precisely and correctly.

## 9. `UploadPanel`'s Sample File link isn't a `<Button>` ŌĆö because `Button` has no `asChild` pattern

The natural first instinct for "a button-styled link" is `<Button asChild><a href={...}>Sample file</a></Button>`, the shadcn/Radix `Slot` pattern. This project's `Button` (`components/ui/Button.tsx`) is hand-written, not generated via shadcn (a deliberate Phase 9 decision to avoid injecting shadcn's default token set) ŌĆö it always renders a native `<button>` and has no `asChild`/`Slot` support at all. Passing `asChild` to it does nothing useful and TypeScript correctly rejects rendering a `<Button>` wrapping an `<a>` as a valid pattern here. The fix was exporting the `cva` variant function itself (`export const buttonVariants = cva(...)`, previously module-private) so any element ŌĆö not just the `Button` component ŌĆö can apply the identical visual classes: `<a href={...} download className={buttonVariants({ variant: "secondary" })}>Sample file</a>`. The same exported `buttonVariants` is reused by `ReportCard`'s new "Go to workflow" `Link`, so both new button-styled non-`<button>` elements this phase share one class source instead of two hand-copied class strings that could drift apart.

## 10. Why the run functions take explicit `File` parameters instead of reading closure state ŌĆö the "Run sample data" refactor

"Run sample data" was added mid-session, after the original plan and its live-browser verification pass. It fetches the workflow's sample file(s) via `fetchSampleFile()`, needs to display them in the upload UI, *and* needs to immediately run the workflow on them ŌĆö all from one button click. The original `handleRunValidation` read `ordersFile`/`productMasterFile` from component state (set by `UploadPanel`'s `onFileChange`). React state updates are asynchronous and batched: calling `setOrdersFile(fetchedFile)` does not make `ordersFile` equal to `fetchedFile` within the same synchronous function body ŌĆö it schedules a re-render for later. If "Run sample data" had called `setOrdersFile(orders); setOrdersFile(productMaster); handleRunValidation()` in sequence, `handleRunValidation` would still see the *previous* (likely `null`) values of `ordersFile`/`productMasterFile`, because the state update hasn't been applied to the component's rendered state yet at the point `handleRunValidation` runs.

The fix was extracting the actual request logic into a separate async function ŌĆö `runValidation(orders: File, productMaster: File)` ŌĆö that takes the files as explicit parameters instead of closing over component state. `handleRunValidation` (the button's `onClick`) now just reads current state and calls `runValidation(ordersFile, productMasterFile)`; `handleRunSampleData` calls `setOrdersFile`/`setProductMasterFile` for *display* purposes (so the upload panels visually update) but calls `runValidation(orders, productMaster)` directly with the freshly-fetched `File` objects, sidestepping the state-timing problem entirely. The same restructuring was applied to Inventory Allocation's `runAllocation` and Payment Aging's `runAging`.

## 11. Live-browser verification tooling: no project run-skill existed yet

This repository had no `.claude/skills/*/SKILL.md` describing how to launch and drive this app ŌĆö the `run` skill's search for one came back empty. Following its fallback pattern for a browser-driven web app, the intended tool is `chromium-cli`, which wasn't available in this environment. The fallback-of-the-fallback (documented directly in the `run` skill's `playwright.md` reference) is to drive the app with the `playwright` npm package directly. It wasn't a project dependency, so it was installed with `npm install --no-save playwright` (no `package.json`/`package-lock.json` change) plus `npx playwright install chromium` to fetch the browser binary (~94MB). Because Node's ESM `import` resolution doesn't consult `NODE_PATH` the way CommonJS `require` does, the driver script had to be physically located inside the project directory (where `node_modules/playwright` actually resolves) rather than run from the scratchpad ŌĆö it was copied in, executed, and deleted immediately after. `playwright` was uninstalled afterward (`npm uninstall playwright`), confirmed via `git diff package.json`/`package-lock.json` showing no change, so the temporary tooling left no trace in version control. If this project accumulates more browser-driven verification work, generating a real project run-skill (via `/run-skill-generator`, as the `run` skill itself suggests when its fallback path required real setup) would save re-deriving all of this next time.

## 12. The verification script's one false test failure ŌĆö and the real bug it accidentally surfaced later

The Playwright verification script asserted the deliberate error-case test (uploading the wrong file into the Orders slot) via `page.getByRole("alert").innerText()`, which came back empty even though the console log showed the expected `400` response. The screenshot from that exact step showed the correct, fully-rendered `BusinessErrorMessage` with the right text ("The uploaded orders file is missing required columns: ... Please check the sample file.") ŌĆö so this was a flaky assertion in the throwaway verification script (a timing/race quirk in when `innerText()` was called relative to the element's content settling), not a real product defect. It was resolved by trusting the screenshot evidence over the flaky assertion rather than debugging the disposable script further, since the script's only job was to produce evidence for this document, not to become a permanent test.

Separately, and unrelated to that flake: shortly after finishing verification, killing the two background dev-server processes as cleanup left the FastAPI backend down while the frontend (a leftover stray process on the same port) kept running. The user's very next real interaction with the live app reproduced exactly the "Could not reach the API server" message `lib/api-client.ts` is designed to show for a network failure ŌĆö which is worth noting precisely because it's *not* a bug: the error path worked exactly as designed, correctly reporting that the backend was genuinely unreachable. Diagnosed with a single `curl -sf http://localhost:8000/api/templates/orders` (returned connection failure, `lsof -i :8000` showed nothing listening) and fixed by restarting the backend ŌĆö no code changed. See `docs/diagnosing-bugs/` if a slug was created for that session; the finding was that no test seam applies to "did the operator restart the dev server," which is an operational concern, not a code defect.

## 13. Context-file wording correction reached all the way into Python

`src/excel_io.py`'s `MissingColumnsError` and `src/inventory_allocation.py`'s `InvalidInventoryDataError`/`InvalidOrderDataError` originally said "Please check the sample template." The grilling session (documented separately in `docs/grilling/phase-10-fastapi-integration/`) settled on "Sample File" as the correct term ŌĆö the committed `sample_data/*.xlsx` files carry the same intentional data-quality issues as the rest of the demo, not a blank starting point, so calling them "templates" implied something false. Fixing only the UI copy (`UploadPanel`'s button label) would have left the actual backend error messages ŌĆö the ones a user reads when something goes wrong ŌĆö saying the old, now-incorrect thing. All three Python exception messages were updated to say "sample file" instead, alongside the matching UI copy in `context/ui-contract-plan.md` and `context/ui-rules.md`'s documented example error strings, so the demo's error copy is consistent end-to-end rather than just at the UI layer.
