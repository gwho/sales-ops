# Explanation — Feature sample-data-enrichment: Sample Data Enrichment

## 1. Why this started, and what "planning input" vs. "schema authority" meant in practice

The user opened `sample_excel_data_requirements/08_agent_instructions_with_skills.md` in the IDE and asked to "read the file and implement what is required." That file is itself a meta-instruction document: it tells an agent to read seven companion planning docs in order, then treat the *current* `src/` modules and tests as schema authority, then run `/architect` before writing any code. The distinction mattered immediately once the companion docs were read (via three parallel Explore agents, one for the requirements docs, one for the current Python schema, one for the TypeScript/UI contract layer): the planning docs describe a customer master file (`customer_id`, credit-status logic), several new optional columns, and a completely different fictional business scenario than what `src/sample_data.py` actually generates today. Per the doc's own repeated warning ("the current Python business layer does not consume this file... do not make customer master upload required without a separate planning decision") and `CLAUDE.md`'s Field Scope Boundary rule, the planning docs were treated as *inspiration and structure*, never as a source of new required columns or business logic. Every column that ended up in the final generator was cross-checked against the actual `INVOICES_REQUIRED_COLUMNS`/`ORDERS_REQUIRED_COLUMNS`/etc. constants in `src/order_validation.py`, `src/inventory_allocation.py`, and `src/payment_aging.py` before being added.

## 2. Plan Mode stood in for `/architect` — and was told so explicitly by the user

`08_agent_instructions_with_skills.md`'s Skill Sequence step 1 is "Invoke `/architect` before coding... do not implement until the user explicitly confirms this plan." The session was already in the harness's built-in Plan Mode (triggered automatically), which has the same shape: research, produce a plan file, block all code changes until `ExitPlanMode` is approved. Rather than invoking a second, redundant `/architect` skill call inside Plan Mode, the plan file stated explicitly that Plan Mode was serving that role, and enumerated exactly which of the doc's later skills (`/feature-docs`, `/project-review`) would still run post-implementation and which (`/architect`, `/grill-with-docs`, `/request-refactor-plan`, `/to-issues`, `/imprint`) would not, with a one-line reason for each. The first `ExitPlanMode` call was rejected by the user with a substantive list of five corrections (see §3–§5 below); the *second* `ExitPlanMode` call was also rejected, but only to ask "would any skills mentioned in instructions be used based on the current plan?" — the answer was already implicitly present in the Context section, but the user wanted it as its own explicit, scannable section rather than buried in a paragraph. This is worth remembering as a pattern: when a plan references an external process document (like the skill sequence here), pull the process-compliance mapping out into its own heading rather than folding it into prose, since it's exactly the kind of thing a reviewer needs to scan quickly rather than read end-to-end.

## 3. The business-scenario pivot: from a generic multi-category catalog to a fictional medical-optics company

`src/sample_data.py` as it existed before this feature had nothing to do with the requirements docs' story. It sold a mix of medical, industrial, office, and electronics products (`Optical Lens Kit`, `Industrial Pressure Valve`, `Ergonomic Office Chair`, `USB-C Cable`) to customers in Hong Kong, Singapore, and Taiwan, warehoused across `HK Warehouse` and `SG Warehouse`. The requirements docs describe a medical-device company serving Hong Kong/Macau/Mainland China from HK/China/Europe warehouses. This is a genuine narrative fork, not a schema question, so it was surfaced via `AskUserQuestion` rather than decided unilaterally — the user chose to adopt the new scenario direction. In a follow-up correction (after the first plan draft), the user sharpened this further: the scenario should be explicitly "inspired by a ZEISS Far East-style operating model," while remaining "fully fictional... no real company names, customer names, product names, pricing, or confidential details." This phrasing was deliberately preserved verbatim in the plan's Context section and in `sample_data/README_sample_data.md`'s first paragraph, because it's the kind of instruction where losing the nuance ("inspired by the *shape* of an operating model" vs. "based on") would materially change what's acceptable to generate. The actual implementation never uses the string "ZEISS" anywhere — it appears only in planning documentation (this file, the plan file, `progress-tracker.md`) as a description of the *inspiration*, never as generated content. This was verified with a case-insensitive grep across `src/sample_data.py`, `sample_data/README_sample_data.md`, and `tests/test_sample_data.py` after implementation, which returned zero matches (exit code 1, meaning grep found nothing).

## 4. The issue-ratio math correction

The first plan draft listed the docs' issue types (duplicate order_id, unknown SKU, inactive SKU, missing required field, invalid quantity ×3 variants, missing delivery date, delivery-before-order, blank payment_terms, invalid priority) as roughly one dedicated row each. For a 30–40 row order set, that's 10+ issue rows — comfortably over the docs' own stated 15–20% target, and closer to "a disguised unit-test matrix" than a plausible operations day, which is exactly what the docs warn against. The user's correction specified a concrete target: "6-8 issue-bearing order rows; some rows may carry multiple intentional issues where realistic." The final design combines issues onto the same row wherever the combination is a plausible single data-entry mistake — for example, `SO-2026-032` is missing its `sku` *and* has a blank `payment_terms`, modeling one sloppy entry rather than two separate ones; `SO-2026-034` combines a negative quantity with an invalid priority value. This produced exactly 7 issue-bearing rows out of 36 total (≈19.4%), landing inside the target band while still exercising every individual issue type the docs listed. A dedicated test, `test_generate_orders_row_count_and_issue_ratio_stay_moderate`, asserts both the row-count band (30–40) and the ratio band (0.15–0.25) directly, so a future edit that quietly inflates the issue count will fail loudly rather than being caught only by manual inspection.

## 5. Resolving the "deterministic generator with a fixed seed" vs. "hand-authored literal rows" mismatch

The requirements docs describe a "deterministic generator... with a fixed seed, e.g. `RANDOM_SEED = 42`." The existing `src/sample_data.py` (and the rewritten version) has never used Python's `random` module at all — every row is a literal dict in a list, which is deterministic by construction with no seed needed. The user's correction made the resolution explicit rather than leaving it ambiguous: "for this moderate scale, explicit deterministic literal rows are acceptable... update the README requirement accordingly." The plan and the shipped `README_sample_data.md` both now say "Deterministic by construction... no `random` module, no seed" rather than describing a seed strategy that doesn't exist in the code. This matters because a reader who trusted the planning docs' wording over the actual `README_sample_data.md` might go looking for a `RANDOM_SEED` constant that was never going to exist — the fix was to make the *shipped* documentation match the *shipped* code, not to retroactively add a seed nobody needs at 36 rows.

## 6. Designing guaranteed allocation outcomes without simulating the whole engine by hand

`src/inventory_allocation.py`'s rules (read during exploration, not modified) sort valid orders by priority → delivery date → order date → order ID (IA-001), compute `allocatable_qty = available_qty - reserved_qty` per (sku, warehouse) row (IA-002), and for each order line pick **one** warehouse — the one with the highest current `allocatable_qty`, ties broken by warehouse name (the IA-007 V1 stand-in) — then fully, partially, or zero-allocate from that single warehouse only; stock is never split across warehouses for one order line. Rather than trying to hand-simulate every one of the 36 order rows against this algorithm, three specific, low-risk guarantees were engineered directly into the inventory data instead of the order quantities:
- **Guaranteed full backorder**: `PART-CAL-015` was given *zero* inventory rows anywhere. IA-005's rule ("backorder when allocatable = 0, or SKU not in inventory at all") means any valid order against it backorders unconditionally, with no dependency on other orders' timing or quantities. This is far more robust than trying to construct a quantity that exactly exhausts stock.
- **Guaranteed partial allocation**: `DIAG-OCT-005`'s HK Warehouse row was stocked at exactly 4 units, and the first (highest-priority) order against it (`SO-2026-005`, High priority, quantity 5) requests one more than that. Because it's the highest-priority order for that SKU, IA-001's sort guarantees it's processed first against the full 4 units, producing a clean 4-allocated/1-backordered partial split.
- **Guaranteed persistent reorder alert**: `DIAG-TONO-007`'s HK Warehouse row starts at `available_qty=2` against `reorder_point=3` (already below reorder point pre-allocation). Its China Warehouse row was deliberately given much higher `allocatable_qty` (8, after reserved_qty) than HK's (2), so the warehouse-pick rule always routes demand to China first, leaving HK's already-low stock untouched through the whole allocation pass — it's still below reorder point *after* allocation, which is IA-008's actual trigger condition.

These three were the only outcomes designed by hand-tracing the algorithm; everything else (which SKUs got how much stock, how many orders touched them) was populated by narrative plausibility and then verified by actually running the real pipeline (see §8) rather than continuing to hand-trace 30+ more rows.

## 7. The duplicate-order-id row also removes its "clean" twin from valid orders — a non-obvious allocation-story consequence

`SO-2026-010` appears twice: once as an ordinary, otherwise-clean order for `PART-BULB-013` (quantity 10), and again — deliberately — reusing the same ID for an unrelated `DIAG-TONO-007` order (quantity 1). OV-002's duplicate detection, confirmed by running the real pipeline, flags **both** occurrences as invalid, not just the second one; there's no way for the validator to know which is the "real" one. This means the `PART-BULB-013` order that looks perfectly clean in isolation never reaches inventory allocation at all — it's filtered out purely because of the ID collision with an unrelated row. This is intentional and mirrors the original (pre-enrichment) file's own pattern (`SO-2026-005` was reused the same way), but it's worth stating explicitly here because it's easy to misread the generator source and assume the first occurrence of a duplicated ID survives validation. It does not. The end-to-end compatibility run (§8) confirmed this: `duplicate_orders: 2` in the validation summary, not 1.

## 8. Why the sanity pass ran the real business modules instead of stopping at pytest

`tests/test_sample_data.py` only asserts properties of the generator's *output shape* (column presence, counts, specific row IDs) — it never calls `validate_orders`, `allocate_inventory`, or `calculate_payment_aging`. Passing pytest alone would not have caught, for example, whether the allocation story actually produced a partial/backorder/reorder-alert spread, or whether payment aging actually produced all 5 buckets once real bucket-boundary logic (not just the data-shape offset checks in the tests) was applied. So after pytest passed, a one-off script (not committed — this was verification tooling, not a permanent artifact) imported the real `validate_orders`, `allocate_inventory`, `calculate_payment_aging`, and all three `report_export` functions and ran them against the freshly generated DataFrames. This surfaced the true numbers: 28 valid / 8 invalid orders across 8 distinct error codes, 26 fully-allocated / 1 partially-allocated / 1 backordered order lines plus 3 supplier follow-ups (one more than the two hand-designed cases — `DIAG-OCT-005`'s HK Warehouse also ended up below its reorder point as a side effect of `SO-2026-005` consuming all 4 units, which is a legitimate emergent case, not a bug), all 5 aging buckets represented, both data-issue types caught, and 17 draft messages generated. All three Excel reports exported without error. This is the actual verification that matters for a "does the richer sample data still work" feature — pytest alone proves the generator's own claims about itself, not that downstream business logic accepts and processes the output believably.

## 9. Reconnecting the UI to the richer sample data — a deliberate, user-directed second pass

The first pass of this feature (§1–§8, §11) explicitly did *not* touch `lib/mock-data.ts`, `scripts/generate_mock_data.py`, or any generated JSON, because at the time `lib/mock-data.ts`'s header comment said the JSON came from `tests/contract_fixtures.py`, and that was true — the two pipelines were genuinely decoupled. Later in the same session, the user asked for a second, explicit step: read `08_agent_instructions_with_skills.md`'s roadmap item ("Refresh contract fixtures and mock JSON if generated output changes") and connect the enriched `sample_data/*.xlsx` to the static UI, using `/architect` to decide the approach before touching code. This was a real architecture decision, not a mechanical follow-up, because there were two defensible designs: keep `tests/contract_fixtures.py` as the UI's data source and just regenerate it with richer values (Option A/B-style, blurring "contract-shape test fixture" and "demo dataset" into one file), or keep the two fully separate and add a new generator that reads the actual sample Excel files (Option C). The user specified Option C directly rather than leaving it to be re-derived, and the `/architect` session (invoked via the `architect` skill, run inline in this same conversation) worked out the remaining mechanical decisions: rewrite `scripts/generate_mock_data.py` in place rather than adding a second script file (one obvious source of truth for `lib/mock-data/*.json`, no risk of someone running the stale one by accident), and add a manual-only `npm run mock-data` alias.

## 10. Why `MOCK_AS_OF_DATE`/`MOCK_GENERATED_AT` had to be pinned, and how the pinning was verified

`calculate_payment_aging(invoices_df, as_of_date=None)` and `report_export.py`'s three export functions (`generated_at: datetime | None = None`) both default to "now" when no explicit value is passed — the same pattern `sample_data.py`'s `write_sample_workbooks(reference_date=None)` already uses for invoice date offsets. Left at their defaults, every re-run of `scripts/generate_mock_data.py` would compute aging buckets and report timestamps relative to whatever day it happened to run, producing a spurious git diff in the committed `lib/mock-data/*.json` even when nothing about the underlying sample data changed. This was not a hypothetical concern: an earlier ad-hoc verification script in this same session called `calculate_payment_aging(invoices)` with no `as_of_date` at all, and its aging-bucket counts (`61-90 Days: 4, 90+ Days: 4`) came out slightly different from the final pinned run (`61-90 Days: 5, 90+ Days: 3`) — one invoice crossed a bucket boundary between the two runs simply because "today" had moved (or more precisely, differed between the sandbox's system clock at each invocation). That discrepancy, observed directly rather than theorized, is the concrete evidence that pinning was necessary, not just tidy. The fix was to add `MOCK_AS_OF_DATE = date(2026, 7, 10)` and `MOCK_GENERATED_AT = datetime(2026, 7, 10, 9, 0, 0)` as module-level constants matching `sample_data/README_sample_data.md`'s own stated "Generated on" date, and thread them explicitly into every call that would otherwise default to "now."

## 11. Loader choice: business-module `load_*` helpers, not raw `pandas.read_excel`

`scripts/generate_mock_data.py` loads each sample file through `load_orders`/`load_product_master`/`load_inventory`/`load_invoices` — the same functions `order_validation.py`, `inventory_allocation.py`, and `payment_aging.py` export for their own required-column validation — rather than calling `pandas.read_excel` directly and trusting the shape. This was a deliberate choice reinforced by the user's framing of the script as a "compatibility smoke test": using the real loaders means a required-column mismatch between `sample_data.py`'s output and a business module's `*_REQUIRED_COLUMNS` constant surfaces as a `MissingColumnsError` when the script runs, exactly as it would for a real upload, rather than surfacing later (or not at all) if the script had its own parallel, more lenient loading logic.

## 12. A real test bug: `paid_amount > invoice_amount` isn't a safe way to detect "overpayment" when negative amounts exist

The first test run failed on `test_generate_invoices_has_exactly_one_deliberate_overpayment_row`, which checked `df["paid_amount"] > df["invoice_amount"]` and expected exactly one match. It found two. The second, unintended match was `INV-2026-025` — the row deliberately created for the *separate* "invalid/negative invoice_amount" data-issue case (`invoice_amount = -500.00`, `paid_amount = 0.0`). Since `0.0 > -500.00` is true, that row trivially satisfied the overpayment condition despite having nothing to do with overpayment. The fix was to exclude rows with invalid (negative) `invoice_amount` from the overpayment check before comparing: `df[df["invoice_amount"] >= 0]` first, then the `paid_amount > invoice_amount` comparison. This is a small but real lesson about combining boolean conditions over deliberately-broken data: a check that's correct for well-formed rows can silently produce false positives once intentionally-invalid rows are mixed into the same DataFrame, and the fix belongs in the test's filtering logic, not in the data (the negative-amount row is correct as designed).
