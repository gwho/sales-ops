# Plan — Feature sample-data-enrichment: Sample Data Enrichment

## What was built

- `src/sample_data.py` — rewritten. Replaced the minimal mixed-category (medical/industrial/office/electronics), HK/Singapore/Taiwan scenario with a fictional regional medical optics/healthcare equipment company (HK + Mainland China customers, HK/China/Europe warehouses/suppliers). `generate_product_master()` now emits 18 SKUs (2 inactive, up from 1). `generate_orders()` now emits 36 rows (up from 12), 7 of which are issue-bearing (~19%), several combining more than one intentional issue per row. `generate_inventory()` now emits 17 rows across HK/China/Europe warehouses (up from 10 across HK/SG), including one SKU (`PART-CAL-015`) with zero inventory rows to guarantee a full backorder. `generate_invoices()` now emits 27 rows (up from 10) covering all 5 aging buckets, all 3 payment states, and a new optional `order_id` column. New `generate_customers()` function emits 17 rows for a new, explicitly reference-only `sample_customers.xlsx`. `write_sample_workbooks()` updated to also write the customers file.
- `tests/test_sample_data.py` — rewritten. Same structural assertions as before (required columns, inactive-SKU count, duplicate order_id, unknown SKU, below-reorder-point inventory, missing-due-date invoice, high-priority invoice shape, date-shift behavior, Excel round-trip) plus new assertions for every newly introduced issue type: inactive-SKU order reference, missing-SKU row, blank-payment_terms row, zero/negative/non-whole quantity rows, missing-delivery-date row, delivery-before-order-date row, invalid-priority row, order row-count/issue-ratio bounds, invalid/negative invoice amount, deliberate overpayment, full aging-bucket-range coverage, and the new `generate_customers()` function (required columns, unique IDs, name overlap with orders/invoices).
- `sample_data/README_sample_data.md` — new. Fictional-data disclosure, generation/determinism approach, file-by-file schema table, demo workflow summary, and a full list of intentional data issues with exact row IDs.
- `sample_data/sample_product_master.xlsx`, `sample_orders.xlsx`, `sample_inventory.xlsx`, `sample_invoices.xlsx` — regenerated from the rewritten generator.
- `sample_data/sample_customers.xlsx` — new, committed, optional/reference-only.
- `context/progress-tracker.md` — updated: new "Sample Data Enrichment (Phase 2 revisit)" progress section, Current Status block, and two new Decisions Made entries.

**Follow-on same session — mock-data pipeline reconnection:**

- `scripts/generate_mock_data.py` — rewritten in place (via `/architect`, user-directed "Option C"). No longer derives `lib/mock-data/*.json` from `tests/contract_fixtures.py`; now loads `sample_data/*.xlsx` via each business module's own `load_*` helper and runs the real `validate_orders` → `allocate_inventory` (independently) `calculate_payment_aging` → `report_export` pipeline. - `context/progress-tracker.md` — updated again: Current Status follow-on paragraph, two more Decisions Made entries.

## Schema changes

None. No `src/contracts.py` TypedDict changed, no business-rule input/output contract changed. `sample_invoices.xlsx` gained one new *optional* column (`order_id`) already anticipated by `payment_aging.py`'s optional-column handling — not a required-column change. `sample_customers.xlsx` is an entirely new file but is read by zero Python modules, so it introduces no schema surface for any business module to satisfy.

## Key invariants

- **`sample_customers.xlsx` must stay reference-only.** No function in `src/` may load, validate, or depend on it. Adding a `load_customers()`/`validate_customers()` function or making any upload panel require it is a scope-expanding change that needs a new ADR first, per `CLAUDE.md`'s Scope Gate.
- **`src/sample_data.py` generation stays deterministic-by-construction (literal hand-authored rows), not randomized.** No `random`/seed module. If the file ever needs to be regenerated with different content, the specific issue-bearing order_ids (`SO-2026-010` dup, `SO-2026-030`–`SO-2026-035`) and invoice_ids (`INV-2026-001`, `INV-2026-024`, `INV-2026-025`, `INV-2026-026`) referenced by `tests/test_sample_data.py` and `sample_data/README_sample_data.md` must be updated together — the tests, the README, and the generator must never drift out of sync on which row carries which issue.
- **`tests/contract_fixtures.py` stays permanently decoupled from `lib/mock-data/*.json`.** It is a small, single-row, contract-shape fixture set used only by `test_contracts.py`/`test_report_export.py` — never import it from `scripts/generate_mock_data.py` again. `lib/mock-data/*.json`, by contrast, is now *coupled* to `sample_data/*.xlsx`: regenerating the sample Excel files without re-running `npm run mock-data` (`scripts/generate_mock_data.py`) leaves the committed UI JSON stale. This reverses the original Phase 8/9 assumption that sample-data changes never require a frontend data refresh — that assumption held only while the two were disconnected, and it no longer does.
- **`MOCK_AS_OF_DATE`/`MOCK_GENERATED_AT` in `scripts/generate_mock_data.py` must stay in sync with `sample_data/README_sample_data.md`'s stated generation date.** If `sample_data/*.xlsx` is ever regenerated with a different `reference_date`, update both constants and the README together, or the committed `lib/mock-data/*.json` will show aging buckets/report timestamps computed "as of" a date that no longer matches the sample data's own documented generation date.
- **`scripts/generate_mock_data.py` is a compatibility smoke test, not just a data pipeline.** If it fails after a change to `src/sample_data.py` or any business module's required-column constants, treat that as equivalent to a failing `tests/test_sample_data.py` assertion — it means the committed sample Excel no longer matches what the Python loaders expect — not as something to silently work around.
- **No real company, product, customer, or pricing data anywhere in `sample_data.py`, the generated `.xlsx` files, or `README_sample_data.md`.** The "ZEISS Far East-style" scenario is tonal/structural inspiration only, grepped clean of the literal string "ZEISS" (case-insensitive) at the end of this feature — re-check with the same grep after any future edit to this file.
- **Field Scope Boundary still applies**: `unit_price`/`default_supplier` were deliberately not added to the product master even though the planning docs suggested them, because no `src/` module consumes them and `reorder_point` already lives per-warehouse in `sample_inventory.xlsx`. Don't add columns to any generated file "for symmetry" with the planning docs — only when a real business module needs to read them.
