## Key invariants

- **The Inverse Surface token family (`surface-inverse`, `surface-inverse-hover`, `text-on-inverse`, `text-on-inverse-muted`) is scoped to exactly two consumers: `SidebarNav` and `Button`'s `dark` variant.** Do not reach for these tokens elsewhere without a new token-change decision — the rest of the app stays on the light `surface`/`background` family.
- **`Button`'s `dark` variant is scoped to exactly two usages: each workflow page's "Download Report" button and `UploadPanel`'s "Sample file" link.** Never apply it to the primary Run action, "Run sample data", or `ReportCard`'s "Go to workflow" link — those three are deliberately unchanged so the button hierarchy stays 3-tier (primary blue / dark navy / bordered secondary), not flattened.
- **`MetricCard`'s prop contract (`label`, `value`, `icon?`, `tone?`) did not change** — only its internal layout did. Any future change to this component must preserve that contract, since ~15 call sites across `/dashboard` and all 3 workflow pages depend on it without their own per-site styling.
- **`/dashboard`'s 5-card Overview row is a deliberate, user-approved content reduction, not the full KPI set.** The dropped KPIs (Duplicate Orders, Invalid SKUs, Missing Fields, Total Order Lines, Partially Allocated, Low Stock SKUs, Total Outstanding, High Priority Count, 90+ Days Amount) are intentionally not shown on `/dashboard` — they remain on each workflow page's own post-run summary grid. Do not silently re-add them to `/dashboard` without the same explicit direction this consolidation had.
- **The 3 workflow pages' KPI summary grids each fit their own card count in one row on desktop** (`lg:grid-cols-6`/`5`/`4` for Order Validation/Inventory Allocation/Payment Aging) — this is intentionally different from `/dashboard`'s capped 5-column row. If a KPI is ever added/removed from a workflow's summary, update that page's own `lg:grid-cols-N` to match its new count; do not apply a shared cap across all KPI grids in the app.
- **`DonutBreakdownChart`'s per-segment `<title>`/text content must be built as a single JS expression before the JSX, never as multi-line JSX children** (e.g. `<title>{a}: {b} ({c}%)</title>` split across lines) — this caused a real React hydration mismatch (see `explanation.md` §3) because the server/client-rendered whitespace inside `<title>` didn't match bit-for-bit.
- **Any full-size absolutely-positioned overlay stacked on top of an interactive SVG/HTML element (e.g. `DonutBreakdownChart`'s center-total display) must carry `pointer-events-none`** unless it is itself meant to intercept input — otherwise it silently swallows hover/focus events meant for the elements beneath it, with no console error or visual sign that anything is wrong.
- **`SidebarNav`'s fixed `w-60` width and lack of a responsive collapse/drawer is a known, pre-existing, out-of-scope limitation** — Phase 10.2 only changed its color and added icons, not its width or breakpoints. Do not treat the mobile content-squeeze this causes as a Phase 10.2 regression; it predates this phase and needs its own planning pass if it's ever addressed.
