# Explanation — Feature phase-10.2-portfolio-ui-polish: Portfolio UI Polish

## 1. Why this was planned via `/architect` before any code was touched

Phase 10.2 was framed from the start as a "token-only visual/hierarchy pass" — no backend, API, contract, or Phase 11 (SQL Reporting) changes. That framing came from the previous session's `/grilling` planning pass, which had already decided Phase 10.2 should exist as a small, scoped step between the completed Phase 10 (FastAPI Integration) and the already-approved-but-queued Phase 11. Given that scope boundary, the risk wasn't "will this work technically" — it was "will this quietly grow into something bigger than a style pass, or drift from what the user actually pictured." That's exactly the failure mode `/architect`'s plan-mode workflow is built to catch: it forces explicit alignment on ambiguous terms (what "compact tile" means, whether new tokens are dedicated or shared) and pins down subjective design decisions (exact Tailwind config key shapes, which buttons get the new dark variant) before a single line of component code changes. The plan file this produced (`/Users/jessejames/.claude/plans/plan-phase-10-2-portfolio-abstract-stream.md`) went through one full review-and-correction round with the user before approval — see `docs/architect/phase-10.2-portfolio-ui-polish/decisions.md` for the ten decisions that came out of that round, several of which (the active-sidebar-link wording, the disabled-dark-button handling, the exact token key shapes) directly shaped the code described below.

What the plan didn't anticipate — and this is a genuinely useful thing to notice about how this feature actually got built — is that the user kept extending scope *during* implementation by sending live screenshots of the running app: first a reference dashboard image and a screenshot of the real, broken `UploadPanel` alignment; then, after the first pass landed, a screenshot of the actual chart-card blank-space bug; then a request to consolidate the dashboard's KPI layout entirely; then more screenshots surfacing a sample-file button sizing bug and asking for sidebar icons and card hover effects. None of these were treated as a second full plan-mode round — each was small enough, and grounded enough in a concrete visual artifact, to implement directly, verify, and fold into the same Phase 10.2 scope. The plan file's addendum section documents this explicitly so a future reader doesn't mistake the original approved plan for the full story of what shipped.

## 2. The shared "Inverse Surface" token architecture, and why the Tailwind config key *shape* mattered as much as the values

The core design decision was whether the new dark-navy tone needed one token family or two: a `SidebarNav`-only family plus a separately-tuned dark color for the new `Button` variant, or one shared family used by both. The user's own framing settled this before implementation started — "the sidebar and dark buttons should look like the same visual system... a shared inverse token family... prevents two separate token families from drifting" — so `--surface-inverse`, `--surface-inverse-hover`, `--text-on-inverse`, and `--text-on-inverse-muted` were added once to `app/globals.css`'s `:root` and consumed by both `SidebarNav.tsx` and `Button.tsx`'s new `dark` variant.

What took a real correction round was the exact shape of the `tailwind.config.ts` entries. This codebase's existing token config mixes two conventions: some color families are nested objects (`surface: { DEFAULT, muted, subtle }`, which Tailwind expands into `bg-surface`, `bg-surface-muted`, `bg-surface-subtle`), while text colors are flat sibling keys (`"text-primary"`, `"text-secondary"`, each mapping to one CSS variable, expanding into `text-text-primary`, `text-text-secondary`). The first implementation pass didn't pin down which shape the two new families should use. The user's review corrected this precisely: `surface-inverse` should be nested (`{ DEFAULT, hover }`, matching the `surface`/`border` precedent, since `hover` is a genuine state-variant of one base color) while `text-on-inverse` and `text-on-inverse-muted` should be two separate flat keys (matching the `text-primary`/`text-secondary` precedent, since they're independently-named text roles, not a base+variant pair). Getting this wrong wouldn't necessarily throw a build error — Tailwind would still generate *some* class name from whatever shape it was given — but it could generate an unexpected one that silently doesn't match what the component files reference, a bug that surfaces as a missing CSS utility class rather than a TypeScript error, and is annoying to trace. The final config: `"surface-inverse": { DEFAULT: "hsl(var(--surface-inverse))", hover: "hsl(var(--surface-inverse-hover))" }` alongside two flat top-level keys, `"text-on-inverse"` and `"text-on-inverse-muted"`, each a direct `hsl(var(...))` string.

One more reuse decision worth naming: `--surface-inverse`'s value (`222 47% 11%`) is the *exact same* HSL triple as the pre-existing `--text-primary`. That's deliberate, not coincidental — rather than picking an arbitrary "looks navy enough" value, this reuses a hue/lightness this palette already uses (as a *text* color on light backgrounds) and repurposes it as a *fill*. It means the new dark surface is provably drawn from the existing palette rather than a new color invented by feel. The same logic applies to `--text-on-inverse: 0 0% 100%`, which reuses `--text-on-accent`'s pure white.

## 3. The React hydration mismatch inside `DonutBreakdownChart`'s SVG `<title>`, and why it was subtle

While adding hover/focus tooltips to the donut chart, each ring `<circle>` gained a nested `<title>` element (for a native browser tooltip and screen-reader text), written initially as:

```tsx
<title>
  {segment.label}: {formatNumber(segment.value)} ({percent}%)
</title>
```

This looked completely ordinary — it's the same multi-line JSX-with-embedded-expressions pattern used throughout the rest of this codebase without issue. But loading `/dashboard` in a real browser (during the Playwright verification pass, by tailing the Next.js dev server log) surfaced a genuine React error: "Hydration failed because the server rendered HTML didn't match the client," with the diff pointing specifically at the `<title>` element inside `DonutBreakdownChart`. React's own hydration algorithm compares the server-rendered HTML text content against what the client's re-render produces, and for reasons rooted in how JSX serializes text nodes split across multiple lines with embedded expressions — the line breaks and indentation between `{segment.label}:`, `{formatNumber(...)}`, and `({percent}%)` become part of the text node structure, and how a browser's SVG/`<title>` handling normalizes or doesn't normalize that whitespace can differ subtly from what React's server-side renderer assumed — the exact string ended up different enough between server and client to trip the mismatch check. The practical consequence, visible to any real user loading the dashboard, was that React would discard and fully remount that part of the tree client-side, causing a visible flash on every page load — not a cosmetic nit, a real defect.

The fix was to stop building the tooltip text as multi-line JSX children and instead compute it as a single JavaScript string *before* the return statement, then reference that one variable as the sole child of `<title>`:

```tsx
const segmentTitle = `${segment.label}: ${formatNumber(segment.value)} (${percent}%)`;
// ...
<title>{segmentTitle}</title>
```

With the text now built as one atomic string with no JSX-formatting whitespace anywhere near it, server and client produce byte-identical output, and the hydration warning disappeared on the next verification pass (confirmed by clearing the dev server log and re-running the same page load — nothing logged). The same `segmentTitle` string is also reused for the circle's `aria-label`, so the fix also removed a small duplication that existed in the first draft (the aria-label and the title text were computed via two separate template literals that could have drifted apart).

## 4. The pointer-events bug: the donut's own center label was silently blocking hover on the ring beneath it

This was found by the verification tooling itself, not by visual inspection. The first attempt to script a Playwright hover test against a ring segment (`circles.first().hover()`) failed with a timeout, and Playwright's own diagnostic trace named the exact blocking element: `<span class="text-xl font-semibold text-text-primary">28</span>` — the donut's center "total" label — reported as intercepting the pointer event before it could reach the `<circle>` underneath.

The root cause was structural: the center-total overlay is `<div className="absolute inset-0 flex flex-col items-center justify-center">`, and `inset-0` stretches that div to cover the *entire* bounding box of its relatively-positioned parent — not just the donut's hollow center, but the full square area the ring itself occupies too. Since this div comes after the `<svg>` in DOM order, it renders on top in the default stacking order, and because it has no `pointer-events` override, any part of its box — including the parts sitting directly over the visible ring — captures mouse and focus events meant for the SVG circles beneath it. This is a real, user-facing bug, not just a test artifact: a real person hovering over the ring in a browser would have had exactly the same problem, unable to trigger a tooltip anywhere the center overlay's invisible box happened to sit on top of it.

The fix was a single class addition: `pointer-events-none` on that overlay div. Once the overlay stopped capturing events, mouse hover worked immediately; the *test script* still needed a separate adjustment afterward, switching from `.hover()` (which targets the geometric center of an element's bounding box — always the donut's empty hole for an annulus shape, never a point actually on the visible stroke) to `.focus()` (which exercises the `onFocus` handler directly, sidestepping the geometry question entirely and testing the more important a11y-relevant behavior anyway). Both fixes were necessary and served different purposes: the `pointer-events-none` fix corrected a real bug affecting real users; the test-script change corrected an unrelated limitation of how Playwright's default hover targeting works against ring-shaped SVG geometry.

## 5. `UploadPanel`'s "alignment" bug was actually a bottom-anchoring bug, and reading the request literally would have fixed the wrong thing

The initial request described the problem as the "Choose a file…" text and "Browse" button not being aligned "cleanly on the same horizontal row." Read at face value, that implies the row's own flex properties are wrong. But `UploadPanel.tsx`'s existing markup for that row was already `flex items-center justify-between gap-3` — correct, already doing exactly what a same-row-alignment fix would produce. Tweaking that row further would have changed nothing visible.

The user then supplied a real screenshot of the live Inventory Allocation page, and the actual mechanism was visible there: three `UploadPanel` cards side by side, each with a different-length "Required columns" list above the drop zone. Because that text wraps to a different number of lines per card (Orders File's long column list wraps to three lines; Product Master's and Inventory File's short lists wrap to one), the drop-zone row sitting in normal document flow beneath it lands at a different vertical height in each card — an *inter-card* misalignment across a grid row, not an *intra-row* one. This is a common shape of bug: a component that looks entirely correct viewed in isolation (one card, one row) only reveals its problem when several instances with genuinely different content lengths sit next to each other.

The fix targeted the actual mechanism: `Card` gained `h-full` (letting it participate in CSS Grid's default `align-items: stretch`, so all three cards in a row match the tallest one's height), and the drop-zone `<label>` plus the Sample-file `<div>` were wrapped in a new `mt-auto flex flex-col gap-3` block, pinning that specific content to the *bottom* of the now-equal-height card regardless of how many lines the required-columns text above it took. This is a standard "stretch the outer box, then push a specific inner block to the far edge" pattern, and it's self-limiting in a useful way: on a narrow (mobile) viewport, where the grid collapses to one column, there's no taller sibling for `h-full` to stretch to, so the fix is inert there rather than introducing a new gap — confirmed directly in the Playwright mobile screenshot rather than assumed from the CSS mental model alone.

## 6. The chart-card sizing bug: CSS Grid's default row-stretch, and why "content plus padding" needed to be made explicit rather than incidental

A later screenshot showed the "Allocation Status" donut card with a large blank area beneath the donut and legend — visually oversized and unbalanced next to its sibling. The two chart `Card`s sit in one `<section className="grid gap-4 lg:grid-cols-2">` row, and CSS Grid's default cross-axis behavior (`align-items: stretch`) makes both grid *items* — the `Card` divs — match the height of whichever is naturally taller, based on their own content. `VerticalBucketBarChart`'s natural content (a subtitle line, a fixed `h-32` bar area, and a bucket-label row beneath it) is taller than `DonutBreakdownChart`'s natural content (just a `h-32` ring plus a shorter legend list), so the shorter donut card was being stretched to match — and because nothing inside it was set up to use that extra space, all of it collected as dead space below the donut.

The fix replaced each chart's plain `<div className="mt-3">` body wrapper with `<div className="mt-3 flex min-h-48 flex-col justify-center">` — an explicit, *identical* minimum height on both cards (`min-h-48`, Tailwind's standard `12rem`/192px scale value, chosen over an arbitrary bracket value to stay on the project's existing spacing scale), with `justify-center` vertically centering whatever content sits inside. This solves the problem at its root rather than patching the symptom: once both chart bodies commit to the same explicit minimum height, CSS Grid's row-stretch has little or nothing left to stretch — the two cards are already near-equal by design, not by chance — and if a future edit does make one chart's content taller than `min-h-48`, `justify-center` ensures any leftover stretched space is distributed evenly rather than dumped entirely below the content, which is the visually broken state that prompted this fix in the first place.

## 7. The dashboard KPI consolidation was a deliberate content decision, not a restyle, and shifted mid-session

The originally approved plan kept `/dashboard`'s three per-workflow KPI groups (Order Validation, Inventory Allocation, Payment Aging — roughly 15 `MetricCard` tiles combined, plus a `SegmentedBar`) intact, only restyling `MetricCard` itself and retuning each group's grid density. Partway through implementation, the user redirected this explicitly: "The current dashboard has too many KPI cards arranged in workflow-specific groups... I want the dashboard to feel more like a modern executive overview: fewer top-level KPI cards." This is a genuinely different kind of change from everything else in this phase — every other edit in Phase 10.2 preserves all existing data and values, restyling only; this one removes KPIs from `/dashboard` entirely.

The user's own message supplied the justification and the boundary condition: dropping a KPI from `/dashboard` is acceptable specifically *because* the same information remains available elsewhere — each workflow page's own post-run summary grid already displays the full, un-trimmed KPI set for that workflow once a user actually runs it. `/dashboard` becomes a curated overview, not a second copy of every summary metric. The five cards chosen (Total Orders, Invalid Orders, Fully Allocated, Backordered, Overdue Amount) were picked to mirror the reference dashboard's own composition — four counts and one dollar amount — giving the row some variety in what kind of number each tile shows, rather than five same-shaped counts. The `SegmentedBar` (Valid vs. Invalid orders) was dropped outright rather than relocated, since its information — the ratio between two numbers already shown as separate KPI cards — is redundant with the new Total Orders/Invalid Orders tiles, and the user's own "preferred structure" for the page only specified two charts in the main insight row (Allocation Status, Payment Aging Buckets), not a third Order Validation visualization.

A second, related correction followed almost immediately: the first pass had applied a shared "cap every KPI grid at 4 columns" rule to *both* `/dashboard` and the three workflow pages' own post-run summary grids. The user's next screenshot (of Inventory Allocation's summary wrapping to 4+1 across two rows) made clear this was wrong for the workflow pages specifically — they wanted each workflow page's summary to fit its own card count in a single row on desktop, distinct from `/dashboard`'s deliberately-capped 5-card row. The fix was to revert each workflow page's `lg:grid-cols-4` back to matching its actual KPI count (`lg:grid-cols-6` for Order Validation's 6 cards, `lg:grid-cols-5` for Inventory Allocation's 5), while leaving `/dashboard`'s new 5-card Overview row at `lg:grid-cols-5` (which happens to already be a single row for exactly 5 cards, so no further change was needed there).

## 8. Two different tooltip implementation strategies for the two charts, and why they didn't need to match

The user's later request for "data-bearing hover" tooltips (not just decorative opacity changes) was implemented differently in each chart component, deliberately:

`DonutBreakdownChart` needed local component state (`useState<string | null>` for `hoveredLabel`), because its tooltip content is *shared and swappable* — a single floating card near the ring, and the always-visible center-total display, both need to know *which* segment (if any) is currently active, and that's inherently cross-element coordination (hovering circle A needs to change what's displayed in a totally different DOM location). That requirement — one piece of UI reacting to hover state on a different, sibling element — can't be expressed in pure CSS without much more elaborate tricks (like modern `:has()` selectors), so the component was promoted to a Client Component (`"use client"`) to hold that state directly. Both the ring segments and the corresponding legend rows read and write the same `hoveredLabel` state, so hovering either one highlights both — the ring segment's opacity change and the legend row's background highlight are driven by the same source of truth.

`VerticalBucketBarChart`, by contrast, needed no shared cross-element state at all — each bar's tooltip is entirely self-contained: hovering bar N should only ever affect bar N's own fill opacity and its own tooltip, never anything about bar N+1. That's exactly the shape of interaction Tailwind's `group`/`group-hover`/`group-focus` utilities solve without any JavaScript: each bar column becomes a `group relative` container holding both the bar itself (with `group-hover:opacity-70`) and an absolutely-positioned tooltip div (with `opacity-0 group-hover:opacity-100`), with no state anywhere. The component stayed a Server Component, since nothing about it requires client-side interactivity beyond what CSS `:hover`/`:focus-within` pseudo-classes already provide. This is a useful contrast to keep in mind for any future chart component: the deciding factor for "does this need `useState`" is not "does it need to react to hover" but specifically "does reacting to hover on element A need to change something rendered somewhere other than element A."

## 9. The `UploadPanel` Sample-file button sizing bug, and the specific flex mechanics behind it

After the bottom-anchoring fix (§5) shipped, a further screenshot showed the "Sample file" button on the 3-column Inventory Allocation page rendering as an oversized, roughly-square block with its text wrapped onto two lines ("Sample" / "file"), visually inconsistent with its single-line rendering on other pages. The row containing it — `<div className="flex items-center justify-between gap-3"><span>...</span><a>...</a></div>` — had neither child given an explicit sizing hint: by default, flex children can both shrink (`flex-shrink: 1` is the browser default) to fit available space, and in a narrow 3-column card, the caption text ("Need a starting point? Use the sample file.") and the button were competing for a tighter row than on wider single- or two-column pages. Because the caption `<span>` had no `min-w-0` and text wraps naturally, and the anchor had no `shrink-0`, both elements shrank together — but shrinking an inline-flex anchor whose content is two words separated by a space eventually forces those words onto separate lines once the available width drops below the content's natural single-line width, which is exactly what happened.

The fix targeted each element's actual role: `min-w-0` on the caption span lets it be the one that yields space by wrapping onto multiple lines (a `<span>` without `min-w-0` can refuse to shrink below its content's intrinsic width in a flex context, which was part of what was squeezing the button in the first place); `shrink-0 whitespace-nowrap` on the anchor guarantees it never participates in the shrink calculation and never wraps its own text, so it renders at its normal single-line compact size on every page regardless of how much room the caption text needs to give up. This required adding a `cn()` import to `UploadPanel.tsx` (it wasn't previously used in that file) to merge the new utility classes with the base `buttonVariants({ variant: "dark" })` output.

## 10. Verification tooling: reusing the project's established throwaway-Playwright pattern, and rediscovering the stray `.next` build-artifact bug

This project has no committed run/test skill for live-browser verification (noted in the Phase 10 session's own memory record), so verification followed the same established, one-off pattern used in that prior session: `npm install --no-save playwright`, a throwaway driver script copied briefly into the project root (ESM `import` resolution needs the script physically inside a directory with access to the project's `node_modules`; running it from the scratchpad directory failed with `ERR_MODULE_NOT_FOUND`), then deleted afterward along with `npm uninstall playwright --no-save` — confirmed via `git diff --stat package.json` showing no trace, matching the precedent from the Phase 10 session.

Along the way, `npm run typecheck` initially failed with two unrelated-looking TypeScript errors inside `.next/types/` (`cache-life.d 2.ts`, `routes.d 2.ts` — note the literal space before "2" in the filenames, a signature of a macOS Finder-created duplicate file, not a real source file). This is the exact same class of bug recorded in the Phase 9.1 session's memory notes, caused by stray Finder-duplicate build artifacts confusing `tsc`'s module resolution when it encounters two files that both look like they define the same route types. The fix was identical to that prior occurrence: `rm -rf .next` (safe — it's build output, already gitignored) followed by re-running the affected command, which regenerates a clean `.next/` directory. One additional wrinkle this time: an already-running `next-server` process (left over from an earlier session, serving from the just-deleted `.next` build) started returning `500` errors on every route once its backing build directory disappeared mid-session; the fix was to kill that stale process and start a fresh `npm run dev`, after which every route returned `200` again.

## 11. The sidebar's mobile responsiveness limitation was surfaced, not fixed — and why that line was held

A mobile-viewport screenshot during verification showed `/dashboard`'s main content column badly squeezed — headings clipped, text overlapping the top bar. The cause is straightforward: `SidebarNav` is a fixed `w-60` (240px) on every viewport, with no hamburger/drawer/collapse behavior anywhere in `AppShell.tsx` or `SidebarNav.tsx`, so on a 390px-wide mobile viewport, the sidebar alone consumes over 60% of the available width. This is real, and visually bad — but it is not a Phase 10.2 regression. Nothing in this phase touched `SidebarNav`'s width, `AppShell`'s grid/flex breakpoints, or introduced any new responsive behavior for the sidebar; Phase 10.2 only recolored `SidebarNav` and added icons to it. The identical squeeze would have existed, pixel-for-pixel, with the *old* light-colored sidebar in the *same* fixed-width layout, on the same viewport, before this phase started.

The decision was to name this clearly to the user and to `context/ui-registry.md` rather than silently building a collapsible/drawer sidebar as an unplanned addition. A responsive navigation pattern is a real feature with its own design questions (does it become an overlay drawer, a bottom tab bar, an icon-only rail — and at what breakpoint) that deserves its own scoping decision, not a quiet scope-creep tacked onto a phase explicitly bounded as "token-only visual polish." This mirrors the same discipline `docs/architect/phase-10.2-portfolio-ui-polish/discussion.md` describes for reference-screenshot handling: a picture of a problem is not itself permission to solve every problem the picture happens to reveal.
