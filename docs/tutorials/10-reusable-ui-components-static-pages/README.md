## How to use an LLM before this tutorial

### Concept 1 — Variant-based styling versus conditional class strings

> "Explain the difference between writing a component's CSS classes as a chain of manual
> `if`/ternary logic (`isPrimary ? 'bg-blue-600' : 'bg-gray-200'`) versus using a dedicated
> 'variants' library that maps a small set of named variant values to fixed class strings declared
> once, up front. Independent variant dimensions (say, `variant` and, separately, `size`) each add
> their own named branch rather than requiring a class string per *combination* — explain that
> distinction precisely, including what still grows if two dimensions genuinely need to interact
> (a `dark` variant that needs a different disabled treatment than `primary`, for instance). Quiz me
> on when a config like this stays simple listing, and when it needs an explicit compound-variant
> entry instead."

*What to listen for:* manual conditional class logic tends to hardcode combinations directly
(`isPrimaryAndLarge ? '...' : isPrimaryAndSmall ? '...' : ...`), so adding a dimension multiplies the
number of branches already written. A declarative variants config avoids that specific failure mode
for *independent* dimensions — each one is listed once, and the library composes the right classes
for whatever combination is actually requested, without a hand-written branch per combination. This
is centralizing and typing the decision, not eliminating combinatorics outright: dimensions that
genuinely interact (two variants needing meaningfully different behavior when combined, not just
stacked) still need an explicit compound-variant entry naming that specific interaction — the
library gives you a place to declare it precisely, it doesn't make the interaction disappear.

*Practice question:* if a `Button` component supports 3 visual variants and, independently, 2 sizes,
and every size behaves identically across every variant with no special-case interaction, how many
variant-config entries does that need — one per variant, one per size, or one per combination? What
would change if the `dark` variant specifically needed different padding at the `sm` size than every
other variant does?

### Concept 2 — Discriminated unions for "this prop only makes sense in this state"

> "Explain a 'discriminated union' (also called a 'tagged union') in a statically-typed language: a
> type that's actually one of several distinct shapes, each identified by a shared 'tag' field with
> a different literal value, where the *other* fields available differ per tag. Give a concrete
> example of a UI component whose props genuinely differ depending on a status/state field — e.g. a
> component that needs a 'result' object only when its state is 'success' and needs nothing else in
> that case. Quiz me on what a non-discriminated version of the same props type would have to do
> instead (e.g. making every field optional) and what bug that invites."

*What to listen for:* a discriminated union lets the type system guarantee that accessing a
state-specific field (like a `manifest` that only exists when `state === "Ready"`) is genuinely
safe, because the compiler narrows the type the instant it sees a check against the tag field. The
non-discriminated alternative — one flat type with every field marked optional — compiles, but
provides zero protection against reading a field that doesn't actually apply in the current state
(nothing stops code from accessing a `manifest` that's `undefined` because the real state is
`"Not Generated"`), pushing a compile-time-catchable mistake into a runtime bug instead.

*Practice question:* if a component's props type marks every field optional instead of using a
discriminated union, what stops a caller from passing `state: "Not Generated"` and a `manifest`
object at the same time — an invalid combination that a discriminated union would make
unrepresentable?

### Concept 3 — Exhaustiveness checking as a design tool, not just a syntax feature

> "Explain how a statically-typed language's `switch` statement (or pattern match) over a value
> whose type is a small, fixed set of literals (e.g. `\"a\" | \"b\" | \"c\"`) can let the compiler
> verify every possible value is handled, catching a missing case at compile time rather than
> letting it silently fall through at runtime. Quiz me on what happens, under this kind of checking,
> if a new literal value is later added to the union type but an existing `switch` statement
> elsewhere in the codebase is never updated to handle it."

*What to listen for:* a `switch` over a closed literal union, written so every branch returns a
value and there's no unhandled fallthrough, effectively becomes a compiler-verified checklist —
adding a new member to the union type later will make every *existing* exhaustive `switch` over
that type fail to compile, until every one of them is updated to handle the new case. This turns
"did I remember to update every place that switches on this value" from a manual audit into an
automatic, compiler-enforced one.

*Practice question:* if a five-value union type gains a sixth value, and three different `switch`
statements elsewhere in the codebase each exhaustively handle the original five, what happens to
each of those three `switch` statements the next time the project is type-checked?

### Concept 4 — Computing derived state during render vs. inside an effect

> "In a UI framework with a 'render' phase (compute what should appear) and a separate mechanism for
> side effects that run *after* rendering (like React's `useEffect`), explain why a piece of state
> that needs to reset whenever a *different* piece of state or prop changes is often better computed
> directly during render — comparing the new value to a remembered previous value, right there in the
> function body — rather than using an effect that runs afterward and calls a state setter. What
> concrete extra render cycle does the effect-based approach cost that the render-time approach
> avoids? Quiz me on when an effect genuinely is the right tool instead."

*What to listen for:* an effect that exists purely to synchronize one piece of state to another
(no real side effect involved, like a network call or a subscription) forces an extra render pass —
render once with stale state, the effect runs after that render commits, the effect calls a setter,
and a second render happens with the corrected state. Computing the same reset directly during the
component's own function body (comparing a prop/derived value against a remembered previous value,
right in the render path) reaches the correct state in the very first render, with no extra pass.
An effect remains the right tool specifically when a *real* side effect is involved — a subscription,
a manual DOM measurement, a network request — not when the only goal is keeping two pieces of
React-owned state in sync with each other.

*Practice question:* if a component needs to reset a `selectedTab` state back to `0` every time its
`items` prop changes to a different list, and this reset is implemented inside a `useEffect` that
watches `items`, how many render passes does the component go through before the user ever sees the
corrected `selectedTab` value?

### Concept 5 — Comparator functions and stable sort

> "Explain what a 'comparator function' is in the context of a general-purpose sort function (like
> `Array.prototype.sort` in JavaScript, or `sorted(key=...)` in Python): a function that takes two
> elements and returns a negative, zero, or positive number indicating their relative order. Explain
> what 'stable' means for a sort algorithm — what property is preserved for elements that compare as
> equal. Quiz me on what a three-way comparator (`va < vb ? -1 : va > vb ? 1 : 0`) actually does for
> two values that are exactly equal, and why that specific return value matters for stability."

*What to listen for:* a stable sort guarantees that elements which compare as *equal* under the
comparator retain their original relative order in the output, rather than being shuffled
arbitrarily among themselves. A three-way comparator returning exactly `0` for equal values is what
signals "these two are tied, don't reorder them relative to each other" to a stable sort
implementation — getting this return value wrong (or omitting the equal case) can silently make an
otherwise-correct sort behave unpredictably for tied values.

*Practice question:* if a comparator mistakenly always returned `1` (never `0` or `-1`) regardless
of the actual values being compared, what would happen to a stable sort's output?

routes, each a bare stub. Phase 9 is where a genuine, reusable component *system* gets built, and
where three of its planned decisions collided with real constraints (a build error, a token
collision, a design tradeoff) that had to be resolved with something more specific than the
original plan:

Key invariants for this phase:

1. **Any page that builds a `DataTable` `columns` prop must be a Client Component.** Discovered as
   a real build failure, not a design choice made in advance (Part 3).
2. **`StatusBadge` never infers tone from the label string.** The identical string `"High"` means
   two different things depending on which contract field produced it, and callers must always pass
   an explicitly-computed tone (Part 4).
3. **`components.json`'s `baseColor: "slate"` must never be exercised.** No primitive in this
   project was ever generated by `npx shadcn add` (Part 1).

## Part 1 — Primitives rewritten against project tokens

Tutorial 09 Part 5 covered the *prediction*: shadcn/ui's own token vocabulary (`--primary`,
`--card`, `--destructive`, and critically its own `--accent`, meaning a subtle gray hover) collides
directly with this project's real tokens (`--accent` meaning the brand blue), and Phase 8 deferred
resolving that collision by shipping zero primitives. Phase 9 is where that prediction was actually
tested — `Button`, `Card`, `Badge`, and `Table` were needed for the first time, and
[`docs/plan/phase-9-reusable-ui-components-static-pages/explanation.md`](../../plan/phase-9-reusable-ui-components-static-pages/explanation.md) §3 records the resolution:
hand-write all four, reading shadcn's public reference implementations only for API *shape* (the
`cva`-for-variants convention, `forwardRef` on `Button`), then rewrite every class list against
`ui-tokens.md`'s actual names.

Open [`components/ui/Button.tsx`](../../../components/ui/Button.tsx) in full:

```tsx
export const buttonVariants = cva(
  "inline-flex items-center justify-center gap-2 rounded-md px-4 py-2 text-sm font-medium transition-colors focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-accent disabled:pointer-events-none disabled:opacity-50",
  {
    variants: {
      variant: {
        primary: "bg-accent text-text-on-accent hover:bg-accent-hover",
        secondary: "border border-border bg-surface text-text-primary hover:bg-surface-muted",
        dark: "bg-surface-inverse text-text-on-inverse hover:bg-surface-inverse-hover disabled:bg-surface-muted disabled:text-text-muted",
      },
    },
    defaultVariants: {
      variant: "primary",
    },
  },
);
```

Every class in every variant traces to a real `ui-tokens.md` name — `bg-accent`,
`text-text-on-accent`, `border-border`, `bg-surface-inverse` — never a shadcn token name like
`bg-primary` or `text-primary-foreground`. This is the `cva` API shape shadcn popularized (a
`variants` object mapping a prop name to named class strings, plus `defaultVariants`), reused
because it's a genuinely good pattern independent of shadcn, with every string inside it rewritten.

> **Design patterns — Variant composition via a declarative style function:** `cva`'s whole job is
> turning "pick classes based on a prop value" from scattered conditional logic into one
> declarative table, read top to bottom. This is the same shape as a strategy pattern implemented
> as a lookup table instead of a class hierarchy: `variant: "dark"` doesn't trigger a branch of
> `if`/`else` logic buried in the component's render body — it's a key into a pre-declared mapping,
> checked and resolved once. `Badge.tsx`'s `tone`-keyed `badgeVariants` and `DOT_TONE_CLASSES`
> follow the identical shape.

**Checkpoint:** `components.json` still has `baseColor: "slate"` even though `npx shadcn add` is
never supposed to run. Is leaving that config in place a trap for a future session that forgets
this project's rule, or is it harmless because nothing reads it unless the CLI is invoked? Would you
delete it, or leave it with a comment?

<details>
<summary>Reveal answer</summary>

It's genuinely inert until the CLI is invoked — `components.json` is read only by the `shadcn`
command-line tool itself, so its mere presence changes nothing about how the app builds, runs, or
renders. But "harmless today" and "not a trap for a future session" are different claims: a future
contributor (human or agent) who runs `npx shadcn add select` because they need a new primitive and
don't already know this project's rule would get exactly the token-collision Part 1's opening
predicted, with no in-the-moment warning from the tool itself. `explanation.md` §3 records the
actual resolution: leave it in place, because a future primitive complex enough to genuinely need
generation speed might still use it deliberately — but that argues strongly for a comment directly
in `components.json` (or a very visible note in `CLAUDE.md`/`context/library-docs.md`, which
already state the rule) rather than relying on institutional memory alone.
</details>

**Checkpoint:** The verification pass used `git diff app/globals.css` returning empty as proof the
collision never happened. What's a scenario where that check would pass even if a real problem had
been introduced — i.e., what does an empty `globals.css` diff *not* tell you?

<details>
<summary>Reveal answer</summary>

An empty `globals.css` diff proves specifically that nothing injected new CSS variables into that
one file — it says nothing about a component elsewhere referencing a shadcn token class
(`bg-primary`, `text-destructive`) that was never defined anywhere, which would compile fine (an
undefined Tailwind color class simply generates no matching CSS rule) and render as invisible or
unstyled, a much harder bug to notice than a loud build failure. It also wouldn't catch a
*different* file — say, a new `tailwind.config.ts` entry — introducing a colliding token definition
without touching `globals.css` at all. The check is real evidence for the specific claim it tests
(no shadcn CLI run touched this one file), not a general guarantee that no token collision exists
anywhere in the codebase.
</details>

**Checkpoint:** If Phase 10 needs a genuinely complex primitive — say, a `Select` with keyboard
navigation and portal-rendered options — does hand-writing still make sense, or is that the point
where the shadcn-CLI-plus-manual-token-cleanup tradeoff flips in the other direction?

<details>
<summary>Reveal answer</summary>

The four primitives this phase hand-wrote (`Button`, `Card`, `Badge`, `Table`) are all small enough
that the "read shadcn's shape, rewrite the tokens" cost was trivial — none of them involve real
interaction logic beyond a click, so there was nothing complex to preserve. A `Select` with keyboard
navigation (arrow keys, typeahead, focus management) and portal rendering (rendering its dropdown
outside the normal DOM tree to escape overflow/z-index constraints) is a different order of
complexity — that behavior is exactly what a library like shadcn (or Radix underneath it) exists to
get right once, correctly, including edge cases a from-scratch rewrite would likely miss or
under-test. At that complexity level, generating via the CLI and then manually reconciling only the
*token classes* (leaving the interaction logic untouched) plausibly costs less than reimplementing
the interaction behavior by hand from a token-cleaned copy of the reference source — the tradeoff
this checkpoint asks about does shift, though this project hasn't yet needed to cross that line.
</details>

**Try it yourself:** Run `grep -rn "bg-primary\|text-primary-foreground\|bg-destructive\|--card\b" components/ app/` from the repo root. Confirm zero matches — direct, current proof that no shadcn-token-vocabulary class has ever been introduced anywhere in this codebase's real component files.

## Part 2 — Component ownership by responsibility

Open [`context/ui-registry.md`](../../../context/ui-registry.md) and note the folder structure
`plan.md`'s own "What was built" section documents: `components/ui/` (primitives — `Button`,
`Card`, `Badge`, `Table`), `components/layout/` (`AppShell`, `SidebarNav`, `TopHeader`),
`components/feedback/` (`EmptyState`, `LoadingState`, `BusinessErrorMessage`),
`components/workflow/` (`StatusBadge`, `MetricCard`, `ReportCard`, `UploadPanel`,
`WorkflowStepper`), and `components/tables/` (`DataTable`).

This is organization by *what a component is responsible for*, not by which page happens to use
it — `StatusBadge` isn't filed under `order-validation/` even though that's the first page that
needs it, because `payment-aging` and `inventory-allocation` need it too. `EmptyState` isn't
filed under `tables/` even though `DataTable` is its only current caller, because it's a general
feedback primitive that any future component could reuse for an empty state that isn't a table.
Each of the five folders answers a different question about a component: is it a raw visual
primitive with no business meaning (`ui/`)? Does it structure the page chrome around content
(`layout/`)? Does it communicate a state about an operation (`feedback/`)? Does it encode
something specific to this project's actual workflows (`workflow/`)? Is it a generic data-display
mechanism (`tables/`)?

Open [`components/feedback/EmptyState.tsx`](../../../components/feedback/EmptyState.tsx) and
[`components/feedback/BusinessErrorMessage.tsx`](../../../components/feedback/BusinessErrorMessage.tsx)
in full — both are short enough to read completely:

```tsx
// Types
type EmptyStateProps = {
  title: string;
  description?: string;
};

// Component
export function EmptyState({ title, description }: EmptyStateProps) {
  return (
    <div className="flex flex-col items-center justify-center gap-2 rounded-xl border border-dashed border-border bg-surface-subtle p-10 text-center">
      <p className="text-sm font-medium text-text-primary">{title}</p>
      {description ? <p className="text-sm text-text-secondary">{description}</p> : null}
    </div>
  );
}
```

```tsx
// Types
type BusinessErrorMessageProps = {
  message: string;
};

// Component
/** Always render business-readable copy here, never a raw exception (see ui-rules.md Error Messages). */
export function BusinessErrorMessage({ message }: BusinessErrorMessageProps) {
  return (
    <div role="alert" className="rounded-xl border border-border bg-danger-subtle p-4 text-sm text-danger">
      {message}
    </div>
  );
}
```

Neither file carries a `"use client"` directive, no state, no event handler — both just render
whatever string props they're given. That makes each one **directive-free and server-compatible**:
nothing in either component *requires* the client bundle. It does not, on its own, make either
component "a Server Component" as a fixed, permanent label — per L5.3
(`docs/teach/lessons/0033-server-and-client-components.html`), whether a directive-free module
actually ends up in the server tree or the client module graph depends entirely on where it's
imported *from*. Checking that for real: `EmptyState` is imported by `components/tables/DataTable.tsx`
(which carries its own `"use client"`, Part 3) and directly by all three Client workflow pages; today,
every one of its real call sites is already inside the client graph. `BusinessErrorMessage` is
imported by `components/dashboard/DashboardLiveSections.tsx` (also `"use client"`) and by the same
three workflow pages — same story. Neither component is wrong to write this way — a directive-free,
server-compatible component genuinely *could* be rendered from a Server parent elsewhere, and
staying directive-free keeps that option open — but "Server Component" isn't a property of the file
in isolation; it's a property of where the module graph actually places it, and today that's
entirely client-side for both. `BusinessErrorMessage`'s own docstring ties it directly back to
`ui-rules.md`'s "Error Messages" section (the bad/good `KeyError` example every earlier Python
tutorial in this series has referenced) — this component is the frontend half of that same
discipline: a caller must already have translated a technical failure into a business-readable
sentence before this component ever sees it; the component itself performs no translation.

**Try it yourself:** Open `context/ui-registry.md` and count how many distinct component entries
exist under each of the five folder headings. Confirm the count under `workflow/` (5) is the
largest — a direct reflection of this project's own domain having the most genuinely
project-specific display logic, versus the more generic, reusable-anywhere primitives in `ui/` and
`feedback/`.

## Part 3 — The Server-to-Client serialization failure

[`docs/plan/phase-9-reusable-ui-components-static-pages/explanation.md`](../../plan/phase-9-reusable-ui-components-static-pages/explanation.md) §1 opens with something
worth reading as it actually happened, not as a tidy design decision made in advance: the
`/architect` plan for this phase expected `DataTable` to be the *only* Client Component among the
new interactive pieces — every page using it was expected to stay a Server Component by default,
per L5.3's "Server Components unless you need state/events/browser APIs" rule. That assumption
broke the first time a real page tried to use `DataTable`:

```text
Error: Functions cannot be passed directly to Client Components unless you explicitly
expose it by marking it with "use server".
```

Open [`components/tables/DataTable.tsx`](../../../components/tables/DataTable.tsx) lines 19–29:

```tsx
export type DataTableColumn<T> = {
  key: string;
  header: string;
  render: (row: T) => ReactNode;
  sortable?: boolean;
  sortValue?: (row: T) => string | number;
  /** Allows this column's cell to wrap — reserve for long free text (error messages, suggested actions). Every other column stays single-line. */
  wrap?: boolean;
  /** Right-aligns header + cell — use for numeric/quantity/amount columns. */
  align?: "left" | "right";
};
```

`render` and `sortValue` are both function types. Per L5.3, props crossing from a Server Component
into a Client Component must be serializable — plain data React can turn into the RSC payload and
ship to the browser — and a JavaScript closure is definitionally not serializable; there is no
representation of "here is a function's actual executable code" that can cross that specific
boundary the way a string or number can. Every page that builds a `DataTableColumn<T>[]` array
(`order-validation`, `inventory-allocation`, `payment-aging`) was, at the moment of the error, still
a plain Server Component function calling `<DataTable columns={ERROR_COLUMNS} ... />` — exactly the
forbidden crossing, since `ERROR_COLUMNS` itself carries `render` functions.

The fix actually applied: mark the three workflow pages themselves `"use client"` — not just
`DataTable`. Open
[`app/(workspace)/order-validation/page.tsx`](../../../app/(workspace)/order-validation/page.tsx)
line 1 and lines 46–77:

```tsx
"use client";
```

```tsx
const ERROR_COLUMNS: DataTableColumn<ValidationErrorRow>[] = [
  { key: "row_number", header: "Row", render: (r) => r.row_number, sortable: true, sortValue: (r) => r.row_number, align: "right" },
  {
    key: "order_id",
    header: "Order ID",
    render: (r) => <span className="font-medium text-text-primary">{r.order_id ?? "—"}</span>,
    sortable: true,
    sortValue: (r) => r.order_id ?? "",
  },
  {
    key: "sku",
    header: "SKU",
    render: (r) => <span className="font-medium text-text-primary">{r.sku ?? "—"}</span>,
    sortable: true,
    sortValue: (r) => r.sku ?? "",
  },
  { key: "error_code", header: "Error Code", render: (r) => <span className="text-text-secondary">{r.error_code}</span> },
  {
    key: "error_message",
    header: "Error Message",
    render: (r) => (
      <span className="block max-w-[280px] truncate" title={r.error_message}>
        {r.error_message}
      </span>
    ),
  },
  {
    key: "severity",
```

Once `order-validation/page.tsx` is itself a Client Component, passing a function-carrying object
to another Client Component (`DataTable`) is unremarkable — L5.3's restriction only applies at the
Server→Client boundary, never Client→Client, since both sides of a Client-to-Client prop pass
already run in the same browser environment with no serialization step involved at all.
`app/(workspace)/dashboard/page.tsx` and `app/(workspace)/reports/page.tsx` never construct a

tracks *what a page actually needs*, not a blanket policy either way.

**Checkpoint:** Walk through exactly what happens, step by step, when Next.js tries to serialize
`ERROR_COLUMNS` (with its `render` functions) to send from a Server Component to `DataTable`. When,
exactly, does that error actually surface — and is it always at `next build` time?

<details>
<summary>Reveal answer</summary>

React's server-rendering machinery, while rendering the Server Component tree and encoding the
resulting RSC payload, walks every prop being passed from a Server Component into a Client
Component and checks whether each value is a serializable type — strings, numbers, booleans, plain
objects/arrays of those, and a small set of other supported types (a render callback or a closure
captured for later use is *not* one of them; this is a different case from a Server Function, which
React ships as a special opaque reference rather than trying to serialize the function's own source
code). The moment it reaches a function value inside `ERROR_COLUMNS` (a `render` closure), it has no
valid serialization for that value and raises immediately, rather than silently dropping it or
passing `undefined`. *When* that render-and-encode step actually runs depends on the route: for a
statically-prerenderable route, Next.js renders it ahead of time as part of `next build`, so the
failure surfaces as a build error. For a route that isn't statically prerendered (a dynamic route,
or any page in `next dev`), the same render-and-encode step happens per-request instead — so the
identical error would surface the first time the route is actually requested, not during `next
build` at all. The three original workflow pages this bug hit were building a statically-prerendered
page at the time, which is why the error surfaced at build time in this project's specific case —
that's a fact about *this route's* rendering mode, not a universal rule that the error can only ever
happen during a build.
</details>

## Part 4 — Status meaning before status color

Open [`components/workflow/StatusBadge.tsx`](../../../components/workflow/StatusBadge.tsx) in
full:

```tsx
// Internal imports
import { Badge } from "@/components/ui/Badge";

// Types
export type Tone = "success" | "warning" | "danger" | "info" | "neutral";

type StatusBadgeProps = {
  status: string;
  tone: Tone;
};

/**
 * Thin display wrapper around Badge. Callers pick the tone using one of the
 * domain helpers below rather than StatusBadge inferring it from the raw
 * string — the same label text ("High") maps to a different tone depending
 * on which contract field it came from (see the StatusBadge mapping table in
 * docs/plan/phase-9-reusable-ui-components-static-pages/plan.md).
 */
export function StatusBadge({ status, tone }: StatusBadgeProps) {
  return <Badge tone={tone}>{status}</Badge>;
}

// --- Domain tone mappings ---

export function severityTone(severity: "Error" | "Warning"): Tone {
  return severity === "Error" ? "danger" : "warning";
}

export function allocationStatusTone(
  status: "Fully Allocated" | "Partially Allocated" | "Backordered",
): Tone {
  switch (status) {
    case "Fully Allocated":
      return "success";
    case "Partially Allocated":
      return "warning";
    case "Backordered":
      return "danger";
  }
}

/** Order/allocation `priority` is an importance ranking, not a problem — High just draws attention. */
export function importancePriorityTone(priority: "High" | "Normal" | "Low"): Tone {
  return priority === "High" ? "warning" : "neutral";
}

export function agingBucketTone(
  bucket: "Current" | "1-30 Days" | "31-60 Days" | "61-90 Days" | "90+ Days",
): Tone {
  switch (bucket) {
    case "Current":
      return "success";
    case "1-30 Days":
      return "info";
    case "31-60 Days":
    case "61-90 Days":
      return "warning";
    case "90+ Days":
      return "danger";
  }
}

/** Payment `follow_up_priority` is an urgency-of-problem ranking — High is the worst case. */
export function followUpPriorityTone(priority: "High" | "Medium" | "Low" | "Watch" | "None"): Tone {
  switch (priority) {
    case "High":
      return "danger";
    case "Medium":
      return "warning";
    case "Low":
      return "info";
    case "Watch":
    case "None":
      return "neutral";
  }
}

export type ReportLifecycleState = "Needs Input" | "Not Generated" | "Processing" | "Ready";

export function reportLifecycleTone(state: ReportLifecycleState): Tone {
  switch (state) {
    case "Ready":
      return "success";
    case "Processing":
      return "info";
    case "Not Generated":
    case "Needs Input":
      return "neutral";
  }
}
```

The real file has two more domain helpers this Part's own prose doesn't discuss in detail
(`severityTone`, `allocationStatusTone`) plus `agingBucketTone` and `reportLifecycleTone` — genuinely
part of "in full," just not each one narrated individually below, since `importancePriorityTone` and
`followUpPriorityTone` are this Part's chosen running example for the two-meanings-of-`"High"`
point.

`StatusBadge` itself is a deliberately dumb wrapper — two props, `status` (the label to render) and
`tone` (already resolved), nothing else. `explanation.md` §2 names the design that was rejected:
a single lookup function inside `StatusBadge` mapping any status string directly to a tone, which
would have to guess (or silently pick one, wrong half the time) whenever the same literal string
appears in more than one contract with more than one meaning — exactly `"High"`'s situation here.
`ValidOrderRow.priority`/`AllocationResultRow.priority` is an *importance* ranking (a high-priority
order just deserves attention, it isn't itself a problem) — `importancePriorityTone` maps it to
`warning` at most, never `danger`. `PaymentAgingRow.follow_up_priority` is an *urgency-of-problem*
ranking (a high-priority follow-up means someone is dangerously overdue) — `followUpPriorityTone`
maps its `"High"` to `danger`, the most severe tone the system has.

> **Type theory — Exhaustiveness checking over a literal union:** `followUpPriorityTone`'s `switch`
> handles all five members of `"High" | "Medium" | "Low" | "Watch" | "None"` with no `default`
> clause, and TypeScript's control-flow analysis confirms every case returns a `Tone` — if a sixth
> value were ever added to `PaymentAgingRow.follow_up_priority`'s type, this `switch` would stop
> compiling until a new `case` handled it. `importancePriorityTone` gets the identical safety
> without a `switch` at all, since a boolean ternary over a two-outcome check (`"High"` or
> anything else within the three-member union) is already exhaustive by construction.

Because each helper's parameter type is the exact literal union of one specific contract field —
`importancePriorityTone(priority: "High" | "Normal" | "Low")`,
`followUpPriorityTone(priority: "High" | "Medium" | "Low" | "Watch" | "None")` — TypeScript itself
prevents calling the wrong helper for the wrong field at a call site. Open
[`app/(workspace)/order-validation/page.tsx`](../../../app/(workspace)/order-validation/page.tsx)
line 140 (inside `VALID_ORDER_COLUMNS`):

```tsx
render: (r) => <StatusBadge status={r.priority} tone={importancePriorityTone(r.priority)} />,
```

Passing `r.priority` (typed `"High" | "Normal" | "Low"`) into `followUpPriorityTone` instead — which
expects the five-member union — would be a TypeScript compile error, not a silent runtime mismatch.

**Checkpoint:** `importancePriorityTone` and `followUpPriorityTone` both take a `"High"` value but
return different tones (`warning` vs. `danger`). What would a bug look like if a future developer
accidentally called the wrong helper for the wrong field — and would TypeScript catch it, or only a
visual review?

<details>
<summary>Reveal answer</summary>

It depends on exactly what gets passed, and this is worth being precise about rather than
overclaiming. `importancePriorityTone`'s parameter type is `"High" | "Normal" | "Low"`;
`followUpPriorityTone`'s is `"High" | "Medium" | "Low" | "Watch" | "None"` — these two unions
actually *do* overlap, sharing the literal members `"High"` and `"Low"`. Passing a whole,
already-typed field value from the wrong contract — `r.priority` (typed as the three-member union)
into `followUpPriorityTone`, or `r.follow_up_priority` (the five-member union) into
`importancePriorityTone` — genuinely is a compile error, since neither whole union is assignable to
the other (`"Normal"` isn't a member of the five-member union, and `"Medium"`/`"Watch"`/`"None"`
aren't members of the three-member union). But a bare, narrowed literal is a different case entirely:
a standalone `"High"` value (a literal type with just that one member, which is what TypeScript
infers for something like a hardcoded string, a destructured single-case branch, or any expression
narrowed down to that one string) is a valid subtype of *both* unions, since `"High"` is a member of
each — TypeScript would happily compile `followUpPriorityTone("High")` even though that value, in
context, might genuinely be an order's `priority`, not a payment's `follow_up_priority`.

The current union shapes catch many accidental *whole-field* swaps — passing `r.priority` where
`r.follow_up_priority` is expected fails loudly, today, with these five real fields. What they
cannot catch is a narrowed literal crossing between the two meanings, because TypeScript's
structural type checking verifies shape compatibility, not semantic provenance — it has no concept
of "this `"High"` came from an importance ranking" versus "this `"High"` came from an urgency
ranking." That distinction is preserved today only by explicit helper names and call-site review,
not by the type system. If this project ever needed stronger protection than that, a *branded* type
(a nominal wrapper distinguishing "importance priority" from "follow-up priority" at the type level,
even though both are structurally just strings) could provide it — this project hasn't needed that
extra machinery so far.
</details>

**Checkpoint:** Why does `StatusBadge` not just accept the raw contract field name (e.g.
`"severity"`) and a value, then do the field→tone lookup internally? What specific ambiguity in
this project's contracts makes that design worse than the explicit-helper-per-caller approach that
was chosen?

<details>
<summary>Reveal answer</summary>

A field-name-based internal lookup would need `StatusBadge` itself to maintain a giant, growing
mapping from every possible field name across every contract to its tone logic — coupling one
small, generic display component to the full, ever-changing shape of every business contract in the
project, and reintroducing exactly the "High means one thing" ambiguity this Part opens with:
`priority` alone isn't even a unique field name (`ValidOrderRow.priority` and
`PaymentAgingRow.follow_up_priority` are differently-named, but a hypothetical internal-lookup
design keyed just on "priority-like" would still have to disambiguate which contract it came from).
The chosen design keeps `StatusBadge` itself contract-agnostic and dumb, and pushes the
one place that genuinely needs to know "which field, therefore which meaning" to the call site,
where that context is already unambiguous (the code is already inside `VALID_ORDER_COLUMNS`, so it
already knows it's rendering `ValidOrderRow.priority`, not any other field).
</details>

**Checkpoint:** `RemainingInventoryRow.reorder_alert === "No"` renders plain text instead of a
badge. Is "no badge for the non-alerting case" a good general rule for this design system, or does
it just happen to work here because there's no interesting "OK" status to call out? Would Order

<details>
<summary>Reveal answer</summary>

The rule generalizes reasonably well but isn't universal — its real justification is that a badge
is a visual interrupt meant to draw the eye to something worth acting on, and "nothing to report"
is, by definition, not that. It works cleanly for `reorder_alert` because `"No"` genuinely has zero
actionable content beyond "this row is fine." It would *not* transfer cleanly to a hypothetical
"no errors" row-level indicator for Order Validation, because `ValidationErrorRow` only exists for
rows that already have at least one error — there's no "this row has severity: none" state in that
contract to begin with (a fully valid order row doesn't appear in the Validation Errors table at
all; it appears in the separate Valid Orders table). The rule is really "don't badge a state that
carries no actionable information," and whether that applies depends on whether such a state
actually exists in the contract being displayed — it does for `reorder_alert`, it structurally
doesn't for `ValidationErrorRow.severity`.
</details>

**Checkpoint:** The mapping table lives in `context/ui-registry.md`, not in code comments or a

<details>
<summary>Reveal answer</summary>

The risk is exactly the same shape as every other documentation-drift case this series has

</details>

## Part 7 — Generic tables with deliberately small behavior

Open [`components/tables/DataTable.tsx`](../../../components/tables/DataTable.tsx) lines 63–78:

```tsx
const [sort, setSort] = useState<SortState>(null);

const sortedData = useMemo(() => {
  if (!sort) return data;
  const column = columns.find((c) => c.key === sort.key);
  if (!column?.sortValue) return data;
  const sortValue = column.sortValue;
  const sorted = [...data].sort((a, b) => {
    const va = sortValue(a);
    const vb = sortValue(b);
    if (va < vb) return -1;
    if (va > vb) return 1;
    return 0;
  });
  return sort.direction === "asc" ? sorted : sorted.reverse();
}, [data, sort, columns]);
```

`explanation.md` §5 names the intended shape this column-def structure was built toward: something
close to what TanStack Table would eventually need, specifically so the *page-level* column
definitions (the part every page author already has to hand-write) wouldn't need rewriting if
Phase 10's real API data ever justified adopting that library — only `DataTable`'s own internals
would change. Sorting is deliberately minimal: one `SortState` in local `useState`, a plain
three-way comparator, no multi-column sort. `explanation.md` names why directly: the mock fixtures
were, at the time, 1–2 rows per table, so there was nothing for more elaborate sort/filter/
pagination behavior to actually demonstrate — building it against data that couldn't exercise it
would mean writing and "testing" behavior with no real way to verify it worked.

> **Algorithms — Comparator-based stable sort:** `[...data].sort((a, b) => ...)` copies the array
> first (`[...data]`), then sorts the copy in place using a three-way comparator — negative means
> "`a` before `b`," positive means the reverse, zero means "leave their relative order alone."
> Modern JavaScript engines guarantee `Array.prototype.sort` is *stable*: two elements whose
> comparator returns `0` for each other keep their original relative order in the result, which is
> exactly why the equal-values branch (`return 0`, not merely omitted) matters — the pre-study's
> Concept 5 covered why getting this specific case right is what makes "stable" a meaningful
> guarantee rather than an accident.

## Part 8 — Display formatting without changing business meaning

Open [`lib/formatters.ts`](../../../lib/formatters.ts) lines 18–26 in full:

```ts
/** Formats a "YYYY-MM-DD" contract date without shifting across timezones. */
export function formatDate(isoDate: string): string {
  const [year, month, day] = isoDate.split("-").map(Number);
  return new Date(year, month - 1, day).toLocaleDateString("en-US", {
    year: "numeric",
    month: "short",
    day: "numeric",
  });
}
```

`explanation.md` §7 documents a subtlety worth taking seriously precisely because it's invisible
until someone in the wrong timezone loads the page: contract dates like `ValidOrderRow.order_date`
are plain `"YYYY-MM-DD"` strings, with no time or timezone component. Per the ECMAScript `Date`
parsing spec, a date-only ISO string is parsed as UTC midnight — so `new Date("2026-03-31")`,
displayed via `.toLocaleDateString()` in any timezone west of UTC (US Pacific, for instance), would
render as March 30, an off-by-one error that a demo running in UTC (or reviewed by someone in a
UTC-adjacent timezone) would never surface. `formatDate` avoids the whole trap by never constructing
a `Date` from the raw ISO string at all — it splits `"2026-03-31"` into `[2026, 3, 31]` by hand and
calls `new Date(year, month - 1, day)`, the local-time constructor overload, which has no UTC-
conversion step to go wrong.

This is display formatting, not a business decision — nothing about *when* an order was actually
placed changes; only how that already-correct date is rendered to a reader changes, the same
presentation-layer discipline Track 3's lessons established for the Python/Excel side (a report
never recalculates, it only formats). `lib/mock-data.ts`'s `ninetyPlusDaysAmount()` — Tutorial 08's
"90+ Days Amount" derived aggregate, finally implemented as real code — follows the identical
discipline in a different direction: it introduces a new *computed number*, but only by summing
fields that already exist, following the exact source-rows/source-fields/rule this tutorial series
has already fully specified, never inventing a new business interpretation.

**Try it yourself:** Open a Node REPL (or a scratch `.ts` file) and run
`new Date("2026-03-31").toLocaleDateString("en-US", { timeZone: "America/Los_Angeles" })`.
Confirm it prints March 30, not March 31 — direct, hands-on proof of the exact off-by-one bug
`formatDate`'s manual parsing was written to avoid.

## Full data flow: one validation error row, from typed mock data to a rendered badge

1. **Typed mock data is imported.** `lib/mock-data.ts`'s `orderValidationResult` (Tutorial 09 Part
   7) — already asserted as `OrderValidationResult`, its `errors` array already typed as
   `ValidationErrorRow[]`.
2. **A page-level column definition names the field.**
   `app/(workspace)/order-validation/page.tsx`'s `ERROR_COLUMNS` (Part 3): the `severity` column's `render`
   function reads `r.severity` and builds `<StatusBadge status={r.severity}
   tone={severityTone(r.severity)} />`.
3. **The page, already a Client Component, passes both the column array and the data into
   `DataTable`.** No Server→Client boundary crossing happens here — `order-validation/page.tsx`
   carries its own `"use client"` directive, so `ERROR_COLUMNS`'s `render` functions never need to
   be serialized at all (Part 3).
4. **`DataTable` calls `column.render(row)` for each cell**, per
   `components/tables/DataTable.tsx`'s render loop — this is where `StatusBadge` actually gets
   instantiated, once per row, inside `DataTable`'s own JSX.
5. **`StatusBadge` delegates to `Badge` with an already-resolved tone.** `severityTone("Error")`
   (Part 4) returns `"danger"`; `StatusBadge` passes `status="Error"` and `tone="danger"` straight
   through to `<Badge tone={tone}>{status}</Badge>`.
6. **`Badge` resolves the tone to real semantic-token classes.**
   `components/ui/Badge.tsx`'s `badgeVariants({ tone: "danger" })` resolves to
   `"bg-danger-subtle text-danger"` — the exact CSS custom properties Tutorial 09 Part 4 traced
   from `app/globals.css`'s `:root` block through `tailwind.config.ts`.

Every step after step 3 stays entirely within the Client boundary — the only place this trace
*could* have crossed a Server→Client line (passing `ERROR_COLUMNS` from a Server-Component page
into `DataTable`) is exactly the crossing Part 3 covers as the reason that page is `"use client"`
in the first place. If `order-validation/page.tsx` were somehow still a Server Component, step 3
would fail at build time with the exact error Part 3 quotes.
