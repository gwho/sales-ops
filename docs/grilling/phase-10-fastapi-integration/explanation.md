# Explanation: Why These Decisions, Not Others

## Statelessness is a property of the whole system, not a slogan

It's easy to say "the API is stateless" and still accidentally build something stateful. The tell in this session was concrete: `GET /api/reports/{report_id}` *looks* like a normal REST endpoint, but walk through what has to be true for it to work. The client calls `POST` something to generate a report, gets an ID back, then later calls `GET /api/reports/{id}` to fetch it. Between those two calls, *something* has to remember what that ID points to — either the actual bytes, or enough state to regenerate them on demand. That "something" is a job store or a file store, full stop. There's no version of that endpoint shape that doesn't require server-side memory of a specific past request.

Compare that to `POST /api/orders/validate/report`: the client sends everything needed to produce the artifact in the same request that asks for it. No step requires the server to remember anything about a *previous* request. That's the actual test for "is this stateless," not whether the word "state" appears anywhere in the code.

The giveaway that this project had drifted toward the two-phase shape without meaning to: `ReportManifest.report_id` was already a deterministic string built from `report_type` and a timestamp — not a randomly generated key you'd use to look something up in a table. Nothing in `src/report_export.py` had ever been built to support fetching by that ID; the field existed for display purposes only. Reading the actual code before accepting the documented endpoint list is what surfaced this — the endpoint list in `build-plan.md` was aspirational, written before anyone had reasoned about the return-typing/persistence mechanics.

## Trusting the client is a security decision, not a convenience trade-off

The report-recompute-vs-trust-client-JSON question looks like a performance question ("why upload the file twice?") but it's really a trust-boundary question. If the report endpoint accepted `{ "summary": {...}, "valid_orders": [...] }` as input and just formatted it into a workbook, then *any* client — not just the real frontend — could send hand-edited numbers and get back a workbook that looks exactly like an official export. There's nothing in that flow that verifies the numbers came from actually running the business rules. Recomputing from the raw source file closes that gap completely: the only way to influence the report is to change the input data, which is the same thing you'd have to do to influence the on-screen result. The server never has to ask "is this JSON I'm about to format actually trustworthy" because it never accepts pre-computed JSON as input in the first place.

This is a specific instance of a general principle: **a system should not accept as authoritative anything it could not have produced itself.** The cost — a duplicate file upload — is cheap here because the files are small and the workflows are demo-scale. In a system processing gigabyte uploads or expensive multi-minute computations, this same principle might point toward a different mechanism (e.g., signed, short-lived tokens that reference a server-side cache of the *validated* result, not raw client JSON) — but the principle itself doesn't change.

## Why `as_of_date` required-and-editable beats optional-with-server-default

`calculate_payment_aging(invoices_df, as_of_date=None)` already has a perfectly good default (`date.today()`). The lazy choice would be to let the live UI omit the field and lean on that default. Two things argued against it. First, a hidden default is a hidden dependency on server clock/timezone — if the demo is ever deployed somewhere with a different local time than the developer's machine, "today" silently means something different than what the developer tested. Second, and more important for a *portfolio* project specifically: the disabled date input was pure decoration. Nothing about the Phase 9 UI let a viewer actually *do* anything with payment aging beyond look at one fixed snapshot. Making the date required and editable turns a static table into something a reviewer can interact with — pick an earlier date, watch buckets shift, watch a "None" priority invoice become "Watch" become "High." That's a more convincing demonstration that the business logic is real and reactive, not baked into fixture data.

## FastAPI's default error shape breaks a contract you didn't know you'd made

The session had already agreed on `{"detail": "string"}` as the uniform error shape before the `fastapi` skill got pulled in. What wasn't obvious yet: FastAPI's *own* validation machinery — the thing that fires automatically when a required `File()` or `Form()` field is missing — returns `detail` as a *list* of structured error objects, not a string. If the frontend's error-handling code assumed `error.detail` was always a string (which it needs to, to hand straight to `<BusinessErrorMessage message={...} />`), a missing-file request would crash that assumption instead of showing a clean error. The fix is a `RequestValidationError` exception handler that intercepts FastAPI's own validation failures and rewrites them into the same single-string shape used everywhere else. The lesson generalizes: a "handle every error the same way" contract has to account for errors the framework raises on your behalf, not just the ones your own code raises.

The sync-vs-async question resolved the same session is related but distinct: FastAPI runs a `def` (sync) route handler in a worker thread pool automatically, so a blocking call like `pd.read_excel` doesn't block the single event loop that's also trying to serve every other concurrent request. Marking that same handler `async def` without actually awaiting anything async inside it would remove that automatic thread-pool offloading — the blocking pandas call would then run directly on the event loop, stalling every other in-flight request for its duration. "Async" is not automatically faster; it's only faster when the work inside is genuinely non-blocking.

## Why `ReportLifecycleState` didn't grow a fifth state

There were two axes being conflated: "how far along is generating this artifact" (`Needs Input` → `Not Generated` → `Processing` → `Ready`) and "did the most recent attempt to generate it fail." It would have been easy to just add `Error` as a fifth value and call it done. The reason not to: `BusinessErrorMessage` already exists as exactly the right component for "something failed, here's a business-readable reason" — reused identically for JSON workflow failures, report failures, and anything else that can fail. If `ReportLifecycleState` also grew an `Error` branch, you'd end up with two different code paths rendering essentially the same kind of failure message, and a future contributor would have to guess which one applies where. Keeping the lifecycle type narrowly about *artifact readiness*, and treating "did this specific request fail" as a completely separate, page-level fork, keeps each piece answering exactly one question.

## A name should describe what a thing is, not what role you wish it played

"Sample Template" implied something it wasn't: a blank, clean starting point. The actual files are full realistic datasets that happen to contain the same intentional errors the business rules are built to catch. Renaming to "Sample File" isn't cosmetic — it's fixing a claim the old name was making that wasn't true. This matters more in a demo/portfolio context than it might elsewhere: the whole point of the demo is "watch realistic-looking data expose real validation logic," and a name suggesting "here's a clean template" undercuts that story before a viewer even opens the file.

## Why only one decision became an ADR

The three-part test — hard to reverse, surprising without context, result of a genuine trade-off — was applied deliberately to every decision, not just the ones that felt "big." Most decisions in this session failed at least one prong. The upload-validation rules, for instance, are easy to change later (nothing else depends on their exact shape) and aren't surprising (checking file extensions before parsing is the obvious thing to do) — so even though real trade-offs were discussed (extension-only vs. content-type enforcement, size caps), it didn't clear the bar. The statelessness decision cleared all three: reversing it later means introducing a job store that other endpoints would need to be redesigned around (hard to reverse); it directly contradicts the endpoint shape already written down in `build-plan.md` (surprising without the ADR's context); and there were genuine alternatives with real costs on both sides (trust/simplicity vs. revisitability). Writing an ADR for every decision in a planning session would dilute the ones that actually need a permanent "why" — the discipline is in saying no to most candidates, not in being thorough for its own sake.
