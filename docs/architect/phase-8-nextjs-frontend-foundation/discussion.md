# Discussion — Phase 8: Next.js Frontend Foundation (`/architect` session)

A teaching record of the full reasoning trail across this session's `/architect` planning for Phase 8. This includes the false starts, the paused session, and the concepts explored — not just the outcome.

## 1. Why the session paused entirely rather than working around a missing Figma connection

The Phase 8 kickoff arrived with an explicit instruction: "Use Figma MCP to inspect these references... If MCP cannot inspect them, stop and report the exact MCP error. Do not silently fall back to screenshots." This is a strong, specific instruction — not "try to use Figma if convenient" but "stop and report" on failure. When the actual link text turned out to contain a literal unfilled placeholder (`<paste link>`), the first move was not to guess a URL or silently drop the Figma step — it was to surface the gap directly via `AskUserQuestion`, because proceeding either way (skipping Figma silently, or fabricating a URL) would have been worse than a thirty-second clarifying question.

That question surfaced something unexpected: the user's answer wasn't a simple choice between two options. It was free text containing a complete, more detailed 10-item instruction block plus four real Figma URLs — as if the user had, in answering the clarifying question, gone back and actually finished composing the message they'd meant to send the first time. This is a pattern worth recognizing: `AskUserQuestion`'s free-text answer field can become the channel through which a user delivers the "real" version of an instruction, not just a yes/no. Treating that embedded text as authoritative — rather than only acting on the literal option label selected — was the right call, because the alternative (extract only "proceed without Figma" and discard the rest) would have thrown away information the user had clearly intended to convey.

With real URLs and MCP genuinely required this time, the next check was whether a Figma MCP connection existed at all. `claude mcp list` is the ground-truth command here — not memory, not assumption. It returned five configured servers (Gmail, Calendar, Drive, Zapier, PostHog), none of them Figma, and a check for `.mcp.json` in the repo confirmed no local override either. This is a materially different situation from "Figma is configured but needs auth" — it's "Figma was never set up in this environment at all." The distinction mattered for what to tell the user: not "please authenticate," but "please install and configure a server first," which is a bigger ask with more steps.

WebSearch confirmed the two legitimate paths — Figma's own remote MCP server (`mcp.figma.com`, OAuth-based) via either the official Claude Code plugin (`claude plugin install figma@claude-plugins-official`) or a manual `claude mcp add`. A detail worth noting: this environment already had the `claude-plugins-official` marketplace configured (visible via `claude plugin marketplace list`, showing it alongside an `openai-codex` marketplace and one other installed plugin, `posthog@claude-plugins-official`). That meant the plugin path required no new marketplace setup — just the install command itself. This was reported precisely, and then the session genuinely stopped: no further planning happened until the user came back reporting MCP was connected. This is the "pause" pattern in its purest form — not a soft pause where planning continues in the background, but a hard stop where the next message from the user is what resumes work.

## 2. Redoing the gate check properly instead of trusting the prior session's caveat

The previous session (Phase 7) had left an honest but weak claim in its memory and progress tracker: the Phase 8 gate had been "spot-checked... not a literal line-by-line traceability audit." That caveat existed for a reason — Phase 7 was documentation work under time pressure, and a full line-by-line audit wasn't the point of that session. But Phase 8's kickoff instructions explicitly asked to "Verify every spec §11/§12 test case line-by-line against the current pytest suite" — this session's job was specifically to close that gap, not to inherit and re-repeat the same caveat a second time.

The audit ran as three parallel background research agents rather than one sequential pass, because the three areas were genuinely independent: (1) mapping all 26 spec-listed test-case rows across three spec files to named test functions in three test files, (2) confirming Phase 6's report-structure test coverage against the specs plus verifying the fixture correctness, and (3) surveying the Phase 8 planning docs (build-plan, ui-contract-plan, ui-tokens, ui-rules, ui-registry, code-standards, library-docs) for anything relevant to a Next.js scaffold session. Running these as separate agents kept the main context clean of the intermediate exploration and let all three run concurrently rather than sequentially.

Agent 1's report was the load-bearing one: an explicit table for each spec, scenario-by-scenario, naming exactly which test function (with line number) covers each row, and calling out zero "NOT COVERED" items across 8+8+10 = 26 required cases. This is meaningfully stronger evidence than a count comparison ("spec lists 8 cases, test file has 23 tests, therefore probably fine") — it's a real 1:1 mapping. Agent 2 independently re-confirmed the Phase 6 report-structure gate and, crucially, verified that the `report_id` fixture fix applied earlier in this same session was actually correct against the real exporter's format string — not just present, but numerically correct for a specific fixed timestamp.

One small but real mistake happened here and is worth recording: an initial `uv run pytest -q 2>&1 | tail -20` run showed only dot-progress lines with no summary ("152 passed in X.XXs"), because `tail -20` happened to cut off exactly at the summary line in that particular terminal wrapping. This looked odd enough (no summary line at all is unusual for pytest) to prompt a second, more careful run — `uv run pytest > /tmp/pytest_out.txt 2>&1; echo "exit code: $?"` — which confirmed both the exit code (0) and the actual summary line. The lesson: when a verification command's output looks incomplete or unexpected, re-run it in a way that can't be truncated, rather than accepting an ambiguous result as confirmation.

## 3. Reading the real Figma Make source, and what it revealed

Once MCP was connected, the first real technical question was how to actually inspect a Figma **Make** file (an AI-prototype product) as opposed to a normal Figma design file. The `get_design_context` tool's own description spells this out: for Make URLs specifically, assume `nodeId="0:1"` — a special case the tool documents directly, distinct from the normal flow of extracting a `node-id` from the URL's query string. `get_metadata` and `get_screenshot`, by contrast, explicitly state they are *not* supported for Make files at all. This meant the "try the Make-context path first" instruction had exactly one real interpretation: `get_design_context` with the fixed `nodeId`, not any of the other read tools.

The result was not a rendered screenshot or a metadata tree — it was a full listing of resource links to the prototype's actual source files (`App.tsx`, component files, CSS, images), because Figma Make prototypes are themselves small React apps. Reading each `App.tsx` via `ReadMcpResourceTool` gave literal TypeScript source: real type definitions, real hardcoded sample data, real business-rule-looking constants (like a `$40,000` threshold in a "Priority Rules" explainer panel). This is a much richer and more falsifiable source of truth than a screenshot would have been — it made a field-by-field diff against `src/contracts.py` possible, rather than requiring visual inference.

That diff surfaced genuine, specific drift, not vague "looks different" observations. The clearest example: `payment_aging.py`'s real high-priority threshold, grepped directly (`outstanding_amount >= 50000`), is $50,000 — but the Figma prototype's static "Priority Rules" panel displayed $40,000. This is exactly the kind of error that would have been invisible from a screenshot alone (a screenshot shows a number, not where the number came from) and would have been easy to copy verbatim into a real Phase 9 component if nobody had checked it against the source. Similarly, `inventory_allocation.py`'s `PRIORITY_RANK = {"High": 0, "Normal": 1, "Low": 2}` — grepped directly, three keys, "Normal" not "Medium" — directly contradicted all three Inventory-Allocation-adjacent Figma screens, which used a four-tier `Critical | High | Medium | Low` vocabulary consistently across all three files. Consistency across multiple independently-generated AI prototypes is not evidence of correctness; it just means the AI that generated them made the same plausible-sounding guess each time.

The `SupplierFollowUpRow` case was the most complete example of Figma inventing structure beyond the actual code. The real contract (`src/contracts.py`) has exactly six fields: `sku`, `warehouse`, `remaining_qty`, `reorder_point`, optional `supplier_name`, optional `lead_time_days`. The Figma prototype's `SupplierRow` interface has `sku`, `shortageQty`, `preferredSupplier`, `suggestedReorderQty`, `followUpPriority` — of which only `sku` and (loosely) `supplier_name`/`preferredSupplier` correspond to anything real. `shortageQty` could in principle be derived arithmetically from `reorder_point - remaining_qty` (which would make it an allowed "derived display-only aggregate" per this project's own established rule), but `suggestedReorderQty` and `followUpPriority` are not derivable from anything the Python core computes — they represent genuinely new business logic (how much to reorder, how to prioritize the reorder) that no spec defines. This is precisely the category of thing the project's Scope Gate exists to catch: something that looks like a reasonable, obvious feature but isn't actually backed by any approved spec.

## 4. Why the chrome-pattern conflict didn't become a blocking question, and why that's different from the mechanical corrections

It would have been easy to lump "which nav chrome should Phase 9 use" in with the other Figma corrections and just declare a winner unilaterally, the same way the priority-vocab and threshold corrections were resolved. But this one genuinely felt like it deserved more scrutiny before being waved through, because unlike a wrong dollar figure (which has one obviously correct answer, the real code), "which visual chrome pattern" is a legitimate design choice with no single objectively correct answer from the code alone.

The reason it still didn't become a blocking `AskUserQuestion` is that a decision already existed — documented in `context/ui-rules.md` and restated in the root `CLAUDE.md` ("light theme, sidebar nav, white cards...") — made before any Figma reference was ever available. That's a materially different situation from "no decision exists, help me choose." The four Figma prototypes didn't even agree with each other (one sidebar, two top-nav-with-stepper, one bare top bar), so there was no coherent alternative position being advocated by "the Figma reference" as a whole — just noise from an AI generating four independent guesses with no shared brief. Given that, defaulting to the pre-existing, deliberately-made decision and noting the conflict in writing (so the user could object during `ExitPlanMode` review if they disagreed) was the right level of caution: visible and reversible, but not blocking.

## 5. The shadcn token collision: how it was found, and why it changed the plan after "Blueprint ready"

This was the one place where implementation diverged from the approved blueprint, and it's worth being honest about why. The original blueprint (approved via `ExitPlanMode`, with the user's own seven guardrail additions layered on) still listed "shadcn init + add button/card/table/badge/input/select primitives" as a literal Phase 8 build step. Nothing in the planning conversation had surfaced a problem with that — the token collision wasn't discovered by thinking harder about the plan, it was discovered by actually reading `context/ui-tokens.md`'s literal `:root` variable list side-by-side with shadcn's well-known default token names while writing the real `tailwind.config.ts`.

The collision is specific and non-obvious: both systems use the token name `--accent`, but they mean opposite things. In this project's `ui-tokens.md`, `--accent` is the brand blue (`221 83% 53%`) — the primary interactive color, used for buttons, links, active states. In shadcn's convention, `--accent` is a subtle, usually-gray hover/highlight background — a much quieter, secondary role. If shadcn's `button.tsx` had been dropped in unmodified, either it would render using undefined CSS variables (broken), or `tailwind.config.ts` would need both token sets present, at which point `colors.accent` would need two different definitions and one would silently shadow the other, depending on object key ordering — a subtle, hard-to-debug bug that might not surface until someone actually used `bg-accent` expecting brand blue and got a gray hover shade instead.

This is exactly the kind of thing a project's own accumulated tooling knowledge is good at catching, and it's why the project's Tailwind skills were checked at this point rather than proceeding from general React/Next.js knowledge alone. Two skills were directly relevant: `tailwind-configuration` (general Tailwind config guidance, no strong opinion here) and `tailwind-best-practices`, which has a rule file tagged **CRITICAL**: "Never Modify Design Tokens... without explicit approval," explicitly calling out `tailwind.config.ts` as a protected file and specifying the correct response to a token gap as "escalate... document the use case... use existing tokens closest to the requirement until approved." That is a direct, on-point match for the exact situation at hand. A third skill, the main `skills/tailwind/SKILL.md`, was also checked and found to be irrelevant — it turned out to document Tailwind v4's browser-runtime patterns for a completely different tool (HyperFrames video composition), explicitly not for `tailwind.config.js`-based v3 projects like this one. Recognizing that mismatch and setting the skill aside, rather than trying to force-fit its v4-specific guidance, mattered — following it would have actively contradicted the project's own v3 pin.

Given the CRITICAL rule, adding shadcn's primitives (which would require adding shadcn's token set) stopped being a routine implementation step and became exactly the kind of token-change decision that rule says needs explicit approval — not something to just proceed with because it was in the original plan. This was surfaced via `AskUserQuestion` with a real recommendation and real trade-off (plumbing-only vs. primitives-plus-a-bridge, where choosing the bridge would itself constitute the required explicit approval), and the user chose plumbing-only. The practical cost of that choice was almost nothing, because Phase 8 never renders a real component anyway — the primitives weren't needed yet, so deferring them cost zero rework, only a documentation note for Phase 9.

## 6. What stayed simple: the decisions that didn't need re-litigating

Not everything in Phase 8 required active reasoning during the session — several decisions were made once, during the pre-implementation plan review (when the user added seven specific guardrails after the first `ExitPlanMode` attempt), and then just executed without further discussion. The scratchpad-then-transplant scaffolding pattern is a good example: it was proposed as a guardrail against `create-next-app` clobbering the repo's real `AGENTS.md`/`CLAUDE.md`/`README.md`, and it turned out to be exactly the right call — the scaffold really did generate its own versions of those three files, and the transplant step correctly left them behind. Verifying this wasn't optional trust; `git status --short AGENTS.md CLAUDE.md README.md` returning empty both before and after the transplant was the actual proof, not an assumption.

The Tailwind v3-vs-v4 question was similarly settled in advance by reading the installed Next.js docs (a step required by this project's own `AGENTS.md`) before writing any config: the default Next.js CSS guide documents v4's CSS-first syntax, but a separate, explicitly-linked `tailwind-v3-css.md` guide exists with the exact recipe this project needed. Following that guide precisely (rather than accepting v4 and downgrading afterward) meant zero rework — `npm ls tailwindcss` and the lockfile both confirmed `3.4.19` with no `@tailwindcss/postcss` (the v4 marker package) present, on the first attempt.
