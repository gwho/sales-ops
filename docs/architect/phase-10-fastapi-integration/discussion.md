# Discussion: Reasoning Behind the Phase 10 Architecture Plan

## Reading existing docs before treating something as an open question

The very first planning question in the session's argument list was "exact FastAPI package/file layout." It turned out not to be a real question at all — `context/architecture.md` had already named `backend/` as the location, months before this session, in a table nobody had revisited. This is worth dwelling on because it's a failure mode that's easy to fall into: treating every item on a list of "planning questions to resolve" as equally open, when some of them were already answered by documentation that exists specifically to prevent re-litigating settled decisions. The fix wasn't clever — it was reading `architecture.md` before opening a design discussion about where files should live. The lesson generalizes past this one decision: before spending conversation turns designing something, check whether the project's own docs already committed to an answer.

## CORS is not just "allow the origin" — headers have to be exposed separately

This is a genuinely non-obvious piece of the browser security model, and the session got it wrong on the first pass. `allow_origins` controls *whether the browser lets JavaScript see the response at all*. It does not control *which headers* JavaScript can read from that response. By default, only a small "CORS-safelisted" set of headers (things like `Content-Type`, `Content-Length`) are readable by client JS on a cross-origin response — everything else, including custom headers like `X-Report-Id`, is invisible to `fetch()` code unless the server explicitly lists it in `Access-Control-Expose-Headers` (FastAPI's `expose_headers` parameter). The response *does* contain the header on the wire — you'd see it in browser devtools' network tab — but `response.headers.get("X-Report-Id")` in application code would silently return `null`. This is a classic "works when I look at it manually, breaks when the actual code runs" bug, exactly the kind that's cheap to catch in a plan review and expensive to debug after the fact once report downloads are visually working except for one missing piece of metadata.

## Why the frontend state boundary isn't "share everything" or "share nothing"

There's a natural pull toward building a single reusable hook the moment you notice three pages are about to repeat a similar pattern (upload → submit → loading → result-or-error). The session started there, with `useWorkflowRequest` as the initial recommendation. But look at what's actually different across the three pages: Order Validation needs two files, Inventory Allocation needs three, Payment Aging needs one file plus a required date field with its own validation. A hook trying to abstract over all three shapes would need either a generic, flexible-enough API (accepting arbitrary form fields, arbitrary file counts) that ends up about as complex as just writing three separate `useState` blocks, or it would need workflow-specific branches inside the "shared" hook, which defeats the point of sharing it.

The resolution the developer redirected toward draws the line at a different, more defensible place: what's shared is the *mechanics* that are byte-for-byte identical regardless of workflow — building a `FormData`, POSTing it, parsing a `{detail: string}` error body into a typed `ApiError`, triggering a blob download from response bytes. None of that cares what the specific fields are. What's *not* shared is the state machine wrapping those mechanics, because that's where the workflows actually differ (what triggers a submit, what the result shape is, what a second "report" action needs as input). This is a useful general heuristic for "should I extract a shared abstraction": share the parts that are identical in *mechanism*, not the parts that are merely similar in *shape*.

## A decision's blast radius can exceed the question that produced it

The dashboard-stays-static decision was one of four explicit planning questions and got resolved cleanly on its own terms. What wasn't anticipated going in: `/reports` uses the *exact same pattern* — a single list of `ReportManifest`s driving `ReportCard`s, sourced from a place that (per the frontend-state decision) no longer exists as a cross-page concept. Nobody had asked "what about `/reports`" as one of the four framed questions, but the answer to "no global store" mechanically applies to it anyway, because `/reports` was already built on an assumption (a shared `reportManifests` list) that the earlier session's static-showcase decisions hadn't yet forced anyone to reconsider.

This surfaced only because the plan was being written carefully enough to trace where `ReportManifest`'s `sheet_names` field would come from on a live path — and the answer was "nowhere," which is what triggered going back to ask about it explicitly rather than silently leaving `/reports` half-addressed. The general practice worth naming: after resolving a decision, it's worth asking "what else in the codebase assumed the *previous* state of this decision was permanent?" rather than assuming a decision's effects are contained to whatever prompted it.

## What a plan-rejection round is actually good for

Four corrections came back on the first `ExitPlanMode` attempt: the `fastapi[standard]` dependency consolidation, the CORS `expose_headers` gap, propagating "sample template → sample file" wording into `src/`'s actual exception messages (not just UI copy), and stating explicitly that `src/` stays framework-free. None of these were disagreements with the plan's *direction* — every one of them was a specific, concrete detail that only becomes visible when you simulate the plan running, not when you're reasoning about it in the abstract. "Will cross-origin JS be able to read this header" is a question you only actually answer by picturing the exact browser request/response cycle, not by thinking about the CORS decision at a policy level. This is the practical value of writing a plan down and having it reviewed as a discrete artifact, separate from the conversation that produced it: a review pass can simulate the concrete mechanics in a way that's easy to skip over mid-conversation when the focus is on settling the bigger architectural question.

## Documenting work that doesn't exist yet is a category error, not a shortcut

Partway through this planning-to-implementation transition, a message arrived asking to run `/feature-docs` to document "the completed Phase 10 FastAPI Integration feature" — while still inside Plan Mode, before `ExitPlanMode` had even been approved, with zero lines of `backend/` code written. The instinctive shortcut would be to just write something plausible-sounding, since the plan already specified exactly what *would* be built. The reason that's wrong isn't pedantry: `/feature-docs`'s entire value is being a faithful record of what actually happened, including the specific problems that came up during implementation (which can't be known before implementation happens) and the actual final shape of the code (which can differ from the plan in small ways even when the plan is good). Fabricating that record ahead of time doesn't just risk being wrong in the moment — it establishes a documentation artifact that looks authoritative but was never checked against reality, which is worse than no documentation at all for someone reading it later trying to understand what was actually built.
