# Decisions — Landing Page Evidence and Technical Credibility

Each section is one architectural decision made during the `/architect` session for adding real
proof content to the public landing page (`app/(public)/page.tsx`, `components/landing/*`), which
until this session described all four workflows only in the abstract (icon + one-sentence blurb)
and never mentioned Phase 12's session-persistence feature anywhere.

## 1. Primary proof artifact: a verified duplicate-order *pair*, not a single row or a table

**What it is:** `SO-2026-010` appears twice in `sample_data/sample_orders.xlsx` (source rows 11 and
37), and both occurrences are flagged by `validate_orders()` with `error_code: "OV-002"` /
`"Duplicate order ID found."`. The landing page will show this exact pair plus the real summary
counts (`duplicate_orders: 2`, `invalid_orders: 8`, `total_orders: 36`) as an input → message →
outcome narrative.

**Why this over the alternatives considered:** four candidates were on the table going in — a real
dashboard screenshot, a verbatim validation example, an input→message→outcome narrative, and a
compact static output table. The narrative won because it's the smallest unit that's still a
complete story (a visitor sees what went wrong, in the tool's own words, and what happened as a
result) without requiring new interactive chrome. A screenshot was ruled out for this pass — not on
principle, but because no real one exists (`context/designs/*.png` is explicitly marked legacy/
do-not-use by its own README) and no browser-control tool was available in this session to capture
one fresh. A standalone output table was rejected in Decision 5 below and folded into this one.

**The mid-session correction that mattered most:** the first draft of this plan cited only the row
11 occurrence, described in the plan as "pluck the `SO-2026-010` / `OV-002` row" — singular. A plan
review caught that `sample_orders.xlsx` actually has *two* matching rows (11 and 37), and that
silently picking one would misrepresent what "duplicate" means — the whole point of a duplicate-ID
error is that two rows collide. The fix wasn't just changing copy to say "twice" — it changed the
build-time invariant (Decision 2) to require *exactly* two matches with distinct row numbers, so the
claim "appears twice" can never silently become true-by-accident of the loader only checking for
"at least one."

## 2. Sync mechanism: a checked data boundary, not a copy-paste or a convenience lookup

**What it is:** `lib/content/landing-evidence.ts` imports the existing typed export from
`lib/mock-data.ts` (itself auto-generated by `scripts/generate_mock_data.py`, which runs the real
`validate_orders()` pipeline against `sample_data/sample_orders.xlsx` — confirmed not hand-written).
It reads its lookup selector (`orderId`, `errorCode`) from the content JSON, not as hardcoded
constants, and **throws at module-eval time** — failing the build — if any of these invariants
don't hold: the selected order ID isn't found; the matching error rows don't number *exactly* two;
those two matches don't have distinct `row_number`s; the message text isn't an exact match; or
`summary.duplicate_orders !== 2` (or `invalid_orders`/`total_orders` is missing/non-numeric).

**Why this over the obvious alternative:** the obvious, faster approach is to run
`npm run mock-data` once, eyeball the output, and hand-copy the row/summary values into
`content/portfolio/sales-admin-automation-toolkit.json` like every other piece of landing copy.
That was explicitly rejected. The whole point of sourcing evidence from tested code is that it stays
true — a hand-copied value is just as capable of going stale as an invented one; it's only *true at
the moment someone copied it*. A `.find()`-style lookup with no invariant is only marginally better:
it would silently render nothing (or the wrong row) if `sample_data/sample_orders.xlsx` is ever
regenerated without this exact duplicate case. A loader that throws turns that failure mode into a
build break someone has to notice and fix, instead of a landing page quietly making a claim that's
no longer true.

**Cost accepted:** this is the first piece of landing content not sourced entirely from
`content/portfolio/...json`. That's a real precedent change from `docs/architect/portfolio-landing-page/decisions.md`
#3, which established that file as the single source for all landing copy. Decision 3 below is the
guardrail that keeps this from spreading further than intended.

## 3. Content ownership split: JSON owns prose, loader owns data, component owns presentation

**What it is:** three strict layers, each forbidden from doing the other two's job.
`content/portfolio/sales-admin-automation-toolkit.json` owns every word of prose and the evidence
*selector* (`orderId`, `errorCode`) — never hardcoded in a component. `lib/content/landing-evidence.ts`
owns data lookup and the invariant checks from Decision 2 — no prose at all. `ValidationEvidence.tsx`
owns presentation only, composing the JSON's prose fragments with the loader's verified numbers —
it never invents a word of its own copy.

**Why this over one file doing everything:** the plan's second draft split "prose" from "data" but
left the exact boundary soft — it said the loader "looks up the `SO-2026-010` row" as if that ID
could live as a constant inside the loader. A later review caught that this quietly reopened the
same problem Decision 2 was trying to close: if `SO-2026-010` is hardcoded in two places (the JSON's
`orderId` field *and* a constant inside the loader), they can drift apart from each other with no
error, defeating the point of having a single verified source. The fix was making the loader read
its selector *from* the JSON, so there is exactly one place in the codebase where "which order ID
are we citing" is decided.

**The locked shape** (`PortfolioValidationEvidence`) breaks the prose into small fragments —
`heading`, `orderId`, `errorCode`, `duplicateStatement`, `messageLead`, `summaryLead`,
`summaryOutcome` — rather than one big string, specifically so the component can interleave the
loader's *verified* values (the real message text, the real counts) between fixed editorial phrases
without ever concatenating unverified prose next to verified numbers in a way that blurs which part
is which.

## 4. Proof strip: durable capability statements, never a metric that will go stale

**What it is:** a short strip under `Hero`, before `WorkflowsSection`, with two fixed statements:
"Three business-rule modules covered by automated tests." and "Python business rules and Excel
exports were verified before the production frontend was built."

**Why this over citing real numbers (e.g. a live test count):** an earlier ideation pass (outside
this `/architect` session, in the review that preceded it) considered citing the real, verified
pytest count (212 passing at the time). That was rejected here specifically because it's *too*
precise to stay true — `context/progress-tracker.md`'s own history shows the test count changing
almost every phase, and the project's own `CLAUDE.md` already flags that number as "last known from
memory... re-run before relying on this." A number that requires an update cadence nobody currently
owns is a worse landing-page claim than a true-forever statement, even though the number *feels*
more impressive. "Three business-rule modules" is stable at the module level (it would only need
updating if a whole module were added or removed — a rare, major event) in a way "212 tests" isn't.

## 5. Persistence highlight: reused, vetted wording — not re-derived from memory

**What it is:** one secondary section, placed after the business-evidence content (not in the
hero), describing Phase 12's session-scoped persistence: "Each browser receives an anonymous session
ID stored in `localStorage`. The latest result for each workflow is stored in Postgres so the
dashboard can restore it without requiring an account."

**Why the wording was pulled from `CONTEXT.md` and the ADR instead of written fresh:** this project
already made — and had to correct — this exact mistake once. `docs/architect/portfolio-landing-page/decisions.md`
#6 records that an earlier draft of the `/dashboard` persistence notice said results are "saved to
this browser only," which is wrong: only the *session ID* lives in the browser (`localStorage`); the
*saved results* live in Postgres, keyed by that ID. Writing new landing copy about the same feature
from memory risked reintroducing that exact error. Instead, this session pulled the distinction
directly from `CONTEXT.md`'s already-reviewed glossary entries for "Anonymous Session ID" and "Saved
Workflow Result," so the new copy inherits precision that was already fought for once rather than
re-deriving it and risking the same mistake twice.

## 6. Output table: rejected as its own component

**What it is:** the idea of a small, static 3-4-row table (real `AllocationResultRow` or similar)
somewhere on the landing page was considered and rejected as a *separate* component. Decision 1's
narrative treatment absorbs the same evidentiary job.

**Why:** `docs/architect/portfolio-landing-page/decisions.md` #1 already made a deliberate call that
this page's visual identity is flat, bordered, editorial, and specifically *not* the dashboard's
`Card`/`Table`-heavy operational style. A real data table — even a small, non-interactive one — is
the single element most likely to blur that line, because it's the one thing on the page that would
look like actual product surface rather than case-study copy. Since the narrative in Decision 1
already delivers "real proof, not decoration" without introducing a `Table` primitive, there was no
remaining job left for a table to do that would justify that risk.

## 7. Dual-audience wayfinding: rejected

**What it is:** the idea (raised in the ideation pass preceding this session) of adding a "for
hiring managers" / "for engineers" split near the hero, so two different readers could jump to
different content. Rejected outright — no change to page structure for this.

**Why:** `PublicHeader.tsx` already provides `#workflows`/`#value`/`#tech` anchor navigation. A
second wayfinding row would likely just restate that in different words unless it were framed
entirely differently (by reader intent, not section name) — and even then, it's solving a problem
that was never confirmed to be real for a page this short (one linear scroll, five-ish sections).
The session's own ideation pass had already flagged this as its weakest-justified idea before the
architect session even started.

## 8. Workflow cards become real links, with a closed route type

**What it is:** `WorkflowsSection.tsx`'s four cards, currently plain non-interactive `<div>`s with a
`hover:bg-surface-muted` affordance but no `href`/`tabIndex`, become full-card `next/link`s to their
matching real routes. The `href` field is typed as a closed union (`PortfolioWorkflowHref`), not a
bare `string`, so an invalid internal destination is a compile error, not a runtime dead link.

**Why this wasn't in the original six decisions:** it surfaced as a side finding during research —
the cards visually promise interactivity (the hover state) but deliver none, which is a small but
real UX/accessibility mismatch (a keyboard user tabbing through the page has nothing to land on
where a mouse user sees a hover response). Once found, fixing it was low-risk and high-value enough
(the four workflow pages already exist as real routes) to fold into this pass rather than defer.

## 9. PublicHeader mobile fix: fully specified, not just "hide the anchors"

**What it is:** below `sm`, the three anchor links (`#workflows`/`#value`/`#tech`) hide entirely
(`hidden sm:flex`), and the brand label swaps from "Sales Admin Automation Toolkit" to "Sales Ops
Toolkit" — two fixed strings, no CSS-truncation fallback. The brand wrapper gets
`min-w-0 whitespace-nowrap`; the CTA gets `shrink-0 whitespace-nowrap`. 320px is a required manual
verification width.

**Why the first draft wasn't enough:** the initial fix ("hide anchors below `sm`") solved the most
visible problem but left an unverified assumption — that the full brand string plus the CTA plus
`px-6` padding would actually fit in the remaining space at the narrowest realistic width (320px).
A plan review pointed out that assumption was never checked, and that leaving the brand treatment as
"a two-string swap, or a CSS-only truncation fallback if rejected at review" was itself an
unresolved decision disguised as a resolved one. The final version picks one approach outright
(the two-string swap) so there's nothing left to decide during implementation.

## 10. Component boundary constraints

**What it is:** `ValidationEvidence.tsx` stays a Server Component; `lib/content/landing-evidence.ts`
must never be imported into a Client Component; the quoted validation message must not use
`role="alert"` or any ARIA live region.

**Why:** the evidence is static, server-rendered content decided at build time, not a live event
happening in front of the user — using `role="alert"`/`aria-live` would misrepresent it as a dynamic
notification (the same category of semantic misuse this project has been careful to avoid elsewhere,
e.g. `PersistenceNotice` deliberately *not* reusing `BusinessErrorMessage`'s `role="alert"` pattern
because a failed save isn't the same kind of event as a validation error). Keeping the loader
Server-Component-only also matches the project's default (`context/code-standards.md`: Server
Components by default, `"use client"` only for state/effects/browser APIs) — there's no state or
browser API involved here, so there's no reason to cross that boundary.

## 11. Pre-implementation `/session-docs` gate

**What it is:** this documentation itself — written before any code changes, from the session
transcript, per the recommendation that closed out the plan review.

**Why:** the six decisions above (plus the four side-findings) involved enough back-and-forth — two
research agents, one `AskUserQuestion` round, three rounds of plan correction — that the reasoning
behind each locked detail (why exactly two matches, why the loader reads its selector from JSON
instead of a constant, why the brand string is two fixed strings and not a truncation fallback)
would be expensive to reconstruct later from the code alone. Writing it down now, while the
transcript is still available, is cheaper than re-deriving it during a future edit.
