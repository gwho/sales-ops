# Discussion — Architect Sessions: Phase 9 (Reusable UI Components and Static Pages) and Phase 9.1 (Visual Alignment Fixes)

This is the deep-dive companion to `decisions.md`. Where that file states what was decided and why in a few sentences, this file walks through the underlying mechanics, the wrong alternatives that were seriously considered, and what would actually have gone wrong if the session had chosen differently.

## 1. "Static showcase" as a general pattern for building UI ahead of a backend

The core tension in Phase 9 was: the UI needs to look and feel like a working application for a portfolio reviewer, but the backend it will eventually talk to (Phase 10's FastAPI layer) doesn't exist yet. There are two honest ways to resolve that tension. One is to build a *simulated* live application — an `UploadPanel` that, when a file is chosen, sets some `status: "processing"` state, waits on a `setTimeout`, then flips to `status: "ready"` and swaps in the mock data as if it had just been computed. The other is a *static showcase* — the mock data is simply always there, rendered immediately, and the interactive pieces (the file picker, the stepper) are present and functional as UI elements but don't gate anything.

The simulated-live approach is seductive because it *feels* more impressive in a live demo — clicking "upload" and watching a spinner resolve into results reads as "it works." But it's building a fake version of exactly the state machine Phase 10 needs to build for real. Every `setTimeout`, every `status` enum transition, every "did the fake upload finish yet" check would need to be torn out and replaced with real `fetch` calls, real loading states driven by actual network requests, and real error states driven by actual HTTP failures — not adapted, replaced. The static showcase, by contrast, asks a narrower and more answerable question: "does this component correctly render this exact shape of real contract data?" That question's answer transfers directly into Phase 10, because the rendering logic doesn't change when the data source changes from a static import to a `fetch` response — only the *plumbing that gets the data there* changes.

The tell that distinguishes a wise deferral from a cop-out is whether the thing being deferred would be *thrown away* or *extended*. Here, `ReportCard`'s three non-`Ready` states were built as prop-driven visual variants — meaning the component's rendering logic is already correct for all four states, and Phase 10 only needs to add the *logic that decides which state to pass in*, not rebuild the states themselves. That's the difference between deferring wisely and deferring by not doing the work.

## 2. Server/Client boundaries: what "can't pass a function as a prop" actually means and why it bites table libraries specifically

This wasn't a Phase 9 *planning* decision — it was discovered during Phase 9 implementation via a real `next build` failure, and it's worth understanding deeply because it shaped a design constraint that the Phase 9.1 sessions had to keep respecting.

React Server Components render on the server and serialize their output — including the props they pass to Client Component children — into a format that gets sent to the browser and "hydrated" into interactive React. That serialization format can represent strings, numbers, booleans, arrays, and plain objects. It cannot represent a JavaScript closure, because a closure is a function *plus the variables it captured from its enclosing scope* — there is no way to serialize "the code plus its captured environment" into JSON-like data and reconstruct it faithfully on the other side. So React enforces a hard rule: a Server Component cannot pass a function as a prop to a Client Component.

This becomes a real, sharp constraint the moment you build a generic, reusable table component with a column-definition API — which is almost every non-trivial table component ever written, including this project's `DataTable`. A `DataTableColumn` needs a `render: (row) => ReactNode` function to say "here's how to draw this cell," and often a `sortValue: (row) => string | number` function to say "here's what to sort this column by." Those functions are closures — they capture things like `formatDate` or `StatusBadge` from their enclosing module scope. The moment a Server Component page tries to build an array of these column objects and hand it to a Client Component `<DataTable columns={...} />`, the build fails, because `columns` contains functions.

The fix that was applied — making the *page*, not just `DataTable`, a Client Component whenever it builds a `DataTable` column config — works because the Server→Client serialization boundary only applies at the actual crossing point. Once a component is already a Client Component, it can pass functions to other Client Components freely, the same as any ordinary React tree with no server rendering involved at all. The alternative considered (and explicitly rejected as not worth the complexity for this phase) was moving each table's column definitions into small, dedicated Client child components that receive only serializable `data` as a prop, letting the *page* stay a Server Component. That's architecturally cleaner in the abstract, but it means one extra file per table section — nine of them, across three pages, for a phase that had already deliberately kept its primitive set minimal. The tradeoff was made explicitly and documented as a standing invariant rather than silently absorbed.

## 3. The same word, two meanings: why `StatusBadge` refuses to guess

There's a general principle here worth naming: a controlled vocabulary that reuses the same literal value across semantically different fields is a trap for any code that tries to interpret the value without knowing which field it came from. `"High"` is a perfectly good value for both `ValidOrderRow.priority` (an *importance* signal — this order matters more, handle it first) and `PaymentAgingRow.follow_up_priority` (an *urgency* signal — this invoice is dangerously overdue, someone needs to act). Both are legitimate, sensible uses of the word "High" in their own domain. The trap is assuming that because the *string* is the same, the *treatment* should be the same.

A naive `StatusBadge` implementation might do something like a single switch statement: `if (status === "High") return "danger"`. This works perfectly for `follow_up_priority` and is subtly, confidently wrong for `priority` — a high-priority *order* isn't a problem, it's just something to fast-track, so coloring it red/danger (the same treatment as "this invoice is critically overdue") sends the wrong signal to anyone reading the UI. The bug wouldn't announce itself with an error; it would just be a color that feels slightly off to anyone who noticed, and easy to dismiss as a design nitpick rather than a real defect.

The fix — small, typed, per-field helper functions, each accepting only the literal union type for its specific field (`importancePriorityTone(priority: "High" | "Normal" | "Low")` vs. `followUpPriorityTone(priority: "High" | "Medium" | "Low" | "Watch" | "None")`) — turns a runtime judgment call into a compile-time impossibility. You cannot accidentally call `importancePriorityTone` with a `follow_up_priority` value, because TypeScript's structural typing would reject `"Watch"` or `"None"` as invalid arguments to a function that only accepts `"High" | "Normal" | "Low"`. The type system does the disambiguation that a single string-keyed lookup function would have to do by guessing.

## 4. Tailwind's static analysis model, and why "it worked when I typed it manually" is a trap

Tailwind CSS (in the version this project uses, and in most of its history) doesn't generate CSS for every class it *could* produce — that would be an enormous stylesheet. Instead, at build time, it scans the project's source files as plain text and looks for substrings that match the shape of a utility class name (`bg-success`, `stroke-warning`, and so on), then generates CSS only for the classes it actually found. This is a *text search*, not a JavaScript evaluation — Tailwind's scanner does not run your components, does not know what a `tone` variable will hold at runtime, and does not follow string concatenation logic.

This means a component that builds a class name via string interpolation — `` className={`stroke-${tone}`} `` — is invisible to the scanner in a specific and insidious way: the scanner sees the literal characters `stroke-${tone}` (or, depending on how the bundler processes it, just the static fragments around the `${...}`), which don't match any real utility class, so nothing gets generated for whatever `tone` might actually be at runtime. In development, depending on the exact tooling and caching involved, this can sometimes *appear* to work if the class happened to get generated for an unrelated reason elsewhere in the codebase — which is the trap: a developer types `` `stroke-${tone}` ``, sees it render correctly against dev-server hot-reload, and ships it, only for it to silently render unstyled once the styles are purged for a real production build where that exact combination never appeared as a literal substring anywhere.

The fix pattern used throughout this project — a `Record<Tone, string>` object where every value is a *complete, literal* class name string (`{ success: "stroke-success", warning: "stroke-warning", ... }`) — works because the full string `"stroke-success"` exists verbatim in the source file, in that object literal, regardless of what runtime value `tone` holds. The scanner finds it there, generates the CSS, and the lookup (`STROKE_TONE_CLASSES[tone]`) is a plain object property access at runtime — no string building involved, just picking one of five pre-existing, fully-formed strings. This is worth understanding as a general rule for any Tailwind project, not just this one: *any* Tailwind class name that will actually be used must appear somewhere in the source as a complete, literal string — computed or interpolated class names are a reliable way to ship invisible bugs.

## 5. Type-only imports: why they don't create the dependency they look like they create

TypeScript's `import type { X } from "./module"` syntax tells the compiler "I only need this for type-checking; erase it entirely from the compiled output." This matters architecturally in a way that's easy to miss if you only think about imports in terms of "this file depends on that file." A component that does `import type { Tone } from "@/components/workflow/StatusBadge"` looks, at a glance, like it depends on `StatusBadge.tsx` — but after compilation, that import line doesn't exist. There is no `require("./StatusBadge")` or `import "./StatusBadge"` left in the JavaScript that actually runs. The dependency is real at type-check time (if `StatusBadge.tsx`'s `Tone` type changes shape, the importing file needs to know) and completely absent at runtime.

This distinction is exactly what resolved the Phase 9.1 review question about whether importing `Tone` into a chart component "from a visual component file" was an architectural smell. The instinct to be suspicious of it was reasonable — "don't let low-level rendering components import from higher-level feature components" is a good general rule, and `StatusBadge` is a workflow-specific component while `DonutBreakdownChart` is meant to be generic. But the actual dependency being created wasn't "the chart renders using StatusBadge's logic" — it was "the chart's TypeScript prop type is defined in terms of a type that happens to live in StatusBadge.tsx." Those are very different claims, and only one of them describes a real coupling problem. The resolution — keep the type-only import, don't create a new file — was reached by actually tracing what gets imported and how it's used, rather than pattern-matching on "component A imports from component B, that's probably bad" without checking whether the import is load-bearing at runtime.

## 6. CSS custom property collisions as a class of bug, and why "just use their tokens" isn't free

The shadcn primitive decision (hand-write, never generate) is worth understanding as an instance of a more general problem: two design systems that use the same *variable names* for *different meanings* cannot coexist by simply merging their token sets, because the merge itself is the bug. This project's `--accent` is HSL `221 83% 53%` — a saturated brand blue, used for primary buttons and active navigation state. shadcn's own default `--accent` convention is a near-white gray, used for hover backgrounds on otherwise-plain elements. These aren't two systems that happen not to overlap; they're two systems that use the identical CSS custom property name to mean something close to opposite. If shadcn's generator were allowed to write its own `--accent` definition into `globals.css` (which is exactly what `npx shadcn add` does when `cssVariables: true`, as this project's `components.json` has it configured), the *last variable declaration wins* — meaning either the project's real brand blue gets silently overwritten by a gray hover token, or (if the project's declaration comes later in the cascade) shadcn's own generated components, which expect `--accent` to mean "subtle gray hover," would render using brand blue in places they never intended, looking broken.

There's no partial-merge solution here — you can't "use half of shadcn's tokens." The only clean resolutions are (a) rename one system's tokens so they don't collide, which means the generated shadcn code needs everywhere `--accent` is referenced to change, defeating the purpose of using a generator, or (b) never run the generator and write the components by hand against the project's real tokens, treating shadcn's public source purely as a style/API reference. This project chose (b), and the "why" is worth generalizing: whenever adopting a third-party component library that ships its own design-token convention, check for name collisions with your own token set *before* deciding whether the library's generator/CLI is safe to run — a naming collision at the CSS-variable layer is not a merge conflict you resolve, it's a semantic incompatibility you have to route around entirely.

## 7. Zero-value guards: defensive code for math that can't fail loudly

JavaScript's arithmetic doesn't throw on divide-by-zero the way many languages do — `5 / 0` evaluates to `Infinity`, and `0 / 0` evaluates to `NaN`, both of which propagate silently through further arithmetic without ever raising an exception. This is a specific hazard for any UI code computing percentages or ratios for layout purposes (bar heights, donut arc lengths, progress-bar fills), because the failure mode isn't a crash with a stack trace pointing at the bug — it's a CSS property set to a string like `"NaN%"` or `"Infinity%"`, which browsers simply refuse to apply, usually rendering as if the property were unset. The visual result is something that looks subtly broken (a bar with no height, a donut ring that doesn't draw) with no console error, no failed test, and no obvious link back to "oh, the underlying data summed to zero."

The two zero-guards added in the Chart Alignment Addendum session are a small case study in matching the guard's *behavior* to what would actually communicate the right thing to a viewer, not just avoiding the crash. A donut chart guarded to render a neutral gray ring with "0" in the center is still telling the truth — there's a real answer, and it's zero, and a viewer scanning the dashboard sees a complete-looking chart that happens to report nothing. A bar chart guarded the same way (all bars flattened to zero height) would instead look like a rendering failure — five bars, all invisible, no obvious "this is intentional" signal. Choosing an `EmptyState` caption instead for that case is a deliberate acknowledgment that "zero bars, all technically present" and "no data to show" read very differently to a human, even though they're mathematically the same input.

## 8. Component-folder taxonomy as a form of documentation

The `components/tables/` vs. `components/charts/` split might look like a bikeshed-level decision, but it's worth taking seriously as a case study in how a folder structure functions as a form of documentation that decays if you're not deliberate about it. `MiniBar`, `SegmentedBar`, and `AgingBucketBars` all live in `components/tables/` because they share a real property: they're small visual elements meant to be composed *into* or *directly beside* a table row or a compact summary card — none of them is a standalone visualization someone would reach for on its own. `DonutBreakdownChart` and `VerticalBucketBarChart` don't share that property; they're whole, self-contained chart cards that happen to appear near tables on the dashboard, but they're not table-adjacent in the same structural sense.

The cost of putting them in `components/tables/` anyway wouldn't have been an immediate bug — the code would work identically regardless of which folder it lived in. The cost would show up later, gradually, the first time a future developer (or a future session of the same AI) needed to find "the chart components" and searched `components/charts/` first because that's the more semantically obvious name, found nothing, and either duplicated work or spent time discovering the actual location. A folder name is a promise about what you'll find inside it; keeping that promise accurate is a small, continuous cost that's much cheaper to pay at creation time than to pay later as a "where did we put that" search.
