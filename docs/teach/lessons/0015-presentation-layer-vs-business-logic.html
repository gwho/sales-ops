Track 3 — Presentation Without Leakage

Presentation Layer vs. Business Logic Layer

You've finished Tutorials 03–05 — three modules that each decide real business facts about a spreadsheet row. Track 3 is about a fourth module that decides nothing at all: report_export.py. This lesson teaches the one distinction the whole track, and Tutorial 06, builds on.

Primary source

No single official external resource teaches this as its own topic — like Lesson 12's greedy-processing gap, it's best learned directly from this repo's own worked example: docs/plan/phase-6-excel-report-export/explanation.md §1, and Tutorial 06 Part 1, once you get there.

Two layers, two jobs

Business logic layerPresentation layer
What it decidesWhat a result is — computes, classifies, validates a real-world fact.How an already-decided result should look to a human reading it.
Examples in this repoorder_validation.py, inventory_allocation.py, payment_aging.py (Tutorials 03–05) report_export.py (Tutorial 06)
Can it be "wrong" in a business sense?Yes — a bug in a rule produces a wrong business fact.No — a bug here can only be a formatting mistake. It has no rules of its own to get wrong.

That last row is the whole point. report_export.py reads a result someone else already computed and writes it to a spreadsheet. It never asks "is this quantity correct," "is this SKU active," or "is this invoice overdue" — those questions were already answered, upstream, by the module that owns them.

What "never recalculates" looks like in real code

Open src/report_export.py and find the start of export_order_validation_report:

def export_order_validation_report(
    result: OrderValidationResult,
    original_orders_df: pd.DataFrame | None = None,
    generated_at: datetime | None = None,
) -> tuple[bytes, ReportManifest]:
    wb = _new_workbook()
    _write_summary_sheet(wb, "Summary", result["summary"])
    _write_detail_sheet(wb, "Valid Orders", result["valid_orders"], VALID_ORDER_COLUMNS)
    _write_detail_sheet(wb, "Validation Errors", result["errors"], VALIDATION_ERROR_COLUMNS)

Every line reads a key straight out of result — the exact dict validate_orders() already returned — and hands it to a generic sheet writer. Nothing here re-parses a date, re-checks a SKU, or re-decides anything. Now open tests/test_report_export.py and find:

def test_inventory_allocation_report_backorders_sheet_uses_precomputed_list_as_is():
    workbook_bytes, _ = export_inventory_allocation_report(
        _inventory_allocation_result(), generated_at=GENERATED_AT
    )
    wb = _load(workbook_bytes)
    ws = wb["Backorders"]
    row = dict(zip(ALLOCATION_RESULT_COLUMNS, [c.value for c in ws[2]]))
    assert row["status"] == "Backordered"

"Backordered" is a fact inventory_allocation.py already decided, back in Tutorial 04 — by comparing allocatable_qty against the order's requested quantity. The report just writes whatever string is already sitting in row["status"].

The failure mode this rule prevents

Imagine _write_detail_sheet instead re-derived status inline — say, "Backordered" if allocated_qty == 0 else "Fully Allocated". It would usually agree with the real module, right up until a future rule change to inventory_allocation.py shipped without an identical change here. At that point, the live dashboard (reading the real result) and a downloaded report generated moments later (reading a stale reimplementation) could show two different answers for the same order line — a report that quietly recomputes and disagrees with the screen. This is exactly why the rule is "never recalculate," not "try to recalculate correctly."
