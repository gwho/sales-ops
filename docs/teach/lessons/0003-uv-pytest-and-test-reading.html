uv, pytest, and Reading a Test File

Two tools, one skill: what uv and pytest each do, and how to read a test file as a plain-English description of behavior — not scary code, just claims with proof attached.

Primary source

pytest — Get Started (official docs). Once this lesson feels solid, also skim uv — Getting Started — the tool this repo's pyproject.toml and uv.lock are built around.

What uv does

Python projects need a specific, reproducible set of installed libraries. uv is the tool that reads pyproject.toml (which library ranges are needed) and uv.lock (the exact versions that were actually tested), and sets up an isolated environment matching that lock file exactly. Two commands cover everything Track 1 needs:

  • uv sync — install everything uv.lock specifies.
  • uv run pytest — run the test suite inside that environment, without you manually "activating" anything first.

You don't need to memorize more than that for now — uv's job is "make sure the right libraries are installed and used," full stop.

What pytest does

pytest is a test runner. Point it at the repo and it will:

  1. Find every file named test_*.py under tests/ (this repo's pyproject.toml says exactly that: testpaths = ["tests"]).
  2. Inside each file, find every function named test_*.
  3. Run each one. If it completes with no error, that's a pass. If it hits a failed assert or an unexpected exception, that's a failure.
  4. Print a summary — how many passed, how many failed, and exactly which line broke for any failure.

Reading an assert statement

assert is the whole mechanism. It's one word, and it does one thing:

assert 2 + 2 == 4        # true — nothing happens, execution continues
assert 2 + 2 == 5        # false — raises AssertionError, the test fails here

A test function is just a sequence of setup followed by one or more assert statements — each one is a claim the author is making about the code, backed by a real example. Here's a real one, from tests/test_excel_io.py:

def test_validate_required_columns_passes_when_all_present():
    df = pd.DataFrame({"order_id": ["SO-1"], "sku": ["MED-LENS-001"]})

    validate_required_columns(df, ["order_id", "sku"], "orders file")

Notice: no assert at all. The claim here is "this call does not raise an error" — if validate_required_columns raised anything, the test would fail right there, without needing an explicit assertion. Compare it to the very next test in the same file, which does expect an error:

def test_validate_required_columns_raises_business_readable_message():
    df = pd.DataFrame({"order_id": ["SO-1"]})

    with pytest.raises(MissingColumnsError) as excinfo:
        validate_required_columns(df, ["order_id", "sku"], "orders file")

    assert "orders file" in str(excinfo.value)
    assert "sku" in str(excinfo.value)

Read with pytest.raises(MissingColumnsError) as excinfo: as: "I expect the indented code below to raise a MissingColumnsError. If it doesn't, that's the test failure. If it does, capture the error as excinfo so I can inspect it afterward." The two assert lines after it check the error's message actually names the file and the missing column — not just that some error happened.

Why this matters

A test function's name is usually a full sentence once you read the underscores as spaces: "test validate required columns raises business readable message." You can often understand what a module does by reading only the test function names in its test file, top to bottom, before reading a single line of the module itself. Try this in tests/test_excel_io.py right now — three names, three behaviors.

Tiny exercise (3 minutes)

If you have the repo set up locally, run:

uv run pytest tests/test_excel_io.py -v

Read the three test names in the output before scrolling. Predict, from the name alone, whether each one is checking a success case or a failure case — then check yourself against what you just read above. If you don't have a local setup yet, do the same prediction exercise by opening tests/test_excel_io.py and reading just the three def test_... lines, ignoring their bodies.
