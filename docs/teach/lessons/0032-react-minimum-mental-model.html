Track 5 — UI Components & Next.js

React's Minimum Mental Model

Just enough React to read this repo's frontend — component functions, JSX, props, state, events, and the one rule that makes all of it move: a state change triggers a re-render. Tiny, invented examples first; this repo's real components wait for Tutorial 10.

Primary source

"Your First Component" and "State: A Component's Memory" on react.dev — the official React documentation, written for exactly this on-ramp.

A component is a function that returns UI

In React, a component is a JavaScript function whose job is to return a description of what should appear on screen, written in a syntax called JSX (HTML-like markup inside JavaScript).

The smallest possible component

function Greeting() {
  return <p>Hello, sales admin.</p>;
}

That's it — a function, named with a capital letter (React's convention for telling components apart from plain HTML tags), returning JSX. Using it elsewhere looks like using an HTML tag: <Greeting />.

Props: how a parent hands data to a child

A component becomes reusable once it can vary based on the data it's given. That data is called props (short for "properties") — read-only values passed in from whatever rendered the component, the same way an HTML tag's attributes configure it.

A component with props

function Greeting({ name }) {
  return <p>Hello, {name}.</p>;
}

// used as:
<Greeting name="sales admin" />
<Greeting name="warehouse coordinator" />

Same component, two different rendered outputs — that's the entire point of props. A component never modifies its own props; if it needs to change, that's what state is for.

State: a value a component remembers between renders

State is data a component owns and can change over time — unlike a prop, which only ever flows in from outside. React's useState gives a component a piece of memory and a function to update it.

A component with state

function Counter() {
  const [count, setCount] = useState(0);
  return (
    <button onClick={() => setCount(count + 1)}>
      Clicked {count} times
    </button>
  );
}

count is the current value; setCount is the only sanctioned way to change it. Calling setCount doesn't just update a variable somewhere — it's a signal to React that this component needs to run again.

Events: user actions become function calls

JSX attaches behavior with camelCase handler props — onClick, onChange — each taking a function to call when that event happens. This is the same "behavior" job JavaScript owned in Lesson 31, just written inline as a JSX attribute instead of a separate addEventListener call.

The one rule that makes it all move: render-after-state-change

The core loop

Call a state setter (like setCount) → React re-runs that component's function → the function returns new JSX describing the updated UI → React updates only the parts of the real page that actually changed. Nothing on screen updates because you "changed a variable" — it updates because you called the state setter, which schedules a re-render.

Correct framing

"The UI updates because I called setCount, which tells React to re-run the component and produce new JSX."

A framing to avoid

"The UI updates because count changed." (This skips the actual mechanism — directly mutating a variable that useState gave you, without calling its setter, does not update the screen. This is a very common early bug.)

Tiny exercise (7 minutes)

Exercise

On paper or in a scratch file, write a component WorkflowStatus({ label }) that renders its label prop inside a <span>. Then add one piece of state, expanded, defaulting to false, and an onClick handler on the span that flips it. Don't worry about actually rendering something different based on expanded yet — the goal is just correctly wiring useState and an event handler together, predicting out loud what happens on each click before you'd run it.

If the render-after-state-change loop still feels like a black box — especially "why does directly mutating state not work" — ask here before Lesson 33. Server Components (next lesson) only make sense once you're confident this loop is a client-side, in-the-browser mechanism.
