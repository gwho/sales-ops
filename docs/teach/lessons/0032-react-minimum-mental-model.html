Track 5 — UI Components & Next.js

React's Minimum Mental Model

Just enough React to read this repo's frontend — component functions, JSX, props, state, events, and what actually schedules a re-render. A state update is the trigger you'll write by hand the most, but it isn't the only one that exists. Tiny, invented examples first; this repo's real components wait for Tutorial 10.

Primary source

"Your First Component" and "State: A Component's Memory" on react.dev — the official React documentation, written for exactly this on-ramp.

A component is a function that returns UI

In React, a component is a JavaScript function whose job is to return React elements — a description React uses to produce and update UI, written in a syntax called JSX (HTML-like markup inside JavaScript) — or to return null when it should render nothing at all. "Returns a description of the UI," not "returns HTML directly," matters here: that distinction is exactly what makes Server and Client Components (Lesson 33) possible later.

The smallest possible component

function Greeting() {
  return <p>Hello, sales admin.</p>;
}

That's it — a function, named with a capital letter (React's convention for telling components apart from plain HTML tags), returning JSX. Using it elsewhere looks like using an HTML tag: <Greeting />.

Props: how a parent hands data to a child

A component becomes reusable once it can vary based on the data it's given. That data is called props (short for "properties") — read-only values passed in from whatever rendered the component, the same way an HTML tag's attributes configure it.

A component with props

function Greeting({ name }) {
  return <p>Hello, {name}.</p>;
}

// used as:
<Greeting name="sales admin" />
<Greeting name="warehouse coordinator" />

Same component, two different rendered outputs — that's the entire point of props. A component never modifies its own props; if it needs to change, that's what state is for.

State: a value a component remembers between renders

State is data a component owns and can change over time — unlike a prop, which only ever flows in from outside. React's useState gives a component a piece of memory and a function to update it.

A component with state

function Counter() {
  const [count, setCount] = useState(0);
  return (
    <button onClick={() => setCount(count + 1)}>
      Clicked {count} times
    </button>
  );
}

count is the current value; setCount is the only sanctioned way to change it. Calling setCount doesn't just update a variable somewhere — it requests a state update, which is one real trigger for React to run this component again (more on the others below).

Events: user actions become function calls

JSX attaches behavior with camelCase handler props — onClick, onChange — each taking a function to call when that event happens. This is the same "behavior" job JavaScript owned in Lesson 31, just written inline as a JSX attribute instead of a separate addEventListener call.

What actually schedules a re-render

The core loop, for this Counter example

Calling setCount requests a state update → React schedules rendering work, calling Counter again with the next state snapshot → the function returns new JSX describing the updated UI → React reconciles that against the existing DOM, updating only the parts of the real page that actually changed. Nothing on screen updates because you "changed a variable" — it updates because you called the state setter.

That's one trigger, not the only one. A component also re-renders because its parent rendered it again, because its props changed, because a context value it consumes changed, or because a subscribed external store reported a change. And even a state-setter call doesn't guarantee a re-render: if the new value is Object.is-equal to the current state, React skips re-rendering that component (and its children) entirely — a real optimization documented in React's own reference, not a hypothetical edge case.

Correct framing

"The UI updates because I called setCount, which tells React to re-run the component and produce new JSX."

A framing to avoid

"The UI updates because count changed." (This skips the actual mechanism — changing some unrelated local variable never requests a React render, and mutating a state object or array directly, without calling its setter, is not a replacement for calling the setter with the next value. This is a very common early bug.)

Tiny exercise (7 minutes)

Exercise — predict, then check against a model answer

On paper or in a scratch file, write a component WorkflowStatus({ label }) that renders a real <button type="button"> — not a <span>: a span isn't keyboard-focusable, doesn't activate on Enter/Space, and exposes no button semantics to assistive tech, and a status toggle needs all three, the same habit Lesson 40 builds on later. Add one piece of state, expanded, defaulting to false. Clicking the button should flip expanded and the component should render something visibly different depending on its value — state with no observable effect gives you nothing to actually check your work against.

Before revealing anything, predict out loud: after the first click, what is expanded, and what does the button (and anything else on screen) show? Then the second click?

Reveal: one correct implementation 
function WorkflowStatus({ label }: { label: string }) {
  const [expanded, setExpanded] = useState(false);

  return (
    <div>
      <button type="button" onClick={() => setExpanded((value) => !value)}>
        {label}: {expanded ? "Hide details" : "Show details"}
      </button>
      {expanded ? <p>Workflow details are visible.</p> : null}
    </div>
  );
}

Notice the updater form, setExpanded((value) => !value), rather than setExpanded(!expanded) — computing the next value from the previous one instead of from the outer expanded variable directly. Not strictly required for this one component, but it's the exact pattern Lesson 36 needs to explain later, so it's worth the habit now rather than unlearning it then.

Check your own attempt against four things: (1) a real <button type="button">, never a <span>; (2) expanded starts as false; (3) each click flips it; (4) the rendered output — the button's own text, and whether the paragraph exists at all — actually looks different between true and false, not just in your head.

If what schedules a re-render still feels like a black box — especially "why doesn't mutating a state object directly work" — ask here before Lesson 33. Server Components (next lesson) only make sense once you're confident this loop is a client-side, in-the-browser mechanism.
