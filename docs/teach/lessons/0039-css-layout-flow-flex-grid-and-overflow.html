CSS Layout: Flow, Flex, Grid, and Overflow

Every lesson so far assumed you could already read a Tailwind class list. This one assumes nothing — it's the CSS prerequisite Tutorial 11 had to half-teach mid-debug. The throughline: for any real layout in this repo, you should be able to identify every size constraint and overflow boundary in an element's ancestor chain, and name exactly which region and axis each one owns, before you ever touch a class.

Primary sources

MDN's "Introduction to CSS layout" for normal flow and the box model; MDN's "Flexbox" for the main/cross axis; MDN's "Grids" for tracks and the fr unit; MDN's min-width reference specifically for the automatic-minimum-size rule the "Intrinsic minimum size" section below depends on; MDN's overflow reference for the precise difference between hidden and clip the "Overflow ownership" section draws on; and Tailwind's "Responsive Design" page for the mobile-first breakpoint convention.

Why this matters

All three mission drivers apply: you can explain in an interview which element owns height and which region owns scrolling for a given axis in a layout you're looking at for the first time; normal flow, flex, grid, and overflow ownership are the same four mechanisms in every CSS codebase, Tailwind or not; and diagnosing a layout bug in this repo by naming the responsible ancestor and axis is faster and safer than guessing at classes until a screenshot looks right.

Normal flow and the box model, fast

Without any layout CSS at all, block-level elements — div, p, section, and similar — stack top to bottom, each one as wide as its parent allows. This is normal flow, the default every other layout method in this lesson deliberately overrides. Every element also has a box model: content, then padding, then a border, then margin, nested like layers. This repo's Tailwind setup uses box-sizing: border-box globally (Tailwind's Preflight reset), so padding and border are included inside whatever width or height you set — a w-64 element with padding doesn't grow past 16rem, the content area just gets narrower to make room:

components/ui/Card.tsx, line 14

"rounded-xl border border-border bg-surface p-6 shadow-sm"

p-6 is 1.5rem of padding on all four sides, inside a 1px border, inside whatever width the caller's own layout gives this Card. Nothing here positions the card relative to anything else — that's normal flow's job, and it's already handling it by default.

Flex: two axes, traced through one real shell

Flexbox lays child elements out along one main axis and lets them stretch or compress along the perpendicular cross axis. flex-row (the default) makes the main axis horizontal; flex-col makes it vertical. AppShell.tsx uses both, nested:

components/layout/AppShell.tsx, lines 55–58, 73–77 (the mobile drawer's JSX, lines 59–72, is omitted — Lesson 40's territory, not layout)

<div className="flex h-screen overflow-hidden bg-background">
  <div className="hidden lg:block">
    <SidebarNav />
  </div>

  <div className="flex h-full flex-1 flex-col overflow-hidden">
    <TopHeader isDrawerOpen={isDrawerOpen} onToggle={() => setIsDrawerOpen((value) => !value)} />
    <main className="flex-1 overflow-y-auto">{children}</main>
  </div>
</div>

The outer div is flex (row by default) — its main axis is horizontal, so the sidebar and the content column sit side by side; its cross axis is vertical. The inner div is flex-col — for its children, the main axis flips to vertical, stacking TopHeader above main. flex-1 means "grow to fill whatever space is left along the main axis": the content column claims whatever horizontal space the sidebar doesn't use, and main claims whatever vertical space TopHeader doesn't use. TopHeader.tsx itself is h-14 shrink-0 — a fixed height, and an explicit refusal to compress even if space runs short, which is exactly why main is the one that absorbs whatever space remains instead.

Grid: explicit tracks instead of one axis

Where flex distributes space along a single axis item-by-item, CSS Grid lets you declare explicit tracks up front — named column and row sizes the whole layout is built around:

components/landing/ValueSection.tsx, line 9

"grid grid-cols-1 gap-8 md:grid-cols-[1fr_2fr] md:gap-16"

At the default (mobile) size, grid-cols-1 is one track — everything stacks in a single column. At the md: breakpoint and up, grid-cols-[1fr_2fr] declares exactly two tracks using the fr ("fraction") unit: the parent weights them 1:2. That's the heading/intro block on the left and the value-list block on the right — but fr distributes available space, not all space, so this is a target ratio, not a content-independent guarantee. If either block's own content needed more than its share (a long unbreakable word, for instance), that intrinsic minimum would be satisfied first, and only what's left would actually split 1:2. Here neither block forces that, so the split lands exactly on ratio — but the mechanism is "weight the remaining space," not "ignore content entirely."

Intrinsic minimum size: why a flex child won't just shrink

A flex item's default minimum size along the main axis isn't zero — by default it won't shrink below its own content's natural size, even if flex-1 or a sibling's shrink-0 is actively squeezing it. UploadPanel.tsx has both the problem and two different real fixes for it:

components/workflow/UploadPanel.tsx, lines 68–71

<span className="truncate">{displayedFileName ?? "Choose a file…"}</span>
<span className="shrink-0 rounded-md bg-accent px-3 py-1 text-xs font-medium text-text-on-accent">
  Browse
</span>

components/workflow/UploadPanel.tsx, lines 87–94

<span className="min-w-0 text-xs text-text-muted">Need a starting point? Use the sample file.</span>
<a
  href={getSampleFileUrl(sampleFileName)}
  download
  className={cn(buttonVariants({ variant: "dark" }), "shrink-0 whitespace-nowrap")}
>
  Sample file
</a>

Both pairs sit in a flex items-center justify-between row with a long text label on one side and a fixed-size button on the other. The buttons are shrink-0 — an explicit refusal to compress, the same class TopHeader used above. The labels take two different real routes to the opposite problem — being allowed to shrink below their own text's full width instead of pushing the button out of the row. The second label sets min-w-0 directly, overriding the default. The first doesn't need to: MDN's own min-width reference documents that once a flex item becomes a scroll container — an element whose overflow would establish one, which overflow: hidden does — its automatic minimum size becomes zero instead of content-based. truncate already sets overflow: hidden as part of what it does, so that span gets the same effect as a side benefit, not a separate rule. (This isn't true of every non-visible overflow value — overflow: clip is a deliberate exception, covered below.)

Overflow ownership: which region scrolls, on which axis

"Overflow" isn't one setting, and a page rarely has just one scroll owner — the actual question is always "for this region and this axis, which element owns it?" overflow-hidden clips content with no visible scrollbar and no ordinary click-and-drag or scroll-wheel access, but it's still a scroll container: tabbing to a clipped focusable element scrolls it into view, and code can still scroll it programmatically (scrollTo(), setting scrollTop). overflow: clip — not used anywhere in this repo, but worth knowing exists — is the one that's genuinely unreachable: no scroll container at all, no programmatic scroll, and a focused descendant stays exactly where it is instead of scrolling into view. overflow-y-auto/overflow-x-auto add a visible scrollbar along one specific axis, only if that axis's content actually exceeds its box. This repo layers several of these at once, at different scopes, inside the same page:

The workspace shell has two independent internal scroll regions, not one. AppShell's outer div is h-screen overflow-hidden — pinned to exactly the viewport height, clipping anything that doesn't fit inside it. Two different descendants each own their own scrolling within that fixed frame — <main>, from the excerpt above, for the workspace content, and:

components/layout/SidebarNav.tsx, line 24

<nav className="flex h-full w-60 shrink-0 flex-col gap-1 overflow-y-auto bg-surface-inverse p-4">

— for the navigation list, independently, if it ever grows taller than the viewport. The sidebar region stays fixed in place — it doesn't slide off-screen when <main> scrolls — but its own contents can still scroll on their own axis, inside that fixed region. The rule was never "one scrollbar for the whole shell"; it's "every region gets exactly the scroll behavior its own content needs," identified one ancestor chain at a time.

The public site scrolls as a page. app/(public)/layout.tsx takes the opposite approach entirely:

app/(public)/layout.tsx, line 8

<div className="flex min-h-screen flex-col bg-background">

min-h-screen, not h-screen — a floor, not a ceiling. This container is free to grow taller than the viewport, and nothing here has overflow-hidden or an internal scroll region, so the browser's own page scrollbar handles it. Marketing/landing pages behave like an ordinary webpage; workspace pages behave like an app shell — same flex-col structure, opposite overflow strategy, because they're different kinds of pages.

A table owns its own horizontal scroll, nested inside both. Table.tsx wraps every table in two layers, each with a different job:

components/ui/Table.tsx, lines 14–19

<div className="rounded-xl border border-border overflow-hidden">
  <div className="overflow-x-auto">
    <table className={cn("w-full border-collapse text-xs", className)} {...props} />
  </div>
</div>

The outer div's overflow-hidden isn't there to scroll anything — a table with square cell corners would otherwise poke past the outer rounded-xl corners, so this just clips them clean. The actual scroll container is the inner div, overflow-x-auto, wrapping only the <table> itself — a wide table with many columns scrolls sideways in its own bounded box, completely independent of whatever vertical scrolling <main> is doing around it. Two axes, two separate owners, at two different levels of nesting.

Mobile-first breakpoints

Tailwind's breakpoint prefixes (sm:, md:, lg:, ...) read mobile-first: an unprefixed class applies at every screen size as the baseline, and a prefixed class overrides it starting at that breakpoint and up — never the other way around. Both examples already shown in this lesson are mobile-first in exactly this sense: AppShell's sidebar is hidden lg:block — invisible by default, shown only once the viewport is at least the lg breakpoint wide, which is why a mobile drawer exists at all. ValueSection's grid-cols-1 md:grid-cols-[1fr_2fr] is the same pattern applied to a track layout instead of visibility: one column below md, two above it.

Correct framing

"Before I change a class, I can point to which ancestor owns this element's height and which descendant, if any, owns its scrolling — then I know whether I'm changing the right layer."

A framing to avoid

"I'll just add a width or height until this screenshot looks right." (Without knowing which ancestor owns height and which owns scrolling, a fix that repairs one viewport can easily break another, or create a second, nested scrollbar nobody asked for. This exact shell has its own reserved historical retrospective — the deferred Phase 9.1 tutorial — for that specific story; this lesson deliberately stops at the general mechanics needed to understand it later, not the story itself.)

Retrieval check

In AppShell's inner flex-col div, why does <main> absorb the leftover space instead of TopHeader?

Why does the first UploadPanel label truncate correctly without an explicit min-w-0?

Why does Table.tsx wrap its table in two nested divs instead of one?

Tiny exercise (7 minutes)

Exercise — predict, then watch it break safely

Before touching anything: predict, out loud, what will visually happen to a workspace page with a lot of table rows (Order Validation, after running sample data, works well) if <main>'s overflow-y-auto disappeared. Specifically: would the whole browser window grow a scrollbar instead, or would something else happen? Base your prediction on which ancestor in AppShell.tsx already has overflow-hidden.

Then make the edit for real — in components/layout/AppShell.tsx, change <main className="flex-1 overflow-y-auto"> to <main className="flex-1"> — save, and look at a long workspace page in the running dev app. You should see the bottom rows simply missing, with no scrollbar capable of reaching the clipped main content: the outer shell's h-screen overflow-hidden (and the content column's own overflow-hidden) still clip everything past the viewport, and nothing else in main's own ancestor chain has overflow-y-auto to take over. The sidebar, notice, is unaffected either way — it's a sibling branch with its own independent overflow-y-auto, not part of main's chain at all, so this edit doesn't touch its scrolling one way or the other. Undo the edit with your editor's Undo (Cmd/Ctrl+Z) — not git checkout --, which would discard any other unstaged changes in the file too — and confirm the page scrolls normally again.

If your prediction was "main's content gets clipped with no scrollbar reaching it," not "the whole page scrolls instead," you've internalized overflow ownership. If you predicted the page-scroll outcome, that's a very natural guess — it's just not what an ancestor with overflow-hidden allows — and it's worth asking about here before Lesson 40, which builds directly on being able to name a component's accessible behavior the same precise way this lesson named its layout.
