Track 5 — Optional Reinforcement · 8 of 9

Frontend Verification Layers

This repository currently uses four verification mechanisms: two repository scripts (npm run typecheck, npm run lint), a production build (npm run build), and a manual curl health diagnostic documented in README.md for confirming the backend is up. None of them opens a real browser. Two of this project's worst UI bugs — a hydration mismatch and a pointer-events dead zone, both from Tutorial 11 — passed all three automated checks anyway. This lesson is about what each mechanism actually proves, what none of them can tell you, and how to pick the cheapest layer that would actually observe a given failure, without installing anything new or pretending this repo has a frontend test suite it doesn't.

Primary sources

The installed Next.js docs' own "Testing" overview, for the Unit/Component/Integration/E2E/Snapshot vocabulary. The Playwright setup guide, for what E2E automation actually is. The TypeScript config reference and ESLint plugin reference, for two precise, version-specific facts about what next build does and doesn't check in this repo's installed Next.js 16. React's own hydrateRoot reference, for what a hydration mismatch actually violates. Playwright's Locator.hover() reference, for what its actionability checks and default target point actually do.

Why this matters

All three mission drivers apply: in an interview, naming exactly which check would (and wouldn't) catch a described bug is a sharper signal than "we have good test coverage"; the layered vocabulary — unit, component, integration, E2E, and everything below it — is identical on any frontend stack; and before adding a new check or reaching for a new tool on this repo, it's worth knowing precisely what its existing four mechanisms already cover and where the real gaps are.

Four mechanisms this repo already uses, and what each one actually proves

All four exist today, verified directly against this repo's own package.json and README.md — none of them is hypothetical. The first three are repository scripts; the fourth is a manual diagnostic a developer runs by hand, not something a workflow or completion gate currently guarantees runs for every change:

package.json, lines 7, 9–10

"build": "next build",
"lint": "eslint",
"typecheck": "tsc --noEmit",

npm run typecheck (tsc --noEmit) checks the source against its declared static types and compiler configuration. It does not prove that unvalidated runtime data — a value from an API response, a file upload, `localStorage` — actually matches those declarations at runtime; it only proves the code is internally consistent with the types it claims. components/ui/Badge.tsx is a real, currently-necessary example of a defect this check can catch:

components/ui/Badge.tsx, lines 39–43

export function Badge({ className, tone, dot = true, children, ...props }: BadgeProps) {
  const resolvedTone = tone ?? "neutral";
  return (
    <span className={cn(badgeVariants({ tone: resolvedTone }), className)} {...props}>
      {dot ? <span className={cn("h-1.5 w-1.5 shrink-0 rounded-full", DOT_TONE_CLASSES[resolvedTone])} /> : null}

tone's type (from cva's VariantProps) includes null/undefined, since it's an optional prop, but DOT_TONE_CLASSES is a plain object keyed only by the five real tone strings. Indexing it with tone directly, instead of the resolved fallback, is a real, checkable type error — this lesson's own exercise below reproduces it and shows the exact compiler output.

npm run lint (plain eslint, no arguments) surfaces findings from the configured rule set — but a successful exit only proves there were no lint errors. ESLint's own CLI returns exit code 0 whenever there's at least one error or (if --max-warnings is set) more warnings than that limit allows; this repo's script sets neither, so warnings alone never fail the command. Lesson 38's own exercise already produced a real one, deleting a useMemo dependency — and it's a warning, not an error:

Real npm run lint output, quoted from Lesson 38's own exercise

78:6  warning  React Hook useMemo has a missing dependency: 'sort'. Either include it or remove the dependency array  react-hooks/exhaustive-deps

Reading that word again matters: npm run lint would exit successfully with this warning present. Only actually reading the output (or adding --max-warnings=0, which this repo's script doesn't) would have surfaced it as a failure.

npm run build (next build) proves production compilation succeeds and exercises whatever rendering work the build itself performs. In this repo specifically, every current route is statically prerendered — a real npm run build run confirms it, printing ○ (Static) prerendered as static content against all six app routes plus /_not-found — so a successful build here does exercise each route's actual build-time render path, not just its compilation. That's still narrower than "every route can render," though: nothing about a runtime request, a real user's state, or a dynamic route path is exercised by a build against pages that take no runtime input. This repo hit a real, build-time-only defect during Tutorial 10 (Part 3): every workflow page was still a plain Server Component the first time it built a DataTable columns array carrying render functions, and the build failed with a real, verbatim error:

Real build-time error, quoted from docs/tutorials/10-reusable-ui-components-static-pages/README.md

Error: Functions cannot be passed directly to Client Components unless you explicitly
expose it by marking it with "use server".

A function prop crossing a Server-to-Client boundary isn't a type error — render: (row: T) => ReactNode typechecks fine on its own — it's a serialization rule Next.js enforces as part of the build's own render step, not something tsc alone would ever flag in isolation.

One precise, version-specific fact worth knowing before assuming these three checks overlap more than they do: this repo's installed Next.js 16 fails next build on TypeScript errors by default ("Next.js fails your production build (next build) when TypeScript errors are present in your project"), so build and typecheck genuinely overlap here — but next build no longer runs ESLint at all. The docs are explicit: "Starting with Next.js 16, next lint is removed," and the old eslint option that used to fail a build on lint errors "is no longer needed." Concretely: Lesson 38's missing-dependency warning above would not have failed npm run build in this repo — only npm run lint, and only if someone actually read its output.

The curl health diagnostic, documented in this repo's own README.md as a manual step to run "before opening the app," proves something narrower still: the backend process accepts an HTTP request and returns the expected response shape. It does not prove a frontend route renders, that any actual workflow endpoint behaves correctly, that the frontend can successfully call the backend, or that a user can complete a workflow — only that one specific process is up and answering:

README.md, lines 46–47

curl -fsS --max-time 3 http://127.0.0.1:8000/health
# expected: {"status":"ok"}

Two real bugs a fully green run of all three automated checks never caught

Both bugs below passed typecheck, lint, and build at the time — Tutorial 11 documents both in full. Neither is hypothetical, and both are already fixed in the current file. (The curl health diagnostic isn't relevant here at all — it checks the backend process, not a frontend chart.)

The hydration mismatch. Adding hover/focus tooltips to the donut chart originally wrote each ring's accessible title as ordinary multi-line JSX (<title>{"{segment.label}"}: {"{formatNumber(segment.value)}"} ({"{percent}"}%)</title>, split across lines). Loading /dashboard in a real browser surfaced a genuine React error: "Hydration failed because the server rendered HTML didn't match the client." React's own docs are explicit about the expectation this violates: "hydrateRoot() expects the rendered content to be identical with the server-rendered content." The line breaks and indentation between the embedded expressions became part of the text node's actual structure, and the server-generated DOM and the client's initial render disagreed about that text structure inside the SVG <title> — a mismatch no type system or linter is positioned to catch, since it's about runtime rendered output, not source code. The fix, still in the current file:

components/charts/DonutBreakdownChart.tsx, lines 72, 92

const segmentTitle = `${segment.label}: ${formatNumber(segment.value)} (${percent}%)`;
// ...
<title>{segmentTitle}</title>

The pointer-events trap. This is actually two distinct findings, worth keeping separate rather than telling as one story. First, the real defect: the donut's center "total" label sat in an inset-0 overlay that stretched across the entire bounding box — not just the hollow center, but the full square area the visible ring occupies too — with no pointer-events override to let clicks or hover pass through to the ring underneath. This was a real bug for real users, not a test artifact: anyone hovering anywhere in that box in an actual browser hit the same dead zone. A scripted Playwright locator.hover() call against a ring segment surfaced it directly, because hover() "wait[s] for actionability checks on the element" before it acts — one of which detects whether the target point is obstructed by another element intercepting the pointer event. The call timed out, and the tool's own diagnostic trace named the exact intercepting element: the overlay. That's a property of Playwright's actionability-checking mechanism, not a claim about automated tests being more "thorough" than a human in general. The fix, still on line 99 of the current file:

components/charts/DonutBreakdownChart.tsx, line 99

<div className="pointer-events-none absolute inset-0 flex flex-col items-center justify-center">

Second, a separate problem that only appeared after that fix, and belongs to the test script rather than the app: hover()'s default behavior is to "hover over the center of the element," and for a hollow ring (an annulus), that geometric center is the empty hole — not any point on the visible colored stroke. Once the overlay stopped intercepting events, hovering the ring worked fine for a real user, but the test itself still couldn't meaningfully assert anything about the ring segment's own hover behavior by calling .hover() at its default point. The actual fix for the test was switching to .focus(), which exercises the onFocus handler directly and sidesteps the "where exactly does this land" question entirely — a test-design fix, unrelated to how the overlay defect itself was originally found.

The formal vocabulary, and where this repo's real checks actually sit on it

The installed Next.js testing guide names five categories, independent of any specific tool:

Quoted directly from the installed Next.js docs' "Testing" overview

Unit Testing involves testing individual units (or blocks of code) in isolation. In React, a
unit can be a single function, hook, or component.

Component Testing is a more focused version of unit testing where the primary subject of the
tests is React components. This may involve testing how components are rendered, their
interaction with props, and their behavior in response to user events.

Integration Testing involves testing how multiple units work together. This can be a
combination of components, hooks, and functions.

End-to-End (E2E) Testing involves testing user flows in an environment that simulates real
user scenarios, like the browser. This means testing specific tasks (e.g. signup flow) in a
production-like environment.

Snapshot Testing involves capturing the rendered output of a component and saving it to a
snapshot file. When tests run, the current rendered output of the component is compared
against the saved snapshot. Changes in the snapshot are used to indicate unexpected changes
in behavior.

This repository has no committed, repeatable frontend test suite in any of these five categories — verified directly, not assumed: no *.test.* or *.spec.* file exists anywhere under app/, components/, or lib/, and package.json defines no test script at all. That's a real, current gap, not something to gloss over — the Python core has a genuine automated test layer (uv run pytest, over 200 passing tests), but nothing analogous exists for the TypeScript/React side yet. That's compatible with a separate fact, though: this repository has performed real frontend testing before, twice — it just doesn't retain any of it as automated regression coverage. What this repo has used instead, twice, is a deliberately temporary stand-in for E2E: install Playwright without saving it to package.json (npm install --no-save playwright), run a throwaway driver script against a real running dev server, then remove it — docs/tutorials/11-portfolio-ui-polish/README.md (Part 8) documents this exact pattern, including the check that git diff --stat package.json shows no trace afterward. Playwright itself is squarely an E2E tool by the docs' own description: "a testing framework that lets you automate Chromium, Firefox, and WebKit with a single API," used "to write End-to-End (E2E) testing." One-off and scripted is still real coverage for the one bug it's aimed at — it just isn't a committed, repeatable suite guarding against regressions later.

Lesson 40's own tab-through exercise sits outside this whole taxonomy on a different axis entirely: a human, not a script, walking an E2E-shaped path by hand. It found real gaps (a focus-invisible file input, an unnamed div in the tab order) that none of this repo's three automated checks were ever positioned to catch — but it isn't repeatable the way a committed test is, since nothing re-runs it automatically before the next change ships.

Choosing the smallest layer that could actually observe the failure

A short, honest decision list, built entirely from the real examples above:

Ask, in order, before reaching for the next layer

Could a declared static type be inconsistent?       -> typecheck
Could a configured source pattern be problematic?   -> lint
Could production compilation/build rendering fail?  -> build
Could the backend process be unavailable?           -> curl /health
Could an API behavior or contract be wrong?         -> call the real endpoint, or an integration test
Could rendered interaction or accessibility fail?   -> a real browser, manual or automated
Could a complete user workflow fail?                -> an E2E path through the running system

This keeps the "smallest layer capable of observing the failure" principle intact while splitting what earlier felt like one "process boundary" step into two genuinely different questions: whether a service is even reachable at all, versus whether what it does once reached is correct. The two named failure modes this lesson is built around: treating a fully green typecheck/lint/build run as proof of accessibility or interactivity — the donut chart passed all three while genuinely broken twice over — and reaching for a heavy scripted browser test to catch something a one-line type or lint rule would already catch, more cheaply and more reliably, every single time it actually runs.

Correct framing

"A green build tells me production compilation succeeded and this repo's statically prerendered routes rendered once at build time without throwing. It tells me nothing about hydration, focus order, or whether a mouse can actually reach an element — those need an actual browser, scripted or by hand."

A framing to avoid

"Typecheck, lint, and build are all green, so the UI works." (Both real bugs in this lesson are the direct counter-example — all three passed, twice, on genuinely broken interactive behavior.)

Tiny exercise (8 minutes)

Exercise — reproduce Badge.tsx's type-level bug in a disposable scratch file

Predict first. Of this repo's four verification mechanisms (typecheck, lint, build, curl health), which would flag DOT_TONE_CLASSES[tone] if the tone ?? "neutral" fallback on components/ui/Badge.tsx line 40 were removed and the raw tone prop were indexed directly? Write down your answer before running anything — and don't edit the real file to check it.

Verify, without touching any application file. Create a scratch file anywhere outside this repo (a temp directory is fine) named badge-repro.ts:

type Tone = "success" | "warning" | "danger" | "info" | "neutral";
const DOT_TONE_CLASSES: Record<Tone, string> = {
  success: "bg-success", warning: "bg-warning", danger: "bg-danger",
  info: "bg-info", neutral: "bg-text-muted",
};
declare const tone: Tone | null | undefined;

const broken = DOT_TONE_CLASSES[tone];          // the bug
const resolvedTone = tone ?? "neutral";
const fixed = DOT_TONE_CLASSES[resolvedTone];   // the real fix

Run it with this repo's own installed TypeScript compiler, not whatever npx might resolve or download on its own — from the repo root:

./node_modules/.bin/tsc --noEmit --strict /absolute/path/to/badge-repro.ts

(Or, equivalently: npx --no-install tsc --noEmit --strict /absolute/path/to/badge-repro.ts — the --no-install flag refuses to fetch a different version, so either command guarantees you're checking against the exact compiler this repo already uses.) The real output, verified against this repo's own installed compiler while writing this lesson:

.../badge-repro.ts(8,33): error TS2538: Type 'null' cannot be used as an index type.
.../badge-repro.ts(8,33): error TS2538: Type 'undefined' cannot be used as an index type.

Compare against your prediction. The whole loop is one scratch file and one command — nothing in this repo changes, and deleting the scratch file afterward leaves zero trace.

If you can look at a described UI change and name, in order, the cheapest check that would actually observe its specific failure mode — not just "we should add tests" — this lesson has landed. It deliberately doesn't install Vitest, Jest, Cypress, or a committed Playwright suite; that's a real, named gap in this repo today, not a decision this lesson makes for you.
