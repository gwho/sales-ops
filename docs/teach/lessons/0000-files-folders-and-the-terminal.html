Before Track 1 — The Absolute Basics

Files, Folders, and the Terminal

The layer underneath everything else in this course. Lesson 1 talks about "folders" and tells you to run a command in a terminal without explaining either — this lesson exists so those words already mean something concrete by the time you get there.

Primary source

Software Carpentry — "The Unix Shell: Introducing the Shell" and its next episode, "Navigating Files and Directories" — a free, official lesson written specifically for people who have never used a command line before. This page only covers the slice you need right now; those two episodes are the right next read if you want more practice typing real commands.

A file is one thing; a folder holds things

A file is a single named piece of content on a computer — text, code, a spreadsheet, an image. A folder (also called a directory — same thing, two names) doesn't hold content itself; it holds other files and folders, giving them a shared home and a shared name.

This repo, sales-ops, is itself just one folder. Opened up, it looks like this (trimmed to what matters right now):

sales-ops/              ← the repo: one folder, holding everything below
  src/                   ← a folder, holding Python code files
    excel_io.py          ← one file, inside the src/ folder
    contracts.py         ← another file, inside the same folder
  tests/                 ← a different folder, holding test files
    test_excel_io.py
  docs/
    tutorials/
      01-python-foundation/
        README.md        ← one file, three folders deep

Nothing here is special to programming — it's the same folders-inside-folders idea as any file browser you've used. The only new thing is that from here on, you'll navigate this structure by typing instead of clicking, which the rest of this lesson covers.

A path is directions to a file

A path is just written-out directions from one place in the folder structure to another, with a slash (/) between each step. Reading the tree above, the path to excel_io.py from the repo root is:

src/excel_io.py

Read left to right: "go into the src folder, then find the file excel_io.py." A path to something three folders deep just keeps chaining steps: docs/tutorials/01-python-foundation/README.md.

One more piece of path syntax you'll see constantly once you start clicking links inside the tutorials — .. means "go up one folder level" instead of down. Here's a real link from Tutorial 3, which lives at docs/tutorials/03-order-validation-core/README.md:

[`src/order_validation.py`](../../../src/order_validation.py)

That path starts three folders deep (docs/tutorials/03-order-validation-core/) and needs to reach src/order_validation.py, which lives off the repo root instead — a completely different branch of the tree. Each ../ climbs up one level (docs/tutorials/03-order-validation-core/ → docs/tutorials/ → docs/ → the repo root), and only then does src/order_validation.py make sense as "go back down into src/ from here." Three steps up, because the link starts three folders deep.

What a file extension tells you

The letters after the final . in a filename are the extension — a label saying what kind of content the file holds, and often which program or tool is meant to open it. You'll meet exactly these six in this repo:

ExtensionMeansExample in this repo
.pyPython codesrc/excel_io.py
.mdMarkdown — plain text with light formatting, human-readable even unrendereddocs/tutorials/01-python-foundation/README.md
.htmlA web page — like this lesson itselfdocs/teach/lessons/0000-files-folders-and-the-terminal.html
.tomlA configuration file (settings, not code that runs)pyproject.toml
.lockAn exact, frozen record of installed versionsuv.lock
.xlsxAn Excel spreadsheetsample_data/sample_orders.xlsx

None of this is a Python rule — it's a general computing convention almost every tool respects. Recognizing a file's job from its extension alone, before opening it, is most of what "reading a repo's structure" actually is.

A terminal is a way to talk to your computer by typing

Every file browser you've used lets you double-click a folder to open it. A terminal (also called a command line or shell) is the same kind of interaction, done entirely by typing: you type a short instruction called a command, press Enter, and the computer carries it out — usually printing some text back as a result.

One command is worth knowing cold before Lesson 1, because its very first exercise uses it: ls (short for "list") prints the names of every file and folder in your current location — wherever your terminal happens to be "standing" right now, which starts out as your repo's root folder if that's where you opened it.

$ ls
CLAUDE.md   context/   docs/   sample_data/   src/   tests/   ...

The $ isn't something you type — it's how documentation (including this lesson) marks "this is a command, typed at the terminal," to distinguish it from ordinary text or output. You only ever type what comes after it.

Why this matters

A command is not magic — it's a short, precise instruction, and most of the ones you'll meet in this course (ls, later uv run pytest) just ask the computer to look at or run something already sitting in the folder structure you now know how to read. The terminal isn't a separate skill from files and paths — it's how you move through them without a mouse.

Tiny exercise (3 minutes)

Open a terminal with this repo as your current location (if you're working alongside an AI coding assistant, it likely already has one open for you — just ask where it is). Run:

ls

Compare what prints against the tree near the top of this lesson — find src, tests, and docs in the output. Then run ls src and confirm excel_io.py and contracts.py both appear — you're reading the same folder structure, just one level deeper, by typing instead of clicking into it.

That's the whole prerequisite. Lesson 1's own exercise (ls docs/plan/phase-1-python-foundation/) is exactly this same command, aimed one path deeper — you now know what every part of it means before you type it.

If "folder" vs. "directory," the .. path syntax, or opening a terminal at all still feels shaky — ask now. Lesson 1 assumes this lesson, not the other way around.
