FormData, UploadFile, and Loader Callbacks

Lesson 31 mapped which endpoint does what. This one rehearses the exact mechanical bridge one workflow request crosses: a browser's FormData key, to a FastAPI parameter name, to a business module's own loader function — three files, two languages, one string that has to match across all of them with nothing enforcing it.

What you're rehearsing

Tutorial 07 Part 1 and Part 4, against the real files: backend/uploads.py, backend/routers/orders.py, and lib/api-client.ts.

The chain, four links

Follow one field, orders_file, across its entire journey:

LinkWhereWhat it is
1app/order-validation/page.tsxformData.set("orders_file", orders) — a plain string key on a browser FormData object
2backend/routers/orders.pyorders_file: Annotated[UploadFile, File()] — a Python parameter whose name must match link 1's key exactly
3backend/uploads.pyread_xlsx_upload(orders_file, "orders file", load_orders) — load_orders passed in as a plain callback, not hardcoded
4src/order_validation.pyload_orders(file_obj) — the actual required-column check, owned entirely by Tutorial 03's module

Nothing links 1 and 2 except that both files were written to agree on the literal string "orders_file". No shared constant, no schema file, no compiler. FastAPI's Annotated[UploadFile, File()] binding works purely by name: it looks at the incoming multipart body for a part named orders_file and binds it to the Python parameter of the same name. If either side's string drifted — a typo, a copy-paste from a different endpoint — the request would still complete a full round trip; the expected field would simply never arrive, and FastAPI's own validation machinery (Lesson 33) would report it as missing.

Multipart boundary failure — a name-adjacent trap, not the name trap itself

A related but different failure lives one layer below the field-name question: never manually set Content-Type: multipart/form-data on a request whose body is a real FormData object. When fetch()'s body is a FormData instance, the browser builds the multipart body and writes the Content-Type header itself — including the exact boundary=... string it chose while constructing that body. Overwrite that header by hand and the real boundary is gone; the server receives a body with no way to tell where one named part ends and the next begins. lib/api-client.ts's postFormData (Tutorial 07 Part 1) never sets Content-Type for exactly this reason — the absence is deliberate, not an oversight.

read_xlsx_upload: a callback, not a copy

Open backend/uploads.py's signature:

def read_xlsx_upload(
    file: UploadFile, label: str, loader: Callable[[object], pd.DataFrame]
) -> pd.DataFrame:

loader is one of load_orders / load_product_master / load_inventory / load_invoices — passed in by whichever router calls it, not owned by backend/uploads.py itself. read_xlsx_upload only ever checks two things on its own: is there a filename at all, and does it end in .xlsx. Every question about which columns are required stays entirely inside whichever load_* function it was handed — the same functions Tutorials 03-05 already tested with zero HTTP machinery involved.

Open backend/routers/payments.py and find its own version of this chain: which FormData key (check app/payment-aging/page.tsx) maps to which Annotated[UploadFile, File()] parameter, and which load_* function from src/payment_aging.py gets passed as loader. Then, on paper only, rename the frontend's FormData key to "invoice_file" (singular) while leaving the backend parameter as invoices_file (plural). Predict, in one sentence, what error a user would actually see — then check your answer against Lesson 33's error-contract material once you've written it down.

Failure mode named: duplicating required-column lists in backend/ instead of delegating to the load_* functions owned by the business modules. The rejected alternative would hardcode a second copy of, say, ORDERS_REQUIRED_COLUMNS inside backend/uploads.py — and a second copy is a copy that can silently drift the moment someone adds a required column to the Python core and forgets the FastAPI-side list even exists.

If "why does this even need a callback instead of four separate functions" still feels unresolved, ask now — Tutorial 07 Part 4's testing-benefit checkpoint answers it directly, and it's a good one to be able to explain unprompted before Lesson 35.
