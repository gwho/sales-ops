Track 2 — Business Rules & Testing Discipline

Business Rules vs. Code Rules

You've finished Track 1 and (per docs/teach/ROADMAP.md) Tutorials 01 and 02 — you can read a contract, a DataFrame, and a test file. Track 2 is about a different skill: reading a whole module as a set of real-world policies, each one proven by a test. This lesson teaches the one distinction everything else in Track 2 builds on.

Primary source

pytest — How to write and report assertions (official docs) — the mechanism this lesson keeps coming back to: an assert is how a business rule gets turned into a checkable fact.

Two kinds of "wrong"

Open src/order_validation.py and tests/test_order_validation.py side by side. Somewhere in there, something can go wrong in two completely different senses:

A business rule violationA programming error
What it meansThe data breaks a real-world policy — a customer name is blank, a quantity is negative.The code is wrong, regardless of what data it receives — a typo'd variable name, an off-by-one.
Who fixes itWhoever owns the data — a sales admin correcting a spreadsheet cell.A developer — the code itself needs to change.
How this repo reports itA structured, business-readable result — see Lesson 9. A crash — a traceback, on purpose, so it gets noticed and fixed.

The whole reason order_validation.py exists is to turn the first kind of "wrong" — a real spreadsheet problem — into something a non-programmer can read and act on, instead of into a crash. Confusing the two is a real design mistake: a module that crashes on a blank cell is treating a routine data problem like a bug in itself; a module that silently swallows an actual code bug as "just bad data" hides something a developer needed to know about.

A test proves a rule on one known example

Open tests/test_order_validation.py and find this:

def test_missing_customer_name_raises_ov001():
    result = validate_orders(_orders_df(_order_row(customer_name="")), _product_master_df())
    errors = _errors_by_code(result, "OV-001")
    assert len(errors) == 1
    assert errors[0]["error_message"] == "Customer name is missing."
    assert errors[0]["severity"] == "Error"
    assert result["valid_orders"] == []

The business rule here — "a customer name must not be blank" — is a sentence a sales admin would recognize instantly, with no Python knowledge at all. This test doesn't prove the rule in general; it proves it on one concrete, known example: build an order row with customer_name="", run it through validate_orders, and check that the result says exactly what the rule promises — one error, the right message, the right severity, and the row excluded from valid_orders. Four separate assert lines, four separate facts, all about the same one example.

Why "on one known example" is the whole point

A rule stated only in English ("names can't be blank") is not yet checkable by a computer. A rule expressed as "given this exact input, the output must be exactly this" is. The gap between those two is what a test closes — and it's why every rule in this repo's three Track 2 specs eventually gets at least one test built the same way this one is: a concrete input, a concrete expected output, an assert connecting them.
