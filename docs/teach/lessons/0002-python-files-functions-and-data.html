Python Files, Functions, and Data

Just enough Python to read src/contracts.py and src/excel_io.py without flinching: what a file is, what a function is, the two data containers you'll see constantly (list, dict), and what those colons and arrows in a function signature mean.

Primary source

The Python Tutorial (official) — written by the language's own maintainers, safe for absolute beginners. This lesson only covers the slice you need right now; the tutorial's "Defining Functions" and "Data Structures" chapters are good next reads once this lesson feels solid.

A file is a module

A .py file is called a module. It's just a text file Python can run top to bottom. Other files pull code out of it with import:

from src.excel_io import load_excel, validate_required_columns

Read this left to right: "from the module src.excel_io, bring in the two names load_excel and validate_required_columns." After this line, this file can call those two functions as if they were written locally. This exact line appears in tests/test_excel_io.py — you'll meet it again in Lesson 3.

A function is a named, reusable recipe

Here's a function, deliberately unrelated to this repo, to isolate the shape:

def greet(name: str, excited: bool = False) -> str:
    if excited:
        return f"HELLO, {name}!"
    return f"Hello, {name}."

Break that first line apart:

  • def — "I'm defining a function." greet is its name.
  • name: str — a parameter called name, and a type hint saying it should be a string. Python won't stop you from breaking this promise, but it tells you (and any tool reading the code) what's expected.
  • excited: bool = False — a second parameter with a default value. Callers can omit it entirely; if they do, it's False.
  • -> str — another type hint, this time for what the function returns.

Calling greet("Ada") returns "Hello, Ada.". Calling greet("Ada", excited=True) returns "HELLO, Ada!". Nothing runs until you call it — a def block on its own just teaches Python the recipe.

Now look at a real signature from src/excel_io.py:

def load_excel(file, sheet_name: int | str = 0) -> pd.DataFrame:
    """Load an Excel file (path or file-like object) into a DataFrame."""
    return pd.read_excel(file, sheet_name=sheet_name)

int | str is a type hint meaning "an integer or a string" — you'll see this exact shape again in Lesson 4. The triple-quoted line right under def is a docstring: a description of what the function does, readable by humans and by tools, that runs as documentation, not as logic.

Two containers you'll see everywhere: list and dict

A list is an ordered sequence, written with square brackets:

required_columns = ["order_id", "sku", "quantity"]

A dict (dictionary) maps names to values, written with curly braces and colons:

order = {"order_id": "SO-2026-001", "quantity": 3, "priority": "High"}

You read a value back out by its key: order["quantity"] gives you 3. A dict is how this repo represents "one record with named fields" — an order, a row, a result — everywhere before it's written to disk or sent over the network. Lesson 4 builds directly on this: TypedDict is nothing more than a dict with its expected keys and types written down in advance.

Both containers nest freely. A list of dicts — [{"sku": "A"}, {"sku": "B"}] — is exactly the shape src/sample_data.py uses to build a table's worth of rows, one dict per row, before Lesson 5 turns that into a real table.

A class, just enough to recognize one

You'll see class exactly once in Track 1, in src/excel_io.py:

class MissingColumnsError(Exception):
    ...

Read this as: "MissingColumnsError is a new, custom kind of error, built out of Python's built-in Exception." You don't need to know how to write a class yet — just recognize that this line is defining a specific, nameable failure (missing spreadsheet columns) instead of leaving the code to fail with a generic, unhelpful error. That's the whole idea for now; the tutorial revisits this specific class in depth.

Tiny exercise (3 minutes)

Open a Python shell — run uv run python in the repo root if you have it set up, or any Python REPL — and type this function, then call it twice with different arguments:

def describe_order(order_id: str, quantity: int = 1) -> str:
    return f"{order_id}: {quantity} unit(s)"

print(describe_order("SO-2026-001"))
print(describe_order("SO-2026-002", quantity=5))

Predict both outputs before you run it. If you don't have Python available right now, just write the two predicted output strings down — that's the exercise.
