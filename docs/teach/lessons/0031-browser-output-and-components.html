Track 5 — UI Components & Next.js

What a Browser Renders, and What a Component Is

Every lesson so far has stayed on the Python side of this repo. Track 5 is where the frontend starts — and before anything about React or Next.js makes sense, three older, framework-free ideas have to be solid: what HTML, CSS, and JavaScript each actually do, and what a "component" means as a concept, independent of any specific tool that implements it.

Primary source

MDN, "Your first website" and "Structuring content with HTML" — the standard, vendor-neutral reference for exactly this split. Read the first if you've never opened an HTML file before; skim the second if you have.

Three jobs, three languages

A browser turns a text file into a picture on screen using three separate technologies. Each has a primary responsibility — not an exclusive one, as the qualifications right after the table show:

LanguageJobAnswers the question
HTMLStructureWhat is this piece of content — a heading, a list, a button, a table?
CSSPresentationHow should it look — color, spacing, size, layout?
JavaScriptBehaviorWhat should happen when the user does something — click, type, scroll?

None of the three is actually confined to its own column, though. HTML elements bring some real behavior of their own, for free — a real <button>, <a>, form control, or <details> element already responds to clicks, focus, and keyboard input before any JavaScript runs. CSS can respond to state through selectors like :hover and :focus-visible and through media queries, and can drive its own animations, with zero JavaScript involved. JavaScript, for its part, routinely reaches into the other two columns — creating and removing HTML structure, changing classes and inline styles. And a browser can render a genuinely useful page with no JavaScript running at all. This split isn't specific to React or Next.js — it's the browser's own division of labor, and it predates every JavaScript framework by decades — but it's a map of where each technology's strength lies, not a wall between them. A React component still ultimately produces HTML, still gets its look from CSS, and still runs JavaScript for behavior; React just changes how those three things get written and combined, not what each one is responsible for.

A component, before any framework exists

A component is a named, reusable UI unit with a defined responsibility and inputs. It may combine structure, presentation, and behavior — a button that shows a label and responds to a click bundles all three, and is a component whether it's written as a hand-rolled HTML/CSS/JS snippet, a React function, or a Next.js file — but it doesn't need all three to count. A component that only renders structure and picks its own presentation, with no user-triggered behavior at all, is still a real component; you'll meet a real one below.

A component without React — plain HTML/CSS/JS, genuinely reusable

<!-- structure, produced by a function, not hardcoded once -->
<script>
function createStatusBadge(label, tone) {
  const el = document.createElement("span");
  el.className = `badge badge--${tone}`;
  el.textContent = label;
  return el;
}
</script>

/* presentation — real project tokens, no hardcoded hex */
.badge { padding: 0.25rem 0.6rem; border-radius: 999px; font-size: 0.85rem; }
.badge--warning { background: hsl(var(--warning-subtle)); color: hsl(var(--warning)); }
.badge--success { background: hsl(var(--success-subtle)); color: hsl(var(--success)); }

<!-- usage: label and tone are real inputs, called as many times as needed -->
<script>
document.querySelector("#demo").append(
  createStatusBadge("Backordered", "warning"),
  createStatusBadge("Fully Allocated", "success"),
);
</script>

Nothing here is React, and this one is genuinely reusable — unlike a single hardcoded id, createStatusBadge takes a label and a tone as real inputs and returns a fresh element every time it's called, so calling it twice never collides the way one fixed ID would. It's a component in every meaningful sense: one name, one defined responsibility, callable with different inputs wherever that same piece of UI is needed. It's also deliberately display-only — no click handler, no behavior at all — which sets up an honest comparison with this project's real components/workflow/StatusBadge.tsx (built in Tutorial 10): both are named, reusable, take a label/tone as input, and render structure plus presentation — and neither one has any user-triggered behavior at all. That's not a gap in either one. A status label isn't a button, and shouldn't have to act like one to count as a component.

Correct framing

"A component is React's way of organizing structure, presentation, and behavior together — the idea predates React."

A framing to avoid

"A component is a React thing." (This makes L5.2 harder — you'll be learning React's syntax for an idea you already understand, not the idea itself.)

Tiny exercise (5 minutes)

Exercise — predict, inspect, then check mechanically

Predict first. Open app/globals.css and find any one CSS rule. Say out loud which of the three jobs (structure, presentation, behavior) it belongs to, and why it couldn't belong to either of the other two.

Then inspect. Open components/workflow/StatusBadge.tsx (you don't need to understand the React syntax yet) and decide, for the whole file: does it contain structure, a presentation mapping, real user behavior, or some combination? Write down your answer before revealing.

Reveal

StatusBadge renders structure (it returns a Badge element wrapping the status text) and selects presentation through its tone input — a caller picks "success"/"warning"/"danger"/etc., and that's what decides the color — but it defines no user event behavior at all: nothing responds to a click, a key press, or any other thing a user does.

Confirm mechanically. Search the file for onClick, onChange, onKeyDown, or any other event prop. There are none — the same real, checkable absence the reveal above just described in words, not something to take on faith. You're building a landmark for L5.2, not analyzing the file in full yet.

If any of the three jobs feels blurry — for instance, whether inline event handling counts as "behavior" even when it's one line inside an HTML-looking tag — ask here before L5.2. That exact blurriness is what JSX (L5.2) deliberately reintroduces on purpose, and it helps to notice the seam before React hides it.
