Track 5 — UI Components & Next.js

What a Browser Renders, and What a Component Is

Every lesson so far has stayed on the Python side of this repo. Track 5 is where the frontend starts — and before anything about React or Next.js makes sense, three older, framework-free ideas have to be solid: what HTML, CSS, and JavaScript each actually do, and what a "component" means as a concept, independent of any specific tool that implements it.

Primary source

MDN, "Your first website" and "Structuring content with HTML" — the standard, vendor-neutral reference for exactly this split. Read the first if you've never opened an HTML file before; skim the second if you have.

Three jobs, three languages

A browser turns a text file into a picture on screen using three separate technologies, each with one job:

LanguageJobAnswers the question
HTMLStructureWhat is this piece of content — a heading, a list, a button, a table?
CSSPresentationHow should it look — color, spacing, size, layout?
JavaScriptBehaviorWhat should happen when the user does something — click, type, scroll?

This split isn't specific to React or Next.js — it's the browser's own division of labor, and it predates every JavaScript framework by decades. A React component still ultimately produces HTML, still gets its look from CSS, and still runs JavaScript for behavior; React just changes how those three things get written and combined, not what each one is responsible for.

A component, before any framework exists

A component is a reusable, self-contained unit of UI — a chunk of structure, presentation, and behavior bundled together and given a name, so it can be reused wherever that same piece of UI is needed. A button that shows a label and responds to a click is a component whether it's written as a hand-rolled HTML/CSS/JS snippet, a React function, or a Next.js file — the concept exists independently of any of those.

A component without React — plain HTML/CSS/JS

<!-- structure -->
<button class="badge badge--warning" id="statusBadge">Backordered</button>

/* presentation */
.badge { padding: 0.25rem 0.6rem; border-radius: 999px; font-size: 0.85rem; }
.badge--warning { background: #fff3cd; color: #7a5b00; }

// behavior
document.getElementById("statusBadge").addEventListener("click", () => {
  console.log("Status badge clicked");
});

Nothing here is React. It's still a component in every meaningful sense: one reusable unit, one name, structure + presentation + behavior bundled together. This project's real components/workflow/StatusBadge.tsx (Track 5's Tutorial 10) does the same job — it just expresses it as a typed React function instead of a raw DOM snippet.

Correct framing

"A component is React's way of organizing structure, presentation, and behavior together — the idea predates React."

A framing to avoid

"A component is a React thing." (This makes L5.2 harder — you'll be learning React's syntax for an idea you already understand, not the idea itself.)

Tiny exercise (5 minutes)

Exercise

Open app/globals.css and find any one CSS rule. Say out loud which of the three jobs (structure, presentation, behavior) it belongs to, and why it couldn't belong to either of the other two. Then open components/workflow/StatusBadge.tsx (you don't need to understand the React syntax yet) and just point at — don't read closely — the parts that look like they're doing structure versus behavior. You're building a landmark for L5.2, not analyzing the file yet.

If any of the three jobs feels blurry — for instance, whether inline event handling counts as "behavior" even when it's one line inside an HTML-looking tag — ask here before L5.2. That exact blurriness is what JSX (L5.2) deliberately reintroduces on purpose, and it helps to notice the seam before React hides it.
