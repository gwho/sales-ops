API Error Contracts and Business-Readable Failures

Every success response in this API has a different shape depending on which endpoint answered. Every failure response, by design, has exactly the same shape no matter which endpoint or which layer produced it. This lesson rehearses that uniform error contract, and a genuine testing gotcha that comes with proving it.

What you're rehearsing

Tutorial 07 Part 7, against backend/errors.py and tests/test_backend_errors.py in full.

One shape, three sources

Every error this API can produce ends up as {"detail": "<business-readable string>"} — but that string can originate from three genuinely different places:

SourceExampleStatus
A router or backend/uploads.py raising HTTPException directlyWrong file extension, missing columns400
FastAPI's own RequestValidationError (a required File()/Form() field is entirely absent)No file submitted at all400
An unhandled Exception — a genuine bugA typo'd variable name inside a route handler500

The first row needs no special handling — HTTPException already produces {"detail": "..."} via FastAPI's default behavior. The other two rows are exactly what backend/errors.py's two registered handlers exist to normalize.

The contract you didn't know you'd made

FastAPI's own validation machinery — the thing that fires automatically when a required field is simply missing — returns detail as a list of structured error objects by default, not a string. If nothing intercepted that, a request missing a required file would return a detail shaped completely differently from every other error this API produces — and <BusinessErrorMessage message={errorDetail} /> on the frontend expects a plain string every time. register_exception_handlers's RequestValidationError handler rewrites FastAPI's own list-shaped error into the same single-string shape everything else uses.

400 vs. 500 — whose fault, not how bad

The split isn't about severity — it's about which side of the request is responsible. 400 means the client sent something this specific request can't proceed with (a missing column, a corrupt file, a malformed date) — fixable by the caller, without touching the server. 500 means something broke on the server's side regardless of what was sent — never the caller's fault, and the response must never explain exactly what broke to a stranger on the internet. backend/errors.py's generic Exception handler enforces that second half directly: no matter what the real exception's message says, the client always receives the same fixed, generic string.

HTTP contract testing: proving shape, not just business results

Open tests/test_backend_errors.py in full. Neither test in this file asserts anything about business logic — no SKU, no order, no aging bucket. Both tests exist purely to prove the HTTP contract itself: status code, header behavior, and body shape. This is a different kind of test from every one in Tutorials 03-06 — those proved "the business logic is correct"; these prove "a caller can rely on the shape of what comes back, independent of what the business logic decided."

def test_unexpected_exception_returns_generic_500_without_leaking_message(monkeypatch):
    def _boom(*args, **kwargs):
        raise RuntimeError("super secret internal stack trace detail")

    monkeypatch.setattr("backend.routers.orders.validate_orders", _boom)

    # raise_server_exceptions=False: exercise the registered Exception handler's
    # actual response instead of TestClient's default debugging behavior of
    # re-raising unhandled exceptions straight into the test.
    error_client = TestClient(app, raise_server_exceptions=False)

The genuine gotcha

Starlette's TestClient defaults to raise_server_exceptions=True — deliberately built so an unhandled exception inside a route propagates straight into the test process as a real traceback, which is exactly what you want while debugging a genuine bug. But that same default bypasses any registered exception handler that exists specifically to convert an exception into a response — including handle_unexpected_error, the exact thing this test needs to observe. Without TestClient(app, raise_server_exceptions=False), this test can never see what a real client actually receives for a genuine server error; it only observes pytest's own exception-propagation behavior, which proves nothing about whether the handler is registered correctly.

Tiny exercise (10 minutes)

Run uv run pytest tests/test_backend_errors.py -v and confirm both tests pass. Then, in tests/test_backend_errors.py, temporarily remove raise_server_exceptions=False from the second test's TestClient(...) call and re-run just that test. Write down, in your own words, what pytest reports instead of a clean assertion failure — then revert the change.

Failure mode named: letting FastAPI's default list-shaped validation errors, or a raw Python exception's message/traceback, leak past this boundary into a client response. Both would silently break the one promise this API makes about every failure: that response.json().detail is always a plain, business-readable string, safe to render directly.

If the "why does a test-client default even exist that could hide a bug like this" question feels worth chasing further, ask now — it's a good example of a testing tool's convenience default working against you in one specific, narrow case.
