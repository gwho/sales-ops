Track 3 — Presentation Without Leakage

Excel Workbooks, Sheets & Manifests

Before reading any more report_export.py code, you need six words this module uses constantly: workbook, sheet, row, cell, header row, and manifest. None of them are exotic — but one of them (manifest) is this project's own term, and the way this module builds it teaches a general lesson about where metadata should come from.

Primary source

openpyxl documentation (official) — the library report_export.py is built on. You don't need to read it cover to cover; skim enough to recognize Workbook, Worksheet, and cell as real objects this library gives you, not this repo's own invention.

Six words, one table

WordWhat it means
WorkbookThe whole .xlsx file — one Python object, openpyxl.Workbook().
Worksheet (sheet)One tab inside the workbook — "Summary", "Valid Orders", and so on. A workbook holds many.
RowOne horizontal line inside a sheet — row 1 is almost always the header row in this project's reports.
CellOne single value at a specific row/column position — the smallest unit this module ever writes to.
Header rowRow 1: the column labels a human reads before any data — see Lesson 19 for why this repo never lets it be an accident.
ManifestThis project's own term (ReportManifest) — a small record describing what a generated report contains: its ID, type, filename, timestamp, and sheet names. Not an openpyxl concept — this repo's own invention, built from a real workbook after the fact.

Building a workbook, one sheet at a time

Open src/report_export.py and find:

def _new_workbook() -> Workbook:
    wb = Workbook()
    wb.remove(wb.active)
    return wb

Workbook() actually ships with one sheet already created, always named "Sheet" — wb.remove(wb.active) deletes it immediately, before any real sheet exists. Every other sheet in this module is added with wb.create_sheet(sheet_name), and every row is added with ws.append([...]) — one list per row, one value per cell in that row.

Deriving the manifest from the real workbook

Open src/report_export.py again, near the end of export_order_validation_report:

    sheet_names = wb.sheetnames
    workbook_bytes = _save_workbook_bytes(wb)
    manifest = _build_manifest(
        "order_validation", "order_validation_report.xlsx", sheet_names, effective_generated_at
    )

wb.sheetnames reads back the workbook's actual sheets, in the exact order every wb.create_sheet(...) call already ran — openpyxl preserves creation order, it never alphabetizes or reorders. That list is handed straight into _build_manifest. There is no second, separately-typed list of sheet names anywhere in this function that the manifest could possibly disagree with.

Single source of truth

The rejected alternative is a hardcoded list, typed out once alongside the wb.create_sheet(...) calls: manifest_sheets = ["Summary", "Valid Orders", ...]. A developer adding a new sheet later would have to remember to update two places — the real calls, and this parallel list — and nothing would stop them from updating one and forgetting the other. Deriving from wb.sheetnames instead makes that whole category of bug structurally impossible: there's only one list, read twice, not two lists that are supposed to agree.

Retrieval check

After three wb.create_sheet(...) calls, in what order does wb.sheetnames list them?

Why does _build_manifest read wb.sheetnames instead of a separately-typed list of names?

Tiny exercise (5 minutes)

Run this, deliberately naming the sheets out of alphabetical order:

uv run python -c "from openpyxl import Workbook; wb = Workbook(); wb.remove(wb.active); wb.create_sheet('B Sheet'); wb.create_sheet('A Sheet'); print(wb.sheetnames)"

Confirm the printed order is ["B Sheet", "A Sheet"] — creation order, not alphabetical — the exact fact this lesson's manifest discussion depends on.
