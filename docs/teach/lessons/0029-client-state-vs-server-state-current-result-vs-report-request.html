Track 4 — Optional Reinforcement · 6 of 7

Client State vs. Server State: currentResult vs. a Report Request

Lesson 30 taught that the server forgets every request the moment it responds. This lesson is the natural follow-up question a learner always asks next: "then why does the page still show my results after I click around?" The answer is the browser remembering, not the server — and the distinction matters more than it sounds like it should.

What you're rehearsing

Tutorial 07 Part 9, against app/order-validation/page.tsx and docs/architect/phase-10-fastapi-integration/decisions.md #4 and #5.

Open app/order-validation/page.tsx's own state declarations:

const [status, setStatus] = useState<RequestStatus>("idle");
const [currentResult, setCurrentResult] = useState<OrderValidationResult | null>(null);
const [errorDetail, setErrorDetail] = useState<string | null>(null);

const [reportStatus, setReportStatus] = useState<ReportRequestState>("idle");
const [reportErrorDetail, setReportErrorDetail] = useState<string | null>(null);

Every one of these lives entirely inside this one page's React component tree. Refresh the browser tab and every one of them resets to its initial value — not because the server forgot anything (Lesson 30: the server never knew any of this existed in the first place), but because the browser tab itself, the only place any of it was ever stored, was thrown away and rebuilt from scratch.

The trap this lesson exists to prevent

It's tempting to reason: "the dashboard still shows my last result, so the server must remember it." That's backwards. currentResult surviving between a "Run Validation" click and scrolling down to read the table isn't evidence of any server memory — it's ordinary React state, living exactly as long as the tab stays open on that page. The moment you navigate away and back, or refresh, that state is gone, and the only way to see results again is to submit a fresh request — because there was never anything on the server side holding onto it.

RequestStatus vs. ReportRequestState — two trackers, not one

Notice the page tracks two separate status enums, not one shared "is anything loading" flag: RequestStatus ("idle" | "submitting" | "succeeded" | "failed") for the JSON workflow request, and ReportRequestState ("idle" | "processing" | "failed") for the report download button. Running validation and downloading a report are two genuinely independent requests — a user can download a report without ever having clicked "Run Validation" on screen first, since the report endpoint recomputes from the raw files regardless (Lesson 30). Merging these into one shared status would make "the report download failed" and "the validation request failed" indistinguishable in the UI, when they're answering two different questions about two different requests.

Forward reference — later, deliberate persistence

A later phase of this project (Phase 12) deliberately adds real server-side state on top of everything Track 4 taught — a Postgres row, keyed by an anonymous browser-held session identifier, that remembers a session's latest saved result per workflow. That's a genuinely new architectural decision, made explicitly and with its own ADR, not something Track 4's design was quietly already doing. Don't try to absorb it here — Track 6 teaches it properly, once the Server/Client Component split (a separate prerequisite) is solid. The only thing worth holding onto for now: Phase 10's state was entirely browser-local; anything server-remembered afterward was added on purpose, later, and is visibly different in kind.

Tiny exercise (10 minutes)

Run npm run dev and uv run fastapi dev backend/main.py, open /order-validation, run validation on the sample data, and confirm results appear. Then open the exact same URL in a second, completely separate browser tab. Predict, before checking: does the second tab show the same results, an empty state, or something else — and why, in one sentence tied to where currentResult actually lives?

Failure mode named: assuming the server remembers a result just because the browser still displays it on screen. The two are unrelated facts that happen to coincide while a tab stays open — conflating them is exactly the misconception that makes Phase 12's real, deliberate persistence layer (the forward reference above) look like "nothing new," when it's actually the first server-side memory this project ever added.

If "so what would it actually take to make results survive a refresh" feels like the obvious next question, hold it — that's precisely Track 6's subject, and it'll land better once you've felt the gap this lesson describes for yourself first.
