Effects, Cleanup, and Async UI

Lesson 36 ended on a promise: order-validation/page.tsx never needs useEffect because every piece of its work is caused by a click, but DashboardLiveSections.tsx genuinely does. This lesson traces that one real Effect end to end — not as a general "run code later" hook, but as React's mechanism for synchronizing with something outside itself.

Primary source

"Synchronizing with Effects" on react.dev — covers both why Effects exist and, in its "Add cleanup if needed" section, the exact cleanup pattern this lesson traces. For the development-only double-invoke behavior specifically, the useEffect API reference's Caveats section is the precise source: Strict Mode runs one extra setup → cleanup → setup cycle before the first real setup, specifically to stress-test cleanup logic.

Why this matters

All three mission drivers apply: you can explain in an interview exactly why one network request — this file's dashboard fetch — belongs in an Effect while a structurally similar one, Lesson 36's runValidation, correctly lives in a click handler instead; that same rendering-caused-vs-click-caused distinction is portable to any React codebase; and recognizing which failure modes a clean build cannot catch is what lets you extend this repo's async UI without silently reintroducing a bug this file already solved once.

What actually needs an Effect: caused by rendering, not by a click

Lesson 36's runValidation already reaches outside React — it calls postJSON, a real HTTP request — and that's completely correct without an Effect, because that request only ever starts in response to one specific click. Reaching outside React isn't what makes an Effect necessary; being caused by rendering itself, rather than by a user action, is. Loading a visitor's saved dashboard results needs one for exactly that reason: the request has to start the moment this component appears on screen — nothing the visitor clicks causes it, so there's no event handler for it to live in. This Effect is synchronization with rendering, not a response to an action:

components/dashboard/DashboardLiveSections.tsx, lines 53–73

useEffect(() => {
    let cancelled = false;
    async function load() {
      try {
        const results = await getDashboardResults();
        if (cancelled) return;
        setLiveOrderValidation(results.order_validation);
        setLiveInventoryAllocation(results.inventory_allocation);
        setLivePaymentAging(results.payment_aging);
        setStatus("loaded");
      } catch (error) {
        if (cancelled) return;
        setErrorDetail(error instanceof ApiError ? error.message : "Something went wrong. Please try again.");
        setStatus("failed");
      }
    }
    void load();
    return () => {
      cancelled = true;
    };
  }, []);

The dependency array, [], tells React "no reactive value read here should ever cause this Effect to re-run — treat it as caused only by mounting." In production, that means it runs exactly once, right after the first render. In development, React's Strict Mode deliberately runs it twice — an extra setup → cleanup → setup cycle meant to surface exactly the kind of cleanup bug the next section covers, before it ever reaches production.

There's a second reason status doesn't belong in that array, beyond timing: status isn't an input to this synchronization, it's this Effect's own output — the Effect is what decides its final value, by calling setStatus. If status were listed as a dependency, the first setStatus("loaded") call would count as a dependency change, and React would clean up this Effect instance and start a new one — redundantly re-fetching a second time. That second run's own setStatus("loaded") call matches the value already there, so React skips the re-render it would otherwise cause, and the cycle stops there: one wasted request, not an unbounded loop. (A truly non-terminating Effect loop is still possible — an Effect with no dependency array at all runs after every render regardless of what changed, so a setter call inside one of those really does retrigger forever — but that's not what adding status here would do.) Either way, [] is the accurate declaration: nothing this Effect reads from render should control when it runs.

The cancelled guard: cleanup for work that might finish too late

An Effect's return value, if it's a function, is its cleanup — React calls it before every re-run of the Effect, and when the component unmounts. This Effect's cleanup does one thing: cancelled = true;. It doesn't cancel the in-flight HTTP request itself — the fetch keeps running regardless — it just marks this particular Effect instance's eventual result as no longer wanted. Both if (cancelled) return; checks exist because getDashboardResults() is asynchronous, so its response can arrive after this specific Effect instance has already been cleaned up — not only because the component unmounted (the visitor navigated away before the fetch finished), but also because React started a newer setup in its place, which is exactly what happens once, deliberately, during development Strict Mode's setup → cleanup → setup cycle mentioned above. Without the guard, an obsolete response could still call setStatus and the other setters, silently applying a stale result on top of whatever the current Effect instance's own request returns.

Why the loading branch returns before deciding sample vs. live

The component's very first render — before the fetch has resolved either way — has to show something. This repo makes a specific, deliberate choice about what:

components/dashboard/DashboardLiveSections.tsx, lines 75–92

// First paint is always a skeleton, never sample-then-swap -- showing
  // sample values immediately and replacing them on fetch return would
  // flash content and blur "is this real or sample?" (ADR 0007).
  if (status === "loading") {
    return (
      <div className="mt-6">
        <LoadingState label="Loading your saved results…" />
      </div>
    );
  }

  if (status === "failed") {
    return (
      <div className="mt-6">
        <BusinessErrorMessage message={errorDetail ?? "Something went wrong. Please try again."} />
      </div>
    );
  }

Once past both early returns, the component computes which sections are sample data by checking whether the fetch actually returned something for each workflow:

components/dashboard/DashboardLiveSections.tsx, lines 94–96

const orderValidationIsSample = liveOrderValidation === null;
  const inventoryAllocationIsSample = liveInventoryAllocation === null;
  const paymentAgingIsSample = livePaymentAging === null;

liveOrderValidation starts as null and only becomes non-null once the fetch succeeds and finds a saved result. If the early return on line 78 didn't exist, this component would reach these three lines while still loading, when every live value is still null for the ordinary reason that nothing has arrived yet — and it would render every KPI as sample data, then silently swap to live data a moment later. The early return is what keeps "haven't fetched yet" from ever being mistaken for "genuinely has no saved result."

Not every derived value here needed an Effect

The three IsSample consts above are computed the same way Lesson 36's canSubmit was: plain, unmemoized, recalculated every render from state that already exists. order-validation/page.tsx's filteredErrors derives a different value the same way, just wrapped in useMemo (reserved for Lesson 38 — its mechanics aren't needed for the point here) with a dependency array of [errors, errorSeverity, errorSearch]. None of that is caused by rendering itself the way this component's dashboard fetch is; it only recomputes because inputs it already has changed, and it never reaches outside React to do it. That's the real dividing line running through this lesson and Lesson 36 together: reaching outside React is necessary for needing an Effect, but never sufficient by itself — runValidation's click-triggered fetch reaches outside React too, and correctly stays in an event handler, because what actually makes an Effect the right tool is that rendering itself, not a click and not an already-available value, is what causes the work.

Correct framing

"This Effect exists because fetching the visitor's saved results is caused by the component appearing, not by anything the visitor clicks — everything else on this page is either plain render-time computation or work a click handler already covers."

A framing to avoid

"I'll just wrap this in useEffect to be safe." (An Effect is not a neutral default. Every Effect this repo doesn't need but adds anyway is one more place a dependency array can be wrong, one more cleanup function to get right, and one more render pass than a plain derived value would have cost.)
