Track 3 — Presentation Without Leakage

Allow-Lists, Hidden State & Safe Defaults

Two small, unrelated-looking decisions in report_export.py turn out to share one instinct: when this module is uncertain, it should fail in the direction that's easiest for a human to notice and hardest for a human to be misled by.

Primary source

No dedicated external resource — a repo-specific design decision, best learned from docs/plan/phase-6-excel-report-export/explanation.md §6–7 and docs/plan/phase-6-excel-report-export/plan.md's Key Invariants 6 and 7.

Safety default 1: an allow-list, not an exclusion

Open src/report_export.py and find:

FOLLOW_UP_PRIORITIES = {"High", "Medium", "Low", "Watch"}

follow_up_rows = [
    row for row in result["aging_rows"] if row["follow_up_priority"] in FOLLOW_UP_PRIORITIES
]

Every priority payment_aging.py actually produces today is covered by this set. The rejected alternative — row["follow_up_priority"] != "None" — would produce the exact same output for every row this repo's tests have ever seen. The difference only shows up for a value neither side has actually anticipated. Open tests/test_report_export.py and find:

def test_payment_aging_report_follow_up_list_excludes_unexpected_priority_string():
    aging_rows = [{**PAYMENT_AGING_ROW_FIXTURE, "follow_up_priority": "Unexpected"}]
    workbook_bytes, _ = export_payment_aging_report(
        _payment_aging_result(aging_rows=aging_rows), generated_at=GENERATED_AT
    )
    wb = _load(workbook_bytes)
    ws = wb["Follow-up List"]
    assert ws.max_row == 1

A row with a priority string this module never expected does not appear on the "needs attention" sheet. Notice: this is still not revalidation, the rule Lesson 15 taught — the row's follow_up_priority value is never touched or second-guessed anywhere. Only its membership in a filtered view is decided.

Allow-list vs. deny-list, generically

An allow-list only accepts values it explicitly recognizes; everything else is excluded by default. A deny-list only rejects values it explicitly recognizes; everything else is accepted by default. For a report a salesperson acts on, the two fail in genuinely different directions: an allow-list's failure mode is a missing row — a human cross-checking totals might eventually notice. A deny-list's failure mode is a wrong-looking, unvetted row quietly included as if it belonged there. Given a choice, this module always picks the direction where the mistake is more likely to be noticed, not the direction where it's silently acted on.

Safety default 2: no hidden global state

Open src/report_export.py and find:

report_id = f"rpt-{report_type}-{generated_at:%Y%m%d%H%M%S}"

This is the entire ID-generation logic. No module-level counter, no database lookup, nothing that remembers how many reports of a given type have been generated before. The hand-authored test fixtures happen to end their example IDs in -001, which reads like a sequential counter — but nothing in this project actually counts anything. A counter would mean a variable living outside any function, remembered between calls — exactly the "hidden global state" this repo's own coding standards rule out everywhere, not just here. It would also make this module's output depend on call order: two identical calls, run in a different sequence relative to other report generation elsewhere in the process, would produce different IDs — a much harder kind of bug to reproduce than anything a pure function like this one can produce.
