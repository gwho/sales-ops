Track 1 — Python, Data-as-Tables, and Contracts

Demo Data vs. Test Data

This repo maintains three separate sources of example data, on purpose: sample_data/*.xlsx, src/sample_data.py's generator functions, and tests/contract_fixtures.py's hand-typed examples. They look similar. They are not interchangeable.

Primary source

pandas — 10 Minutes to pandas (same as Lesson 5) — both demo data and test fixtures in this repo are expressed as DataFrames or dicts, so the same mental model applies; this lesson is about purpose, not new syntax.

Three things, three jobs

SourceBuilt byJob
sample_data/*.xlsx Generated, then committed to git A believable demo a reviewer opens directly — realistic, mostly clean, with a few deliberate imperfections.
src/sample_data.py Hand-authored Python code The generator that produces the xlsx files above — pure functions, no file writing themselves.
tests/contract_fixtures.py Hand-typed dict literals One believable example value per contract (Lesson 4) — proves the shape can hold real-looking data, nothing more.

Why not derive the fixtures from the generated data?

It would be technically possible to write a small function that reads sample_orders.xlsx and computes a real validation-error example from it. This repo deliberately doesn't. Two reasons, both worth holding onto:

  1. The rule doesn't exist yet. Computing a real ValidationErrorRow means running order-validation logic — but that logic is a later phase's job. Writing a throwaway version now just to fill in a fixture is that later phase's work, done early and without its real design.
  2. Independence is the whole point. If a fixture were computed from the real logic, a bug in that logic would produce a fixture that faithfully reproduces the same bug — the "proof" and the thing it's supposed to prove would trace back to one source, so they could never disagree. A fixture typed by hand, from reading the spec directly, has an independent chance of catching a mistake the real code makes later.

Look at the docstring in tests/contract_fixtures.py — it says this outright:

"""Realistic example values for each Phase 1 output-contract family (Contract Fixtures).

Hand-authored, not computed by src/sample_data.py or any business-rule module —
order_validation.py, inventory_allocation.py, and payment_aging.py don't exist
until Phases 3-5. These only prove the contract shapes can hold believable demo
data; they are not evidence that any business rule is implemented correctly.
"""

Tests that count properties, not exact values

Because src/sample_data.py's generator functions are just regular functions (Lesson 2) that build and return a DataFrame (Lesson 5), they can be tested directly — call the function, check something true about what comes back. Here's the pattern from tests/test_sample_data.py:

def test_generate_orders_has_exactly_one_duplicate_order_id():
    df = generate_orders(generate_product_master())

    duplicate_mask = df["order_id"].duplicated(keep=False)
    duplicated_ids = set(df.loc[duplicate_mask, "order_id"])

    assert duplicated_ids == {"SO-2026-010"}
    assert duplicate_mask.sum() == 2

This doesn't check "row 34 equals this exact dict" — it checks a property: exactly one order_id is duplicated, and it's specifically SO-2026-010. That's deliberately in between two extremes: too loose (just "some duplicate exists somewhere," which would miss a second, accidental duplicate creeping in later) and too rigid (hardcoding the whole row, which would break on any unrelated edit to the sample data).

The boundary that's being protected

Just like Lesson 5's validate_required_columns, these tests never compute a real business result (an aging bucket, an allocation outcome) — only properties of the raw input data (an amount, a date gap, a count). That line — testing what the data looks like, never what a not-yet-built rule would decide about it — is the exact same "don't reimplement a later phase early" boundary from this lesson's first half, just showing up again inside a test file instead of inside a fixture file.
