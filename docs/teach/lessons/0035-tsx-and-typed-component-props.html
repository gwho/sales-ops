TSX and Typed Component Props

You've finished Tutorials 08–11 — you can read this repo's components and pages. This lesson doesn't teach new React; it teaches the TypeScript layer sitting on top of the JSX Lesson 32 already covered, so a component's prop list reads as an enforceable contract instead of a pile of names you copy without checking.

Primary source

"Using TypeScript" on react.dev — the official React docs' own page on typing props and children. For the two TypeScript mechanics this lesson leans on hardest, the TypeScript Handbook is more precise: "Object Types" (optional properties) and "Everyday Types" (literal types, union types).

Why this matters

All three mission drivers apply here: you can name and defend a component's prop contract in an interview; the same typed-props pattern is portable to any other React/TypeScript codebase, not just this one; and reading a prop type correctly is what lets you extend this repo's components without accidentally passing something the compiler would have caught for you.

TSX: JSX with a type checker watching

Every component file in this repo you've already read — MetricCard.tsx, UploadPanel.tsx, every page under app/ — has a .tsx extension, not .jsx. TSX is exactly the JSX syntax Lesson 32 covered (HTML-like markup inside a JavaScript function), compiled through TypeScript instead of plain JavaScript. Nothing about how JSX itself works changes. What's new is that every prop a component declares now has a type, and TypeScript checks every call site against it before the app is even allowed to build.

A prop type is a contract, not a comment

Track 1's Lesson 4 covered this exact idea on the Python side: a TypedDict declares a data shape up front, and code elsewhere is checked against it rather than trusted to get it right by convention. A component's prop type is the same idea, one layer up — it's the API contract for that component, and the compiler enforces it on every caller, not just the ones a human reviewer happens to read closely.

components/workflow/MetricCard.tsx, lines 11–20

type MetricCardProps = {
  label: string;
  value: string | number;
  icon?: ReactNode;
  tone?: Tone;
  /** Dashboard-only (Phase 12): true when this card's value came from the
   * static sample fallback rather than the visitor's own Saved Workflow
   * Result -- see docs/adr/0007-session-scoped-workflow-result-persistence.md. */
  sample?: boolean;
};

Five fields, three of them optional (the ?). value: string | number is a small union — this repo passes it a formatted string almost everywhere (Track 4's formatNumber/formatAmount helpers), but the type honestly allows a raw number too, since nothing about rendering a number inside a <span> requires it to already be a string.

Named alias vs. inline type: same mechanism, a size judgment call

MetricCardProps is a named type alias, declared once above the component and referenced once below it. Compare that with how this repo's own route-group layouts type their single most common prop:

app/(workspace)/layout.tsx, lines 5–7 (the complete function)

export default function WorkspaceLayout({ children }: { children: ReactNode }) {
  return <AppShell>{children}</AppShell>;
}

app/(public)/layout.tsx opens with the identical signature line, verbatim: export default function PublicLayout({ children }: { children: ReactNode }) { (line 6) — its body differs (it also renders PublicHeader/PublicFooter), but the prop type is exactly the same inline shape.

Neither layout bothers declaring a WorkspaceLayoutProps type — one field, used once, written inline. MetricCardProps and UploadPanel.tsx's UploadPanelProps get a named alias because they carry several fields worth naming as a unit and documenting together. Both are the exact same TypeScript feature — an object type describing a component's props — the choice between inline and named is a judgment call about whether the shape is worth a name of its own, not a rule this repo enforces mechanically.

ReactNode: "anything React can render," typed

children in both layouts above is typed ReactNode — the type for "whatever can legally appear inside JSX": a string, a number, another JSX element, an array of those, or nothing at all. MetricCard.tsx's icon?: ReactNode uses the same type for a different purpose — not children in the JSX-nesting sense, just "a prop that holds renderable content," most often a small icon element like <ClipboardList size={16} />.

Optional props: what ? actually promises

UploadPanelProps has four optional fields:

components/workflow/UploadPanel.tsx, lines 13–31

type UploadPanelProps = {
  label: string;
  requiredColumns: string[];
  accept?: string;
  /** Allowlisted backend/routers/templates.py key, e.g. "orders" -- omit to hide the Sample file link. */
  sampleFileName?: string;
  /** Hands the selected File to the parent page so it can be submitted with the Workflow Request. */
  onFileChange?: (file: File | null) => void;
  /**
   * Displayed filename, controlled by the parent's own file state. Pages hold
   * the File as the source of truth (native pick or "Run sample data" both
   * flow through the same onFileChange -> parent state), so passing this down
   * keeps the panel's display in sync regardless of which path set the file --
   * "Run sample data" bypasses the native <input>, so without this the panel
   * would keep showing "Choose a file..." even after a sample file was loaded
   * and submitted.
   */
  selectedFileName?: string | null;
};

A trailing ? means the prop's real type is string | undefined (or whatever the base type is, unioned with undefined) — it does not mean "this prop has a default value." A default only exists if the component's own destructuring supplies one:

components/workflow/UploadPanel.tsx, lines 39–46

export function UploadPanel({
  label,
  requiredColumns,
  accept = ".xlsx",
  sampleFileName,
  onFileChange,
  selectedFileName,
}: UploadPanelProps) {

accept gets a default (".xlsx") — omit it at a call site and the component sees ".xlsx". sampleFileName, onFileChange, and selectedFileName get no default — omit them and the component sees literally undefined, which its own logic then has to handle (the sampleFileName ? check that hides the "Sample file" link entirely).

This repo's two real <MetricCard> call sites show the same choice from the caller's side, for MetricCardProps' optional sample?: boolean:

Same prop, omitted vs. supplied

// app/(workspace)/order-validation/page.tsx, lines 369–374 — sample omitted
<MetricCard
  label="Total Orders"
  value={formatNumber(currentResult.summary.total_orders)}
  icon={<ClipboardList size={16} />}
  tone="info"
/>

// components/dashboard/DashboardLiveSections.tsx, lines 112–118 — sample supplied
<MetricCard
  label="Total Orders"
  value={formatNumber(validation.total_orders)}
  icon={<ClipboardList size={16} />}
  tone="info"
  sample={orderValidationIsSample}
/>

Both typecheck. The workflow page never uses the dashboard's automatic sample fallback (Phase 12) — an uploaded file and a "Run sample data" click are both freshly processed by the same validation call, so this card has nothing meaningful to pass for sample and relies on MetricCard's own default, sample = false. The dashboard genuinely might be showing that Phase-12 fallback instead of the visitor's own saved result, so it always supplies the real answer.

Literal unions: a fixed vocabulary instead of open-ended string

tone?: Tone on MetricCardProps isn't tone?: string. Tone is a literal union — a type built from specific allowed string values, not the general shape "some string":

components/workflow/StatusBadge.tsx, line 5

export type Tone = "success" | "warning" | "danger" | "info" | "neutral";

Passing tone="info" typechecks because "info" is one of the five listed values. Passing tone="urgent" would fail to compile — not at runtime, not in a code review, but the moment TypeScript checks the call site — because "urgent" is not a member of that union. A plain string type would have silently accepted the typo and let it reach CHIP_TONE_CLASSES[tone], where it would have looked up a key that doesn't exist.

Callback prop signatures

onFileChange?: (file: File | null) => void types a prop whose value is itself a function: it takes one argument (File | null — a native browser type, not React-specific), and returns nothing. This is exactly the "hand the parent a way to react" pattern — the parent page owns the actual file state, and UploadPanel just calls whatever function it was given whenever the native <input> fires:

app/(workspace)/order-validation/page.tsx, lines 314–320 — caller and callee, both Client Components

<UploadPanel
  label="Product Master File"
  requiredColumns={["sku", "product_name", "active"]}
  sampleFileName="product-master"
  onFileChange={setProductMasterFile}
  selectedFileName={productMasterFile?.name ?? null}
/>

Why this doesn't contradict Lesson 33

Lesson 33 showed a function prop that couldn't cross the Server→Client boundary, because a Server Component's function value can't be serialized into something the browser can receive. This callback prop is fine for a different reason: order-validation/page.tsx starts with "use client", and so does UploadPanel.tsx — both already run in the browser, so nothing is crossing a Server→Client boundary at all. The rule was never "function props are illegal," it was "function props can't cross that one specific boundary."

Forward reference — generics, previewed only

components/tables/DataTable.tsx is typed DataTable<T> — a component whose prop types depend on a type parameter chosen at each call site, so its columns and data props stay correctly linked to whatever row shape a given page passes in. That's a genuinely different, bigger idea than anything above, and it gets its own lesson — Lesson 38 in this same reinforcement sequence, not yet generated. Nothing here requires understanding it yet.

Correct framing

"tone?: Tone is a compiler-enforced contract — passing anything outside those five strings fails the build before the app ever runs."

A framing to avoid

"The types are basically documentation — as long as I know what values make sense, it doesn't really matter what I write in the type." (This is exactly backwards: an inaccurate type is worse than no type, because it makes TypeScript actively vouch for call sites that are actually wrong. The type isn't advisory — it's the thing the compiler checks against.)

Tiny exercise (7 minutes)

Exercise — predict, then let the compiler confirm

Before touching anything: look at order-validation/page.tsx lines 369–374 above and predict, out loud, whether each of these three edits would typecheck — (1) changing tone="info" to tone="urgent"; (2) deleting the label line entirely; (3) changing icon={<ClipboardList size={16} />} to icon="a string instead".

Then make exactly one of those three edits for real, save the file, and run npm run typecheck in the terminal. Read the error TypeScript reports — does it name the field, the expected type, and the value you gave it? Undo the edit with your editor's Undo (Cmd/Ctrl+Z) rather than a git command — this file may already have other unstaged changes of yours, and git checkout -- would discard all of them, not just this edit. If you do want to confirm with git instead, run git diff -- "app/(workspace)/order-validation/page.tsx" first and check that the only line shown is the one you just changed. Confirm npm run typecheck is clean again before moving on.

If predicting all three edits felt confident and the actual error message matched what you expected, this lesson is solid. If any of the three surprised you, reread whichever section above covers that specific field — optional props, literal unions, or ReactNode — before continuing to Lesson 36.
