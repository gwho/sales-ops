Track 3 — Presentation Without Leakage

Serialization Boundaries & Safe Cell Values

Python has several different ways to represent "nothing." Excel's .xlsx format doesn't. This lesson is about the function that reconciles the two — and about a genuinely surprising fact you have to actually observe to believe, not just read about.

Primary sources

openpyxl documentation (official) — for how a workbook is actually written and reloaded. Python — io.BytesIO (official docs) — the in-memory buffer that makes an entirely disk-free save possible, and the thing standing between "an object in memory" and "a file a real person could open."

Four ways to mean "nothing"

Before this lesson's code, notice how many different values can mean "there's no real data here," depending on where a value came from:

ValueWhere it comes from
NonePython's own null — e.g. a NotRequired field's default.
float('nan')A pandas DataFrame column with mixed numbers and blanks, upcast to float64.
pd.NaTThe same upcasting, but for a blank cell in a datetime column.
pd.NApandas's own newer, type-agnostic missing-value marker.

Four different runtime representations, all meaning the same business fact: "this cell has nothing in it." A serialization boundary — any place Python data is about to become some other format's data (a spreadsheet cell, a JSON field, a database column) — is exactly where this kind of mismatch has to get resolved, once, in one place.

One function, one place

Open src/report_export.py and find:

def _safe_cell_value(value: Any) -> Any:
    if value is None:
        return ""
    if isinstance(value, float) and math.isnan(value):
        return ""
    try:
        if pd.isna(value):
            return ""
    except (TypeError, ValueError):
        pass
    return value

Three checks, each catching one of this lesson's "nothing" shapes, all converging on the same output: an empty string. Every single cell this module writes passes through this function first — no sheet-writing code anywhere else in this file re-implements its own "is this blank" check.

The surprise: a saved workbook is not the same as the object in memory

Here's the part you have to actually run to believe. Open tests/test_report_export.py and find a comment sitting right above an assertion:

# openpyxl round-trips a written "" as a genuinely blank cell -> None on reload.
    assert row["sku"] is None

_safe_cell_value wrote the literal empty string "". This assertion checks for None. That is not a contradiction — it's a real, documented fact about the .xlsx format: an empty-string cell has no distinct representation from "no value here at all," so once a workbook is saved and reloaded, a written "" genuinely comes back as None. The in-memory Workbook object, before it's ever saved, would still show "" — it's only after the save/reload round trip that the difference disappears.

Round-trip testing, as a general principle

This is why every test in tests/test_report_export.py calls load_workbook(io.BytesIO(workbook_bytes)) before asserting anything about a cell — never inspecting the in-memory Workbook object directly. A test against the unsaved object only proves "Python assigned the value I expected," which is true but uninteresting. A test against the saved-and-reloaded object proves what a real person opening the downloaded file in Excel would actually see — the only version of the question worth asking. The general rule extends past this one project: whenever code produces an artifact (a file, an API response, a database row), test the artifact after it's been through its real save/load path, not just the object still sitting in memory.

Retrieval check

After writing "" to a cell, saving to bytes, and reloading, what does cell.value show?

Why does _safe_cell_value convert None, NaN, NaT, and pd.NA all to the same output?

Why do this project's tests always save() and load_workbook() before checking a cell's value?

Tiny exercise (5 minutes)

Run this four-line reproduction yourself:

uv run python -c "import io; from openpyxl import Workbook, load_workbook; wb = Workbook(); wb.active['A1'] = ''; buf = io.BytesIO(); wb.save(buf); print(load_workbook(buf).active['A1'].value)"

Confirm it prints None, not an empty string. Then, without changing the command, explain in one sentence why io.BytesIO() is the piece of this line that makes "save" possible without ever touching a real file on disk.
