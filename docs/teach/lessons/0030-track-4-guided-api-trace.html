Trace 1 — "Run Validation"

Predict each numbered step yourself first — what function runs, what it does, what it hands to the next step — then reveal.

  1. User selects both files; clicks "Run Validation."
  2. handleRunValidation() runs.
  3. runValidation(orders, productMaster) builds a FormData.
  4. postJSON("/api/orders/validate", formData) sends the request.
  5. FastAPI's validate_orders_endpoint binds the two files by parameter name.
  6. _load_and_validate runs the adapter.
  7. validate_orders(orders_df, product_master_df) runs.
  8. The route returns OrderValidationResult.
  9. postJSON resolves in the browser.
  10. The page re-renders from currentResult.
Reveal: the full chain, with the file/function behind every step

1. Both File objects already sit in app/order-validation/page.tsx's ordersFile/productMasterFile state (Lesson 36). 2. handleRunValidation() checks canSubmit and calls runValidation(ordersFile, productMasterFile) — explicit parameters, not read from state inside runValidation itself (Tutorial 07 Part 9's timing-bug reasoning). 3. runValidation sets status = "submitting", builds a FormData with keys orders_file/product_master_file — the exact strings backend/routers/orders.py's parameters expect (Lesson 32). 4. postJSON calls postFormData, which does a plain fetch(..., { method: "POST", body: formData }) — no manual Content-Type (Lesson 32's multipart-boundary warning). 5. FastAPI matches the multipart part names to Annotated[UploadFile, File()] parameters of the same name (Lesson 32). 6. Both files go through read_xlsx_upload(file, label, loader) (Lesson 32) — extension check, then load_orders/load_product_master (Tutorial 03). 7. The exact tested function from Tutorial 03 runs — zero new business logic anywhere in backend/ (Lesson 35). 8. -> OrderValidationResult is the real TypedDict, serialized by Pydantic v2 with no parallel schema (Lesson 35). 9. postJSON's promise resolves with the parsed JSON; runValidation calls setCurrentResult(result) and setStatus("succeeded") — ordinary React state, forgotten by the server the instant step 8's response was sent (Lesson 30, Lesson 36). 10. The Summary/Validation Errors/Valid Orders sections re-render from currentResult — a React state update, not a page reload. Nothing from this chain is stored anywhere server-side; the next click reruns all ten steps from scratch.

Trace 2 — "Download Inventory Allocation Report"

Same exercise, a different endpoint shape (Lesson 31's "Report Artifact," not "Workflow Result") and a chained business call (Tutorial 07 Part 6). Predict first.

  1. User already has all three files selected; clicks "Download Report."
  2. handleDownloadReport() runs.
  3. A fresh FormData is built from the same three files.
  4. postReport("/api/inventory/allocate/report", formData, ...) sends it.
  5. FastAPI's allocate_inventory_report_endpoint receives it.
  6. _run_allocation runs — two business functions, not one.
  7. export_inventory_allocation_report(result, ...) runs.
  8. The route returns raw .xlsx bytes plus headers.
  9. postReport() reads the blob and headers.
  10. downloadBlob(...) triggers the save.
Reveal: the full chain, with the file/function behind every step

1. The same File objects already sitting in app/inventory-allocation/page.tsx's component state — nothing re-selected. 2. handleDownloadReport() confirms all three files are present, sets reportStatus = "processing" (Lesson 36's separate ReportRequestState). 3. A completely independent FormData is built — not a reference to whatever was submitted to POST /allocate earlier (Lesson 31's Report Artifact contract: it never accepts a prior result). 4. postReport sends it through the same postFormData mechanics as any other request (Lesson 32, Lesson 34). 5. allocate_inventory_report_endpoint (backend/routers/inventory.py) calls _run_allocation. 6. _run_allocation re-runs the full chain from raw files: validate_orders() first, producing a fresh valid_orders_df, then allocate_inventory(valid_orders_df, inventory_df) (Tutorial 07 Part 6) — genuinely recomputed, not reused from the earlier POST /allocate call, because the server has no memory of that call ever happening (Lesson 30, Lesson 36). 7. Tutorial 06's report-export function runs completely unchanged — this endpoint is the first place in the codebase that calls it from something other than a test. 8. Response(content=workbook_bytes, headers={"Content-Disposition": ..., "X-Report-Id": ..., "X-Generated-At": ...}) — never a TypedDict, never JSON (Lesson 31). 9. postReport() reads exactly the three headers expose_headers allows cross-origin JS to see (Lesson 34). 10. URL.createObjectURL → hidden <a download> → .click() → URL.revokeObjectURL (Lesson 34) — the browser's save dialog appears, and nothing about the generated file is stored anywhere server-side once step 8's response was sent.

Interview drill (90 seconds, out loud or timed in writing)

Answer, in under 90 seconds: "Why not generate a report once, save it, and let the user fetch it later by ID?" A complete answer names all four of: what has to be stored and where; who is responsible for cleanup/expiry of that storage; what serves as the report's identity and whether it needs any access check; and how the trust-boundary problem (Lesson 30, Tutorial 07 Part 3) reappears the moment a client's own copy of a result could be substituted for a fresh recomputation. If your answer doesn't mention all four, reread Lesson 30 and Tutorial 07 Part 2/3 before trying again — this is the single question most likely to come up if this project's architecture is ever discussed in an interview.

That's every concept Track 4 set out to teach: the endpoint map (31), the FormData/UploadFile bridge (32), the uniform error contract (33), CORS's two separate permissions (34), the adapter/business-rule boundary (35), and browser-owned vs. server-owned state (36) — all reassembled here into two real requests, start to finish. You should now be able to explain, from memory and without notes, why this backend has no run_id, why report endpoints re-accept files, why backend/ stays thin, and why browser details like multipart boundaries and CORS exposed headers were never trivia.
