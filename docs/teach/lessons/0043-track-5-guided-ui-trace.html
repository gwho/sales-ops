Trace 1 — A Validation Error Row, From Contract to Table Cell

Predict each numbered step yourself first — what holds the data, what derives from what, what actually reaches the DOM — then reveal.

  1. types/index.ts defines ValidationErrorRow.
  2. runValidation() resolves and calls setCurrentResult(result).
  3. const errors = useMemo(() => currentResult?.errors ?? [], [currentResult]).
  4. filteredErrors — a second useMemo, applying the severity dropdown and search box.
  5. <DataTable columns={ERROR_COLUMNS} data={filteredErrors} getRowKey={...} />.
  6. Inside DataTable<T>, sortedData — a third useMemo.
  7. Each row: <TableRow key={getRowKey(row)}>.
  8. column.render(row) fills each <TableCell>.
  9. The sortable header itself — what element is it, and how is sort direction actually conveyed?
Reveal: the full chain, with the file/function behind every step

1. ValidationErrorRow (types/index.ts, lines 22–29) is copied verbatim from context/ui-contract-plan.md — snake_case, no camelCase adapter, a decision Lesson 41's Server/Client material never had reason to revisit. This is a declared static contract, not a runtime guarantee — worth stating precisely after Lesson 42's own correction: TypeScript checks that this frontend's own code uses ValidationErrorRow values consistently with the declaration, but lib/api-client.ts's postJSON<T> parses the real response with (await response.json()) as T — a type assertion, not runtime schema validation. A malformed backend response could violate this contract without producing a compile error anywhere. 2. The entire OrderValidationResult becomes page state (app/(workspace)/order-validation/page.tsx, line 154 declares currentResult; line 184's setCurrentResult(result) is what actually sets it) — not just its errors field pulled out early. That's a source-of-truth/derived-state decision, not evidence of Lesson 36's snapshot rule by itself: the snapshot rule is that any one render sees one fixed currentResult until an update schedules the next; storing the whole object rather than splitting it into separate fields is a separate state-shape choice, made so errors, valid orders, and every summary figure can all be derived from one source instead of drifting independently. 3. errors (line 236) is derived, not independently stored — React recomputes the memoized expression when currentResult changes and otherwise reuses the same cached array reference; "recomputed on every render" would be wrong, since that's precisely what useMemo exists to avoid. 4. filteredErrors (lines 239–249) recomputes specifically when errors, errorSeverity, or errorSearch is not Object.is-equal to its value from the previous render — Lesson 38's precise dependency-array rule, not a vaguer "when something changes." 5. At the call site (line 425), TypeScript combines the typed columns={ERROR_COLUMNS}, data={filteredErrors}, and the contextual type of the getRowKey callback to infer T as ValidationErrorRow — columns alone is a contributor, not the sole source of the inference, and no explicit <ValidationErrorRow> argument is written at the call site either way. 6. sortedData (components/tables/DataTable.tsx, lines 65–78) builds [...data].sort(...) — an immutable copy, never data.sort() in place (Lesson 38). 7. getRowKey={(r) => `${r.row_number}-${r.error_code}`} (line 428) is stable across re-renders but — Lesson 38's own real, still-unfixed finding — not always unique: a real test (test_ov001_emits_one_error_per_missing_field_in_fixed_order) proves one row can emit two same-code errors, which this composite key can't tell apart. 8. Every cell renders inside real semantic markup (components/ui/Table.tsx) — an actual <table>, <thead>, <tbody>, <tr>, <th>, <td>, not styled <div>s (Lesson 40's "semantic HTML before ARIA" rule, satisfied by construction here). 9. It's a real <button> (DataTable.tsx, lines 99–109) — keyboard-operable for free — but the current sort direction is conveyed only by a visible ▲/▼ character in the button's own text, never an aria-sort attribute on the header cell, exactly the real, still-open gap Lesson 40 named and didn't fix.

Trace 2 — The Dashboard's Live Sections, From Server Shell to Accessible Chart

Same exercise. This one crosses the Server/Client boundary twice, not once — predict both crossings, and predict what actually happens to a Server Component sandwiched in between them.

  1. GET /dashboard renders app/(workspace)/layout.tsx first, then the page.
  2. That layout returns <AppShell>{children}</AppShell> — is AppShell a Server or Client Component?
  3. app/(workspace)/dashboard/page.tsx (a Server Component) is passed to AppShell as children — does that pull it into AppShell's client module graph?
  4. DashboardPage renders its own static shell, then <DashboardLiveSections />.
  5. DashboardLiveSections.tsx opens with its own, separate "use client".
  6. First client render of that component: status starts as "loading".
  7. After mount, the Effect fires: getDashboardResults() runs.
  8. Inside it, getOrCreateSessionId() reads/writes localStorage.
  9. GET /api/dashboard is sent, with or without an X-Session-Id header.
  10. On success: three separate setLive*(...) calls, then setStatus("loaded").
  11. Live-or-sample resolution: liveOrderValidation ?? orderValidationResult, independently per workflow.
  12. The Effect's own cleanup: what does cancelled actually protect against, and when does it matter?
Reveal: the full chain, with the file/function behind every step

1. app/(workspace)/layout.tsx is a plain Server Component, per Lesson 41's route-group trace. 2. components/layout/AppShell.tsx opens with "use client" (line 1) — it is a real Client Component, not a passthrough wrapper. It owns real client state and browser-only behavior: mobile-drawer isDrawerOpen state, usePathname() route tracking to auto-close the drawer on navigation, an Escape-key listener, and moving focus into the drawer's first link on open — none of which could run during a server render. 3. This is the exact "Interleaving Server and Client Components" pattern the installed Next.js docs describe: "You can pass Server Components as a prop to a Client Component... A common pattern is to use children to create a slot in a <ClientComponent>." DashboardPage is rendered on the server first, and only its already-rendered output is handed to AppShell as children — the docs are explicit that this "does not apply" the client-bundling rule: "Server Components passed as children or other props... are not imported into the Client Component's module graph. They are rendered on the server and passed to the Client Component as rendered output." DashboardPage stays a genuine Server Component the whole time, even though it visually renders inside a Client Component. 4. Its own heading, intro paragraph, and the Reports/Workflows sections render as static server output; only <DashboardLiveSections /> needs anything more. 5. That's a second, independent "use client" boundary (line 1 of a completely different file) — nested inside AppShell's already-client tree, but drawing its own separate line for its own separate reason: Lesson 33's rule, applied at two different, deliberately narrow points, not one single boundary for the whole page. The component's own comment names that reason: the Anonymous Session ID lives only in localStorage, a capability that doesn't exist during a server render. 6. Before the Effect below has ever run, status === "loading" renders only a <LoadingState /> — Lesson 37's point that an Effect's result is never available on the render that schedules it. This is the literal code behind a deliberate product decision the component's own comment names directly: never flash sample values first and swap them for live ones after the fetch resolves, since that would blur "is this real or sample?" for a real visitor (ADR 0007). 7. The empty dependency array (DashboardLiveSections.tsx, lines 53–73) is Lesson 37's "runs once per mount" case, synchronizing with something genuinely outside React — a network request — not a value read from props or state. 8. getOrCreateSessionId() (lib/session-id.ts) is the actual browser-only capability driving step 5's Client boundary — reading window.localStorage, creating a UUID with crypto.randomUUID() on first use, and returning null (never throwing) if storage is unavailable. (A small, real, current gap worth naming: this function's own docstring claims it "always returns a non-empty string," which the actual signature — string | null — and these two real null-returning paths both contradict. The code and callers are correct either way; the comment is simply stale.) 9. getDashboardResults() (lib/api-client.ts, lines 179–192) sends the session ID as X-Session-Id only if one exists — an empty header is never sent, since the backend treats an empty string as malformed, not absent. 10. Three independent setLive* calls (never one combined state object) is a state-modeling decision, not the state-snapshot rule itself: it lets each persisted workflow result be absent or present independently. The snapshot rule is the narrower, separate fact that any one render sees one fixed value for each of those three variables until an update schedules the next. 11. resolvedInventoryAllocation = liveInventoryAllocation ?? inventoryAllocationResult (and the matching lines for the other two workflows) resolve independently — a visitor who's only run Order Validation sees real data there and sample data everywhere else, never all-or-nothing. 12. cancelled (Lesson 37) guards against a slower, now-stale request finishing after a newer Effect instance has already started — an unmount, or a development Strict Mode remount — overwriting state that a fresher instance already owns. This connects back to Lesson 42, precisely stated: the three existing automated checks (typecheck, lint, build) cannot prove this trace's complete runtime behavior, but that doesn't mean only a real browser could ever catch a regression here — a wrong state type is still a typecheck catch, a missing Effect dependency is still an ESLint catch, and the cancelled race itself is exactly the kind of thing a unit or integration test with controlled, delayable promises could exercise without a browser at all. The real lesson is choosing the smallest layer that can actually observe a given regression, not reaching for the heaviest one by default.

Reveal: the five-point rubric, a model answer, and how to score yourself

Score one point for each of these your answer genuinely touched, not just gestured at:

  1. Source of truth and derived values — is this behavior's data genuinely re-derivable each render from something else, or does it need its own state slot? (Lesson 36)
  2. External synchronization and Effects — does anything here need to synchronize with a system outside React itself — a fetch, a subscription, a timer — the actual trigger for an Effect, not just "this runs after render"? (Lesson 37, Trace 2)
  3. Browser-only capabilities and Client boundaries — does the behavior depend on something that simply doesn't exist during a server render — localStorage, window, a real user event — the actual trigger for a Client boundary, not "it has some interactivity"? (Lesson 33, Lesson 41, Trace 2 and 3)
  4. Serializable Server-to-Client values — if a Server Component needs to hand this value to a Client Component as a prop, can it actually make that trip, or is it a function or class instance that can't? (Lesson 35, Tutorial 10 Part 3)
  5. Observable risk and the smallest sufficient layer — what's genuinely at risk if this breaks, and what's the cheapest check positioned to actually observe that specific failure — not automatically the heaviest one available? (Lesson 42)

Model answer, compressed: "First I ask whether the value is genuinely new information or just derivable from something I already have — derived stays out of state entirely. If it needs to live in state, I ask whether anything must synchronize with the world outside React; if not, it's an event handler, not an Effect. Separately, I ask whether the behavior touches a browser-only capability — that's what actually forces a Client Component boundary, not merely 'this is interactive.' If a Server Component needs to hand this value to a Client Component, I check it's actually serializable first. And last, I ask what would actually break and what the cheapest check is that could catch that specific break — a type, a lint rule, and a build already cover a lot; I only reach for a real browser, scripted or by hand, for the part those three genuinely can't see."

Scoring: 5/5 — you're not overcorrecting toward "just add a Playwright test" the way Lesson 42 warned against; jumping straight there without touching the other four is the single most common miss on this drill. Lesson 42's own two real bugs did need an actual browser (or scripted browser automation) to surface — that part is real, not a strawman — but the accurate lesson from that pair was narrower than "always test in a real browser": this project didn't need to add a permanent new test dependency just to diagnose them, since a temporary, uninstalled-afterward Playwright script was enough. 3–4/5 — solid, but check which point got skipped and reread that lesson specifically. 0–2/5 — worth rereading Lessons 33, 35, 36, 37, and 42 in sequence before moving on, since this drill is built to need all five at once, not any one of them alone.
