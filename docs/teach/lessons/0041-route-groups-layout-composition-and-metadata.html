Route Groups, Layout Composition, and Metadata

The portfolio landing page added a structure Tutorial 09 never had to teach: app/(public)/ and app/(workspace)/ — two folders in parentheses that reorganize every route in this app without changing a single URL. This lesson traces exactly what those parentheses do, why one root layout still sits above both of them, and why a page's own <title> tag can come from a file that isn't even the page rendering it.

Primary sources

The installed Next.js docs' own "Route groups" section, for what the parentheses convention actually does. The "Layouts and Pages" page, for the root layout's html/body requirement and how nested layouts wrap each other. The "Metadata and OG images" page, for the exact Server-Component-only rule. The "generateMetadata" reference's Ordering/Merging/Inheriting sections, for what actually happens to a route's <title> when only some of its segments define metadata. WCAG 2.2's "Understanding Page Titled", for why a unique title is a navigation/accessibility need, not only an SEO one.

Why this matters

All three mission drivers apply: in an interview, "route groups organize files, they don't change the URL or add a security boundary" is a precise, checkable sentence, not a vibe; the root-layout/nested-layout/metadata-inheritance rules are identical in any App Router project, not specific to this repo; and this repo's real file layout — one root layout, two route groups, one page-level metadata override — is exactly what you'll read the first time you open app/ to extend it.

Route groups organize files, they don't route

The Next.js docs state the rule for a parenthesized folder plainly: "This indicates the folder is for organizational purposes and should not be included in the route's URL path." Two such folders exist right now, holding a very uneven number of routes each — (workspace) holds five, (public) currently holds just one:

Folder path → URL, verified against the real app/ tree

app/(public)/page.tsx                       -> /
app/(workspace)/dashboard/page.tsx          -> /dashboard
app/(workspace)/order-validation/page.tsx   -> /order-validation
app/(workspace)/inventory-allocation/page.tsx -> /inventory-allocation
app/(workspace)/payment-aging/page.tsx      -> /payment-aging
app/(workspace)/reports/page.tsx            -> /reports

Notice what's missing from every URL on the right: neither public nor workspace appears anywhere. docs/architect/portfolio-landing-page/decisions.md records the actual reason this repo added these two groups — a portfolio visitor landing on / shouldn't see the operational sidebar chrome built for repeated internal use before they've even opened the demo, and the group split gets that without moving /dashboard to /workspace/dashboard. The same decision doc also names a consequence worth knowing: navigating from / to /dashboard via <Link> or the router stays one client-side transition, no full page reload — because this app defines only one root layout, shared by both groups. (Next.js's own route-groups reference names a full-reload caveat, but it applies specifically to navigating between two different root layouts in a multiple-root-layout setup — a real, documented use case for route groups that this repo simply doesn't use.) That's the more precise way to say it: a route group is an organizational and layout-selection boundary — its documented use cases include opting a subset of routes into a shared layout, or even defining separate root layouts entirely — but it is never a URL boundary and never an authorization boundary. Nothing about (workspace)'s name restricts who can load those routes — this repo simply has no login step at all, a fact Track 4's statelessness material already established, not something the folder name implies.

One root layout, two chrome layouts underneath it

Every route in this app still passes through exactly one root layout first. The docs are specific about what that file is required to contain: "The root layout is required and must contain html and body tags." This repo's root layout does exactly that, and nothing more:

app/layout.tsx, lines 16–33

// Global providers only -- (public) and (workspace) route groups each supply
// their own chrome (public header/footer vs. AppShell's sidebar/top header).
export default function RootLayout({
  children,
}: Readonly<{
  children: React.ReactNode;
}>) {
  return (
    <html lang="en" className={inter.variable}>
      <body
        className="min-h-screen bg-background font-sans text-text-primary"
        suppressHydrationWarning
      >
        {children}
      </body>
    </html>
  );
}

The component's own comment states the division of labor this lesson is built around: the root layout owns document-wide concerns only — the <html> element, the <body> element, the global font variable, the base background/text color. It renders {children} and stops. Every route's real visible chrome — header, footer, sidebar — comes from whichever nested layout sits between the root and the page. The docs describe nesting itself as automatic: "By default, layouts in the folder hierarchy are also nested, which means they wrap child layouts via their children prop." This repo has exactly two nested layouts, one per route group, and both are short enough to read in full:

app/(public)/layout.tsx, lines 1–14 (complete file)

import { type ReactNode } from "react";

import { PublicHeader } from "@/components/landing/PublicHeader";
import { PublicFooter } from "@/components/landing/PublicFooter";

export default function PublicLayout({ children }: { children: ReactNode }) {
  return (
    <div className="flex min-h-screen flex-col bg-background">
      <PublicHeader />
      <main className="flex-1">{children}</main>
      <PublicFooter />
    </div>
  );
}

app/(workspace)/layout.tsx, lines 1–7 (complete file)

import { type ReactNode } from "react";

import { AppShell } from "@/components/layout/AppShell";

export default function WorkspaceLayout({ children }: { children: ReactNode }) {
  return <AppShell>{children}</AppShell>;
}

Neither file touches <html>, <body>, or the font variable — that would be duplicating a global provider the root layout already supplies, one of this lesson's named failure modes. Each one instead renders exactly the one real UI decision its route group needs: a plain header/footer wrapper for a public visitor, the existing AppShell (sidebar + top header, from Lesson 34/Tutorial 09-10) for the operational workspace. Composing all three files for /dashboard reads, from the outside in: RootLayout (html, body, font) wraps WorkspaceLayout (AppShell) wraps DashboardPage. For /, swap in PublicLayout in that same middle slot.

Static metadata is a Server-Component-only export

This repo's <title> tags come from a plain exported metadata object, not JSX, and the docs restrict where that export is legal: "The metadata object and generateMetadata function exports are only supported in Server Components." That restriction holds regardless of when the metadata actually resolves — the generateMetadata reference explains why it exists at all: metadata "must be resolved on the server before the page component is rendered." What that looks like in practice differs by case, though. Every metadata export in this app is the plain static object form — verified by grep, no generateMetadata function exists anywhere in app/ — and every route here is prerendered, so Next.js resolves that object at build time and writes it straight into the initial HTML your grep will check below. A dynamic generateMetadata function, which this app doesn't use, behaves differently on a dynamically-rendered route: the docs' own streaming-metadata section states Next.js can stream its resolved result in after the initial UI, rather than blocking the whole response on it first. Static-and-prerendered versus dynamic-and-streamed are two different delivery timings for the same Server-Component-only rule — not two different rules, and not a distinction this app's all-static metadata ever needs to navigate. The root layout defines the whole app's fallback:

app/layout.tsx, lines 10–14

export const metadata: Metadata = {
  title: "Sales Admin Automation Toolkit",
  description:
    "Order validation, inventory allocation, payment aging, and report export dashboard.",
};

The landing page overrides both fields for its one route:

app/(public)/page.tsx, lines 16–20

export const metadata: Metadata = {
  title: "Sales Admin Automation Toolkit | Portfolio Case Study",
  description:
    "Order validation, inventory allocation, and payment aging workflows automated with Python and a Next.js dashboard — a portfolio case study.",
};

Every one of the five (workspace) pages, by contrast, exports no metadata object at all. For three of them, the current page.tsx module genuinely can't — it's a Client Component, and that export is Server-Component-only. order-validation/page.tsx states its own reason in its opening comment:

app/(workspace)/order-validation/page.tsx, lines 1–5

"use client";

// This page must be a Client Component: its DataTable column definitions
// contain render functions, and functions cannot cross the Server->Client
// props boundary. Phase 10: also owns live Workflow Request/Result state.

inventory-allocation/page.tsx and payment-aging/page.tsx carry the identical "use client" directive for the same reason (Lesson 33's Server/Client rule, applied). This is a fact about the current module, though, not an inherent limit of the route — nothing stops /order-validation from having both real metadata and interactive client behavior at the same time. The correct move, if this app ever wanted that, would be exactly what the docs show: keep page.tsx itself a Server Component that exports the metadata, and push the interactive logic into a separate imported Client Component. This lesson's failure mode is reaching for a metadata export directly on a module you've already marked "use client", not the (false) belief that the route could never have one.

This repo hasn't made that Server-wrapper move for any workspace page, and it's worth being precise about why not: it isn't a documented decision. Nothing in docs/architect/portfolio-landing-page/decisions.md or any other architecture doc discusses page-level SEO intent for workspace routes, and this app has no robots/noindex mechanism enforcing anything either way — every workspace page is just as publicly reachable and indexable as /. Sharing one title across five routes is this app's current behavior, not a considered trade-off, and it's worth naming as a real, low-risk improvement rather than a gap to leave unexamined. Unique titles matter for more than search engines, too: WCAG's Page Titled criterion exists because titles "identify the current location without requiring users to read or interpret page content" — exactly what a keyboard or screen-reader user needs when several of this app's own tabs are open side by side.

dashboard/page.tsx and reports/page.tsx are different: both stay plain Server Components (no "use client" anywhere in either file) — they simply choose not to export a metadata object. That choice has a real, precise consequence, not an accidental one. The generateMetadata reference's own worked example shows the general rule: a child segment's metadata is "shallowly merged" with its parent's, and where the child doesn't set a field at all, that field is "inherited" from the parent unchanged. dashboard/page.tsx sets no title/description fields, so nothing overrides the root layout's — the real, built output confirms this exactly:

Prerendered HTML, .next/server/app/index.html vs. .next/server/app/dashboard.html

index.html:     <title>Sales Admin Automation Toolkit | Portfolio Case Study</title>
dashboard.html: <title>Sales Admin Automation Toolkit</title>

/dashboard's title is the root layout's string, untouched — not empty, not re-derived, not the landing page's suffixed version. It's the single concrete case behind this lesson's tangible win: given a page's folder path and whether it exports its own metadata, you can predict its rendered <title> without opening a browser.

Correct framing

"(public) and (workspace) decide which layout wraps a page, not what its URL is or who's allowed to see it. The URL comes from the folder path with the parenthesized parts removed; access control, if this app ever needed it, would be a completely separate mechanism layered on top."

A framing to avoid

"The workspace routes are protected because they're inside (workspace)." (No code anywhere checks which group a route belongs to at request time — the parentheses are stripped from the URL and never seen again after routing resolves. This repo has no auth layer at all, workspace routes included.)

Tiny exercise (7 minutes)

Exercise — predict a <title>, then verify it against a real build

Predict first. Before running anything: reports/page.tsx, like dashboard/page.tsx, is a Server Component that exports no metadata object. Write down what you expect its rendered <title> to be, and name the one rule from this lesson that justifies your answer (ordering, merging, or inheritance).

Verify. Run npm run build from the repo root (a disposable, gitignored rebuild of .next/ — nothing here is committed or permanent). Then run:

grep -o '<title>[^<]*</title>' .next/server/app/reports.html

Compare the output against your prediction and against this lesson's own index.html/dashboard.html comparison above. Nothing here edits any application file — the entire loop is a build command and a grep, both fully reversible by construction. One caveat worth naming honestly: .next/server/app/ is this installed Next.js version's own internal build-output layout — real and inspectable right now, verified against this repo's actual build, but not a documented, version-stable public API. Treat it as good for a one-off local check like this one, not something to script permanently into other tooling.

If you can state, for any page under app/, its real URL, its full layout stack from root down to the page, and what its rendered <title> will be — all three without opening a browser — this lesson has landed. It deliberately stops short of loading.tsx/error.tsx/not-found.tsx route-level failure UI, which this repo doesn't currently use at all; that's flagged as a further, deliberately optional topic in the task file behind this lesson, not something to chase down here.
