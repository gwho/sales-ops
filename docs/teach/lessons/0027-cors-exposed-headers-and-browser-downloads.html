CORS, Exposed Headers, and Browser Downloads

A request can succeed completely — status 200, correct body, visible in DevTools — and still "fail" from your own JavaScript's point of view. This lesson rehearses the one CORS distinction that explains it, and how a report download turns a header into a saved file.

What you're rehearsing

Tutorial 07 Part 8, against backend/main.py's CORS config and lib/api-client.ts's postReport()/downloadBlob().

Two separate permissions, easy to conflate

Open backend/main.py:

app.add_middleware(
    CORSMiddleware,
    allow_origins=_cors_origins(),
    allow_methods=["GET", "POST"],
    allow_headers=["*"],
    expose_headers=["Content-Disposition", "X-Report-Id", "X-Generated-At", "X-Persisted"],
)
SettingControlsGet it wrong and...
allow_originsWhether the cross-origin request is permitted at allThe browser blocks the whole request — no body, nothing
expose_headersWhich response headers, beyond a small browser safelist, cross-origin JS may readThe request succeeds fully, but response.headers.get(...) silently returns null for anything not listed

These are not two phrasings of the same rule. A response can be a complete, correct 200 — visible whole in the Network tab, including every header — while your own calling JavaScript still can't read one specific header, because expose_headers is a separate, independent permission from "was the request allowed to happen."

Headers as part of the API contract, not incidental metadata

Content-Disposition, X-Report-Id, and X-Generated-At aren't decoration on top of the "real" response (the file bytes) — they're load-bearing parts of the contract. lib/api-client.ts's postReport() reads all three directly off the response and returns them as part of its own result type. If any one of them were missing from expose_headers, the request would still "work" in the sense of downloading real bytes — but the filename, report ID, and generation timestamp the UI displays would silently be blank or fall back to a default, with no error anywhere to point at the actual cause.

From header to saved file: postReport → downloadBlob

export async function postReport(
  path: string,
  formData: FormData,
  fallbackFilename: string,
): Promise<ReportDownload> {
  const response = await postFormData(path, formData);
  const blob = await response.blob();
  return {
    blob,
    filename: parseFilename(response.headers.get("content-disposition"), fallbackFilename),
    reportId: response.headers.get("x-report-id"),
    generatedAt: response.headers.get("x-generated-at"),
  };
}

export function downloadBlob(blob: Blob, filename: string): void {
  const url = URL.createObjectURL(blob);
  const link = document.createElement("a");
  link.href = url;
  link.download = filename;
  document.body.appendChild(link);
  link.click();
  document.body.removeChild(link);
  URL.revokeObjectURL(url);
}

postReport turns the response body into an in-memory Blob — not yet a downloaded file. downloadBlob is the part that actually triggers a save: URL.createObjectURL(blob) makes a temporary blob: URL the browser can navigate to, a hidden <a download> element gets created and clicked programmatically, then removed, and URL.revokeObjectURL frees the temporary URL afterward. This exists because there's no simpler browser API for "save these already-fetched bytes as a file" — a Blob isn't something you can just assign to window.location.

Contrast this with the "Sample file" link elsewhere in the UI — a plain <a href download> that never touches fetch() at all. That's a normal browser navigation; no JavaScript ever receives the response to read headers or a body from, so CORS never enters the picture. CORS only governs what cross-origin JavaScript is allowed to read — and a plain navigation never hands the response to any JavaScript in the first place.

Tiny exercise (10 minutes)

With both dev servers running (uv run fastapi dev backend/main.py and npm run dev), open /order-validation, run validation on the sample data, then click "Download Report." In DevTools' Network tab, open the /api/orders/validate/report request's Response Headers and confirm X-Report-Id and Content-Disposition are genuinely present. Then, purely on paper, remove "X-Report-Id" from backend/main.py's expose_headers list (don't actually save the change) and predict exactly what value postReport()'s reportId field would hold on the next request.

Failure mode named: seeing a header correctly present in DevTools' Network tab and concluding the frontend code must have a bug, when the actual gap is a missing entry in expose_headers. The header genuinely exists on the wire — DevTools reads the raw HTTP response directly, bypassing the browser's CORS-JS boundary entirely — so "I can see it in DevTools" is not evidence that JavaScript can read it too.

If it's still unclear why the browser bothers hiding headers from JavaScript at all (rather than just blocking or allowing the whole response), ask now — it's worth understanding as a security design choice, not an arbitrary inconvenience, before moving on.
