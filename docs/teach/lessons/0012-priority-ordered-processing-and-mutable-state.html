Track 2 — Business Rules & Testing Discipline

Priority-Ordered Processing and Mutable State

Your direct preparation for Tutorial 04. Inventory allocation is the first module in this repo where order matters — filling one order line changes what the next line sees. This lesson teaches the plain-language idea before you meet the code.

Building on

No dedicated external resource — this is best taught directly against sales_admin_automation_toolkit_specs/02_demo_inventory_allocation.md (Rule IA-001), which is already the primary source. Lesson 2's material on dicts is extended here with something Track 1 never covered: changing a dict after it's built.

A limited-stock problem

Picture 10 units of a SKU in stock, and two orders wanting it: a High-priority order for 8 units, and a Normal-priority order for 8 units. There are two very different ways to "fairly" split this:

  • Share proportionally — both orders get some fraction of what's left.
  • Serve by priority, in full — the High order gets everything it asked for (8 of 10) first; the Normal order gets only what's left (2 of 10).

Rule IA-001 picks the second one, explicitly: sort every order by priority (then delivery date, then order date, then order ID) before allocating anything, and process strictly in that order. This is not a detail — it's the entire meaning of "priority" in this module. Sharing proportionally would make the High order wait its turn like everyone else, which defeats the point of calling it High at all.

One line at a time, never revisited

Open src/inventory_allocation.py and find _allocate_order_line:

chosen["allocatable_qty"] -= allocated_qty
chosen["allocated_qty"] += allocated_qty

These two lines are the whole mechanism. chosen is a dict representing one SKU/warehouse's stock. -= and += don't build a new dict — they mutate (change in place) the exact dict object that's already sitting in this module's shared inventory tracking. The very next order line that wants the same SKU looks up that same dict again and sees the already-reduced number — nobody has to remember to "write the update back" anywhere, because there was only ever one dict, not a copy.

Mutating vs. rebinding — the distinction Track 1 skipped

Track 1 taught you to read a dict (row["sku"]) but never to change one. There are two different things "changing a dict" could mean: chosen = {"allocatable_qty": 5} (rebinding — chosen now points at a brand-new, different dict; anyone else still holding the old dict sees nothing change) versus chosen["allocatable_qty"] = 5 (mutating — the same dict object is edited, so everyone holding a reference to it sees the new value). -=/+= on a dict key are mutation, not rebinding — that difference is exactly what makes the second order line automatically see the first line's effect.

Watching it happen in a real test

tests/test_inventory_allocation.py proves this directly:

def test_warehouse_choice_switches_as_stock_depletes():
    orders = _orders_df(
        _order_row(order_id="SO-2026-001", quantity=15),
        _order_row(order_id="SO-2026-002", quantity=15, requested_delivery_date=date(2026, 7, 16)),
    )
    inventory = _inventory_df(
        _inventory_row(warehouse="HK Warehouse", available_qty=20),
        _inventory_row(warehouse="SG Warehouse", available_qty=10),
    )
    result = allocate_inventory(orders, inventory)
    first = _result_by_order_id(result, "SO-2026-001")
    second = _result_by_order_id(result, "SO-2026-002")
    assert first["warehouse"] == "HK Warehouse"
    assert second["warehouse"] == "SG Warehouse"

Both orders want the same SKU. The first order picks HK Warehouse — it starts with more stock. Filling it drops HK's remaining stock low enough that, by the time the second order runs, SG Warehouse is now the better choice — a genuinely different answer than if you asked "which warehouse has more stock" once, up front, for both orders at the same time. That's only possible because the first order's allocation actually changed the shared numbers before the second order looked at them.
