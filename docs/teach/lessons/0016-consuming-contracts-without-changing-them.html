Track 3 — Presentation Without Leakage

Consuming Contracts Without Changing Them

Lesson 15 taught the rule: report_export.py never recalculates. This lesson is about a narrower, related question: when a presentation module needs to read a contract someone else produced, where does that contract's type live, and what is the consumer allowed to do with it?

Primary source

Builds directly on Track 2's own reference doc, Result Envelope Pattern — if the three envelope shapes (OrderValidationResult, InventoryAllocationResult, PaymentAgingResult) don't feel solid, revisit that page first. The new material here comes from docs/plan/phase-6-excel-report-export/explanation.md §2 — a real mistake caught during development, not a decision that was obvious from the start.

Producers and consumers

Tutorials 03–05 are producer modules — each one builds its own envelope (OrderValidationResult, and so on) and returns it. Every producer module owns the shape of its own output. report_export.py is this repo's first consumer module — it never builds an envelope, it only ever receives one as a parameter and reads from it.

The ImportError that teaches the rule

Open src/report_export.py and find its import block:

from src.contracts import ReportManifest
from src.inventory_allocation import InventoryAllocationResult
from src.order_validation import OrderValidationResult
from src.payment_aging import PaymentAgingResult

Only ReportManifest comes from src/contracts.py. The three envelope types each come from the module that actually produces them. This wasn't obvious from the start: the first draft of this file imported all four types from src.contracts, and the test suite failed immediately with ImportError: cannot import name 'InventoryAllocationResult' from 'src.contracts' — because that type has never lived there. contracts.py only holds the shared, spec-defined row/summary shapes; each envelope is defined locally, once, by the module that builds it.

Reading a contract without changing it

Once a consumer imports a producer's type, it's read-only in a real sense: a consumer module never adds a field to someone else's envelope just because it would be convenient. When report_export.py's Original Orders sheet needed data OrderValidationResult doesn't contain — the raw uploaded rows — it did not add a raw_orders key to that contract. Instead, export_order_validation_report takes a second, completely separate parameter:

def export_order_validation_report(
    result: OrderValidationResult,
    original_orders_df: pd.DataFrame | None = None,
    generated_at: datetime | None = None,
) -> tuple[bytes, ReportManifest]:

Why not just add the field?

OrderValidationResult is also read by every other caller of validate_orders() — a future FastAPI route, a future test, anything that only cares about validation and never generates a report. Adding a raw_orders field to the shared contract would mean every one of those callers now carries a payload only one consumer actually needs. A separate parameter keeps the contract exactly what Tutorial 03 defined it as, untouched by one particular consumer's presentation-layer needs.
