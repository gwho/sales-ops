Track 2 — Business Rules & Testing Discipline

Structured Errors and Result Envelopes

Lesson 8 drew the line between a business rule violation and a programming error. This lesson is about the shape a business module hands back once it's sorted every row into one bucket or the other — no printing, no crashing, no loose tuple of DataFrames.

Building on

No new external source this time — this lesson is Lesson 4's (Track 1) TypedDict/contract material, applied to three real modules instead of one bare definition. Reread Lesson 4 first if TypedDict itself feels shaky.

The rule, stated plainly

context/code-standards.md states this repo's rule directly:

Do not print from business logic; return structured errors or results.

Two alternatives this repo deliberately avoids: a function that print()s a problem straight to the console (nothing downstream — a future web page, a report — can use printed text), and a function that just crashes on the first bad row (Lesson 8's crash-vs-report distinction, now at the scale of a whole batch instead of one cell). The chosen alternative: process every row, and hand back one structured value describing everything that happened.

One dict, several named parts

Each of the three Track 2 modules defines its own small TypedDict — a result envelope — bundling everything that one call produces. Compare the three:

# src/order_validation.py
class OrderValidationResult(TypedDict):
    summary: ValidationSummary
    valid_orders: list[ValidOrderRow]
    errors: list[ValidationErrorRow]

# src/inventory_allocation.py
class InventoryAllocationResult(TypedDict):
    summary: AllocationSummary
    allocation_results: list[AllocationResultRow]
    backorders: list[BackorderRow]
    remaining_inventory: list[RemainingInventoryRow]
    supplier_follow_ups: list[SupplierFollowUpRow]

# src/payment_aging.py
class PaymentAgingResult(TypedDict):
    summary: PaymentAgingSummary
    aging_rows: list[PaymentAgingRow]
    data_issues: list[PaymentDataIssueRow]
    draft_messages: list[DraftMessageRow]

Three different shapes — Order Validation only has two lists, Inventory Allocation has four — but the same idea every time: one function call, one dict, every output the module produces bundled under named keys. A caller runs validate_orders(...) once and gets back everything at once — a summary, the rows that passed, and the rows that didn't — instead of needing to call three separate functions and stitch their results together by hand.

Why not just return a tuple?

A tuple like (valid_orders_df, errors_df) works, but it's positional — the caller has to remember "the first one is valid orders, the second is errors," with nothing in the code itself enforcing that memory. A named dict key can't be silently mixed up the way tuple position can. This is the same "give the promise a name" idea Lesson 4 introduced for TypedDict fields generally, just applied to a whole function's return value instead of one row.

Summary and rows are different questions

Notice every envelope's first key is summary — a small object with counts (how many valid, how many errors) — separate from the rows themselves. "How many problems are there" and "what exactly are they" are different questions a caller might want answered independently — a KPI card needs the first; a detail table needs the second. Bundling both under one envelope means neither question requires a second function call.
