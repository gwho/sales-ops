Track 3 — Presentation Without Leakage

Explicit Columns & Empty Reports

A tempting shortcut for writing a sheet's header row is to just read whatever keys the first row of data happens to have. It works — right up until the data doesn't cooperate. This lesson is about two real ways it stops working, and why a report a human reads needs to survive both.

Primary source

No dedicated external resource — like Lesson 15, this is a repo-specific design decision, best learned from docs/plan/phase-6-excel-report-export/explanation.md §5. It builds directly on docs/teach/lessons/0004-contracts-typeddict-and-json-shapes.html's NotRequired discussion — revisit that lesson first if "a field that's sometimes just absent from the dict" doesn't feel familiar.

The tempting shortcut

An early instinct for writing a detail sheet's header row: derive it from whatever the first row's keys happen to be — list(rows[0].keys()). Less code, and for a non-empty list of uniform rows, it produces the exact same output as an explicit list of column names. Two real scenarios break it.

Scenario 1: there is no first row

Open tests/test_report_export.py and find:

def test_order_validation_report_empty_errors_sheet_has_header_only():
    workbook_bytes, _ = export_order_validation_report(
        _order_validation_result(errors=[]), pd.DataFrame(), generated_at=GENERATED_AT
    )
    wb = _load(workbook_bytes)
    ws = wb["Validation Errors"]
    assert ws.max_row == 1
    assert _headers(ws) == VALIDATION_ERROR_COLUMNS

A validation run that found zero errors still needs a real, correctly-labeled Validation Errors sheet — ws.max_row == 1 means exactly one row exists, and it's the header. rows[0].keys() has nothing to read here at all; it would need a special case just for "the list happens to be empty," or the sheet would render with zero header cells — arguably worse than a header with zero data rows.

Scenario 2: two rows of the same type, different keys present

Recall NotRequired fields (Lesson 4): a key can be legitimately absent from one row's dict and present in another's, even though both rows are the same TypedDict. Open the same test file and find:

def test_order_validation_report_missing_notrequired_field_renders_blank():
    error_without_sku = {
        "row_number": 3,
        "error_code": "OV-001",
        "error_message": "Customer name is missing",
        "severity": "Error",
    }
    # ... exported, then:
    row = dict(zip(VALIDATION_ERROR_COLUMNS, [c.value for c in ws[2]]))
    assert row["sku"] is None

If headers were derived from "whichever row happens to be at index 0," the rendered column set would depend on which row a report happened to put first — not a property a report a human reads and files away should ever have.

The fix: headers as a property of the contract, not the data

Open src/report_export.py and find:

VALIDATION_ERROR_COLUMNS = ["row_number", "order_id", "sku", "error_code", "error_message", "severity"]
def _write_detail_sheet(wb, sheet_name, rows, columns):
    ws = wb.create_sheet(sheet_name)
    ws.append(columns)
    for row in rows:
        ws.append([_safe_cell_value(row.get(col, "")) for col in columns])
    ...

The header row comes from a plain Python list, declared once, that exists whether or not rows has anything in it. row.get(col, "") — for every column, on every row — is what makes this indifferent to which fields any individual row happens to omit.

Human-facing stability

A report's structure — its column set, in a fixed order — is part of what makes it readable across many runs. A sales admin opening this month's report and last month's shouldn't see different columns just because this month happened to have zero errors, or because one row was missing a field another row had. Humans read structure as much as they read data — a header row that survives an empty result is not a wasted formality, it's what makes the report recognizable as the same report every time.
