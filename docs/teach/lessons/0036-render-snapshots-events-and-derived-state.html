Render Snapshots, Events, and Derived State

Lesson 32 gave you the core loop: call a setter → React re-runs the component → new JSX. This lesson doesn't repeat that loop — it opens up what happens to your variables in between those two steps, using the real state in order-validation/page.tsx instead of invented examples.

Primary source

"State as a Snapshot" on react.dev — the official explanation of what this lesson's first section covers. For the derived-value section, "You Might Not Need an Effect" — read only its early sections on computing values during rendering; the Effect-specific parts of that page wait for Lesson 37.

Why this matters

All three mission drivers apply: you can explain in an interview why a value read immediately after a setter call doesn't show the new value yet; the "compute it during render, don't copy it into state" instinct is portable to any React codebase; and recognizing which of this repo's values are genuinely stateful versus merely derived is what keeps you from adding a redundant, driftable copy when you extend it.

Render is a function of state — a snapshot, not a live variable

Inside any single render, every state variable a component reads is a plain, fixed binding — not a live reference to some mutable box that updates the instant you call its setter. Calling a setter doesn't change the value in the function that's currently running; it schedules another render with the new value. That binding is fixed for this render, but if the value is an object, the object itself isn't automatically frozen — mutating its fields directly would change it without ever handing React a new value to render, which is exactly why this repo always replaces state objects wholesale instead of editing them in place (more on that below). Here's the shared helper both submission paths call to actually run Order Validation:

app/(workspace)/order-validation/page.tsx, lines 172–191

async function runValidation(orders: File, productMaster: File) {
    setStatus("submitting");
    setErrorDetail(null);
    setPersisted(null);
    try {
      const formData = new FormData();
      formData.set("orders_file", orders);
      formData.set("product_master_file", productMaster);
      const { data: result, persisted: persistedOutcome } = await postJSON<OrderValidationResult>(
        "/api/orders/validate",
        formData,
      );
      setCurrentResult(result);
      setPersisted(persistedOutcome);
      setStatus("succeeded");
    } catch (error) {
      setErrorDetail(error instanceof ApiError ? error.message : "Something went wrong. Please try again.");
      setStatus("failed");
    }
  }

If a debug line read console.log(status) immediately after setStatus("submitting"); on line 173, it would print whatever status was before this call — not "submitting". status here is the value React handed this render's copy of runValidation when it was created; calling setStatus schedules a fresh render with a fresh value, it doesn't reach backward and edit the constant this function already closed over.

The same function shows the companion idea, immutable updates: setCurrentResult(result) replaces the entire previous result with a brand-new object parsed fresh from the response. Nowhere in this repo does a handler do something like reassigning a field on the old currentResult in place — every update is "here is a whole new value," never "mutate the one you already had."

A genuine event, not an effect

runValidation contains event-driven work, but it isn't itself the event handler — it's a shared async helper. handleRunValidation and handleRunSampleData are the actual handlers, each wired to a click, and each calling runValidation once it has the files it needs:

app/(workspace)/order-validation/page.tsx, lines 324–326

<Button onClick={handleRunValidation} disabled={!canSubmit || status === "submitting" || sampleDataLoading}>
  {status === "submitting" ? "Validating…" : "Run Validation"}
</Button>

Notice this entire file's imports: import { useMemo, useState } from "react"; — no useEffect, anywhere in the page. That's not an omission; it's correct. Every piece of work on this page — running validation, fetching sample files, downloading a report — is caused by one specific user action. Nothing here needs to happen just because the component appeared, or because some value changed regardless of what caused the change. That second kind of work — synchronizing with something outside React whenever a condition holds, not because of one specific click — is what an Effect is for, and it's what Lesson 37 (not yet generated) covers next, tracing DashboardLiveSections.tsx, a component that genuinely needs one.

Derived values that don't need to be state

Not everything a component computes has to live in useState. This page derives several values fresh on every render, straight from state it already has:

app/(workspace)/order-validation/page.tsx, lines 169–170

const canSubmit = ordersFile !== null && productMasterFile !== null;
  const currentStep = status === "succeeded" ? 2 : canSubmit ? 1 : 0;

app/(workspace)/order-validation/page.tsx, lines 263–264

const errorFiltersActive = errorSeverity !== "All" || errorSearch !== "";
  const orderFiltersActive = orderPriority !== "All" || orderSearch !== "";

None of these four are useState. They're plain consts, recalculated every time the component re-renders, directly from state that already exists (ordersFile, productMasterFile, status, errorSeverity, errorSearch, and so on). Because they're recomputed every render instead of stored, there is exactly one source of truth for each — a value like canSubmit can never fall out of sync with the files it's checking, because it's never anything other than "the answer, computed right now."

Correct framing

"canSubmit is computed fresh every render from ordersFile and productMasterFile — there's nothing to keep in sync, because it isn't a separate copy of anything."

A framing to avoid

"I'll give canSubmit its own useState and update it with an Effect whenever the files change." (This creates a second, Effect-synchronized copy of information the component already has. It costs an extra render pass every time the files change, and if the Effect's dependency array is ever written slightly wrong, that copy can drift out of sync with the files permanently — a bug a plain derived const cannot have, because it's recalculated from scratch every render.)
