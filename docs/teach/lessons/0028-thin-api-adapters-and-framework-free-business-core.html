Track 4 — Optional Reinforcement · 5 of 7

Thin API Adapters and the Framework-Free Business Core

Every lesson in this reinforcement block has quietly depended on one rule: backend/ translates HTTP into calls on already-tested Python, and never decides anything itself. This lesson makes that rule the direct subject, and asks you to prove you can apply it to a workflow that doesn't exist yet.

What you're rehearsing

Tutorial 07 Part 5, against all three routers (backend/routers/orders.py, backend/routers/inventory.py, backend/routers/payments.py) and one business module you already know well, src/order_validation.py (Tutorial 03).

Two layers, one rule

context/architecture.md's System Boundaries table already settled this before Phase 10 wrote a single route: backend/ owns "FastAPI orchestration over tested modules" and must not own "duplicated business logic." Two concrete disciplines make that real rather than aspirational:

  1. No fastapi import ever enters src/. Every business module (Tutorials 03-06) raises its own plain pandas/openpyxl/business exceptions exactly as it does for a direct Python caller — it has no idea an HTTP layer even exists.
  2. Every route handler's return type is a real src/-module TypedDict, never a parallel Pydantic BaseModel. OrderValidationResult is imported directly from src.order_validation — not redefined, not wrapped, not subsetted.

Open backend/routers/orders.py's _load_and_validate once more, this time reading it purely for what it doesn't do:

def _load_and_validate(
    orders_file: UploadFile, product_master_file: UploadFile
) -> tuple[OrderValidationResult, pd.DataFrame]:
    orders_df = read_xlsx_upload(orders_file, "orders file", load_orders)
    product_master_df = read_xlsx_upload(
        product_master_file, "product master file", load_product_master
    )
    return validate_orders(orders_df, product_master_df), orders_df

There is no if statement anywhere in this function whose condition is about business meaning — nothing checking "is this SKU active" or "is this order missing a delivery date." Every real decision is one function call away, inside validate_orders(), which owns it. This function's entire job is describable purely in terms of where data comes from (an HTTP upload) and where it goes (a tested Python function) — never what the data means.

How to tell which layer a piece of code belongs in

Read any function and ask: does its correctness depend on a real-world policy (an adapter can be "correct" and still produce a business-wrong answer only if the layer it calls is wrong), or does it decide something about the data itself? An adapter's job is fully described by faithfully carrying data across a boundary. A business-rule module's job is fully described by applying the right real-world policy. If an adapter function ever grows a business-meaning if, that's the smell — the logic is living one layer too shallow.

Retrieval check

Why does every route handler return the real OrderValidationResult TypedDict instead of a separate Pydantic BaseModel?

A function converts an uploaded file into a DataFrame, and also checks "does this SKU exist in the product master" before returning it. Which layer does that SKU check belong in?

What would importing fastapi into src/order_validation.py immediately cost, even if the import was never actually used?

Tiny exercise (20-25 minutes) — design a fourth workflow, don't implement it

Assume a fourth, already-tested business function exists: process_returns(orders_df, returns_df) -> ReturnsProcessingResult in a new src/returns_processing.py. Design, on paper only, the thin adapter layer for POST /api/returns/process, following Phase 10's exact pattern:

  • What would the new backend/routers/returns.py's _load_and_process-style helper call, in what order?
  • Which existing, unchanged functions from backend/uploads.py would it reuse — would read_xlsx_upload need any modification?
  • What would lib/api-client.ts need — a new function, or would postJSON/postReport already cover it unchanged?
  • Name one thing that would be a design smell if your plan added it: a new shared helper that duplicates something backend/uploads.py or lib/api-client.ts already generalizes, or a business rule written directly inside the new router.

Once you've written your own answer, compare it against Tutorial 07's own "Extend it — Challenge 2," which asks the identical question.

Failure mode named: adding business rules directly to FastAPI route handlers, or importing FastAPI into src/. Either one creates a second place a business fact could live — and per every earlier tutorial's recurring lesson, two copies of the same fact are two copies that can silently disagree.

If your fourth-workflow design invented a new shared helper instead of reusing read_xlsx_upload/postJSON/postReport unchanged, ask now whether that's solving a real new problem — it's worth checking before assuming the existing generalization doesn't already cover it.
