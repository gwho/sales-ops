Track 2 — Business Rules & Testing Discipline

Independent Rule Evaluation and Data Issues

Two shapes a "check several things about one item" loop can take: collect every finding, or stop at the first match and commit to one answer. Both are correct designs — for different questions. This lesson teaches you to tell them apart on sight.

Building on

No new external source — this lesson contrasts real code from two modules you'll meet in Tutorials 03 and 05. Lesson 8's business-rule/programming-error distinction is the foundation underneath both shapes below.

Shape 1: collect every finding

Open src/order_validation.py and find the OV-001 loop:

for field, label in OV001_REQUIRED_FIELDS:
    if _is_blank(row.get(field)):
        errors.append(
            _make_error(row_number, "OV-001", f"{label} is missing.", "Error", order_id, sku)
        )

This loop never stops early. A row missing both customer_name and priority gets two separate error entries, not one combined message — the loop checks all eight required fields every time, regardless of how many already failed.

The same shape reappears in a completely different module. Open src/payment_aging.py:

# PA-006 / PA-007 — evaluated independently; a row can have both issues.
row_issues: list[PaymentDataIssueRow] = []
if due_date_parsed is None:
    row_issues.append(_build_data_issue(invoice_id_value, customer_name_value, "PA-006", "Due date is missing."))
if invoice_amount is None or invoice_amount < 0:
    row_issues.append(_build_data_issue(invoice_id_value, customer_name_value, "PA-007", "Invoice amount is missing or invalid."))

Same idea, different rule numbers: a missing due date and an invalid amount are two independent facts about the same invoice row, and both checks run regardless of what the other one found. tests/test_payment_aging.py proves a row can carry both at once:

def test_row_with_missing_due_date_and_invalid_amount_produces_two_data_issues():
    result = calculate_payment_aging(
        _invoices_df(_invoice_row(due_date=None, invoice_amount=-10.0)), as_of_date=AS_OF
    )
    assert result["aging_rows"] == []
    codes = {issue["error_code"] for issue in result["data_issues"]}
    assert codes == {"PA-006", "PA-007"}
    assert len(result["data_issues"]) == 2

Both examples share the same reasoning: a row genuinely can have several unrelated problems simultaneously, and collapsing them into "the first one found" would throw away real, actionable information — a person fixing the spreadsheet wants to know about every problem in one pass, not one round of fixing per re-upload.

Shape 2: first match wins

Now open src/payment_aging.py again, a different function:

def _follow_up_priority(outstanding_amount: float, days_overdue: int) -> str:
    if outstanding_amount <= 0:
        return "None"
    if days_overdue > 60 or outstanding_amount >= 50000:
        return "High"
    if 31 <= days_overdue <= 60:
        return "Medium"
    if 1 <= days_overdue <= 30:
        return "Low"
    if -7 <= days_overdue <= 0:
        return "Watch"
    return "None"

This is the opposite shape: the first condition that matches returns immediately, and nothing after it ever runs for that invoice. An invoice can't be both "High" and "Medium" — follow_up_priority is fundamentally one label, not a list, so there's exactly one correct final answer, and the order of the checks is a real design decision (Paid is checked first, so nothing can override "nothing to follow up on").

The question each shape answers

Shape 1 answers "what are all the true facts about this item?" — a list, possibly empty, possibly long. Shape 2 answers "what is the single label for this item?" — always exactly one value. Before reading a loop or a chain of ifs, ask which question it's trying to answer — that tells you whether "stops at the first match" is a bug or the entire point.
