Track 2 — Business Rules & Testing Discipline

Date Arithmetic, Boundaries, and Signed Values

Your direct preparation for Tutorial 05. Payment aging is built almost entirely out of one operation — subtracting two dates — and one decision: whether the result is allowed to be negative. Both turn out to matter more than they first look.

Primary source

Python — datetime, official docs — specifically the timedelta section, for what subtracting two dates actually produces.

Subtracting two dates

In Python, subtracting one date from another doesn't give you a number directly — it gives you a timedelta, an object representing a duration. To get a plain integer day count, you ask it for .days:

from datetime import date
gap = date(2026, 7, 9) - date(2026, 6, 19)
gap.days   # 20

Real code from src/payment_aging.py does exactly this:

days_overdue = (effective_date - due_date_parsed.date()).days

If due_date_parsed is later than effective_date — the invoice isn't due yet — the subtraction still works, and .days comes back negative. Nothing crashes; a negative day count is just as valid a timedelta as a positive one.

The decision: keep the sign, or floor it at zero?

PA-003's rule text says "if days_overdue <= 0, the invoice is not overdue" — describing what to do with a non-positive value, not saying the value itself should become 0. This repo keeps the real, signed number. An invoice due in 5 days reports days_overdue = -5, not 0.

Here's why that choice matters, concretely. PA-005's "Watch" rule is "not overdue but due within 7 days" — a genuine range: -7 <= days_overdue <= 0. If every not-yet-due invoice were floored to 0, an invoice due tomorrow and an invoice due in eight months would report the exact same value — and there would be no way, from that one field alone, to tell them apart. Watch would become impossible to express correctly with the field this module already has.

Signed vs. floored, as a general idea

Flooring a quantity at zero is the right call when nothing downstream needs to know how far past zero the real value would have gone. It's the wrong call the moment some later rule genuinely needs that distance — which is exactly Watch's situation here. Before flooring any computed value "to keep things simple," ask whether anything downstream would lose information it actually needs.

Why test exactly at 30, exactly at 31

Open tests/test_payment_aging.py and look again at the parametrized boundary test from Lesson 10:

(30, "1-30 Days", "Low"),
(31, "31-60 Days", "Medium"),

These two cases are one day apart on purpose. A bucket rule like _aging_bucket()'s if days_overdue <= 30: return "1-30 Days" has an edge — the exact value where the answer flips. The classic bug here (an "off-by-one") is writing < 30 where <= 30 was meant, or vice versa — a mistake that only shows up for inputs sitting exactly on the boundary. A test using days_overdue = 45 ("comfortably inside 31-60 Days") would never catch that kind of bug; only testing at 30 and at 31 does.
