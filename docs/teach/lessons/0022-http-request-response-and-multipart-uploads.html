Track 4 — HTTP APIs & Statelessness

HTTP Request/Response and Multipart Uploads

Two short concept lessons (L4.1, L4.2) stand between you and Tutorial 07, where this repo's tested Python core gets wrapped in a real HTTP API. This one is the vocabulary: what a request and a response actually are, what a status code promises, and why uploading a file needs a different kind of request body than a login form. Deliberately light on code — Tutorial 07 is where these ideas meet real FastAPI/TypeScript.

A request, a response — the whole model

Every HTTP interaction is exactly one round trip: a client (a browser, a mobile app, another server) sends a request to a URL; a server sends back exactly one response. That's the entire shape — no hidden third message, no server-initiated message the client didn't ask for.

A request hasA response has
AlwaysA method (what kind of action), a URL (what it's about) A status code (what happened)
UsuallyHeaders (metadata), sometimes a body (data being sent) Headers, usually a body (data being returned)

GET vs. POST, in one sentence each

MethodMeans
GET"Give me something" — no body needed, safe to repeat, safe to bookmark or cache.
POST"Here's something, do something with it" — carries a body, not assumed safe to repeat blindly (submitting a form twice can create two orders).

A URL you can paste into a browser's address bar and press Enter is always a GET — that's literally all a browser address bar knows how to send.

Status codes as one-word verdicts

A status code is the response's headline — before reading a single byte of the body, it tells you which of four broad outcomes happened:

CodeVerdict
200OK — worked, here's the result.
400Bad Request — the client sent something the server can't use (a missing field, a malformed file). Fixable by the caller.
404Not Found — this URL doesn't correspond to anything.
500Internal Server Error — something broke on the server's side. Not the caller's fault, and the server should never explain exactly what broke to a stranger on the internet.

Notice the split: 4xx codes mean "you (the caller) did something this request can't proceed with"; 5xx codes mean "I (the server) failed regardless of what you sent." Tutorial 07 leans on this split directly — a business-rule problem like a malformed spreadsheet is always a 400, never a 500.

What "an API endpoint" actually is

Every URL you've ever visited in a browser returns HTML — a page meant to be rendered. An API endpoint is just a URL that returns something else instead: structured data (almost always JSON today) or a raw file, meant to be read by code, not rendered by a browser. Same request/response model, same status codes — only the body's shape changes.

Multipart uploads — why a file needs a different body shape

A simple form field ("What's your name?") fits naturally as plain text in a request body. A file doesn't — it's binary data, potentially large, and a request often needs to send several things at once (a file and a couple of text fields). The body shape built for this is called multipart/form-data: the body is split into named "parts," each with its own small header, separated by a boundary marker the browser generates automatically. You never construct this by hand — in browser code, wrapping your data in a FormData object and handing it to fetch() is what makes the browser build a correct multipart body, boundary and all.

Why this detail matters later

Tutorial 07 names a real, easy-to-hit mistake: manually setting a request's Content-Type header when sending a FormData body. The browser needs to write that header itself, because it has to include the exact boundary string it chose while building the body — overwrite it by hand and the boundary is gone, and the server can no longer tell where one part ends and the next begins.
