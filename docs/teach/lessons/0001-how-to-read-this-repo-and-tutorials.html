Track 1 — Python, Data-as-Tables, and Contracts

How to Read This Repo and Its Tutorials

Before any Python syntax, you need a map. This repo has a specific filing system — for code, for decisions, and for the tutorials you're about to read. Five minutes here saves you from getting lost in docs/tutorials/01-python-foundation/README.md.

Primary source

Michael Nygard — "Documenting Architecture Decisions" — the original post defining the ADR pattern. This repo's docs/adr/ folder follows it directly. You'll appreciate it fully once you reach Track 8, but the shape it describes — a short, numbered document capturing one decision and the reasoning behind it — is worth recognizing now, because you'll see it every time you open a docs/adr/000N-*.md file.

The four folders that matter for Track 1

This repo has many top-level folders (backend/, app/, components/ — those belong to later tracks). Right now, only four matter:

FolderWhat lives thereWhy you'd open it
src/ The actual Python code — business logic, data shapes, helpers. This is the ground truth. Everything else describes or tests this.
tests/ Files named test_*.py, plus shared example data in contract_fixtures.py. Proves src/ does what it claims — and doubles as runnable documentation of expected behavior.
docs/plan/<phase>/ Three files per phase: plan.md, explanation.md, ai-discussion-topics.md. The raw record of what was built and why, written right after the code — the source material tutorials are generated from.
docs/tutorials/<NN-slug>/ One long README.md per phase, generated from that phase's docs/plan triad. Your actual reading material — a teaching pass over the plan docs and the real code.

There's a fifth folder, context/, holding living reference docs (context/architecture.md, context/code-standards.md, etc.) — these describe standing rules for the whole project, not one phase. You won't need them for Track 1; they'll matter once you're reading or writing new code yourself.

Worth noticing

docs/plan and docs/tutorials are not the same thing, even though they cover the same phases. docs/plan/phase-1-python-foundation/plan.md is a terse, three-file record written immediately after Phase 1 shipped. docs/tutorials/01-python-foundation/README.md is a much longer document generated from that record later, specifically to teach it — with code excerpts, questions, and exercises woven in. Read the tutorial first; drop into the plan docs when you want the original, unfiltered decision record.

The anatomy of a generated tutorial

Open docs/tutorials/01-python-foundation/README.md now (you don't need to read it yet — just look at its shape). Every one of these tutorials repeats the same four-beat pattern, over and over, once per concept:

  1. Concept primer. A short section under "How to use an LLM before this tutorial" — a prompt you could paste into a chat, before touching any code, to warm up the general idea (e.g. "what is structural typing?"). It deliberately does not mention this repo yet.
  2. Code excerpt. A "Part N" section quotes real code — verbatim, with a file path and line numbers — and explains why it's written that way, not just what it does.
  3. Checkpoint. A question testing whether you understood the why, followed by a <details> block you click open to reveal the answer — exactly like the box below.
  4. Trace. A "Full data flow" section near the end walks one real value through every layer, step by step, tying every earlier Part together into one concrete example.

Why this order matters for you

The four-beat pattern is deliberate, not decorative. The primer loads a general mental model with no repo-specific noise, so when the code excerpt arrives you're pattern-matching a familiar idea onto unfamiliar code — instead of learning both at once. The checkpoint forces you to state the "why" in your own words before it's handed to you. The trace is the payoff: it only makes sense once every earlier Part's vocabulary is already yours.

This means: don't skip to the trace. If Part 3's checkpoint stumped you, the trace at the bottom will too — it's built entirely out of ideas the Parts already introduced.
