API Endpoint Map and Route Boundaries

This first one is the map: read the whole endpoint list as a business workflow map, not a pile of interchangeable URLs.

No new external resource — this lesson exercises Tutorial 07's own endpoint list against backend/main.py and the four router files (backend/routers/orders.py, backend/routers/inventory.py, backend/routers/payments.py, backend/routers/templates.py). Three kinds of endpoint, not one

Every endpoint in this API returns HTTP, but "returns HTTP" is where the similarity ends. Read each one for what it actually hands back, and three genuinely different shapes fall out:

EndpointShapeWhat comes back
POST /api/orders/validateWorkflow Result (JSON)An OrderValidationResult TypedDict, serialized to JSON
POST /api/inventory/allocateWorkflow Result (JSON)An InventoryAllocationResult TypedDict
POST /api/payments/agingWorkflow Result (JSON)A PaymentAgingResult TypedDict
POST /api/orders/validate/reportReport Artifact (bytes)Raw .xlsx bytes + Content-Disposition/X-Report-Id/X-Generated-At
POST /api/inventory/allocate/reportReport Artifact (bytes)Raw .xlsx bytes, same header set
POST /api/payments/aging/reportReport Artifact (bytes)Raw .xlsx bytes, same header set
GET /api/templates/{template_name}Sample-file downloadA committed sample_data/*.xlsx file, byte-for-byte, from an allow-list

Two structural facts fall straight out of that table, and both are things Tutorial 07 already proved rather than just asserted. First: every Workflow Result endpoint has a paired Report Artifact endpoint with the same URL prefix plus /report — this is a deliberate, uniform naming convention, not three unrelated design choices. Second: the Report Artifact endpoints never accept a report_id or any prior result as input — each one re-accepts the same raw source file(s) its Workflow Result sibling takes (Lesson 30, Tutorial 07 Part 2/3). A sample-file download is a third, simpler thing entirely: it doesn't run any business module at all, it just serves a byte-identical file off disk from a fixed allow-list (backend/routers/templates.py's TEMPLATE_FILES dict).

Why the distinction matters, not just as taxonomy

A "Workflow Result" endpoint and its paired "Report Artifact" endpoint can look almost identical in a request log — same URL prefix, same uploaded files, same 200 status. But they answer two different questions. The Workflow Result endpoint answers "what does the business logic conclude about this data, in a shape the UI can render into KPIs and tables?" The Report Artifact endpoint answers "give me an official, downloadable file of that same conclusion." Confusing the two — for instance, assuming the report endpoint could be skipped because "the JSON already has the data" — misses that the report endpoint recomputes independently and produces a completely different kind of response (bytes, not JSON) meant for a human to open in Excel, not for a browser to parse and render.

GET /api/dashboard — the one that doesn't fit the pattern (on purpose)

backend/routers/dashboard.py's GET /api/dashboard breaks two of the patterns above at once: it's a GET, not a POST, and it returns a Workflow-Result-shaped JSON body without any file upload at all. This is a later, deliberate Phase 12 addition — not part of Track 4's stateless design, and not something to absorb deeply here. The short version: it reads back this browser's own most recently saved result per workflow, keyed by an anonymous session identifier, falling back to sample data per workflow independently. File it as "later, deliberate persistence" for now — Track 6 teaches it properly.

Tiny exercise (5 minutes)

Open backend/main.py and list, from memory first, all seven real endpoints this API exposes (not counting /health) — then check yourself against backend/routers/orders.py, backend/routers/inventory.py, backend/routers/payments.py, and backend/routers/templates.py. For each one, write one word: "result," "report," or "file." Then, without looking anything up, answer: if this project added a POST /api/returns/process workflow tomorrow, how many new endpoints would it need to follow this exact pattern, and what would the second one's URL be?

Failure mode named: treating every endpoint as "just another URL" that happens to live at a different path. The URL shape (/validate vs. /validate/report) isn't decoration — it's the only signal, at a glance, of which of the three fundamentally different contracts (Workflow Result, Report Artifact, sample-file download) a given request is actually making. Missing this distinction is exactly how someone ends up asking "why do I have to upload the file twice" without realizing the second upload is going to a categorically different kind of endpoint, not a redundant call to the same one.

If the "why three shapes and not just one flexible endpoint" question still feels open, ask now — Lesson 30's statelessness/trust-boundary reasoning and Tutorial 07 Part 2/3 are the two places that answer it most directly, and it's worth being able to state the answer in your own words before Lesson 37's capstone trace.
