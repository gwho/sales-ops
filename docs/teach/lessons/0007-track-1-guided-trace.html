The row

The comment directly above it in the real file says: "Unknown SKU not present in product master — the required invalid-SKU imperfection."

Step 1 — Name its table shape

This dict is one row out of the DataFrame generate_orders() returns (Lesson 5). Before revealing: what are the column names of this table, in the order they appear in the row above?

Reveal: the column list

order_id, order_date, customer_name, customer_region, sku, product_name, quantity, requested_delivery_date, priority, payment_terms, sales_owner — eleven columns, one per key in the row dict. Every other row in generate_orders() shares this exact same column set; only the values differ.

Step 2 — Name the contract it will eventually become

This row's sku, "VIS-SCOPE-099", doesn't exist in generate_product_master()'s output — it's fictional, on purpose. Order validation (Phase 3, not yet built as of Track 1) will eventually look at this row and produce a result. Two contracts from src/contracts.py are candidates: ValidOrderRow and ValidationErrorRow (Lesson 4). Which one, and why?

Reveal: which contract, and why

ValidationErrorRow. An unknown SKU is exactly the kind of problem that contract exists to describe — recall its shape from Lesson 4:

class ValidationErrorRow(TypedDict):
    row_number: int
    order_id: NotRequired[str]
    sku: NotRequired[str]
    error_code: str
    error_message: str
    severity: str

This row has a real order_id ("SO-2026-030") and a real sku (the invalid one), so a future validation result for it would likely populate both optional fields, plus an error_code, a human-readable error_message, and a severity. It would not become a ValidOrderRow — that contract is for orders that passed validation, and this one's whole reason for existing is that it shouldn't pass.

Step 3 — Predict the test assertion

Recall Lesson 6: this repo's sample-data tests check properties of generated data, never a not-yet-built rule's output. Given that this is "the required invalid-SKU imperfection" (per the code comment) — meaning exactly one row in the whole file is supposed to have this problem — write, in your own words, what a test named something like test_generate_orders_has_exactly_one_unknown_sku would need to check to prove that.

Reveal: the real test 
def test_generate_orders_has_exactly_one_unknown_sku():
    product_master_df = generate_product_master()
    orders_df = generate_orders(product_master_df)

    known_skus = set(product_master_df["sku"])
    unknown_sku_rows = orders_df[orders_df["sku"].notna() & ~orders_df["sku"].isin(known_skus)]

    assert len(unknown_sku_rows) == 1
    assert unknown_sku_rows.iloc[0]["order_id"] == "SO-2026-030"

Notice the shape: build the set of known_skus from the product master (Lesson 5's column-access skill), filter orders_df down to rows whose sku both exists and isn't in that known set (Lesson 5's filtering skill), then assert two things — exactly one such row exists, and it's specifically SO-2026-030 (Lesson 6's "count and name it, don't just detect it" pattern). Nothing here computes a validation error message or severity — this test only proves the input data has the shape a future rule will need, exactly as Lesson 6 predicted.
