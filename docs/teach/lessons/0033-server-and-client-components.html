Track 5 — UI Components & Next.js

Server and Client Components

The single rule that recurs across more phases of this repo than almost anything else in Track 5 — where a component's code actually runs, what's allowed to cross the boundary between the two worlds, and why one specific browser API is the reason the dashboard has to break its own "Server Component by default" pattern. This lesson must be solid before Tutorial 09, and gets revisited in Tutorials 10 and 15.

Primary source

This repo's installed copy of the Next.js docs, node_modules/next/dist/docs/01-app/01-getting-started/05-server-and-client-components.md — read that file directly for the authoritative version of everything below. Public mirror: nextjs.org/docs/app/getting-started/server-and-client-components.

Two places code can run

A Next.js App Router page is built from components, and every component runs in exactly one of two environments:

EnvironmentRuns whereCan doCannot do
Server Component (the default) On the server, before the page reaches the browser Fetch data close to its source, use secrets/API keys safely, send less JavaScript to the browser Use state (useState), event handlers (onClick), or any browser-only API
Client Component (opt-in via "use client") In the browser (after an initial server-rendered pass) State, event handlers, lifecycle hooks, browser-only APIs like localStorage Directly read secrets/API keys without exposing them to every visitor

Every layout and page in this repo's app/ directory is a Server Component unless its file starts with the "use client" directive on its own line, above every import.

Why the split exists at all

This isn't an arbitrary Next.js rule — it reflects two genuinely different places where JavaScript can execute, with genuinely different capabilities. A server process has no window, no localStorage, no mouse to click — it produces markup once and sends it. A browser tab has all of those, but can't safely hold a database password without exposing it to whoever opens dev tools. Server Components let you write code that stays where its capabilities make sense, rather than shipping every byte of JavaScript to every visitor whether it's needed there or not.

What's allowed to cross the boundary

A Server Component can render a Client Component and pass it data as props — but only data that can be serialized: turned into a plain data format (effectively, JSON) that can travel from the server process to the browser. Strings, numbers, booleans, plain objects, and arrays of those are serializable. A live JavaScript function is not — there's no way to turn "a function that runs on the server" into a JSON value the browser could receive and actually call.

Why a function prop can't cross the boundary

// Server Component — this will fail:
function OrdersPage() {
  const columns = [
    { header: "Status", render: (row) => formatStatus(row.status) }, // a function!
  ];
  return <DataTable columns={columns} />; // DataTable is a Client Component
}

render is a function value — it exists as compiled code on the server. Sending it to the browser would mean sending server-side source code across the network and asking the browser to run it as if it were local JavaScript, which Next.js has no mechanism for and will reject at build time.

Correct framing

"A function can't be serialized, so it can't cross from a Server Component into a Client Component as a prop — the data has to already be a plain value by the time it crosses."

A framing to avoid

"Next.js just doesn't support passing functions as props." (True as a symptom, but it obscures the real reason — the boundary is a serialization boundary, the same kind of boundary Track 3's _safe_cell_value lesson covered on the Python/Excel side.)

Why localStorage specifically forces a Client Component

localStorage is a browser-only API — it's a storage mechanism that belongs to one specific browser tab, and it simply does not exist on a server process. A Server Component runs before any browser is involved, so there is no localStorage for it to read at that point in time — not "not allowed," but genuinely nonexistent in that environment. Any component that needs to read or write localStorage must be a Client Component, because only a Client Component's code runs somewhere localStorage actually exists.

Forward reference — worth remembering for Tutorial 12

This repo's own dashboard (Phase 12) stores each anonymous visitor's session identity in localStorage. That single design choice is what structurally forces the part of the dashboard that reads it into a Client Component — not a style preference, a direct consequence of this lesson's rule. You don't need to trace that file yet; just recognize the shape of the argument when Tutorial 12 revisits it.

Tiny exercise (5 minutes)

Exercise

Open app/layout.tsx and confirm out loud that it has no "use client" line — it's a Server Component. Then open any file under components/ that you'd guess needs interactivity (a button, a table, an upload panel) and check whether it starts with "use client".
