Track 5 — UI Components & Next.js

Server and Client Components

The single rule that recurs across more phases of this repo than almost anything else in Track 5 — where a component's code actually runs, what's allowed to cross the boundary between the two worlds, and why one specific browser API is the reason the dashboard has to break its own "Server Component by default" pattern. This lesson must be solid before Tutorial 09, and gets revisited in Tutorials 10 and 15.

Primary source

This repo's installed copy of the Next.js docs, node_modules/next/dist/docs/01-app/01-getting-started/05-server-and-client-components.md — read that file directly for the authoritative version of everything below. Public mirror: nextjs.org/docs/app/getting-started/server-and-client-components. Also: MDN's Web Storage API for what localStorage actually is, and React's Server Functions reference for the one narrow exception to "functions can't cross the boundary."

Two module graphs, and what happens on first load

A Next.js App Router page is built from components split across two module graphs with genuinely different capabilities — not simply "every component runs in only one physical place," since a Client Component's own module still takes part in an initial server-rendered pass, as the lifecycle below shows:

Module graphCapabilitiesCannot do
Server Component (the default) Fetch data close to its source, use secrets/API keys safely, send less JavaScript to the browser Use state (useState), event handlers (onClick), or any browser-only API
Client Component (opt-in via "use client") State, event handlers, lifecycle hooks, browser-only APIs like localStorage Directly read secrets/API keys without exposing them to every visitor

On a first page load, both graphs actually take part, in this order: Server Components render into a React Server Component (RSC) payload; Next.js uses that payload together with the Client Components' own instructions to prerender the initial HTML the browser first sees; then, once that HTML and its JavaScript arrive, React hydrates the Client Components — attaching real event handlers to the already-visible markup, making it interactive. Server Components never hydrate; they have no client-side life at all once that first payload is rendered. On a later navigation, Next.js can fetch and reconcile a fresh RSC payload again without a full page reload.

"use client" doesn't mark "this one file needs it," though — it declares a boundary between the two module graphs. Add it to the entry file that actually needs the client (state, an event handler, a browser API), and every module that file imports beneath that boundary joins the client graph automatically, with no need to repeat the directive in each one. One further pattern completes the picture: a Server Component can still be rendered inside a Client Component, passed in as children (or another prop) rather than imported — that Server Component keeps rendering on the server and stays out of the Client Component's own module graph, even though it visually appears nested inside the client tree. This repo's own components/layout/AppShell.tsx does exactly this — this lesson's exercise traces it, and Lesson 43 gives it a full guided trace.

Why the split exists at all

This isn't an arbitrary Next.js rule — it reflects two genuinely different places where JavaScript can execute, with genuinely different capabilities. A server process has no window, no localStorage, no mouse to click — it renders Server Components into an RSC payload, which Next.js uses to produce the initial HTML preview, and can send a fresh payload again on a later request to that same route; "produces markup once and sends it" undersells what's actually a payload-based, sometimes-streamed, sometimes-repeated process. A browser tab has all the capabilities the server lacks, but can't safely hold a database password without exposing it to whoever opens dev tools. Server Components let you write code that stays where its capabilities make sense, rather than shipping every byte of JavaScript to every visitor whether it's needed there or not.

What's allowed to cross the boundary

A Server Component can render a Client Component and pass it data as props — but only data that's serializable by React, a broader set than plain JSON: strings, numbers, booleans, plain objects and arrays of those, plus a handful of specific built-ins React knows how to encode — Date, Map, Set, and promises. An ordinary JavaScript function is not on that list, and neither is an arbitrary class instance — there's no React-serializable representation of "a function that runs on the server" the browser could receive and actually call.

Why a function prop can't cross the boundary

// Server Component — this will fail:
function OrdersPage() {
  const columns = [
    { header: "Status", render: (row) => formatStatus(row.status) }, // a function!
  ];
  return <DataTable columns={columns} />; // DataTable is a Client Component
}

render is a function value — it carries executable behavior, closures over whatever variables it captured, and an execution environment (the server), none of which the RSC payload has any way to represent as a plain prop. It isn't really about "sending source code": even if the browser somehow received that function's text, it still wouldn't have the closed-over server-side values and environment the function depends on to run correctly. The fix is to move that behavior inside the Client Component itself, or pass plain data and let the Client Component decide what to do with it. (One narrow, deliberate exception exists: a specially-declared Server Function can cross this exact boundary, but not as an ordinary callback — React serializes a reference to it, and calling that reference from the browser sends a request back to the server to actually run it there. That's a separately-taught mechanism, not a loophole in the rule above.)

Correct framing

"A function can't be serialized by React, so it can't cross from a Server Component into a Client Component as a prop — the data has to already be a React-serializable value by the time it crosses."

A framing to avoid

"Next.js just doesn't support passing functions as props." (True as a symptom, but it obscures the real reason — the boundary is a serialization boundary, the same kind of boundary Track 3's _safe_cell_value lesson covered on the Python/Excel side.)

Why localStorage specifically forces a Client Component

localStorage is a browser-only API whose storage is associated with the document's origin — not one specific tab. It persists across browser sessions (surviving the browser being closed and reopened), and every same-origin tab shares the same storage area; tab-scoped storage is a different API entirely, sessionStorage. What actually matters for this lesson, though, is simpler than the scoping details: localStorage does not exist at all in the Server Component environment. A Server Component runs before any browser is involved, so there is no localStorage for it to read at that point in time — not "not allowed," but genuinely nonexistent in that environment. Any component that needs to read or write localStorage must execute on the client, because only a Client Component's code runs somewhere localStorage actually exists.

Forward reference — worth remembering for Tutorial 12

This repo's own dashboard (Phase 12) stores each anonymous visitor's identity as browser-profile/origin-local state in lib/session-id.ts's localStorage call — shared across every same-origin tab that visitor opens, not scoped to just one of them. That single design choice is what structurally forces the part of the dashboard that reads it into a Client Component — not a style preference, a direct consequence of this lesson's rule. You don't need to trace that file yet; just recognize the shape of the argument when Tutorial 12 revisits it.

Tiny exercise (5 minutes)

Exercise — predict, inspect, then trace one real composition

1. Predict why app/layout.tsx can stay a Server Component. What would it actually need — state, an event handler, a browser API — to be forced otherwise? Then open it and confirm: no "use client" anywhere in the file.

2. Predict why components/layout/AppShell.tsx establishes a client boundary instead. What does it actually do that a Server Component couldn't? Then open it and check line 1.

3. Inspect one file AppShell.tsx imports near the top of the file. Does that imported module repeat its own "use client" directive, or does it simply come along for free once it's imported beneath AppShell's boundary?

4. Trace one thing AppShell.tsx does not import: the actual page content it renders through its own children prop. Does that content stay a Server Component even though it visually renders inside a Client Component?

Reveal

app/layout.tsx only supplies <html>/<body>, global font/style setup, and fallback metadata — no state, no event handler, no browser API, so nothing forces it out of the default. AppShell.tsx opens with its own "use client" (line 1) because it owns real client-only behavior: mobile-drawer state, usePathname() route tracking, an Escape-key listener, and moving focus into the drawer on open — none of which can run during a server render. Its own imports (SidebarNav, TopHeader) never need their own directive; once a module sits beneath AppShell's boundary, it's already part of the client graph. And the page content AppShell renders through {children} is never imported by AppShell.tsx at all — it's rendered on the server first and handed in as already-rendered output, so it keeps its own, separate Server Component status the whole time. This exact composition has a real name in the installed Next.js docs — "interleaving Server and Client Components" — and gets a full guided trace in Lesson 43.
