Track 2 — Business Rules & Testing Discipline

Reading Tests as Specifications

The practical "how to read the proof" lesson before Tutorial 03. Track 1's Lesson 3 covered assert and pytest.raises — this lesson covers three things it didn't: builder-style helper functions, @pytest.mark.parametrize, and reading a test file's own section comments as a table of contents.

Building on

Track 1's pytest reading cheat sheet already covers assert, pytest.raises, and property-style assertions — this lesson doesn't repeat them. If those feel shaky, revisit that reference first.

Arrange, act, assert

Nearly every test in tests/test_order_validation.py, tests/test_inventory_allocation.py, and tests/test_payment_aging.py has the same three-beat shape. Here's one, labeled:

def test_overpayment_clamps_outstanding_to_zero():
    # ARRANGE — build the one input this test is about
    result = calculate_payment_aging(
        _invoices_df(_invoice_row(invoice_amount=100.0, paid_amount=150.0, due_date=_due(20))),
        as_of_date=AS_OF,
    )
    # ACT already happened above — calculate_payment_aging() IS the act
    row = result["aging_rows"][0]
    # ASSERT — check the facts this test exists to prove
    assert row["outstanding_amount"] == 0.0
    assert row["follow_up_priority"] == "None"

In this repo's style, arrange and act are usually one expression — building the input and calling the function happen together, since the function call itself is what's being tested. Assert is always separate, at the end, checking every fact this specific test cares about.

Builder helpers with overrides

Every test file starts with small helper functions like this one, from tests/test_order_validation.py:

def _order_row(**overrides) -> dict:
    row = {
        "order_id": "SO-2026-001",
        "order_date": date(2026, 7, 1),
        "customer_name": "Bright Medical Trading Ltd",
        ...
        "priority": "High",
        "payment_terms": "30 days",
    }
    row.update(overrides)
    return row

**overrides is new syntax: it captures any keyword arguments a caller passes — _order_row(customer_name="") captures {"customer_name": ""} as overrides — as a plain dict. Line 2 builds a complete, valid-looking default row; row.update(overrides) (a dict method — Lesson 2's material, extended) overlays whatever the caller specified on top of those defaults, leaving everything else untouched. The payoff: every test can construct "a normal row, except for the one field I'm testing," in one short line, instead of retyping all eleven fields every time.

Not a pytest fixture — a plain helper function

pytest has its own, separate mechanism for shared test setup, called a fixture (a function decorated @pytest.fixture). None of the three Track 2 test files use that mechanism — _order_row, _invoices_df, and their siblings are ordinary functions this repo's own test authors wrote, called directly. Worth knowing the real pytest feature exists (see the pytest fixtures docs if you want to go deeper later), but don't call these "fixtures" when describing this repo's tests — that would misname what's actually happening.

Running one test body many times: parametrize

Open tests/test_payment_aging.py and find this:

@pytest.mark.parametrize(
    "days_overdue,expected_bucket,expected_priority",
    [
        (30, "1-30 Days", "Low"),
        (31, "31-60 Days", "Medium"),
        (60, "31-60 Days", "Medium"),
        (61, "61-90 Days", "High"),
        (90, "61-90 Days", "High"),
        (91, "90+ Days", "High"),
    ],
)
def test_aging_bucket_and_priority_boundaries(days_overdue, expected_bucket, expected_priority):
    result = calculate_payment_aging(
        _invoices_df(_invoice_row(due_date=_due(days_overdue))), as_of_date=AS_OF
    )
    row = result["aging_rows"][0]
    assert row["aging_bucket"] == expected_bucket
    assert row["follow_up_priority"] == expected_priority

Read the decorator as a table: the first string names the three parameters (days_overdue, expected_bucket, expected_priority); each tuple below is one row of that table, run through the same test body once per row — pytest reports six separate pass/fail results, not one. This is how you get six boundary cases checked without writing six nearly-identical functions.

Section comments are a table of contents

Open any of the three Track 2 test files and scroll — you'll hit lines like this, over and over:

# --- Spec section 12 test cases -------------------------------------------
...
# --- IA-001 sort order edge cases ----------------------------------------
...
# --- Malformed / missing required valid order data ------------------------

These aren't decoration. Before reading a single test body, scroll through just the section comments — you'll get an accurate outline of everything the module handles: the spec's own required cases, then every edge case someone decided was worth a dedicated section. This is the fastest way to answer "what does this module actually cover" without reading 300+ lines top to bottom.

Retrieval check

In _order_row(**overrides), what does row.update(overrides) actually do?

How many separate pass/fail results does pytest report for a test with 6 parametrize tuples?

Are _order_row and _invoices_df real pytest @pytest.fixture functions?

Tiny exercise (4 minutes)

Open tests/test_inventory_allocation.py and scroll through only the # --- section comments, top to bottom — don't read any test bodies. Write down the section names in order. Then find the one test in that file decorated with @pytest.mark.parametrize and name the two values being varied across its cases.
