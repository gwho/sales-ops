Track 5 — UI Components & Next.js

Design Tokens and Semantic Styling

The last Track 5 prerequisite before Tutorial 08 — and the token foundation used most directly in Tutorial 10 — covering what a design token is, how a CSS custom property becomes a Tailwind utility class, and why this repo's rule against hardcoded hex values and raw Tailwind palette classes is a real engineering constraint, not a style preference someone could reasonably ignore under a deadline.

Primary source

Tailwind CSS v3's own docs — "Theme Configuration" and "Customizing Colors". This repo pins Tailwind 3.4 deliberately (see context/library-docs.md) — read the v3 docs, not v4's, since the two configuration systems differ.

A CSS custom property: one value, one name

CSS supports custom properties — informally, "CSS variables" — which give a reusable value a name, declared once and referenced anywhere:

A raw CSS custom property

:root {
  --accent: 221 83% 53%;
}

.button {
  background: hsl(var(--accent));
}

Change --accent's value once, and every rule referencing var(--accent) updates together. This is plain CSS — nothing here is Tailwind-specific yet. A custom property on its own is just this storage mechanism, though — it becomes part of this project's actual token system only once the project assigns it a stable design role and maps components to that role, which is the rest of this lesson. A property named --blue-600 would still be a perfectly valid CSS custom property; it just wouldn't be a semantic one, for reasons the next two sections make concrete.

Tailwind's job: turn a variable into a utility class

Tailwind CSS doesn't replace CSS custom properties — it generates small, single-purpose utility classes (bg-accent, text-accent) that reference them, so you style elements by composing class names in markup instead of writing new CSS rules by hand. Tailwind's theme.extend.colors config is where a project tells Tailwind which custom property each color name should resolve to.

This repo's actual wiring — three files, one value

/* app/globals.css */
:root {
  --accent: 221 83% 53%;
}

// tailwind.config.ts
colors: {
  accent: {
    DEFAULT: "hsl(var(--accent))",
  },
}

// any component
<button className="bg-accent">Save</button>

Three files, one value: the CSS variable holds the actual color, the Tailwind config maps a class name to that variable, and the component only ever writes the class name. See context/ui-tokens.md for this repo's full token list and tailwind.config.ts for the real mapping — both files this exact excerpt is drawn from.

What makes a token semantic

A design token is a named value standing in for a raw design decision — a color, a spacing size, a font size — so code refers to the name, never the raw value directly. A token is semantic when its name describes the token's role (accent, surface, danger) rather than its literal appearance (blue-600, white, red-200). Tailwind ships with a large built-in palette of appearance-named classes — bg-blue-600 is valid Tailwind out of the box — but this repo's token layer exists specifically to replace that built-in palette with role-named alternatives.

Why this repo forbids both hardcoded hex and raw Tailwind palette classes

Two different things are banned, for two related but distinct reasons:

BannedWhy
A hardcoded hex value (style={{ color: "#a8351f" }}) Bypasses the token system entirely — nothing connects this color to any named concept, so a future rebrand or dark-mode pass can't find or update it systematically.
A raw Tailwind palette class (bg-blue-600) Names an appearance, not a role — it says nothing about why this element is blue, so a reviewer can't tell whether it's meant to be "the accent color" or just happened to look right once. The problem isn't that bg-blue-600 is hard to find — a text search finds it instantly. It's that the literal doesn't preserve intent: the same class may be doing several unrelated jobs across the codebase, one role may need several different literal appearances across its own states, and neither a global search-and-replace nor a reviewer scanning class names can tell which occurrences share a role and which just share a color. Tokens make a change centralized and semantically safe — not merely searchable, which raw text already is.

Correct framing

"bg-accent says why this element is that color — because it's this project's accent role. Change --accent once, and every usage mapped to that exact token updates together — its hover, subtle-background, and on-color pairings each live in their own token, and changing the whole accent family means updating all of them, together and deliberately, not as a side effect of touching one value."

A framing to avoid

"It's basically the same either way — bg-blue-600 and bg-accent both just make something blue." (True today, by coincidence — this repo's accent token currently resolves to a blue. The whole point of a semantic name is that this coincidence isn't load-bearing; a raw palette class hardcodes the coincidence, a semantic token doesn't.)

One more real limit worth naming: tokens centralize a decision, but they don't automatically make it accessible. A semantic name like accent says nothing about whether --accent and --text-on-accent actually meet real contrast requirements against each other — if the accent family ever changes, its default, hover, subtle, and on-color pairings all need their contrast rechecked, the same way any color change would (Lesson 40 covers this in full).

Tiny exercise (5 minutes)

Exercise — trace one real token, then check against a reveal

Open context/ui-tokens.md and find --danger. Trace it by hand through the three-file pattern above: its declaration in app/globals.css, its color-name mapping in tailwind.config.ts, and then search the codebase with this repo's standard search tool for one real component using the resulting utility classes:

rg "bg-danger-subtle" components/

Say out loud what role that color is playing at the component you found, and predict whether "danger" actually fits that role, before revealing.

Reveal

app/globals.css declares --danger: 0 84% 60%; and --danger-subtle: 0 86% 97%;. tailwind.config.ts maps danger.DEFAULT and danger.subtle to those two variables. components/feedback/BusinessErrorMessage.tsx uses bg-danger-subtle text-danger on a role="alert" element that renders business-readable error copy — a real error state, exactly the role "danger" claims to play, not a coincidental red.

Tutorial 10 covers a real, harder edge of this same rule — a domain word like "High" priority meaning two visually different things in two different contexts, both legitimately using the token system. If the idea of one word needing two separate tone mappings sounds strange right now, that's expected; ask here if it still doesn't click after Tutorial 10 covers it directly.
