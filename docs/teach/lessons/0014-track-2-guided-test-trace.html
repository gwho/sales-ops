The four questions, every time

For each test below, you'll answer the same four things, in this order:

  1. What input row(s) is the test constructing?
  2. What business rule (Lesson 8) is actually under test?
  3. What output contract (Lesson 9) should change as a result?
  4. What assertion proves the behavior actually happened?

Test 1 — Order Validation

Open tests/test_order_validation.py and find:

def test_invalid_sku_not_in_master_raises_ov003_unknown_sku():
    result = validate_orders(_orders_df(_order_row(sku="MED-LENS-999")), _product_master_df())
    errors = _errors_by_code(result, "OV-003-UNKNOWN_SKU")
    assert len(errors) == 1
    assert errors[0]["error_message"] == "SKU does not exist in product master."
    assert result["summary"]["invalid_skus"] == 1
1. Reveal: the input row

One order row, built from _order_row(**overrides)'s defaults (Lesson 10) with a single override: sku="MED-LENS-999" — a SKU that doesn't appear anywhere in _product_master_df()'s rows. Every other field (customer name, quantity, dates, priority) is left at its normal, valid default.

2. Reveal: the business rule

OV-003 — a SKU on an order must actually exist in the product master. This is the "unknown SKU" sub-case specifically (as opposed to "inactive SKU," a different OV-003 sub-case for a SKU that exists but is flagged inactive).

3. Reveal: the output contract

ValidationErrorRow — one new entry, error_code="OV-003-UNKNOWN_SKU". Also ValidationSummary.invalid_skus, incremented by one. This row does not become a ValidOrderRow — an unknown SKU is Error severity (Lesson 8's business-rule-violation category), which excludes the row from valid_orders entirely.

4. Reveal: the assertions

Three facts, about the one row: exactly one OV-003-UNKNOWN_SKU error exists (not zero, not two); its message is the exact business-readable sentence a sales admin would read; and the summary's invalid_skus count reflects it. Same "collect every fact about one known example" shape Lesson 8 introduced.

Test 2 — Inventory Allocation

Open tests/test_inventory_allocation.py and find:

def test_high_priority_allocated_before_normal():
    orders = _orders_df(
        _order_row(order_id="SO-2026-002", priority="Normal", quantity=10),
        _order_row(order_id="SO-2026-001", priority="High", quantity=10),
    )
    result = allocate_inventory(orders, _inventory_df(_inventory_row(available_qty=10)))
    assert [r["order_id"] for r in result["allocation_results"]] == ["SO-2026-001", "SO-2026-002"]
    assert _result_by_order_id(result, "SO-2026-001")["status"] == "Fully Allocated"
    assert _result_by_order_id(result, "SO-2026-002")["status"] == "Backordered"
1. Reveal: the input rows

Two order rows — notice the file order: SO-2026-002 (Normal priority) is listed first, SO-2026-001 (High priority) second. Only one inventory row, with just enough stock (10 units) to fully cover exactly one of the two 10-unit orders.

2. Reveal: the business rule

IA-001's sort order (Lesson 12) — priority beats upload/file order entirely. Even though the Normal order was typed first in the spreadsheet, the High order must be processed first.

3. Reveal: the output contract

AllocationResultRow — both orders get one row each, but with different status values: "Fully Allocated" for the High order, "Backordered" for the Normal one — a direct consequence of only one of them being able to claim the single available warehouse's 10 units first.

4. Reveal: the assertions

The first assertion checks order itself: result["allocation_results"] lists SO-2026-001 before SO-2026-002, proving the output reflects IA-001's sort, not file order. The next two check each order's individual status — together, they prove both the sort and its consequence (who actually got the stock).

Test 3 — Payment Aging

Open tests/test_payment_aging.py and find:

def test_large_outstanding_amount_forces_high_priority_at_low_day_count():
    result = calculate_payment_aging(
        _invoices_df(_invoice_row(due_date=_due(5), invoice_amount=60000.0)), as_of_date=AS_OF
    )
    row = result["aging_rows"][0]
    assert row["aging_bucket"] == "1-30 Days"
    assert row["follow_up_priority"] == "High"
1. Reveal: the input row

One invoice row, due 5 days ago (_due(5)) with an unusually large invoice_amount of 60000.0 — well above the 50,000 threshold PA-005 names — and the default paid_amount=0.0, so nothing is paid off yet.

2. Reveal: the business rule

PA-005's High-priority condition: days_overdue > 60 or outstanding_amount >= 50000 — an unconditional or, so a large enough amount can force High priority completely independent of how few days overdue the invoice actually is.

3. Reveal: the output contract

PaymentAgingRow — specifically, two of its fields computed independently of each other (Lesson 11's Shape 2 idea, applied twice in one row): aging_bucket (purely duration-based) and follow_up_priority (which this rule lets the amount override).

4. Reveal: the assertions

aging_bucket == "1-30 Days" — only 5 days overdue, so the purely duration-based bucket rule places it in the earliest overdue bucket, completely unsurprising on its own. But follow_up_priority == "High" — the two fields disagree, on the same row, proving the amount-driven override this test exists specifically to lock in.
