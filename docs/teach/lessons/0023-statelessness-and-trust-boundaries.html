Track 4 — HTTP APIs & Statelessness

Statelessness and Trust Boundaries

Lesson 29 covered the mechanics of one request/response round trip. This lesson covers what happens (or rather, deliberately doesn't happen) between round trips, and a closely related idea: why a server should think twice before believing anything a client tells it.

Primary source

No single beginner-friendly external primer covers this as cleanly as this repo's own real decision — like Lesson 12's greedy-processing gap, the clearest teaching source is a genuine worked example: docs/grilling/phase-10-fastapi-integration/explanation.md and docs/adr/0006-stateless-fastapi-workflow-and-report-exports.md, which Tutorial 07 will walk through in real code.

What "stateless" means for a server

A stateless server processes exactly one request, sends back exactly one response, and then forgets everything about that request happened at all. Nothing about it survives in the server's memory for the next request — not who sent it, not what it asked for, not what the server computed. If two requests need to feel connected to each other, something has to deliberately carry that connection: a database row, a file on disk, an identifier the client itself resends every time. Statelessness isn't "the server has no memory anywhere" — it's "the server has no memory by default, and adding memory is always a deliberate, visible decision," never an accident.

A concrete test for statelessness

Here's a test that works on any API design, not just this repo's: if completing this request requires the server to remember something about an earlier request, it is not stateless. Not "the word 'state' doesn't appear in the code" — an actual dependency on server-side memory of the past.

Why this test matters, not just as a definition

A design can look like an ordinary, reasonable API and still fail this test. A two-step flow — "generate a report, get an ID back, fetch the report by that ID later" — reads as completely normal REST design. But walk through what has to be true for step two to work at all: the server has to have stored the actual bytes, or enough information to regenerate them, keyed by that ID, somewhere that survives between the two separate requests. That "somewhere" is a real piece of infrastructure — a job store, a file store, a database table — that didn't need to exist before. The tell isn't in how the endpoint reads; it's in what has to be true for it to function at all.

Trust boundaries: why a server should recompute, not just believe

A trust boundary is the line between "data this system produced and verified itself" and "data that arrived from the outside and could say anything." Anything a client sends — a form field, a file, a claimed result — crossed that boundary from the outside, and a careless server can end up treating it as more authoritative than it's earned.

Concretely: imagine an API that computes something expensive once, shows the result to a user, and later offers "download an official report of that result" — but the download step just accepts whatever result JSON the client currently has and formats it, instead of recomputing from scratch. Nothing stops a client from editing that JSON before asking for the download. The server would hand back a real, official-looking file built from numbers nobody ever actually verified.

The general principle behind the fix: a system should not accept as authoritative anything it could not have produced itself. If the server always recomputes the result from the original raw input rather than trusting a client's copy of a previous answer, there's nothing to edit — the only way to influence the output is to change the same raw input that would have influenced it in the first place.

Tiny exercise (5 minutes)

Design, on paper, a two-step "generate report → fetch by ID" API for some made-up domain (a grade report, a shipping label — your choice). List every new piece of state it would force into existence: what gets stored, keyed by what, for how long, and who's responsible for cleaning it up. Then compare that list to how many of those questions a single "recompute and return it now" endpoint would need to answer.
