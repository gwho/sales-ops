Track 5 — Optional Reinforcement · 6 of 9

Accessible Components as Contracts

Not a final styling pass before shipping — accessibility is part of what a component promises to do, the same way a prop type is. Seven real components try to keep that promise, in seven different ways, with mixed real results: UploadPanel, TopHeader, SidebarNav, BusinessErrorMessage, DataTable, and both chart components. This lesson traces exactly how each one actually behaves — not just the ones that get it right.

Primary sources

MDN's "ARIA" overview, for the first rule of ARIA use — prefer a native element's built-in semantics over adding ARIA to a generic one. The W3C WAI ARIA Authoring Practices Guide's "Names and Descriptions" page, for what an accessible name actually is, how it's computed, and which roles are prohibited from having one. The APG's "Alert" pattern and sortable-table example, for role="alert" and aria-sort respectively. WCAG's "Focus Visible" and "Use of Color" understanding pages, for the objective tests this lesson's focus and color sections are held to.

Why this matters

All three mission drivers apply: you can explain in an interview what a specific real control's accessible name and keyboard behavior are, not just gesture at "accessibility" in the abstract; semantic-HTML-first, accessible naming, and color-plus-text status signaling are portable to any component library; and treating this as a contract — something a component can honor or break, like a prop type — is what keeps you from silently regressing it while extending this repo's UI.

Semantic HTML before ARIA

MDN states it as a first rule, not a suggestion: prefer a native HTML element or attribute with the behavior you need already built in, before reaching for ARIA to bolt that behavior onto a generic element. TopHeader.tsx's drawer toggle is a real <button>, which means keyboard reachability, Enter/Space activation, and a focusable default all come free, with zero extra code:

components/layout/TopHeader.tsx, lines 17–26

<button
  type="button"
  onClick={onToggle}
  aria-expanded={isDrawerOpen}
  aria-controls="mobile-sidebar-drawer"
  aria-label={isDrawerOpen ? "Close navigation menu" : "Open navigation menu"}
  className="-ml-2 rounded-md p-2 text-text-primary hover:bg-surface-muted lg:hidden"
>
  {isDrawerOpen ? <X size={20} aria-hidden="true" /> : <Menu size={20} aria-hidden="true" />}
</button>

Skipping semantic HTML has a real cost, not just a theoretical one. VerticalBucketBarChart.tsx's bar segments try to earn keyboard reachability and a name by hand, because a plain <div> has neither natively:

components/charts/VerticalBucketBarChart.tsx, lines 51–56

<div
  key={datum.label}
  tabIndex={0}
  aria-label={`${datum.label}: ${formatAmount(datum.value)} (${percent}% of total)`}
  className="group relative flex h-full flex-1 cursor-default flex-col items-center justify-end gap-1 focus:outline-none"
>

tabIndex={0} genuinely works — it puts this div into normal tab order. aria-label does not: a plain <div> with no ARIA role has the implicit role generic, and the ARIA specification is explicit that elements with that role have accessible naming prohibited — "authors MUST NOT use the aria-label or aria-labelledby attributes to name the element." Browsers and screen readers are free to ignore this aria-label entirely, which means this div can be a real, focusable, silent stop in the tab order: reachable, but potentially unnamed. This is a genuine, current gap, not a hypothetical one — tabIndex alone doesn't fix what ARIA can't fix either.

A real fix exists one file over. DonutBreakdownChart.tsx names its segments the same way, with one crucial difference — see the next section. A native <button> would be the wrong fix here, though: these segments don't perform an action, they display a value, so the corrected version still wouldn't be a button — it would need a nameable role (role="img", an accessible <figure> with a text summary, or an accompanying data table instead of adding every value to tab order individually). Worth noticing while you're here: DataTable.tsx's sortable column headers are real <button>s, so clicking and keyboard-activating one both work for free — but the current sort direction is only conveyed by a visible ▲/▼ character in the button's own text, never by an aria-sort attribute on the header cell itself, which is what the WAI's own sortable-table pattern actually calls for. Another real, current gap.

A working contrast: two charts, two different real outcomes

DonutBreakdownChart.tsx names its segments with the same intent as the bar chart, but a different, nameable role:

components/charts/DonutBreakdownChart.tsx, lines 83–85

tabIndex={0}
role="img"
aria-label={segmentTitle}

role="img" changes this circle's implicit role from whatever an unstyled <circle> would otherwise have to an explicitly nameable one — img permits aria-label the same way an <img> tag's alt text does. Same intent as the bar chart, same tabIndex/aria-label pairing, but this one actually names the element, because the role underneath it allows a name at all.

The donut chart isn't fully clean either, though. Its legend renders a second, independently focusable element for the very same datum:

components/charts/DonutBreakdownChart.tsx, lines 117–119

<div
  key={segment.label}
  tabIndex={0}

This legend row has no role and no aria-label — it's a plain, focusable generic div again, relying on a screen reader choosing to read its visible child text rather than on any computed accessible name. So for every single data point, a keyboard user tabs through two stops — the named circle, then the unnamed-but-textful legend row — for information that's logically one thing shown twice. Whether that's a deliberate, useful redundancy (both hover targets have always mirrored each other) or just double the tab stops for no real benefit is a fair question this lesson isn't going to resolve for you; the exercise below asks you to feel it out yourself, with a keyboard.

Accessible names: visible label text, or aria-label when there's none

The APG defines an accessible name as the short string assistive tech announces for a control, and its preferred source is visible text a sighted user would also read. UploadPanel.tsx renders two pieces of text that look, to a sighted reader, like they'd both describe the file input — but only one of them actually does:

components/workflow/UploadPanel.tsx, line 54

<span className="text-sm font-semibold text-text-primary">{label}</span>

components/workflow/UploadPanel.tsx, lines 64–74

<label
  htmlFor={inputId}
  className="flex cursor-pointer items-center justify-between gap-3 rounded-md border border-dashed border-border-strong bg-surface-subtle px-3 py-2 text-sm text-text-secondary hover:border-accent"
>
  <span className="truncate">{displayedFileName ?? "Choose a file…"}</span>
  <span className="shrink-0 rounded-md bg-accent px-3 py-1 text-xs font-medium text-text-on-accent">
    Browse
  </span>
  <input
    id={inputId}
    type="file"

The label prop — "Orders File," "Product Master File," whatever the caller passes — is that first, plain <span>. It's never inside any <label> element and never referenced by aria-labelledby, so it contributes nothing to the input's accessible name. The actual <label> element is the second excerpt, further down, and its text is "Choose a file…" (or the current filename) plus "Browse" — useId() only guarantees the htmlFor/id pairing is unique per instance, it does nothing to make the name unique. The real, current consequence: order-validation/page.tsx renders two UploadPanels, "Orders File" and "Product Master File," and both compute to essentially the identical accessible name — a screen reader user navigating by form control would hear "Choose a file, Browse" twice, with nothing distinguishing which is which. The visually obvious heading text never reaches assistive tech at all. The input itself is sr-only (visually hidden, not display: none), so it does stay in the tab order and keyboard-operable — that part works — but nothing in UploadPanel.tsx's className list includes a focus-within: variant on the visible label, so tabbing to that 1px, clipped, visually-hidden input produces no visible focus indicator anywhere on screen. Two separate real gaps, both in the same component.

TopHeader's icon-only toggle button, from the first excerpt in this lesson, gets the naming half of this right: it has no visible text at all — an icon alone isn't a name — so it supplies one directly and correctly with aria-label={isDrawerOpen ? "Close navigation menu" : "Open navigation menu"}, on a real <button>, a role that (unlike generic) permits a name.

ARIA that describes real state, computed from the same state as the visuals

Every stateful ARIA attribute in this repo is bound to the same variable driving what's already on screen — never hardcoded, never guessed. (Not every ARIA attribute is stateful, though — aria-hidden="true" on a decorative icon and aria-controls="mobile-sidebar-drawer"'s literal ID string are both intentionally fixed values, not derived from anything that changes.) SidebarNav.tsx marks the active page:

components/layout/SidebarNav.tsx, line 29

const active = pathname === item.href;

components/layout/SidebarNav.tsx, line 35

aria-current={active ? "page" : undefined}

aria-current="page" tells assistive tech "this link represents the page you're currently on" — the exact same active boolean that also drives the link's visible highlight styling a few lines later, so the two can never silently disagree. TopHeader's toggle button, from the first excerpt, does the same thing for its one genuinely stateful attribute: aria-expanded={isDrawerOpen} announces whether the thing this button controls is currently open. BusinessErrorMessage.tsx uses a different kind of ARIA — not a description of state at all, but a promise about timing:

components/feedback/BusinessErrorMessage.tsx, line 10

<div role="alert" className="rounded-xl border border-border bg-danger-subtle p-4 text-sm text-danger">

role="alert" is an implicit live region — the APG's own description is that content appearing inside one is announced automatically by most screen readers, without the user needing to have focus anywhere near it. The general rule is broader than this one component: live regions announce dynamic changes — content that's genuinely new or changed after the page has already loaded, whether that's React mounting a whole new element (what happens here, since the error text doesn't exist in the DOM until there's a real error) or an existing live region's own text being updated in place. What a live region does not do is announce content that was already sitting in the initial markup before the page finished loading — screen readers don't retroactively narrate what was already there.

Decorative icons: aria-hidden so screen readers skip the noise

An icon sitting next to text that already says the same thing adds nothing for a screen reader — worse, announcing it is noise. Several real icons in this repo are correctly hidden from assistive tech with aria-hidden="true": TopHeader's menu/close icons (first excerpt) sit right next to aria-label text that already says "Open/Close navigation menu"; SidebarNav.tsx's per-item icons, line 43 (<Icon size={16} className="shrink-0" aria-hidden="true" />), sit next to each link's own visible label text; and VerticalBucketBarChart.tsx hides its horizontal guide lines outright, since they carry no information a screen reader could use at all:

components/charts/VerticalBucketBarChart.tsx, lines 41–46

<span
  key={percent}
  aria-hidden="true"
  className="absolute inset-x-0 border-t border-border"
  style={{ bottom: `${percent}%` }}
/>

The rule, where it's applied: if removing an element would cost a screen reader user zero information, it should be aria-hidden; if it would cost them the only copy of some information, it needs a real accessible name instead. But "where it's applied" is doing real work in that sentence — MetricCard.tsx doesn't apply it at all:

components/workflow/MetricCard.tsx, lines 35–44

{icon ? (
  <span
    className={cn(
      "flex h-7 w-7 shrink-0 items-center justify-center rounded",
      CHIP_TONE_CLASSES[tone],
    )}
  >
    {icon}
  </span>
) : null}

The component's own comment calls this icon "decorative only," and the icon it renders — a Lucide component like <ClipboardList size={16} /> — is a real, visible <svg> that nothing here marks as aria-hidden. It isn't hidden automatically either: this repo's installed Lucide version builds its SVG output from a fixed set of default attributes that doesn't include aria-hidden, so unless a caller passes it explicitly as a prop, the icon stays exposed to assistive tech. In practice this is a small, real gap — a screen reader visiting a KPI tile may announce an unlabeled graphic alongside the number and label it sits next to — not a large one, but the same category of gap as the bar chart's aria-label and the missing aria-sort: something this lesson names rather than quietly working around.

Visible focus: replacing the outline, never just deleting it

"Is there a visible focus indicator" isn't purely a matter of taste — WCAG's Focus Visible guidance gives three concrete, checkable tests: is the focused element clearly identifiable; does the indication stay visible for as long as focus remains there; and is it actually distinguishable from the unfocused state, not just technically different. Judge every example below against those three, not against whether it looks conventional.

UploadPanel.tsx's file input fails the first test outright, per the gap named above: its visible label has no focus-within: styling at all, so tabbing to the sr-only input produces no identifiable change anywhere on screen. That one isn't a judgment call — there's simply nothing there.

The bar-segment div from this lesson's first section is a genuinely closer case. It also removes the browser's default focus ring — focus:outline-none, in that same class list — but two sibling elements react to that same focus with group-focus: classes: the tooltip becomes visible (group-focus:opacity-100), and the bar itself dims (group-focus:opacity-70), the identical pair of changes that already happen on hover. Run the three tests yourself: is the focused bar identifiable (a tooltip does appear); does it stay while focus remains (yes, nothing times it out); is it distinguishable from the unfocused state (a 30%-opacity dim and a tooltip, rather than a ring). DonutBreakdownChart.tsx's circles do the same trade — outline removed, replaced by the identical hover-opacity change. Its legend rows remove the outline too, but replace it differently: onFocus sets the same hoveredLabel state onMouseEnter does, which both applies a background highlight (isHovered ? "bg-surface-muted" : "") and reveals an extra percentage figure next to the value. Three different real replacements for one deleted outline, across three components — this lesson states the test and the facts for each; which of them pass it by your own judgment is the exercise below.

Color is never the only signal

WCAG's Use of Color criterion states this plainly: color must never be the only visual means of conveying information. StatusBadge.tsx never renders a tone without the word that earned it — <Badge tone={tone}>{status}</Badge> always passes the real status string as the badge's visible content, never color alone:

components/ui/Badge.tsx, lines 42–45

<span className={cn(badgeVariants({ tone: resolvedTone }), className)} {...props}>
  {dot ? <span className={cn("h-1.5 w-1.5 shrink-0 rounded-full", DOT_TONE_CLASSES[resolvedTone])} /> : null}
  {children}
</span>

Every status this repo shows — "Fully Allocated," "High," "90+ Days" — carries its tone as a colored dot and colored text in addition to the word itself, never as a replacement for it. A colorblind visitor, or anyone on a low-contrast screen in bright sunlight, still gets the actual answer by reading, not by correctly guessing which shade of a status color they're looking at.

Correct framing

"This control's accessible name comes from its visible label text; if I remove that text, I've broken the same contract a missing required prop would break — it just fails silently for a screen reader user instead of failing a typecheck."

A framing to avoid

"I'll add accessibility once the feature is visually done." (Every real example in this lesson is accessibility built into the component's first version, not bolted on after — aria-current and aria-expanded are one line each, computed from state the component already has. Treating it as a separate final pass is what makes it feel expensive; it's cheap when it's part of the same change.)

Tiny exercise (8 minutes)

Exercise — apply the three-part test yourself, with a keyboard

Part 1 — the objective case. Predict first: on Order Validation, tabbing into the "Orders File" upload control, will you see any visible change at all? Then click somewhere else on the page, press Tab until focus should be on that file input, and look. Run WCAG's three checks: identifiable, persistent, distinguishable. This one should fail outright — there's genuinely nothing to see, which is the point of calling it an objective gap rather than a judgment call.

Part 2 — the judgment calls. On the Dashboard page (run sample data first if nothing's loaded), find the "Outstanding by Aging Bucket" bar chart and the "Allocation Status" donut chart, side by side. Predict, then tab through each: for the bar chart, watch one bar segment; for the donut, watch what happens across both a circle segment and its matching legend row — you should feel two distinct stops for the same data point. Apply the same three-part test to all three (bar segment, donut circle, donut legend row) and decide, for each, whether it passes. Also judge the donut's doubled stops on their own terms: redundant, or a reasonable way to make the same information reachable two ways? Nothing to undo in either part — this exercise doesn't touch any code.
