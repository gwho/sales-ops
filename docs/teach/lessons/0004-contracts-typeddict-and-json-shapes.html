Track 1 — Python, Data-as-Tables, and Contracts

Contracts, TypedDict, and JSON Shapes

The single idea every later layer of this repo turns out to be a projection of: a dict holds data, a TypedDict writes down what shape that dict is promised to have, and an output contract is this repo's name for a TypedDict that's shared across modules as a stable handoff point.

Primary source (reference, not a first read)

PEP 589 — TypedDict — the actual language proposal defining what TypedDict is. Skim it after this lesson, once dict basics (Lesson 2) feel solid; it's precise but dense.

Recall: a dict has no promises

From Lesson 2, a dict looks like this:

row = {"total_orders": 12, "valid_orders": 9}

Nothing about this line stops you from typing {"toatl_orders": 12} — a typo — or from leaving out valid_orders entirely. Python won't complain until something later tries to read the missing key and fails, possibly far away from where the mistake was made. A dict is data with no attached promise about its own shape.

TypedDict is the promise, written down

Open src/contracts.py and find this:

class ValidationSummary(TypedDict):
    total_orders: int
    valid_orders: int
    invalid_orders: int
    duplicate_orders: int
    invalid_skus: int
    missing_field_count: int

This reads as: "a ValidationSummary is a dict that must have exactly these six keys, and each one must hold an integer." Crucially — and this is the part that surprises people coming from other languages — at runtime, this is still just a plain dict. TypedDict adds nothing extra when the program actually runs; it exists purely so a type-checking tool (or a careful human) can catch a missing or misspelled key before running anything, by reading the code.

A real value satisfying this contract looks identical to any other dict:

summary: ValidationSummary = {
    "total_orders": 12,
    "valid_orders": 9,
    "invalid_orders": 3,
    "duplicate_orders": 2,
    "invalid_skus": 1,
    "missing_field_count": 0,
}

The : ValidationSummary after summary is a type hint (Lesson 2's syntax again) — a note to any checking tool that this particular dict is claimed to match that shape.

NotRequired: an optional promise

Some fields aren't always there. src/contracts.py handles this explicitly:

class ValidationErrorRow(TypedDict):
    row_number: int
    order_id: NotRequired[str]
    sku: NotRequired[str]
    error_code: str
    error_message: str
    severity: str

order_id and sku are wrapped in NotRequired[...] — this row is allowed to omit them (an error that isn't tied to a specific order or SKU), while row_number, error_code, error_message, and severity must always be present.

Why bother, if it's "just a dict" anyway?

Because this dict is going to cross a real boundary: a Python function builds it, and eventually a web layer will convert it to JSON — a plain-text format almost every programming language and every web browser can read, built out of exactly the same shapes you already know: objects with named keys (like a dict) and arrays (like a list). A plain Python dict converts to JSON for free, with zero extra work — which is precisely why this repo picked TypedDict over heavier alternatives: it gets the "write the promise down" benefit with none of the "now I need extra conversion code" cost.

The term to know

This repo calls a TypedDict that's shared as a stable handoff between modules (Python business logic → a future web layer → a future UI) an output contract. ValidationSummary, PaymentAgingRow, and every other class in src/contracts.py is one. The name is doing real work: like a contract between two parties, it's a promise about shape that every consumer can rely on without re-checking it themselves.
