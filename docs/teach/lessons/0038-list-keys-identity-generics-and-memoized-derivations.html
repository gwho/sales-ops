Track 5 — Optional Reinforcement · 4 of 9

List Keys, Identity, Generics, and Memoized Derivations

Two forward references finally get paid off here. Lesson 35 previewed DataTable<T>'s generic type parameter without teaching it. Lesson 37 reserved useMemo for this lesson. Both turn out to depend on the same underlying idea as list keys and immutable sorting: knowing which value is which, across a render, a sort, or a call site.

Primary source

"Rendering Lists" on react.dev, for keys. The TypeScript Handbook's "Generics" page, for how a type parameter carries a relationship between input and output types. The useMemo API reference, specifically its point that useMemo is a performance optimization only — code must stay correct with it removed.

Why this matters

All three mission drivers apply: you can explain in an interview why a bad key or a missing memo dependency produces a bug that neither TypeScript nor a green build catches; the generic-component pattern and the "memoization must not change semantics" rule are both portable to any React/TypeScript codebase; and picking a correct row key or dependency array is exactly the kind of small, easy-to-get-wrong decision you'll face extending this repo's own tables.

Generics: one component, many row shapes

order-validation/page.tsx renders two completely different row shapes through the same component. Each starts by fixing what T is, at the column list itself:

app/(workspace)/order-validation/page.tsx, lines 46, 48–54

const ERROR_COLUMNS: DataTableColumn<ValidationErrorRow>[] = [
  {
    key: "order_id",
    header: "Order ID",
    render: (r) => <span className="font-medium text-text-primary">{r.order_id ?? "—"}</span>,
    sortable: true,
    sortValue: (r) => r.order_id ?? "",
  },

Writing DataTableColumn<ValidationErrorRow>[] fixes T to ValidationErrorRow for this entire array — which is why r inside render and sortValue is typed ValidationErrorRow automatically, with no annotation written by hand. Try renaming r.order_id to a field that doesn't exist on ValidationErrorRow and TypeScript rejects it at this exact line — the same enforcement Lesson 35 covered for a plain object type, just carried through a type parameter instead of hardcoded.

Where T actually gets supplied is the call site, not the component definition:

app/(workspace)/order-validation/page.tsx, lines 425–428

<DataTable
  columns={ERROR_COLUMNS}
  data={filteredErrors}
  getRowKey={(r) => `${r.row_number}-${r.error_code}`}

Nothing here writes DataTable<ValidationErrorRow> explicitly — TypeScript infers it from what columns and data already are. Inside DataTable.tsx itself, column.render(row) calls that same render function back with whatever row is at this call site — still ValidationErrorRow, the whole way through. That's the entire point of a type parameter: one component definition, and the compiler re-derives the correct concrete type at every call site instead of you writing DataTable once per row shape.

Keys: identity among siblings, not a position label

A key has to satisfy two separate requirements, not one: it must stay stable for a given row across renders, and it must be unique among its current siblings. DataTable.tsx gets a candidate key from a function the caller supplies, not from position in the array:

components/tables/DataTable.tsx, line 119

<TableRow key={getRowKey(row)} className={index % 2 === 1 ? "bg-surface-muted" : undefined}>

Notice index is right there, already destructured from the same .map((row, index) => ...) call, and it's deliberately used only for the zebra-stripe class, never the key — index satisfies neither requirement above once a list can be sorted, since it's only stable as long as nothing ever moves. order-validation/page.tsx supplies getRowKey={(r) => `${r.row_number}-${r.error_code}`} instead, built from the row's own data. For this specific table, the concrete risk of an index key isn't that a cell would display the wrong text — every cell here renders a plain value straight from its row prop, so React would still paint the correct error_message in the correct position after a sort. The real danger with position-based keys is anything that carries identity beyond what's freshly rendered each time: an uncontrolled input's typed value, focus, a selection checkbox, an in-progress animation, or state a row component keeps locally. None of that exists in this particular table today, which is exactly why an index key's bug would stay latent here — but the moment any cell in a reusable, sortable DataTable<T> gains that kind of state, an index key would silently reattach it to whichever row now happens to sit at that position, not the row a visitor was actually interacting with.

The composite key above is stable, but it isn't actually guaranteed unique — a real test proves it:

tests/test_order_validation.py, lines 136–145

def test_ov001_emits_one_error_per_missing_field_in_fixed_order():
    result = validate_orders(
        _orders_df(_order_row(customer_name="", priority="")), _product_master_df()
    )
    ov001_errors = _errors_by_code(result, "OV-001")
    assert [error["error_message"] for error in ov001_errors] == [
        "Customer name is missing.",
        "Priority is missing.",
    ]
    assert result["summary"]["missing_field_count"] == 2

Both of those errors share the same row_number and the same error_code ("OV-001") — only error_message differs, because OV-001 emits one error per missing field on the same row. That makes getRowKey={(r) => `${r.row_number}-${r.error_code}`} produce a genuine duplicate key whenever a row is missing two or more required fields at once: stable, but not unique among siblings. Folding error_message into the key (`${r.row_number}-${r.error_code}-${r.error_message}`) would resolve this specific collision as an interim fix — but the more robust repair is a dedicated identity field the Python core emits per error, which would mean amending ValidationErrorRow's contract, a spec/ADR-level change (this repo's Field Scope Boundary rule) and not something a frontend lesson gets to invent. This repo's real key, exactly as it stands today, has the latent defect this section just described — this lesson is naming it, not fixing it.

The same identity-not-position principle shows up outside tables entirely — DonutBreakdownChart.tsx keys each chart segment by segment.label, which genuinely is unique among that chart's small, fixed set of category labels.

A new array, then reordered: immutable sort in the wild

Lesson 36 established that this repo replaces state wholesale rather than mutating it in place. Sorting is where that rule meets a genuinely mutating method, Array.prototype.sort, head-on:

components/tables/DataTable.tsx, lines 70–77

const sorted = [...data].sort((a, b) => {
      const va = sortValue(a);
      const vb = sortValue(b);
      if (va < vb) return -1;
      if (va > vb) return 1;
      return 0;
    });
    return sort.direction === "asc" ? sorted : sorted.reverse();

.sort() reorders an array in place and returns that same array — it does not produce a new one. [...data] exists specifically to give it a fresh array to mutate, so the data prop the caller passed in is never touched. If this line read data.sort(...) instead, sorting the table would silently reorder filteredErrors itself — a value order-validation/page.tsx still owns and reasons about elsewhere, now changed by a side effect three components away from where it's defined.

useMemo: caching a computation, never changing what it computes

sortedData is the value being cached. React decides whether to reuse the cached array or recompute by comparing each dependency-array entry to its value from the previous render with Object.is — the same reference-equality check Lesson 37 used for state. For objects and arrays, "changed" means a different reference, not "different-looking contents":

components/tables/DataTable.tsx, line 78

}, [data, sort, columns]);

Applied to this exact array: columns (ERROR_COLUMNS) is declared with const at module scope, outside the component, so it's the same reference on every render — it can never be the reason this memo recomputes. data (filteredErrors) is itself already wrapped in its own useMemo back on the page, so it only changes reference when its dependencies change. sort is the one that actually moves: toggleSort calls setSort with a freshly built object literal, { key: column.key, direction: ... }, on every single click — a brand-new reference every time, regardless of whether the key or direction actually differ from before. In practice, clicking a sortable header is the only thing that ever legitimately changes what this memo depends on.

That's exactly why sort has to be listed. If it were left out of the array, clicking a column header would still update the sort state and still flip the header's ▲/▼ indicator (read directly from state on every render, independent of this memo) — but sortedData would normally keep returning the array cached from before the click, since none of its listed dependencies changed reference. The rows on screen would normally stay in whatever order they were already in. "Normally" matters here: the useMemo reference itself is explicit that React may still discard a cached value for its own reasons, and that the calculation can run an extra time under development Strict Mode — useMemo is a cache with no durability guarantee, never a promise about exactly when code runs. The underlying point stands regardless: memoization is only ever a cache in front of a calculation, never a substitute for a correct dependency list — dropping a real input doesn't skip "extra" work, it risks serving a stale answer.

Three different "identity" ideas, not one mechanism

This lesson leaned on the word "identity" repeatedly, and it's worth being precise that the three uses aren't the same mechanism wearing different names. Generics are a compile-time relationship — TypeScript tracking which concrete type T stands for at a given call site, checked and then erased before any code runs. A key is a runtime claim about which rendered item a specific business row is, independent of where it currently sits. A useMemo dependency array is a third thing again: a reference-equality check — whether a value is the very same object in memory as last render, not whether it represents the same real-world thing. All three matter to this one file, and all three get called "identity" in casual conversation about React — confusing any two of them is a fast way to reach for the wrong fix.

Correct framing

"useMemo here is only caching — if I deleted the useMemo(...) wrapper entirely and left the sort logic running plain on every render, the table would still be correct, just doing unnecessary work on renders where nothing sort-related changed."

A framing to avoid

"I don't need to list sort as a dependency — it's not that important." (Every value the memoized function actually reads has to be listed, or the memo can silently return a stale result. There's no such thing as an optional dependency; there's only a dependency array that's either accurate or wrong.)
