CORS and Downloads Cheat Sheet

The two separate CORS permissions, this project's exact header list, and the two different download mechanisms it uses. Companion to Lesson 34.

A note on this document's status

Pre-study, same status as this workspace's other reference docs — a lookup tool, not yet evidence of internalized understanding.

Two separate permissions

SettingControlsFailure symptom
allow_originsWhether the cross-origin request is permitted at allBrowser blocks the whole request — no body reaches your code
expose_headersWhich response headers, beyond a small safelist, cross-origin JS may readRequest succeeds fully; response.headers.get(...) returns null for anything unlisted

These are independent. A response can be a complete, correct 200 while a specific header is still invisible to your own JavaScript.

This project's real CORS config

app.add_middleware(
    CORSMiddleware,
    allow_origins=_cors_origins(),
    allow_methods=["GET", "POST"],
    allow_headers=["*"],
    expose_headers=["Content-Disposition", "X-Report-Id", "X-Generated-At", "X-Persisted"],
)

(X-Persisted is a later, Phase 12 addition — Phase 10 shipped with the first three only.)

Headers as contract, not metadata

HeaderRead byPurpose
Content-DispositionparseFilename()The suggested download filename
X-Report-IdpostReport()Display metadata for the generated report
X-Generated-AtpostReport()Display metadata — when the report was generated

Two download mechanisms, genuinely different

fetch() + Blob (report downloads)Plain <a href download> (sample files)
JavaScript touches the response?Yes — reads headers and body directlyNo — a normal browser navigation
CORS applies?Yes — both allow_origins and expose_headers matterNo — nothing to restrict, no script reads the response
MechanismURL.createObjectURL(blob) → hidden <a download> → .click() → URL.revokeObjectURLDirect browser navigation to the URL with download attribute
Used forReport Artifacts — server-computed on demand, must be fetched via JS to inspect headers firstSample-file downloads — always the exact same committed file, no computation needed

Debugging checklist: "the header is in DevTools but my code sees null"

  1. Confirm the header is genuinely present in the Network tab's Response Headers (not just the request).
  2. Check expose_headers lists that exact header name.
  3. Remember: DevTools reads the raw HTTP response directly — its visibility there is not evidence JavaScript can read it too.
