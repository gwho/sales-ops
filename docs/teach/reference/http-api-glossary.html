HTTP API Glossary

The terms Lessons 29, 31, and 32 introduce, compressed for lookup. Companion to Lesson 29, Lesson 31, and Lesson 32.

A note on this document's status

This is a pre-study glossary, same status as every earlier track's — terms defined so you can start using them, not yet evidence you've internalized them. A term only gets promoted into this workspace's permanent, evidence-gated glossary once you've demonstrated understanding of it in conversation or an exercise.

Terms

Request / Response:
The entire shape of one HTTP interaction — a client sends a request to a URL, a server sends back exactly one response. No hidden third message, no server-initiated message the client didn't ask for.

Status code:
The response's one-word verdict, readable before any body content. 200 OK, 400 Bad Request (the client's fault), 404 Not Found, 500 Internal Server Error (the server's fault, message never leaked).

API endpoint:
A URL that returns structured data or a raw file, meant to be read by code, instead of HTML meant to be rendered by a browser. Same request/response model, same status codes — only the body's shape changes.

Multipart/form-data:
A request-body encoding that splits the body into named "parts" separated by a boundary string, used whenever a request needs to carry binary data (a file) or several fields at once. The boundary is generated by whatever builds the request (a browser's fetch() given a FormData object) and must be declared in the Content-Type header — never set that header by hand for a FormData body.

Workflow Result:
This project's term for a JSON response from a POST /api/.../validate|allocate|aging endpoint — an already-tested Python module's output TypedDict, serialized directly. See backend/routers/orders.py.

Report Artifact:
This project's term for the raw .xlsx bytes returned by a POST /.../report endpoint, alongside Content-Disposition, X-Report-Id, and X-Generated-At headers. Always recomputed from raw source files, never built from a client-supplied result (Lesson 30).

Sample-file download:
GET /api/templates/{template_name} — serves a committed sample_data/*.xlsx file byte-for-byte from a fixed allow-list. Runs zero business-module code.

FormData field name ↔ parameter name binding:
FastAPI's Annotated[UploadFile, File()]/Annotated[str, Form()] binds a request's multipart parts to Python parameters purely by matching the part's name to the parameter's name — no schema, no compiler check links the two sides. A mismatch means the expected field simply never arrives (Lesson 32).

Loader callback:
A business-module load_* function (load_orders, load_product_master, load_inventory, load_invoices) passed into backend/uploads.py's read_xlsx_upload(file, label, loader) as a parameter, so the upload boundary never owns its own copy of required-column knowledge (Lesson 32).
