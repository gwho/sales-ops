Test Reading Patterns

Everything needed to read the three Track 2 test files fluently. Extends Track 1's pytest reading cheat sheet — that page's assert, pytest.raises, and property-style-assertion material isn't repeated here. Companion to Lesson 10.

Arrange / Act / Assert

def test_overpayment_clamps_outstanding_to_zero():
    # ARRANGE + ACT (usually one expression in this repo's style)
    result = calculate_payment_aging(
        _invoices_df(_invoice_row(invoice_amount=100.0, paid_amount=150.0, due_date=_due(20))),
        as_of_date=AS_OF,
    )
    row = result["aging_rows"][0]
    # ASSERT
    assert row["outstanding_amount"] == 0.0
    assert row["follow_up_priority"] == "None"
BeatWhat it does
ArrangeBuild the one input this test is about — usually via a builder helper (below).
ActCall the function under test. In this repo, arrange and act are almost always the same line.
AssertCheck every fact this specific test exists to prove — always separate, always last.

Builder helpers with **overrides

def _order_row(**overrides) -> dict:
    row = {"order_id": "SO-2026-001", ..., "priority": "High", ...}
    row.update(overrides)
    return row

# usage:
_order_row(customer_name="")        # everything normal, except a blank name
_order_row(priority="Urgent")       # everything normal, except an invalid priority

**overrides captures any keyword arguments as a dict; row.update(overrides) overlays them on top of a complete set of valid defaults. Every test file in this repo has its own builder(s) — _order_row/_orders_df, _inventory_row/ _inventory_df, _invoice_row/_invoices_df — always in pairs: one row builder, one function turning a list of rows into a DataFrame.

Not a pytest fixture. None of these are decorated @pytest.fixture — they're plain functions this repo's authors wrote. See the real pytest fixture docs if you want the actual mechanism later; don't conflate the two when describing this repo's tests.

@pytest.mark.parametrize

@pytest.mark.parametrize(
    "days_overdue,expected_bucket,expected_priority",
    [
        (30, "1-30 Days", "Low"),
        (31, "31-60 Days", "Medium"),
        ...
    ],
)
def test_aging_bucket_and_priority_boundaries(days_overdue, expected_bucket, expected_priority):
    ...

The decorator's first argument names the parameters as a comma-separated string; each tuple in the list is one row of that table, run through the same test body once per row. Pytest reports one pass/fail per tuple, not one combined result — six tuples means six separately-named test results.

Reading section comments as a table of contents

# --- Spec section 12 test cases -------------------------------------------
...
# --- IA-001 sort order edge cases ----------------------------------------
...
# --- Malformed / missing required valid order data ------------------------

Scroll a test file reading only these comments before reading any test body — you get an accurate outline of every rule and edge case the module handles, in the order the test author organized them. Fastest way to answer "what does this module cover" without reading top to bottom.

Reading a test name as a sentence

test_large_outstanding_amount_forces_high_priority_at_low_day_count → "test: large outstanding amount forces high priority at low day count." Every test name in this repo is written to be read this way.
