Boundary Case Checklist

Every boundary this repo's three Track 2 modules actually test, in one table — plus the general pattern for spotting a boundary before you go looking for its test. Companion to Lesson 13.

The general pattern

Any rule shaped like "if value <= N, do X; otherwise, do Y" has exactly one input where the answer flips: N itself. A thorough test of that rule checks at least two points, not one:

PositionWhy it matters
Exactly at the edge (N)Catches < written where <= was meant, or vice versa.
One step past the edge (N + 1 or N - 1)Confirms the answer genuinely flips right where it should, not one step early or late.

A test using a value comfortably inside a range (e.g. "45 days overdue" to prove the "31-60 Days" bucket) proves the bucket exists — it proves nothing about where its edges actually are.

A real gap, left visible on purpose

Watch's other edge, days_overdue == -7 exactly, has no dedicated test in the current suite — only -5 (comfortably inside) and 0 (the near edge) are directly tested. This table doesn't paper over that; if you're looking for practice writing a new boundary test from scratch, this is a real, honest place to start.

Reading a boundary straight from the code

if -7 <= days_overdue <= 0:
    return "Watch"

A range check written as LOW <= value <= HIGH has two edges to test, not one — both LOW and HIGH are candidates for an off-by-one mistake, independently of each other.
