Business Rule & Testing Glossary

The terms Lessons 8 and 11 introduce, compressed for lookup. Companion to Lesson 8 and Lesson 11.

A note on this document's status

This is a pre-study glossary, same status as Track 1's contracts glossary — terms defined so you can start using them, not yet evidence you've internalized them. A term only gets promoted into this workspace's permanent, evidence-gated glossary once you've demonstrated understanding of it in conversation or an exercise.

Terms

Business rule:
A real-world policy the data must satisfy — "a customer name must not be blank," "a quantity must be positive." Violating one is a data problem, fixable by whoever owns the data, not a bug in the code that checks it.
Avoid: "validation" alone as a substitute term — validation is the act of checking rules; a business rule is the policy itself.

Programming error:
A mistake in the code itself, wrong regardless of what data it receives (a typo'd variable, an off-by-one). Correctly surfaces as a crash during development, not as a business-readable message — hiding one as "bad data" conceals a real bug.

Row-level error (Order Validation's term, ValidationErrorRow):
A business rule violation scoped to one input row, carrying enough context (which row, which field, what's wrong) that a person can act on it without re-reading every other row.

Data issue (Payment Aging's term, PaymentDataIssueRow):
The Payment Aging module's equivalent of a row-level error — a per-row problem (a missing due date, an invalid amount) reported softly, excluding that row from the aging output without crashing the whole batch. Same shape and role as ValidationErrorRow, kept as a separate contract per the Field Scope Boundary (different spec, different module).

Independent rule evaluation (Shape 1, Lesson 11):
Checking every applicable rule against an item regardless of what earlier rules found, so the item can accumulate several simultaneous findings. Used when the output is a list of possibly many true facts (order validation errors; payment data issues).

First-match-wins dispatch (Shape 2, Lesson 11):
A fixed, ordered sequence of checks where the first match returns immediately and later checks never run. Used when the output is fundamentally a single, mutually exclusive label (a follow-up priority; an aging bucket) — chain order becomes a real design decision.
Avoid: calling this "validation" — it's a classification, not a check for correctness.

Guard clause:
A disqualifying condition checked early in a loop, using continue or return to skip the rest of that iteration's processing. Structurally guarantees an item can't accidentally end up processed by both the "disqualified" path and the "normal" path.

Boundary case:
An input sitting exactly at the edge where a rule's answer flips (e.g. days_overdue = 30 vs. 31). The classic bug this catches — an "off-by-one" — only shows up for inputs exactly on the edge, never for inputs comfortably inside a range. See the Boundary Case Checklist.

Clamping:
Forcing a computed value to stay within a valid range by replacing an out-of-range result with the boundary itself (e.g. max(value, 0.0)). Trades away the ability to tell how far past the boundary the real computation would have gone, in exchange for a simpler, always-valid output.

Signed value:
A quantity that keeps a meaningful negative sign instead of being floored at zero — this repo's days_overdue is the running example: negative means "not yet due," and that distance-past-zero is exactly what a rule like Watch needs to work at all.

Mutation (contrast with rebinding):
Changing an existing object's contents in place (chosen["qty"] -= 5), visible to everyone else already holding a reference to that same object — different from rebinding a variable to point at a brand-new object, which nobody else's reference notices.
