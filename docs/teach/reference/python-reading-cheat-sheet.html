Reading a function definition

def load_excel(file, sheet_name: int | str = 0) -> pd.DataFrame:
    """Load an Excel file (path or file-like object) into a DataFrame."""
    return pd.read_excel(file, sheet_name=sheet_name)

The None-sentinel default pattern

You'll see this shape repeatedly (src/sample_data.py uses it for every date-dependent function):

def generate_invoices(reference_date: date | None = None) -> pd.DataFrame:
    if reference_date is None:
        reference_date = date.today()

Why not just write reference_date: date = date.today() directly? Because a default argument value is computed exactly once, when the def line itself runs — not on every call. A literal date.today() default would freeze to whichever day the module happened to first load, and silently go stale after that. The None sentinel defers the real computation into the function body, which reruns every call.

A minimal class you'll recognize

class MissingColumnsError(Exception):
    def __init__(self, file_label: str, missing_columns: list[str]) -> None:
        ...
        super().__init__(message)

Read as: "a new, specific kind of error, built out of Python's built-in Exception." __init__ is the special method that runs when you create one (e.g. MissingColumnsError("orders file", ["sku"])); super().__init__(message) hands the final message text up to the built-in Exception machinery so str(error) returns it.

List comprehension, read left to right

missing = [column for column in required_columns if column not in df.columns]

Say it as: "Build a new list called missing. For each column in required_columns, include it only if that column is not in df.columns."
