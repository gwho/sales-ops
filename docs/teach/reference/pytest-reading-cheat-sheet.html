pytest Reading Cheat Sheet

How to read a test file as a plain-English list of claims. Companion to Lesson 3.

Commands

CommandWhat it does
uv syncInstall every dependency uv.lock pins, into an isolated environment.
uv run pytestRun the whole test suite inside that environment.
uv run pytest tests/test_excel_io.pyRun only that one file.
uv run pytest -vVerbose — print every test's name and pass/fail, not just a summary count.
uv run pytest -k unknown_skuRun only tests whose name contains "unknown_sku."

How pytest finds tests

Per this repo's pyproject.toml:

[tool.pytest.ini_options]
testpaths = ["tests"]

Pytest looks under tests/, opens every file named test_*.py, and inside each one runs every function named test_*.

assert — the whole mechanism

StatementResult
assert 2 + 2 == 4Condition is true — nothing happens, execution continues.
assert 2 + 2 == 5Condition is false — raises AssertionError, the test fails right there.

Reading a "no assertion" test

def test_validate_required_columns_passes_when_all_present():
    df = pd.DataFrame({"order_id": ["SO-1"], "sku": ["MED-LENS-001"]})
    validate_required_columns(df, ["order_id", "sku"], "orders file")

No assert line at all — the claim being tested is "this call does not raise anything." If it did raise, the test fails there without needing an explicit assertion.

Reading an expected-error test

with pytest.raises(MissingColumnsError) as excinfo:
    validate_required_columns(df, ["order_id", "sku"], "orders file")

assert "orders file" in str(excinfo.value)
assert "sku" in str(excinfo.value)
LineMeaning
with pytest.raises(X) as excinfo:"I expect the indented block below to raise X. If it doesn't, that's the failure. If it does, capture it as excinfo."
str(excinfo.value)The error's message text, as a string.
"orders file" in ...Substring check — is this text present anywhere in the message?

Property-style assertions (this repo's house style for generated data)

duplicate_mask = df["order_id"].duplicated(keep=False)
duplicated_ids = set(df.loc[duplicate_mask, "order_id"])

assert duplicated_ids == {"SO-2026-010"}   # exactly this one, named
assert duplicate_mask.sum() == 2          # exactly two rows involved

Deliberately in between "some duplicate exists" (too loose — would miss a second, accidental duplicate) and "row 34 equals this literal dict" (too rigid — breaks on any unrelated edit). Names the exact expected value and the exact expected count.

Reading a test name as a sentence

test_generate_orders_has_exactly_one_unknown_sku → "test: generate_orders has exactly one unknown sku." Test names in this repo are written to be read this way — scan just the def test_... lines in a file before reading any bodies, and you'll often already know what the module does.
