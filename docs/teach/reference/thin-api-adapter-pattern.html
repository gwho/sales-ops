Thin API Adapter Pattern

The adapter/business-rule boundary, generalized past this one repo, plus the two concrete disciplines that keep it real instead of aspirational. Companion to Lesson 35.

The smell test

Read any function. Does it contain an if whose condition is about business meaning ("is this SKU active," not "did this file arrive with the right field name")? If yes, that logic is living one layer too shallow — it belongs one function call away, inside the module that owns that decision.

Two disciplines that make the rule real

  1. No framework import ever enters the business-rule layer. This repo: zero import fastapi anywhere under src/. Verified mechanically with grep -rn "^import fastapi\|^from fastapi" src/ — should return nothing.
  2. Route/handler return types are the real business-module types, never a duplicate schema. This repo: every router returns the actual TypedDict (OrderValidationResult, etc.) directly — no parallel Pydantic BaseModel redefining the same fields a second time.

Why duplicating a schema is worse than it looks

A second, hand-maintained definition of the same shape has no mechanism forcing it to agree with the original. The moment the real type gains a field, the duplicate silently falls behind — and nothing fails loudly when that happens, because both definitions still compile and run; they just quietly disagree.

Applying the pattern to something new

Before adding a new adapter-layer helper, check whether an existing one already generalizes: backend/uploads.py's read_xlsx_upload(file, label, loader) and lib/api-client.ts's postJSON/postReport were built generic specifically so a new workflow could reuse them unchanged. If a design invents a new shared helper instead, ask whether it's solving a genuinely new problem — or just not noticing the existing generalization already covers it.
