API Error Contract Pattern

The contract

Every error response, from any endpoint, any layer: {"detail": "<business-readable string>"}. No lists, no nested error objects, no raw exception text.

400 vs. 500 decision rule

Ask: whose fault is this, independent of severity? The client sent something this request can't proceed with (fixable by the caller) → 400. Something broke on the server regardless of what was sent (never the caller's fault, message never disclosed) → 500.

Why FastAPI's own validation needed a handler

FastAPI's automatic validation (fires when a required File()/Form() field is entirely absent) returns detail as a list of structured error objects by default — a different shape than every other error this API produces. A frontend component expecting detail to always be a string would break exactly at the moment a request was most simply malformed. backend/errors.py's RequestValidationError handler exists solely to rewrite that list into the same single-string shape.

The raise_server_exceptions testing gotcha

Starlette's TestClient defaults to raise_server_exceptions=True — an unhandled exception propagates straight into the test as a real traceback, useful for debugging a genuine bug. But that same default bypasses any registered exception handler meant to convert the exception into a response. To actually test what a real client sees for an unexpected server error, use a dedicated client:

error_client = TestClient(app, raise_server_exceptions=False)

Reserve this only for the specific test(s) exercising the generic-error handler itself — every other test benefits from the convenient default, since it only changes behavior at the exact moment an exception would otherwise propagate uncaught.
