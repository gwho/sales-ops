Presentation Layer Glossary

The terms Lessons 15, 16, and 20 introduce, compressed for lookup. Companion to Lesson 15, Lesson 16, and Lesson 20.

A note on this document's status

This is a pre-study glossary, same status as Track 1's and Track 2's — terms defined so you can start using them, not yet evidence you've internalized them. A term only gets promoted into this workspace's permanent, evidence-gated glossary once you've demonstrated understanding of it in conversation or an exercise.

Terms

Business logic layer:
The part of a system that decides what a result is — computes, classifies, validates a real-world fact. In this repo: order_validation.py, inventory_allocation.py, payment_aging.py. Can be "wrong" in a business sense — a bug in a rule produces a wrong business fact.

Presentation layer:
The part of a system that formats an already-decided result for a human to read. In this repo: report_export.py. Cannot be wrong in a business sense — it has no rules of its own to break, only formatting to get right or wrong.
Avoid: treating "presentation layer" as a synonym for "UI code" — any module whose job is formatting an already-computed result, including a backend Excel exporter, is playing this role.

Recalculation leak:
The failure mode a presentation layer exists to prevent — quietly recomputing a business fact instead of trusting the value it was handed, risking disagreement with whatever else in the system computed the same fact (a dashboard, an earlier report). Named directly in Lesson 15.

Producer module:
A module that builds and owns the shape of its own output — order_validation.py, inventory_allocation.py, and payment_aging.py are this repo's three producers.

Consumer module:
A module that only ever receives another module's output as a parameter and reads from it, never building an envelope of its own. report_export.py is this repo's first consumer module.

Allow-list:
A filter that only accepts values it explicitly recognizes; everything else is excluded by default. This repo's FOLLOW_UP_PRIORITIES set is the running example — fails safer than its alternative, because an unrecognized value goes missing rather than being silently included.

Deny-list / exclusion-list:
The opposite filter shape — only rejects values it explicitly recognizes; everything else is accepted by default. The rejected alternative for Follow-up List (!= "None") is this repo's own concrete near-miss example — it would have behaved identically for every value the code has ever actually produced, and differently only for a value nobody anticipated.

Presentation-safe filtering (contrast with revalidation):
Deciding whether an already-trusted value belongs in a derived, filtered view — without ever touching or second-guessing the value itself. Different in kind from revalidation, which would mean recomputing whether the value is even correct. A presentation layer may do the first; never the second.

Hidden global state:
A value that lives outside any function and is remembered between calls — a module-level counter is the classic shape. This repo's coding standards rule it out everywhere: it makes a pure function's output depend on call order, and makes bugs far harder to reproduce than anything a side-effect-free function can produce.
