The statelessness test

Ask one question about any endpoint: does completing this request require the server to remember something about an earlier request? Not "does the word 'state' appear in the code" — an actual dependency on server-side memory of the past.

DesignPasses the test?Why
POST /api/orders/validateYesEvery input arrives in this one request; nothing is looked up from an earlier call
POST /api/orders/validate/reportYesRe-accepts the same raw files; recomputes from scratch, never reads a prior result
GET /api/reports/{report_id} (rejected design)NoRequires something stored between the generating request and this fetch — a job store or file store, full stop

What a two-phase "generate → fetch by ID" design always requires

Before accepting any endpoint shaped like this, name all five, concretely — not in the abstract:

  1. Storage: where do the bytes (or enough to regenerate them) live between the two calls?
  2. Identity: what generates the ID, and what guarantees it's unique?
  3. Expiry: how long does a stored artifact stay available before it's cleaned up, and who owns that job?
  4. Miss handling: what does a GET for an expired or nonexistent ID actually return?
  5. Authorization: should any client holding the ID be able to fetch it, or only the client that generated it?

All five are restatements of the same underlying problem — persistence, identity, and lifecycle — asked five different ways. That repetition is itself the evidence: it's one real architectural commitment, not five small ones.

The trust-boundary test

A trust boundary is the line between data a system produced and verified itself, and data that arrived from outside and could say anything. Apply this test to any endpoint that produces an "official" artifact from a previously-computed result:

Could this endpoint's output be influenced by something the server itself never verified? If a client-supplied JSON result (not raw input) is accepted as authoritative, the answer is yes — and the fix is always the same: recompute from the raw input the server can verify, rather than trusting a client's copy of a previous answer.

The general principle

A system should not accept as authoritative anything it could not have produced itself. Recomputing from raw input means the only way to influence the output is to change the same input that would have influenced the on-screen result too — nothing left to edit undetected.
