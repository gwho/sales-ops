Contracts & TypedDict Glossary

The terms Lesson 4 introduces, compressed for lookup. Companion to Lesson 4.

A note on this document's status

This is a pre-study glossary — terms defined so you can start using them, not yet evidence that you've internalized them. Per this workspace's teaching conventions, a term only gets promoted into the workspace's permanent, evidence-gated glossary once you've actually demonstrated understanding of it in conversation or an exercise. Treat this page as scaffolding you'll eventually outgrow, not the final authority.

Terms

dict (dictionary):
A Python container mapping keys to values, written {"key": "value"}. Carries no built-in promise about which keys must be present or what type each value is.

Type hint:
An annotation on a variable, parameter, or return value (e.g. quantity: int) stating the expected type. Checked by external tools and readers, not enforced by Python itself at runtime.
Avoid: type annotation used loosely to mean "runtime validation" — it isn't one.

TypedDict:
A Python construct (from the typing module) that declares the expected keys and value types of a dict, for type-checking purposes only. At runtime, a TypedDict-typed value is an ordinary dict — indistinguishable from any other, with no extra validation or wrapper object.
Avoid: "TypedDict class" implying it's a distinct runtime object type — it produces plain dicts.

NotRequired:
A wrapper inside a TypedDict field's type hint (e.g. sku: NotRequired[str]) marking that key as allowed to be absent from a given instance, rather than merely allowed to hold None.

JSON (JavaScript Object Notation):
A plain-text data format built from the same two shapes as Python's dict and list — objects with named keys, and ordered arrays — readable by nearly every programming language and web browser. A plain Python dict converts to JSON with no extra code.

Output contract (this repo's term):
A TypedDict in src/contracts.py that's shared as a stable handoff shape across modules — built by Python business logic, and (in later phases) carried across a JSON boundary to a web layer and UI. Named for the same reason a legal contract is: a promise about shape every consumer can rely on without re-verifying it themselves.
Avoid: "schema" as a casual synonym in this workspace — this repo's own docs (CONTEXT.md) reserve "Output Contract" as the specific term; use it.

Field Scope Boundary (this repo's term, forward reference):
The rule that an output contract may only contain fields its originating spec explicitly defines — contracts are allowed to be asymmetric across modules; matching another contract's fields "for symmetry" is not a valid reason to add one. Not covered in depth until later tracks — noted here because it's the direct consequence of taking "contract" seriously as a promise.

Contract Fixture (this repo's term):
A hand-authored, believable example value for one output contract, kept in tests/contract_fixtures.py. Proves the contract's shape can hold realistic data; deliberately not computed from real business-rule code, so it can't silently inherit that code's bugs. See Lesson 6.
