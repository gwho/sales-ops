Contract Consumer Pattern

The producer side of a contract, and the consumer side, side by side. Companion to Lesson 16; extends Track 2's Result Envelope Pattern from the producer's point of view to the consumer's.

Two roles, one contract

RoleWhat it doesExample in this repo
ProducerBuilds an envelope, owns its shape, defines its type locally in its own module.order_validation.py defines and returns OrderValidationResult.
ConsumerReceives an envelope as a parameter, reads from it, never redefines or extends its shape.report_export.py imports and reads OrderValidationResult, never modifies it.

What a consumer is allowed to do

  • Read any field the contract already defines. result["valid_orders"], result["errors"] — straightforward.
  • Import the producer's type from the producer's module. Never from contracts.py unless the type actually lives there (see Track 2's Result Envelope Pattern reference — envelopes live with their producing module).
  • Filter or derive a view over already-trusted data — e.g. Follow-up List's allow-list filter (Lesson 20) — without ever changing what a field means or recomputing its value.
  • Accept extra parameters the contract doesn't carry, kept entirely separate from the envelope, when a consumer needs data the producer never had reason to include — e.g. original_orders_df alongside result.

What a consumer is never allowed to do

  • Add a field to the producer's contract just because one consumer would find it convenient — every other caller of the producer would carry a payload it never asked for.
  • Recompute a value the contract already provides (Lesson 15) — that's a recalculation leak, not consumption.
  • Redefine the producer's type inside the consumer's own module — the type has exactly one home, the module that produces it.

The "separate parameter" escape hatch

When a consumer genuinely needs data the contract doesn't carry, the fix is a second, independent function parameter — never a new field bolted onto the existing envelope. export_order_validation_report(result, original_orders_df=None, ...) is the concrete example: OrderValidationResult stays exactly what Tutorial 03 defined, and the report-specific need is satisfied by a parameter no other caller of validate_orders() is ever forced to carry.

When would a type actually move?

The tell, per Lesson 16's checkpoint: a producer's envelope type is worth promoting into contracts.py only if it turns out to be structurally identical to another module's envelope (duplication worth collapsing), or if a consumer module starts needing to treat multiple envelopes polymorphically — one generic function operating on "any envelope" rather than one hand-written function per producer. Neither has happened in this repo yet: all three envelopes have different field counts, and report_export.py still has one explicit export_*_report function per producer.
