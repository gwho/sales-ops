Result Envelope Pattern

The one shape all three Track 2 modules share, side by side. Companion to Lesson 9.

The three envelopes

ModuleEnvelope typeFields
order_validation.py OrderValidationResult summary, valid_orders, errors
inventory_allocation.py InventoryAllocationResult summary, allocation_results, backorders, remaining_inventory, supplier_follow_ups
payment_aging.py PaymentAgingResult summary, aging_rows, data_issues, draft_messages

What every envelope has in common

  • One function, one call, one dict back. No caller ever needs to call a second function and stitch results together by hand.
  • Always a summary key first — counts and totals, separate from the detail rows. A KPI card and a detail table can each read only the part they need.
  • Defined locally in the module, not in src/contracts.py. Each envelope is a bundle of contracts already defined there — it isn't a new business-facing shape of its own, just a transport container for one function's return value.
  • Field names read naturally for that module's domain — valid_orders vs. aging_rows vs. allocation_results. The three envelopes don't share a naming convention, and that's fine: nothing in this repo consumes them generically (by reflecting over "every list field"), only ever by their own specific keys.

Why not a plain tuple?

The spec files for all three modules originally suggested plain tuples of DataFrames (-> tuple[pd.DataFrame, pd.DataFrame]). A tuple is positional — nothing in the code itself enforces "the first one is valid rows, the second is errors." A named dict key can't be silently transposed the way tuple position can.
